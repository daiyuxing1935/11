# 管理者-执行者模式（Manager-Worker）

## 引言

在现实世界中，几乎所有高效的团队都遵循一个基本原则：有人做决策和分工，有人专注执行。软件开发团队有Tech Lead和开发工程师，建筑工队有工头和工人，军队有指挥官和士兵。这种分工模式在人类协作中经历了数千年的验证，其核心理念同样适用于AI多智能体系统。

**管理者-执行者模式（Manager-Worker Pattern）** 是多智能体系统中最经典、最广泛使用的架构模式。它通过引入一个专门的"管理者"Agent来分解任务、分配工作、追踪进度、汇总结果，而多个"执行者"Agent则专注于完成各自的子任务。

### 基本架构

```
                           │                  │

                    ┌───────────────┼───────────────┐
                    │               │               │
              ┌─────▼─────┐  ┌─────▼─────┐  ┌─────▼─────┐
              │ Worker A  │  │ Worker B  │  │ Worker C  │
              │           │  │           │  │           │
              │ 专注领域A │  │ 专注领域B │  │ 专注领域C │
              │ 工具集A   │  │ 工具集B   │  │ 工具集C   │
              └───────────┘  └───────────┘  └───────────┘
```

### 角色定义

| 角色 | 职责 | 能力要求 | 典型配置 |
|------|------|---------|---------|
| Manager | 任务分解、分配、追踪、汇总、异常处理 | 强推理、全局视野、组织能力 | 强模型 + 无工具限制 |
| Worker | 接收指定任务、专注执行、汇报结果 | 专业深度、专注力 | 领域特定模型 + 领域工具集 |

## Manager的职责详解

Manager是整个模式的核心，其决策质量直接决定系统输出的质量。Manager需要具备"指挥官的头脑"。

### 1. 任务分解（Task Decomposition）

Manager首先需要将用户的复杂任务分解为多个可独立执行的子任务。

```python
class ManagerAgent:
    def decompose_task(self, user_request: str) -> list[SubTask]:
        """
        将用户请求分解为子任务列表。

        分解策略：
        1. 识别任务的独立维度（如：技术分析、市场分析、财务分析）
        2. 识别任务的阶段依赖（如：必须先调研才能写报告）
        3. 评估每个子任务需要的技能和工具
        """
        decomposition_prompt = f"""
        你是一个项目管理专家。请将以下任务分解为可独立执行的子任务。

        原始任务：{user_request}

        分解要求：
        1. 每个子任务应该是一个独立、完整的工作单元
        2. 标注子任务之间的依赖关系
        3. 估算每个子任务的复杂度（low/medium/high）
        4. 标注每个子任务需要的专业技能

        输出JSON格式：
        {{
            "subtasks": [
                {{
                    "id": "task_1",
                    "description": "...",
                    "required_skill": "research",
                    "complexity": "medium",
                    "dependencies": [],
                    "estimated_tokens": 1000
                }}
            ]
        }}
        """
        response = self.llm.invoke(decomposition_prompt)
        return self._parse_subtasks(response)
```

### 2. Worker匹配与任务分配

分解任务后，Manager需要将每个子任务分配给最合适的Worker。

```python
class ManagerAgent:
    def __init__(self):
        self.worker_registry = {}  # skill -> [worker_ids]

    def register_worker(self, worker_id: str, skills: list[str],
                        capacity: int):
        """注册Worker及其能力"""
        for skill in skills:
            if skill not in self.worker_registry:
                self.worker_registry[skill] = []
            self.worker_registry[skill].append({
                "id": worker_id,
                "capacity": capacity,
                "current_load": 0
            })

    def assign_task(self, subtask: SubTask) -> str:
        """为子任务选择最合适的Worker"""
        required_skill = subtask.required_skill
        candidates = self.worker_registry.get(required_skill, [])

        if not candidates:
            # 没有完全匹配的Worker，寻找最接近的
            return self._find_best_fallback(subtask, required_skill)

        # 选择负载最低的Worker
        best_worker = min(candidates,
                         key=lambda w: w["current_load"])
        best_worker["current_load"] += 1
        return best_worker["id"]
```

### 3. 进度追踪（Progress Tracking）

Manager需要实时监控各Worker的执行状态，及时发现并处理异常。

```python
from enum import Enum
from datetime import datetime
import asyncio

class TaskStatus(Enum):
    PENDING = "pending"
    ASSIGNED = "assigned"
    IN_PROGRESS = "in_progress"
    BLOCKED = "blocked"
    COMPLETED = "completed"
    FAILED = "failed"

class TaskTracker:
    def __init__(self):
        self.tasks = {}  # task_id -> TaskInfo
        self.dependency_graph = {}  # task_id -> [dependent_task_ids]

    def update_status(self, task_id: str, status: TaskStatus,
                      metadata: dict = None):
        """更新任务状态"""
        task = self.tasks[task_id]
        task.status = status
        task.updated_at = datetime.now()
        if metadata:
            task.metadata.update(metadata)

        # 检查依赖任务的解锁
        if status == TaskStatus.COMPLETED:
            self._unblock_dependents(task_id)

    def _unblock_dependents(self, completed_task_id: str):
        """当任务完成时，解除依赖它的任务的阻塞状态"""
        dependents = self.dependency_graph.get(completed_task_id, [])
        for dep_id in dependents:
            dep_task = self.tasks[dep_id]
            # 检查所有依赖是否都已完成
            if all(self.tasks[d].status == TaskStatus.COMPLETED
                   for d in dep_task.dependencies):
                dep_task.status = TaskStatus.PENDING

    def get_overall_progress(self) -> dict:
        """获取整体进度报告"""
        total = len(self.tasks)
        completed = sum(1 for t in self.tasks.values()
                       if t.status == TaskStatus.COMPLETED)
        failed = sum(1 for t in self.tasks.values()
                    if t.status == TaskStatus.FAILED)
        return {
            "total": total,
            "completed": completed,
            "failed": failed,
            "in_progress": total - completed - failed,
            "completion_pct": (completed / total * 100) if total > 0 else 0
        }
```

### 4. 结果汇总（Result Aggregation）

这是Manager最体现价值的工作——将零散的Worker输出编织成连贯的整体。

```python
class ManagerAgent:
    def aggregate_results(self, task_id: str,
                          worker_results: dict[str, str]) -> str:
        """
        汇总多个Worker的执行结果。

        关键挑战：
        1. 处理结果间的矛盾和不一致
        2. 消除冗余信息
        3. 填充空白和缺口
        4. 统一风格和格式
        """
        aggregation_prompt = f"""
        你是最终报告的整合编辑。以下是不同分析师的独立输出，请将它们整合为一份
        连贯、一致、无冗余的最终报告。

        各Worker输出：
        {self._format_worker_outputs(worker_results)}

        整合原则：
        1. 发现并解决事实矛盾（标注存疑点）
        2. 按逻辑结构重组内容
        3. 统一术语和表达风格
        4. 标注需要人工确认的地方
        5. 保留所有关键发现，不遗漏重要信息

        输出格式：Markdown
        """
        return self.llm.invoke(aggregation_prompt)

    def handle_conflicts(self, result_a: str, result_b: str) -> str:
        """当两个Worker的输出矛盾时的冲突解决"""
        conflict_prompt = f"""
        以下是两个分析师的输出，它们在某些方面存在矛盾。请分析矛盾点并给出综合结论。

        分析师A输出：{result_a}

        分析师B输出：{result_b}

        请：
        1. 列出所有矛盾点
        2. 对每个矛盾，判断哪个结论更可靠（附理由）
        3. 如果有无法判断的，标注为"待人工确认"
        """
        return self.llm.invoke(conflict_prompt)
```

### 5. 异常处理（Exception Handling）

```python
class ManagerAgent:
    async def handle_worker_failure(self, failed_task: SubTask,
                                     error_info: str):
        """处理Worker执行失败的情况"""
        strategies = [
            "retry_same_worker",    # 同Worker重试
            "reassign_new_worker",  # 换Worker重试
            "task_simplify",        # 简化任务
            "task_skip",            # 跳过（非关键任务）
            "escalate_to_human",    # 升级给人类
        ]

        # 根据错误类型选择策略
        if self._is_transient_error(error_info):
            return await self._retry_task(failed_task)
        elif self._is_skill_mismatch(error_info):
            return await self._reassign_task(failed_task)
        elif failed_task.is_critical:
            return await self._escalate(failed_task)
        else:
            return self._skip_and_compensate(failed_task)
```

## Worker的职责详解

Worker的设计哲学是"少即是多"——每个Worker应该尽可能聚焦和专业化。

### 1. 专注执行

Worker接收清晰定义的任务，不应参与任务分解或策略决策。

```python
class WorkerAgent:
    def __init__(self, name: str, role: str, expertise: str,
                 tools: list, model: str):
        self.name = name
        self.role = role
        self.expertise = expertise
        self.tools = tools
        self.model = model

        self.system_prompt = f"""
        你是{name}，担任{role}角色。
        你的专业领域是：{expertise}

        工作原则：
        1. 只做你被要求做的事，不越界
        2. 遇到不确定的信息，标注为"不确定"而非猜测
        3. 输出结构化结果，便于Manager整合
        4. 如果任务超出你的能力范围，立即报告
        """

    async def execute(self, task: SubTask) -> WorkerResult:
        """执行一个子任务"""
        try:
            # 验证任务是否在自己的能力范围内
            self._validate_task_scope(task)

            # 执行任务
            result = await self._do_work(task)

            return WorkerResult(
                worker_id=self.name,
                task_id=task.id,
                status="success",
                output=result,
                metadata={
                    "tools_used": self._get_used_tools(),
                    "confidence": self._assess_confidence(result)
                }
            )
        except OutOfScopeError as e:
            return WorkerResult(
                worker_id=self.name,
                task_id=task.id,
                status="out_of_scope",
                output=str(e),
                metadata={}
            )
        except Exception as e:
            return WorkerResult(
                worker_id=self.name,
                task_id=task.id,
                status="failed",
                output=str(e),
                metadata={"error_type": type(e).__name__}
            )

    def _validate_task_scope(self, task: SubTask):
        """验证任务是否在Worker的能力范围内"""
        if task.required_skill not in self.skills:
            raise OutOfScopeError(
                f"任务需要 {task.required_skill} 技能，"
                f"但 {self.name} 擅长的是 {self.skills}"
            )
```

### 2. 汇报结果

Worker需要按照约定格式汇报结果，使Manager能够高效整合。

```python
class WorkerResult:
    """Worker返回结果的标准化格式"""
    def __init__(self, worker_id: str, task_id: str, status: str,
                 output: str, metadata: dict):
        self.worker_id = worker_id
        self.task_id = task_id
        self.status = status
        self.output = output
        self.metadata = metadata
        self.timestamp = datetime.now()

    def to_dict(self) -> dict:
        return {
            "worker_id": self.worker_id,
            "task_id": self.task_id,
            "status": self.status,
            "output": self.output,
            "confidence": self.metadata.get("confidence", "unknown"),
            "tools_used": self.metadata.get("tools_used", []),
            "token_usage": self.metadata.get("token_usage", 0),
            "duration_ms": self.metadata.get("duration_ms", 0),
            "timestamp": self.timestamp.isoformat()
        }
```

### 3. 自我评估

高级Worker应具备自我评估能力，对结果质量进行评分。

```python
class WorkerAgent:
    def _assess_confidence(self, output: str) -> str:
        """评估输出置信度"""
        assessment = self.llm.invoke(f"""
        请评估以下输出内容的置信度：

        输出内容：{output}

        评分标准：
        - high: 所有信息有坚实依据，无推测成分
        - medium: 大部分有依据，少量推测但已标注
        - low: 信息不完整或存在较大推测空间

        请只返回一个词：high/medium/low
        """)
        return assessment.strip().lower()
```

## 优缺点分析

### 优势

| 优势 | 详细说明 |
|------|---------|
| **结构清晰** | 角色分明，谁负责什么都一目了然，降低认知负担 |
| **质量控制** | Manager可以审查Worker的产出，退回不合格的结果 |
| **动态调整** | Manager可以根据中间结果动态改变后续策略 |
| **责任明确** | 出现问题时容易定位——是Manager分解不当还是Worker执行不力 |
| **天然支持流水线** | 多阶段任务（研究→分析→写作→审校）天然适配 |
| **新手友好** | 与人类团队管理经验高度吻合，容易理解和设计 |

### 劣势

| 劣势 | 详细说明 | 缓解策略 |
|------|---------|---------|
| **Manager单点瓶颈** | Manager的推理能力限制整个系统的上限 | 使用最强的模型作为Manager |
| **通信开销大** | Manager与每个Worker都需要来回通信 | 批量分配+异步通信 |
| **Manager负载过高** | 任务分解、分配、追踪、汇总全部压在Manager身上 | Manager本身可有多实例 |
| **层级僵化** | Manager固定在工作流顶端，缺乏灵活性 | 允许Worker自主发起协作 |
| **信息衰减** | Manager作为信息中转站，可能遗漏或曲解信息 | 结构化消息+共享上下文 |
| **扩展性受限** | Worker数量增加时，Manager的协调负担线性增长 | 引入子Manager（分层管理） |

## 变体模式

### 1. 分层Manager-Worker（Hierarchical Manager-Worker）

当Worker数量过多时，引入中间管理层：

```
                 ┌──────────────┐
                 │ 总Manager    │
                 └──────┬───────┘
          ┌─────────────┼─────────────┐
    ┌─────▼─────┐ ┌─────▼─────┐ ┌─────▼─────┐
    │子Manager A│ │子Manager B│ │子Manager C│
    │(研究领域)  │ │(分析领域)  │ │(写作领域)  │
    └─────┬─────┘ └─────┬─────┘ └─────┬─────┘
      ┌────┴────┐    ┌────┴────┐    ┌────┴────┐
   Worker  Worker  Worker  Worker  Worker  Worker
```

### 2. Manager-Reviewer-Worker

引入独立的Reviewer Agent，减轻Manager的质量审查负担：

```
Manager → 分配任务 → Worker → 产出结果
                              ↓
                          Reviewer → 审查通过？→ Yes → Manager汇总
                              ↓ No
                          退回Worker修改
```

### 3. 轮值Manager

不同阶段由不同的Agent担任Manager：
- 分解阶段：战略分析师担任Manager
- 执行阶段：技术Leader担任Manager
- 整合阶段：编辑/撰稿人担任Manager

---

## 最佳实践

1. **Manager用最强模型**：Manager的推理能力是系统质量的瓶颈
2. **Worker够用就好**：根据子任务复杂度选择模型，节省成本
3. **明确的信息接口**：定义清楚Manager和Worker之间的消息格式
4. **超时和重试**：Worker可能卡住，Manager需要有超时处理
5. **分级日志**：记录每个决策和通信，便于调试
6. **渐进式复杂度**：从2-3个Worker开始，验证通过后再扩展
7. **人工回环**：关键决策点允许人工介入审批

---

## 小结

Manager-Worker模式是多智能体系统中最成熟、最实用的协作模式。它的核心思想——"有人决策，有人执行"——与人类组织管理的经验高度吻合，因此特别容易理解和上手。

**适用判断**：
- 任务有明显的层级结构？用Manager-Worker
- 需要一个Agent统揽全局？用Manager-Worker
- Worker数量不超过10个？直接用
- Worker数量超过10个？考虑分层Manager-Worker
- Manager成为瓶颈？用最强模型或引入子Manager

记住：Manager-Worker虽好，但不要滥用。如果任务本身不需要分解和协调，单Agent就是更优方案。
