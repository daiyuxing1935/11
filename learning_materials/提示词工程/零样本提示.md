# 零样本提示（Zero-shot Prompting）

## 什么是零样本提示

零样本提示（Zero-shot Prompting）是提示词工程中最基础也是最常用的技术。它的核心思想非常简单：**不提供任何示例，直接向大语言模型描述任务要求**，让模型依靠其预训练阶段获得的知识和能力来完成任务。

"零样本"（Zero-shot）这个术语源于机器学习中的"零样本学习"——模型在没有见过特定类别训练样本的情况下，依然能对该类别进行识别或推理。在大语言模型的语境下，"零样本"意味着你不需要给模型"看例题"，只需要清楚地告诉它"你要做什么"。

一个直观的理解：把LLM想象成一个读过海量书籍、文章、代码的超级学者。零样本提示就像你直接对这个学者说："请帮我把这篇文章翻译成英文。"你不需要先给他看几个翻译的例子，因为他已经"读过"无数的翻译文本，知道翻译是怎么回事。但正如一个学者需要清晰的指令才能高效工作，零样本提示的质量很大程度上取决于你如何描述任务——这就是提示词工程的核心价值所在。

## 为什么零样本提示重要

在现代LLM应用中，零样本提示是使用最广泛的提示方式，原因包括：

**经济性**：Few-shot提示需要在每次请求中附带示例，增加了token消耗。对于按token计费的模型，3-5个示例可能增加数百甚至上千token的成本。在高频调用场景下（如客服系统每天处理10万次请求），零样本相比少样本可以节省大量成本。

**灵活性**：零样本提示不需要为每个新任务准备示例。当任务需求频繁变化时，零样本是唯一可行的选择——你不可能为每个可能的用户问题预先准备示例。

**模型能力足够强**：GPT-4、Claude 3.5、Gemini等新一代模型具有极强的指令遵循能力。对于大多数常见任务，精心设计的零样本提示已经能取得非常好的效果。

**避免示例偏差**：不当的示例可能误导模型。如果示例覆盖不全或存在偏差（如情感分析的示例中只有正面和负面而缺了中性），模型可能过度拟合示例的模式，反而降低泛化能力。

## 零样本提示的核心要素

一个完整的零样本提示通常包含以下四个要素：

```
[角色设定] + [任务描述] + [输入数据] + [输出格式要求]
```

### 要素一：角色设定（Role Setting）

通过设定特定角色或身份，激活模型训练数据中与该角色相关的知识和语言模式。

```
没有角色设定：
"解释一下过拟合是什么。"

有角色设定：
"你是一位教了20年机器学习的大学教授，善于用生活化的比喻解释复杂概念。
请向大一新生解释什么是过拟合。"
```

角色设定的效果来自训练数据中不同角色文本的分布——设定"教授"角色时，模型更倾向于采样与学术教学相关的表达模式。常见有效的角色类型包括领域专家、教师/导师、特定职业从业者、特定风格的作家等。

### 要素二：任务描述（Task Description）

这是零样本提示最核心的部分。好的任务描述应做到**具体、清晰、结构化**。

**关键原则：**
- **具体胜过抽象**：说"写500字"比"写一些"强；说"列出5个步骤"比"说明怎么做"强
- **正面指令胜过负面指令**：说"请使用正式语气"比"不要用口语"更有效，因为"不要口语"仍然激活了"口语"这个概念
- **约束适度**：约束太少输出不可控，约束太多则可能矛盾。一般3-7条约束为最佳范围
- **使用分隔符**：用 `###`、`---`、`"""` 等分隔不同部分，帮助模型清晰解析指令结构

### 要素三：输入数据（Input Data）

明确标注输入数据的位置和格式，推荐使用三引号（`"""`）、反引号或XML标签（`<text>...</text>`）来包裹输入数据，让模型清晰区分"指令"和"数据"。这在输入数据本身包含指令性语言时尤为重要，可防止提示注入。

### 要素四：输出格式约束（Output Format）

明确要求输出的格式，在需要程序化处理输出时至关重要。常见方式包括JSON格式（适合程序解析）、Markdown表格（适合数据对比）、列表格式（适合步骤说明）、特定模板（适合报告和邮件等固定格式）。

## 十大任务类型零样本提示实战示例

以下是覆盖常见应用场景的完整零样本提示示例，每个都可以直接复制使用。

### 1. 文本摘要（多种策略）

**策略一：限定字数摘要**
```
请将以下文章用不超过150字的中文进行摘要，保留核心观点和关键数据，
去除修饰性语言和重复内容。

文章：
"""
[在此粘贴原文]
"""
```

**策略二：结构化摘要**
```
请对以下技术文档生成结构化摘要，包含：
- 一句话总结（不超过30字）
- 三个核心要点（每点不超过50字）
- 关键数据和指标（如有）
- 目标读者应关注的行动建议

技术文档：
"""
[在此粘贴原文]
"""
```

### 2. 文本分类（二元和多类）

**二元分类：**
```
判断以下新闻标题是否属于"标题党"（夸张、误导、煽动性）。

只回答"是"或"否"，不要任何解释。

新闻标题：震惊！男子每天喝一杯这个，医生都惊呆了的体检结果！
```

**多类分类：**
```
将以下客户咨询归类到以下部门之一：技术故障、账单问题、产品咨询、
投诉建议、账户管理。只输出分类名称。

客户咨询："我昨天充值了100元，余额显示只有80元，钱去哪了？"
```

### 3. 情感分析

```
你是一位社交媒体舆情分析师。请对以下商品评论进行多维情感分析。

分析维度：
1. 总体情感倾向：正面 / 负面 / 中性
2. 情感强度：1（极弱）到 5（极强）
3. 涉及方面：品质、价格、物流、客服、包装（可多选）
4. 是否含讽刺或反语：是 / 否

输出为JSON格式。

评论：
"""
买了三次才买到满意的。第一次发错颜色，第二次包装破损，
第三次总算对了而且质量确实不错。客服态度一直都挺好，
就是仓库那边需要加强管理。
"""
```

### 4. 翻译（带语气和风格控制）

```
请将以下中文翻译为英文。要求：
- 保持商务信函的正式语气
- 使用被动语态（passive voice）以符合英文商务惯例
- 保留公司名称、产品名称的原文
- 数字格式转换为英文习惯（如"1,000元"转换为"CNY 1,000"）

中文原文：
"""
尊敬的客户，根据贵司2025年第三季度的采购记录，我方已对价格方案
进行了优化调整。新方案将为贵司节省约15%的采购成本，详情见附件。
如有疑问，请于7个工作日内与我方联系。
"""
```

### 5. 创意写作（故事生成、诗歌）

**故事生成：**
```
你是一位擅长写"微型小说"的作家。请创作一篇微型小说。

设定：
- 主角：一位深夜便利店的中年女店员
- 事件：一个雨夜，一位陌生老人走进店里，购买了
  三样看似毫无关联的物品
- 结尾：需要有一个让读者回味的反转
- 字数：500-800字
- 风格：白描为主，对话精炼，避免内心独白式的抒情
```

**诗歌创作：**
```
请以"程序员的深夜"为题创作一首七言律诗。
要求：内容需包含写代码、调试bug、喝咖啡三个元素，
押韵工整，平仄合规。
```

### 6. 代码生成（含错误处理要求）

```
你是一位资深Python后端工程师，代码风格遵循PEP 8规范。
请编写一个文件读取工具函数。

要求：
1. 函数名：safe_read_json
2. 输入：文件路径（str）
3. 返回：解析后的字典或列表；若失败返回包含错误信息的字典
4. 必须处理的异常：
   - 文件不存在（FileNotFoundError）
   - JSON格式错误（json.JSONDecodeError）
   - 权限不足（PermissionError）
   - 文件编码非UTF-8（UnicodeDecodeError）
5. 包含完整的类型注解和docstring
6. 包含异常日志记录（使用logging模块）
7. 附3个pytest格式的单元测试

直接输出代码，不要额外解释。
```

### 7. 数据提取（结构化JSON输出）

```
从以下合同文本中提取关键信息，严格按指定JSON格式输出。
如果某项信息未提及，该字段值设为null。

提取字段：
- 甲方全称、乙方全称
- 合同金额（数字，单位为"万元"）
- 合同起始日期、终止日期（格式：YYYY-MM-DD）
- 付款方式（一次性/分批）
- 违约责任条款摘要（不超过50字）
- 争议解决方式（仲裁/诉讼/协商）

合同文本：
"""
[在此粘贴合同原文]
"""

输出格式：
{
  "甲方": "...",
  "乙方": "...",
  ...
}
```

### 8. 问答系统

```
你是一位精通中国历史的学者。请回答以下问题。

回答要求：
- 先给出简明的直接答案（1-2句话）
- 再提供3-5条支撑性史实证据
- 如果学术界对此问题有争议，列出至少两种主流观点
- 最后给出2-3本推荐深入阅读的书目（作者+书名+出版年份）
- 如果问题涉及的内容在史料中没有明确记载，请诚实说明

问题：明朝万历年间张居正推行的"一条鞭法"改革，
对当时的农民究竟是减轻了负担还是加重了负担？
```

### 9. 邮件撰写

```
请以"客户成功经理"的身份，使用专业、温暖但不过分热情的语气，
撰写一封邮件给试用期即将结束的企业客户。

邮件应包含：
1. 感谢对方试用产品
2. 提及对方在试用期间的使用亮点（此处留出占位信息，
   标记为[使用亮点]便于我后续填充）
3. 自然地引出续费方案（不强制推销）
4. 提供明确的下一步行动链接
5. 署名格式：姓名 | 客户成功经理 | 公司名 | 联系方式

主题行控制在20字以内。正文控制在150字以内。
```

### 10. 会议纪要整理

```
请将以下会议录音转写文本整理为标准会议纪要。

要求：
- 用表格列出"议题-讨论要点-决议-负责人-截止日期"五个维度
- 重点标注有分歧的议题（用"⚠️"标记）
- 提取所有被明确分配的任务（Action Items），单独列为"待办事项"列表
- 纪要末尾附"下次会议预告"（时间、地点、主要议题）
- 隐去发言人姓名，统一用"某同事"替代
- 使用客观中立的语言，不添加主观评价

会议转写文本：
"""
[在此粘贴转写文本]
"""
```

## 有效提示 vs 无效提示对比

理解好与坏的差距，比记住规则更重要。以下是五个任务类型的详细对比。

### 对比一：文本摘要

| | 无效提示 | 有效提示 |
|---|---|---|
| **提示文本** | "帮我总结一下这篇文章。" | "将以下文章压缩为100字以内的摘要，保留核心论点（作者立场、关键数据、最终结论），去除举例说明和修饰性语言。优先使用原文中的关键名词。" |
| **差距分析** | 没有字数限制，没有明确"什么是重要的"，模型可能抓不住重点或写得过长。 | 明确了字数、需要保留的要素、需要去除的部分，并给出了提炼策略（使用原文关键名词），模型有清晰的执行边界。 |

### 对比二：代码生成

| | 无效提示 | 有效提示 |
|---|---|---|
| **提示文本** | "写一个排序函数。" | "用Python写一个名为`quick_sort`的排序函数。输入为`list[int]`，返回升序排列的`list[int]`。需要包含：(1)完整的类型注解 (2)docstring说明算法和时间空间复杂度 (3)对空列表和单元素列表的边界处理 (4)3个assert测试用例。直接输出代码。" |
| **差距分析** | 模型可能用任意语言、任意算法实现，无测试覆盖，无边界处理。 | 锁定了语言、算法、接口签名、异常处理、测试要求。可预测、可验证。 |

### 对比三：情感分析

| | 无效提示 | 有效提示 |
|---|---|---|
| **提示文本** | "判断这条评论是正面还是负面。" | "分析以下用户评论的情感倾向。输出JSON：{"sentiment": "正面/负面/中性", "confidence": 0.0-1.0, "key_words": [...], "sarcasm_detected": true/false}。注意区分"表面正面但实际讽刺"的评论。如果评论同时包含正负两面，综合判断并附简要理由。" |
| **差距分析** | 只有二元分类，忽略中性、混合情感和反语。无法用于程序化处理。 | 输出结构化，覆盖边界（中性、讽刺、混合情感），给出置信度。程序可直接解析。 |

### 对比四：翻译

| | 无效提示 | 有效提示 |
|---|---|---|
| **提示文本** | "翻译成英文。" | "将以下中文技术文档翻译为英文。保持技术术语的准确性，首次出现的术语需在括号中标注中文原文。保持原文的编号结构和代码块不变。如果遇到中文特有的表达（如成语、俗语），优先意译而非直译，并在末尾以注释形式说明翻译策略。" |
| **差距分析** | 未指定风格和目标受众，术语可能翻译不一致，格式可能被破坏。 | 明确了领域（技术文档）、术语处理规则、格式保存要求、文化差异处理策略。结果可直接使用。 |

### 对比五：邮件撰写

| | 无效提示 | 有效提示 |
|---|---|---|
| **提示文本** | "帮我写一封回复客户的邮件。" | "你是一位专业的客户成功经理。请针对客户提出的"产品功能缺失"问题写一封回复邮件。结构：(1)感谢对方反馈 (2)承认该问题 (3)说明已在哪个版本的路线图中（或诚实地表示暂无计划）(4)提供当前可用的替代方案 (5)邀请对方参与beta测试。语气：诚恳而不卑微，专业而不冷漠。主题行不超过15字。正文不超过200字。" |
| **差距分析** | 没有上下文，不知道客户问了什么、什么语气合适、回复目的是什么。模型只能瞎猜。 | 提供了完整的沟通框架、角色定位、语气要求、字数限制。每个段落的功能都被明确指定。 |

**核心规律**：好的提示在三个维度上做到了极致——**边界清晰**（模型知道输出应该满足什么条件）、**目标明确**（模型知道任务的最终目的是什么）、**歧义消除**（模型不需要"猜"你的意图）。

## 零样本提示进阶技巧

### 技巧一：情感刺激（Emotional Stimulus）

在提示中加入情感性的激励语句，可以显著提升模型输出的质量。这不是玄学——研究表明，某些情感暗示会引导模型更认真地"对待"任务。

```
基础版：
"请检查以下代码中的安全漏洞。"

增强版：
"请检查以下代码中的安全漏洞。这段代码将在下周部署到
生产环境，关系到公司核心业务数据的安全——这对我非常重要。"
```

常用情感刺激语句："这关系到我的晋升评审"、"这份报告将被CEO直接审阅"、"这个答案将影响一个重要的商业决策"。注意：情感刺激应真实反映任务的重要性，避免夸张和滥用。

### 技巧二：格式优先设计（Format-First Design）

在提示中先定义输出格式，再描述任务内容。这利用了模型对"结构化指令"的高度敏感性。

```
请按以下结构组织你的回答：
[结论] → [理由1] → [理由2] → [理由3] → [总结]

现在，请分析远程办公对团队协作的影响。
```

相比于"请分析远程办公对团队协作的影响，回答中要包含结论、理由和总结"，格式优先设计把结构要求放在了最显眼的位置，模型更不容易遗漏。

### 技巧三：约束叠加（Constraint Stacking）

将多个简单约束叠加到一个提示中，每个约束都是明确、可验证的。约束之间应互补而非矛盾。

```
请写一篇产品介绍文案。
约束①：目标读者是35-50岁的职场中层管理者
约束②：长度在200-250字之间
约束③：每段不超过3句话
约束④：至少使用2个数据支撑观点
约束⑤：不使用"非常""极其""无与伦比"等夸张词汇
约束⑥：以一个问题开头，吸引读者注意
```

约束叠加的关键是每个约束都独立可验证、彼此不冲突。六个明确的小要求远比一个"写得好一点"的模糊要求有效。

### 技巧四：负面约束（Negative Constraints）

与要素二中的"正面指令胜过负面指令"原则结合使用——以正面指令为主，负面约束为辅。

```
请写一篇关于气候变化的科普文章。

应该做的：
✓ 从日常生活现象切入主题
✓ 使用具体数字说明趋势
✓ 结尾给出读者可以个人行动的3条建议

不应该做的：
✗ 不要使用引发焦虑的灾难性语言
✗ 不要提及争议性政治话题
✗ 不要超过1200字
```

正面指令定方向，负面约束划红线。两者配合使用效果最佳。

### 技巧五：输出模式规范（Output Schema Specification）

当需要精确的机器可读输出时，直接提供完整的schema定义。

```
请分析以下用户反馈并输出。

输出schema（严格遵循）：
{
  "category": "<必须从以下选一：bug_report/feature_request/general_feedback>",
  "urgency": "<必须从以下选一：critical/high/medium/low>",
  "affected_module": "<string, 如未明确提及则填 'unknown'>",
  "is_actionable": <boolean>,
  "suggested_owner_team": "<string | null>"
}

用户反馈：
"""
你们的APP更新后每次打开都要重新登录，烦死了！
而且暗黑模式下的字体颜色太浅，根本看不清。
"""
```

Schema不仅规定了字段名和类型，还通过尖括号注释明确了每个字段的合法取值范围。

### 技巧六：零样本自洽性（Self-Consistency in Zero-shot）

对于关键决策场景，对同一个零样本提示多次调用（使用temperature>0），取出现频率最高的结果：

```python
def zero_shot_self_consistency(llm, prompt, n=5, temperature=0.7):
    results = []
    for _ in range(n):
        response = llm.invoke(prompt, temperature=temperature)
        results.append(response.strip())
    from collections import Counter
    return Counter(results).most_common(1)[0][0]
```

这种方式虽然增加了调用成本，但在需要高可靠性的场景（如医疗文书分析、合同审查）中值得投入。研究表明，在算术推理任务中，5次采样的自洽性能将准确率提升5-15个百分点。

### 技巧七：后退提示（Step-Back Prompting）

对于复杂问题，先让模型退一步思考"这个问题到底在问什么"以及"解决这个问题需要哪些前置知识"，再进行回答。

```
在回答以下问题之前，请先完成两步后退思考：

后退步骤1：确定这个问题属于哪个知识领域？
它考察的核心概念是什么？

后退步骤2：解决这类问题通常需要哪些前提条件或公式？
列出你需要的所有基础知识。

完成以上两步后，再回答原问题：
"如果地球自转速度突然减慢一半，赤道地区的重力加速度
会发生怎样的变化？请给出定量分析。"
```

后退提示让模型先建立"知识框架"，再填充具体内容，显著减少了在复杂推理任务中"跑偏"的情况。

## 零样本 vs 少样本：选择指南

| 判断维度 | 用零样本 | 用少样本 |
|---------|---------|---------|
| 任务新颖性 | 常见任务 | 少见或定制化任务 |
| 模型能力 | 强模型（GPT-4/Claude 3.5+） | 弱模型（GPT-3.5等） |
| 输出格式 | 简单（单标签、短文本） | 复杂（特定模板、特殊风格） |
| 成本敏感度 | 高 | 低 |
| 稳定性要求 | 中低 | 极高（医疗、法律、金融） |
| 输出多样性 | 需要多样性（创意写作） | 需要一致性（批量分类） |

**混合策略建议**：先用零样本提示快速验证可行性。如果效果不够好，逐步加入1-2个精选示例。很多时候，1个好的示例比5个平庸的示例更有效。

## 常见错误与解决方案

**错误一：指令过于笼统。**"写一篇关于机器学习的文章"——模型有太多自由度，输出可能完全偏离预期。**解决**：明确目标读者、字数、风格、结构。例如"请为计算机专业大一新生写一篇介绍机器学习的科普文章，1200-1500字，用轻松易懂的类比方式，先讲应用故事、再讲核心概念、最后推荐3个学习资源。"

**错误二：指令互相矛盾。**"请详细解释量子计算，要非常深入，但要简短"——"深入"与"简短"直接矛盾。**解决**：给出可兼容的约束，如"请用300字简要介绍量子计算的核心概念。末尾推荐2本深入阅读的书籍。"

**错误三：过度依赖负面约束。**连续七八个"不要"让模型束手束脚。**解决**：以正面指令为主体，搭配2-3个关键的负面约束。正面指令告诉模型该做什么，负面约束划定红线。

**错误四：忽略边界情况。**不做空值处理、异常格式处理，导致不可预测的输出。**解决**：明确列出边界规则——"如果输入为空，返回…"、"如果遇到无法识别的内容，用[特定标记]标注"。

## 迭代优化流程

写好零样本提示不是一蹴而就的：

1. 写一个基础版本
2. 在10-20个测试用例上运行
3. 分析失败案例，找出共同模式（模型是否理解核心目标？输出格式是否一致？有无幻觉？）
4. 修改提示词（添加约束、澄清歧义、调整角色设定）
5. 重复2-4步，直到满意
6. 在留出的独立测试集上做最终验证

## 总结与进阶方向

零样本提示是提示词工程的地基。它不依赖示例、成本可控、灵活应变，是日常与AI协作中最常用的方式。写好零样本提示的核心不是记住模板，而是培养一种思维方式——每次写提示之前问自己：**模型需要知道什么才能准确完成这个任务？我是否消除了所有可能的歧义？**

掌握了零样本提示之后，可以继续学习：
- **少样本提示（Few-shot Prompting）**：用精选示例引导模型行为，适合复杂和定制化任务
- **思维链提示（Chain-of-Thought）**：让模型展示逐步推理，大幅提升复杂推理准确率
- **思维树（Tree of Thoughts）**：多路径推理与回溯搜索，适合需要规划和探索的开放性问题
- **自动提示优化**：使用DSPy等框架系统化地搜索和优化提示词

零样本提示是你与AI每天对话的默认语言。把它练成肌肉记忆，你就掌握了高效利用AI的第一步。
