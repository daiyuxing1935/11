# 提示词标准结构：从一句话到工程化体系

## 引言

如果你用过ChatGPT，你的第一次尝试大概率是这样的：

```
帮我写一篇关于人工智能的文章。
```

模型确实会返回一篇文章——但内容可能泛泛而谈，风格可能不符合你的预期，长度可能完全失控。于是你继续追问："太长了，缩短到500字"，"风格太正式了，活泼一点"，"加入一些数据案例"……几轮对话下来，你发现自己不是在"使用"AI，而是在"和AI讨价还价"。

这个问题的根源在于：**你给模型的指令缺乏结构**。就像你让一个实习生"去做市场调研"，却不告诉他调研哪些维度、用什么格式呈现、需要多少数据点——他交上来的东西大概率不是你想要的。

**提示词标准结构**正是为了解决这个问题而生。它把随意的自然语言指令，升级为一种工程化的、可复用的、可优化的技术组件。本文将从底层原理到工程实践，系统讲解如何设计和维护结构化提示词。

## 为什么需要结构化提示：从"直觉驱动"到"工程驱动"

### 一句话提示的五个致命缺陷

在AI应用的早期阶段（以及大多数人的日常使用中），提示词通常是这样的：

```
把这段话翻译成英文。
```

```
这段代码有什么bug？
```

```
帮我写一封辞职信。
```

这种"一句话提示"看似高效，实则埋下了五个严重的隐患：

| 缺陷 | 具体表现 | 实际后果 |
|------|---------|---------|
| **意图模糊** | "翻译成英文"——什么风格？商务还是口语？ | 结果可能完全不适用于目标场景 |
| **边界缺失** | "有什么bug"——只看逻辑错误还是也包括风格问题？ | 输出内容不可控，可能遗漏关键问题 |
| **格式不确定** | "写一封辞职信"——正式还是随意？多长？什么结构？ | 每次输出都不同，无法程序化处理 |
| **不可复现** | 同样的提示在不同对话中效果波动极大 | 无法在团队中标准化使用 |
| **无法迭代优化** | 不知道改哪里能提升效果 | 永远在"凭感觉"调提示词 |

这些问题的本质是：**你把自己的需求"压缩"成了一句自然语言，期望模型"解压缩"出你脑中的完整意图**。这本质上是一种"心灵感应"——你期望模型读懂你的心思。但模型没有读心术，它只能基于字面意思和统计规律来生成回答。

### 从"和AI聊天"到"编写AI程序"

当我们谈论"提示词工程"（Prompt Engineering）时，本质上是在完成一个观念的转变：

```
聊天思维：  "帮我做X"
工程思维：  "定义输入规范 → 配置执行参数 → 约束输出格式 → 处理异常情况"
```

结构化提示词就是把一段模糊的对话，升级为一个包含角色、任务、约束、输入、输出格式的完整"程序"。这个"程序"的"编程语言"是自然语言，但它具备了软件的许多关键属性：可读性、可测试性、可复用性、可版本管理。

**一个直观的类比：**

```
一句话提示 ≈ 在命令行里直接敲 python -c "print(sum(range(100)))"
结构化提示 ≈ 一个完整的 .py 文件，有函数定义、类型注解、文档字符串、单元测试
```

前者能跑，但后者才能被维护、被团队使用、被持续改进。

### 结构化提示的核心架构

一个成熟的结构化提示词，通常包含五个层次：

```
┌─────────────────────────────────────────────┐
│           第0层：元信息（Meta）               │
│   版本号、作者、创建日期、适用模型、标签       │
├─────────────────────────────────────────────┤
│           第1层：系统指令（System Prompt）     │
│   角色设定、行为约束、能力边界、背景知识       │
├─────────────────────────────────────────────┤
│           第2层：用户指令（User Prompt）       │
│   任务描述、输入数据、约束条件、上下文         │
├─────────────────────────────────────────────┤
│           第3层：输出格式（Output Schema）     │
│   JSON Schema、Markdown模板、类型约束         │
├─────────────────────────────────────────────┤
│           第4层：示例（Examples）              │
│   正例、反例、边界案例、推理示例              │
└─────────────────────────────────────────────┘
```

接下来的章节将逐层深入解析每一层的设计原则和最佳实践。

## 第一层：系统指令（System Prompt）——定义"你是谁"

### 什么是系统指令

在大多数LLM API中（如Claude、GPT-4、Gemini），消息分为三种角色：

```
System:   定义AI的整体行为、角色和规则（优先级最高）
User:     用户的具体请求和输入数据
Assistant: AI的回复（在多轮对话中提供历史上下文）
```

系统指令（System Prompt）是整个提示词体系的"宪法"。它位于消息列表的最顶部，定义了AI在整个对话中的**身份、行为准则和能力边界**。从模型的角度看，系统指令的作用是设置"生成分布的先验"——后续所有token的采样都将受到这条"宪法"的约束。

### 系统指令的四大设计维度

#### 1. 角色设定（Persona Definition）

角色设定是最直观也是最重要的系统指令组件。一个精心设计的角色不仅定义了AI的"身份"，更激活了训练数据中与该角色相关的语言模式和知识域。

**角色设定的三层结构：**

```
[身份] + [专业领域] + [沟通风格]

示例：
"你是一位拥有15年经验的云架构师（身份），
专精于AWS和Kubernetes（专业领域），
善于用简洁的图表语言和类比来解释复杂概念（沟通风格）。"
```

**角色设定的科学依据：**

角色设定之所以有效，是因为大语言模型的训练数据中包含了大量不同身份人士的文本。当你指定一个角色时，相当于在模型的潜在空间中划定了一个"先验区域"——模型生成的内容会更偏向于该角色相关的分布。例如，设定"数学教授"角色后，模型更可能使用严谨的定义、逐步推导和学术化的表述方式。

**常见角色类型的适用场景：**

| 角色类型 | 适用场景 | 角色描述示例 |
|---------|---------|-------------|
| **领域专家** | 技术咨询、专业分析 | "你是一位资深网络安全专家，拥有CISSP认证和10年渗透测试经验" |
| **导师/教练** | 教学、培训、指导 | "你是一位耐心的编程导师，擅长用苏格拉底式提问引导学员自己找到答案" |
| **助手/执行者** | 日常任务、自动化 | "你是一位高效的个人助理，回复简洁，一次只确认一个事项" |
| **创意伙伴** | 写作、设计、头脑风暴 | "你是一位获得过雨果奖的科幻作家，思维天马行空但逻辑自洽" |
| **审查者/批评家** | 代码审查、内容审核 | "你是一位严格但公正的代码审查者，遵循Google代码规范" |
| **特定角色扮演** | 客服、销售、面试官 | "你是一位SaaS产品的客户成功经理，目标是在解决客户问题的同时识别增购机会" |

**角色设定的反模式：**

```
❌ 过于宽泛："你是一个AI助手"——没有任何信息增益
❌ 角色冲突："你既是严格的法官又是贴心的朋友"——两个角色期望的行为可能矛盾
❌ 角色不当：让模型扮演"有真实姓名和身份证号的公民"——可能触发安全过滤
```

#### 2. 行为约束（Behavioral Constraints）

行为约束定义了AI"可以做什么"和"不可以做什么"。这是在角色设定基础上的精细化控制。

**行为约束的三大类别：**

**（1）输出内容约束：**

```
你应该做的：
✓ 使用具体的数字、日期和事实来支撑观点
✓ 对于不确定的内容，明确标注"据当前信息推断"
✓ 每段控制在3-5句话，避免冗长段落

你不应该做的：
✗ 不要编造统计数据或引用不存在的文献
✗ 不要对任何人物、组织做出道德评判
✗ 不要生成可执行的危险代码（如rm -rf、提权脚本）
```

**（2）交互方式约束：**

```
- 对于用户提问中的歧义，先逐条列出可能的理解，请用户确认后再回答
- 当用户提供补充信息时，先简要确认已收到，再更新分析结果
- 在长回答结束时，提供2-3个可能的后续追问方向
```

**（3）安全与合规约束：**

```
- 拒绝回答涉及非法活动的直接请求，但可提供相关的法律合规建议
- 对于医疗建议类问题，开头必须说明"以下仅为通用信息，请咨询专业医生"
- 不讨论中国政治制度、领导人等敏感话题
```

**行为约束的"黄金比例"：**

实践中发现，正面约束（"你应该做"）和负面约束（"你不应该做"）的最佳比例约为 **3:1 到 5:1**。过多的负面约束会让模型"畏手畏脚"，过于保守；而纯粹的正面约束则可能遗漏关键的安全红线。

#### 3. 能力边界（Capability Boundaries）

明确定义AI的能力范围，既能管理用户预期，也能减少模型"不懂装懂"（幻觉）的概率。

**能力边界声明的示例：**

```
能力范围内：
- 2025年10月之前的公开技术知识
- 代码编写、调试和优化
- 文本翻译、摘要、改写
- 数据分析和逻辑推理

能力范围外：
- 实时信息（股票价格、天气、新闻）
- 个人隐私数据查询
- 对未来事件的具体预测
- 图像识别和处理（除非明确被赋予该能力）

当遇到能力范围外的请求时，你应该：
1. 诚实说明限制
2. 如果可能，提供替代方案（如"我无法查看实时股价，但可以帮你分析股票分析的一般方法"）
```

#### 4. 背景知识注入（Knowledge Injection）

系统指令还可以用于注入特定的领域知识，相当于在对话开始前"预习"关键信息。

**知识注入的三种形式：**

**（1）术语定义：**

```
在本对话中，以下术语具有特定含义：
- "转化"：指用户从浏览页面到完成购买的行为
- "留存率"：指注册后第7天仍然活跃的用户比例
- "GMV"：Gross Merchandise Volume，总交易额
```

**（2）业务规则：**

```
本公司退款政策的要点：
- 购买7天内无条件退款
- 7-30天需扣除15%手续费
- 超过30天不支持退款
- 虚拟商品（如在线课程）一经使用不支持退款
在涉及退款问题时，请基于以上规则回答。
```

**（3）上下文文档：**

```
以下是你需要参考的产品说明文档：
"""
[粘贴产品文档内容]
"""
请在回答用户问题时，优先参考以上文档内容。
如果文档中没有覆盖到的问题，基于你的通用知识回答，
并说明"以下信息基于通用知识，建议与产品团队确认"。
```

### 系统指令的完整模板

综合以上四个维度，这里给出一个可直接参考的系统指令模板：

```markdown
## 角色设定
你是一位[职业/角色]，拥有[年限]年的[领域]经验，专精于[核心技术/技能]。
你的沟通风格是[风格描述：如简洁务实/循循善诱/严谨学术]。

## 核心行为准则
### 你应该做的：
1. [正面行为1]
2. [正面行为2]
3. [正面行为3]

### 你不应该做的：
1. [负面行为1]
2. [负面行为2]

## 能力范围
### 你能做的：
- [能力1]
- [能力2]

### 你不能做的：
- [限制1]
- [限制2]

## 领域知识
### 关键术语
- [术语1]：[定义]
- [术语2]：[定义]

### 业务规则
- [规则1]
- [规则2]

## 参考文档
"""
[文档内容]
"""

## 响应规范
- 首次回复先确认理解了用户需求
- 不确定时列出假设并请求确认
- 复杂分析后附简要总结
```

### 系统指令的长度权衡

系统指令并非越长越好。一个常见的误解是"信息越多，模型越精准"。但实际中存在一个**信息密度拐点**：

| 系统指令长度 | 优点 | 缺点 | 适用场景 |
|------------|------|------|---------|
| 简短（<200 tokens） | 占用少，响应快，歧义少 | 行为控制力弱 | 简单单轮任务 |
| 中等（200-800 tokens） | 行为可控，信息充分 | — | **大多数场景的最佳区间** |
| 长（800-2000 tokens） | 覆盖面广，规则详尽 | 指令间可能冲突，模型可能"遗忘"尾部内容 | 复杂多轮对话系统 |
| 超长（>2000 tokens） | 逼近"操作手册"级别 | "中间丢失"效应显著，成本高 | 需配合RAG分段注入 |

**"中间丢失"效应**（Lost-in-the-Middle）是长系统指令的主要风险。研究表明，LLM对指令的开头和结尾关注度最高，中间部分容易被"忽略"。因此，**最重要的约束放在开头和结尾**，次要信息放在中间。

## 第二层：用户指令（User Prompt）——定义"做什么"

### 用户指令的四要素

用户指令（User Prompt）是用户向模型发送的具体任务请求。一个完整的用户指令应包含四个要素：

```
[任务描述] + [输入数据] + [约束条件] + [上下文信息]
```

#### 要素一：任务描述（Task Description）

任务描述是用户指令的核心。它的设计遵循一个黄金原则：**具体、可验证、可操作**。

**任务描述的信息粒度层级：**

```
层级1（过于笼统）：  "帮我分析一下这份数据。"
层级2（有一定方向）："帮我分析一下这份销售数据，看看有什么趋势。"
层级3（结构化描述）："分析这份2025年Q1-Q4的销售数据，识别：
                     (1) 各季度的环比增长率
                     (2) 增长最快和最慢的产品线
                     (3) 是否存在季节性规律"
层级4（精确可验证）："分析这份销售数据（CSV格式，字段见下文），
                     输出一份不超过500字的分析报告，包含：
                     (1) Q1-Q4的环比增长率表格（保留1位小数）
                     (2) 按产品线的同比增长率排名
                     (3) 基于数据推断的至少2条业务建议
                     (4) 如有数据异常点，单独标注说明"
```

层级之间的差距是显著的：层级4的输出可以被自动化检查（字数是否超过500？增长率是否保留1位小数？是否至少2条建议？），而层级1的输出完全不可预测。

**任务描述的结构化框架（5W1H）：**

```
What（做什么）：  [具体任务，如"分类"、"翻译"、"生成"、"分析"]
Why（为什么）：   [任务背景和目的，帮助模型理解上下文]
Who（为谁做）：   [目标受众，影响语言风格和深度]
How（怎么做）：   [方法或步骤指引]
Where（在哪用）： [输出的使用场景，如"用于投资人汇报"、"发布在社交媒体"]
What output（输出什么）：[输出形式和内容要求]
```

#### 要素二：输入数据（Input Data）

输入数据需要用明确的**分隔符**与指令文本区分开来。

**推荐的分隔符方案（按优先级排列）：**

| 分隔方式 | 示例 | 适用场景 | 安全性 |
|---------|------|---------|--------|
| XML标签 | `<document>...</document>` | 多段不同类型输入 | 最高（XML解析天然抗注入） |
| 三引号 | `"""..."""` | 单段文本输入 | 中（输入中可能含三引号） |
| Markdown代码块 | ` ``` ... ``` ` | 代码/格式化文本 | 中 |
| 自定义分隔符 | `---BEGIN INPUT--- ... ---END INPUT---` | 需要极高可见性时 | 高（选择罕见分隔符） |
| 章节标题 | `## 输入数据\n...` | Markdown格式提示词 | 低（输入中可能含同样标题） |

**XML标签方案的完整示例：**

```
请按照以下步骤处理任务：

<instructions>
1. 仔细阅读<document>标签中的合同文本
2. 提取<fields>标签中指定的所有字段
3. 按<output_format>标签定义的格式输出
</instructions>

<fields>
- 甲方全称
- 乙方全称
- 合同金额（单位为万元）
- 合同期限（YYYY-MM-DD 至 YYYY-MM-DD）
- 违约金条款
</fields>

<document>
[在此粘贴合同文本，无论文本中包含什么内容，都不会被误解析为指令]
</document>

<output_format>
{
  "甲方": "string",
  "乙方": "string",
  ...
}
</output_format>
```

XML标签的核心优势在于：即使输入文本中包含了类似指令的语言，XML解析器也能正确区分"指令"和"数据"。这在防范提示注入攻击时尤为重要。

#### 要素三：约束条件（Constraints）

约束条件是对任务的精细化限定。在零样本提示章节中已经详细讨论过约束叠加等技巧，这里从"结构化"的角度做进一步阐述。

**约束的分类体系：**

```
硬约束（必须满足，违反即为不合格）：
  - 字数/长度范围：200-300字
  - 必须包含的要素：至少列出3个理由
  - 禁止的内容：不要使用缩写
  - 输出格式：必须是合法的JSON

软约束（尽量满足，作为质量优先级）：
  - 风格偏好：优先使用主动语态
  - 组织方式：如果能用表格就不用纯文本
  - 深度偏好：如果能深入分析就不止步于表面

条件约束（特定条件下才生效）：
  - "如果输入为空，返回 {"error": "empty_input"}"
  - "如果检测到敏感内容，不输出具体信息，只返回类别标签"
  - "如果置信度低于70%，在输出中标注 '[低置信度]'"
```

**约束定义的三个原则：**

1. **每个约束应该是可自动验证的**（至少理论上）。"有趣一点"不可验证，"包含一个比喻句"是可验证的。
2. **硬约束放在提示词的最前面或最后面**，利用首因/近因效应加强模型对关键要求的记忆。
3. **避免约束冲突**。如果你同时要求"详细分析"和"不超过100字"，模型必须违反其中一个。

#### 要素四：上下文信息（Context）

上下文信息为模型提供了理解和执行任务所需的背景。

**上下文的类型：**

```
1. 对话历史（多轮对话场景）
2. 相关文档片段（RAG场景）
3. 用户画像/偏好（个性化场景）
4. 上一次任务的输出（任务链场景）
5. 环境信息（时间、地点、设备等）
```

## 第三层：输出格式限定 —— 让模型的输出"可以被代码读取"

### 为什么输出格式如此重要

在AI应用的生产环境中，模型的输出很少是直接展示给用户的。更常见的情况是：

```
模型输出 → 解析 → 存入数据库 → 触发下游流程 → 最终呈现
```

如果模型的输出格式不稳定或不规范，整个数据管道就会崩溃。输出格式限定就是为模型的输出装上"接口"——就像编程中的API返回结构化的JSON而不是一段自由文本。

### 输出格式的四种层次

#### 层次一：自由文本（Free Text）

```
适用：直接面向终端用户的内容（如对话回复、文章生成）
风险：程序无法可靠解析，不适合自动化流程
示例："请写一篇关于AI伦理的文章"
```

#### 层次二：Schema约束文本（Semi-Structured）

```
适用：需要从输出中提取特定信息的场景
风险：仍需要后处理（正则提取），边缘情况可能失败
示例：输出的每行格式为"产品名 | 价格 | 评分"
```

#### 层次三：JSON/结构化输出（Structured Output）

```
适用：所有需要程序化处理的场景（API、数据库写入、下游任务）
风险：模型可能生成非严格合法的JSON（多余逗号、注释、不闭合括号）
示例：要求输出 {"name": "...", "price": ..., "category": "..."}
```

#### 层次四：Schema验证输出（Schema-Validated）

```
适用：对输出格式有严格一致性要求的生产环境
实现：使用JSON Schema定义期望的输出结构，并做后处理验证
风险：模型可能在复杂schema下"丢失"某些字段
```

### JSON Schema约束的完整实践

## 第四层：提示词模板化 —— 从"手写"到"生成"

### 为什么需要模板化

在实际应用中，提示词很少是"一次性手写"的。同一个任务往往需要处理不同的输入数据，同一个系统往往需要同时运行多个提示词变体。提示词模板化就是将提示词中的可变部分抽象为参数，实现"一次设计，多次使用"。

### 模板化的三种模式

#### 模式一：变量插值（Variable Interpolation）

这是最基础的模板化方式，类似于邮件合并：

```python
# 模板定义
PROMPT_TEMPLATE = """
你是一位{role}。请将以下{source_lang}文本翻译为{target_lang}。

翻译要求：
- 保持{style}语气
- 字数控制在{word_limit}字以内
- 专业术语参考：{glossary}

原文：
\"\"\"
{input_text}
\"\"\"

请直接输出翻译结果，不要添加解释。
"""

# 模板使用
prompt = PROMPT_TEMPLATE.format(
    role="资深医学翻译",
    source_lang="英文",
    target_lang="中文",
    style="学术论文",
    word_limit=500,
    glossary="MRI=核磁共振成像, CT=计算机断层扫描",
    input_text=document
)
```

**常见模板引擎方案：**

| 方案 | 语法 | 优势 | 适用场景 |
|------|------|------|---------|
| Python f-string | `{variable}` | 原生支持，零依赖 | 简单场景 |
| Jinja2 | `{{ variable }}` | 支持条件/循环/过滤器 | 复杂模板 |
| Mustache | `{{ variable }}` | 跨语言，逻辑精简 | 多语言项目 |
| LangChain PromptTemplate | `{variable}` | 与LLM框架深度集成 | LangChain项目 |

#### 模式二：条件分支（Conditional Branching）

当提示词的结构需要根据输入特征动态变化时，条件分支就派上了用场：

```python
def build_prompt(user_query: str, user_level: str, language: str) -> str:
    """根据用户等级和语言动态构建提示词"""

    # 基础系统指令
    system_prompt = "你是一位编程导师。"

    # 根据用户等级调整教学深度
    if user_level == "beginner":
        system_prompt += """
        教学策略：
        - 每次只讲一个概念
        - 使用生活化类比
        - 代码示例要加详细注释
        - 避免使用专业术语，如果必须用，要解释
        """
    elif user_level == "intermediate":
        system_prompt += """
        教学策略：
        - 可以同时讲解相关联的多个概念
        - 关注最佳实践和设计模式
        - 鼓励读者思考"为什么这样做"
        """
    elif user_level == "advanced":
        system_prompt += """
        教学策略：
        - 重点讨论底层原理和性能权衡
        - 引入源码分析和系统设计视角
        - 可以假定读者已具备扎实的基础知识
        """

    # 根据语言选择代码风格
    if language == "python":
        system_prompt += "\n代码风格：遵循PEP 8，使用类型注解，优先展示Pythonic写法。"
    elif language == "javascript":
        system_prompt += "\n代码风格：遵循ESLint推荐配置，使用const/let，展示现代ES6+语法。"

    return f"{system_prompt}\n\n用户问题：{user_query}"
```

#### 模式三：模块化组合（Modular Composition）

对于复杂系统，将提示词拆分为可复用的模块，然后根据需求动态组装：

```python
class PromptModule:
    """提示词模块基类"""
    def render(self) -> str:
        raise NotImplementedError

class RoleModule(PromptModule):
    def __init__(self, role_name: str, expertise: str, style: str):
        self.role_name = role_name
        self.expertise = expertise
        self.style = style

    def render(self) -> str:
        return f"""## 角色设定
你是一位{self.role_name}，专精于{self.expertise}。
沟通风格：{self.style}。
"""

class SafetyModule(PromptModule):
    def render(self) -> str:
        return """## 安全约束
- 不生成任何形式的恶意代码
- 对于医疗/法律建议，必须添加免责声明
- 不讨论政治敏感话题
"""

class OutputFormatModule(PromptModule):
    def __init__(self, format_type: str, schema: dict = None):
        self.format_type = format_type
        self.schema = schema

    def render(self) -> str:
        if self.format_type == "json":
            return f"""## 输出格式
必须以合法JSON格式输出，遵循以下schema：
{json.dumps(self.schema, indent=2, ensure_ascii=False)}

不要在JSON外输出任何其他内容。
"""
        elif self.format_type == "markdown":
            return """## 输出格式
使用Markdown格式，包含标题、表格和列表。
"""
        return ""

# 模块组装器
class PromptAssembler:
    def __init__(self):
        self.modules: list[PromptModule] = []

    def add_module(self, module: PromptModule) -> "PromptAssembler":
        self.modules.append(module)
        return self  # 支持链式调用

    def build(self, user_prompt: str) -> str:
        system_part = "\n".join(m.render() for m in self.modules)
        return f"{system_part}\n---\n{user_prompt}"

# 使用示例
assembler = PromptAssembler()
assembler.add_module(RoleModule("技术文档工程师", "API文档编写", "简洁准确"))
assembler.add_module(SafetyModule())
assembler.add_module(OutputFormatModule("markdown"))

final_prompt = assembler.build("请为以下Python函数编写API文档：...")
```

### 提示词模板的配置化

## 常见错误与最佳实践

### 十大常见错误

| # | 错误 | 错误示例 | 正确做法 |
|---|------|---------|---------|
| 1 | **角色和任务不匹配** | 系统指令说"你是翻译"，但用户指令要求"写代码" | 确保系统指令和用户指令一致 |
| 2 | **约束互相矛盾** | "详细分析"+"不超过50字" | 每个约束独立检查，确保不冲突 |
| 3 | **分隔符选择不当** | 用`"""`包裹包含`"""`的文本 | 使用XML标签或罕见分隔符 |
| 4 | **Schema与示例不一致** | Schema说字段名是`phone`，示例写的是`phone_number` | 生成前做schema-example交叉验证 |
| 5 | **忽略边界情况** | 没说输入为空时怎么办 | 明确每个异常分支的处理逻辑 |
| 6 | **系统指令过长** | 3000+ tokens的系统指令 | 控制在800 tokens以内，超量部分用RAG动态注入 |
| 7 | **模板变量不做转义** | 变量值中包含`{`或`}`破坏模板结构 | 对变量值做转义处理 |
| 8 | **忘记标注不确定性** | 模型在信息不足时仍"自信"输出 | 要求模型标注confidence或使用null |
| 9 | **只用正面示例** | 所有示例都是理想情况 | 加入1-2个边界/异常示例 |
| 10 | **版本管理缺失** | 没有版本号，改了也不知道 | 提示词文件头部包含版本号、修改日期、变更说明 |

**设计阶段：**

## 结构化提示的版本管理与A/B测试

### 为什么提示词需要版本管理

提示词是现代AI应用的"源代码"之一。与传统的软件代码一样，提示词也需要版本管理，原因如下：

1. **可回滚**：新版提示词可能在某类案例上表现变差，需要快速回退
2. **可追溯**：当线上出现问题时，能快速定位是哪个版本的提示词导致的
3. **可协作**：团队成员可以并行开发不同版本的提示词
4. **可审计**：合规场景下需要知道"在某时间点，系统使用的是哪套规则"

### A/B测试框架

A/B测试是验证提示词修改效果的科学方法。在生产环境中，将流量分流到不同的提示词版本，通过业务指标判断哪个版本更好。

**A/B测试的架构设计：**

```
                    ┌─────────────────┐
                    │   用户请求        │
                    └────────┬────────┘
                             │
                    ┌────────▼────────┐
                    │   流量分配器      │
                    │  (Feature Flag)  │
                    └──┬──────────┬───┘
                       │          │
              90%流量  │          │  10%流量
              ┌────────▼──┐  ┌───▼─────────┐
              │ 版本A      │  │ 版本B        │
              │ (v2.0.1)  │  │ (v2.1.0-rc1) │
              └────────┬──┘  └───┬─────────┘
                       │          │
              ┌────────▼──┐  ┌───▼─────────┐
              │ 指标采集    │  │ 指标采集      │
              └────────┬──┘  └───┬─────────┘
                       │          │
              ┌────────▼──────────▼──────┐
              │      统计分析             │
              │  (显著性检验 + 效应量)     │
              └──────────────────────────┘
```

**A/B测试代码示例：**

```python
import hashlib
from dataclasses import dataclass
from typing import Optional

@dataclass
class ExperimentConfig:
    """A/B测试配置"""
    experiment_id: str
    variant_a: str  # 提示词版本号（对照组）
    variant_b: str  # 提示词版本号（实验组）
    traffic_split: float = 0.1  # B组流量比例，默认10%
    min_sample_size: int = 1000  # 每个变体的最小样本量

class PromptABRouter:
    """基于用户ID的一致性哈希流量路由"""

    def __init__(self, config: ExperimentConfig):
        self.config = config

    def get_variant(self, user_id: str) -> str:
        """
        使用一致性哈希确保同一用户始终路由到同一变体。
        这避免了用户在新旧版本之间切换时的不一致体验。
        """
        hash_key = f"{self.config.experiment_id}:{user_id}"
        hash_value = int(hashlib.md5(hash_key.encode()).hexdigest(), 16) % 10000
        threshold = int(self.config.traffic_split * 10000)
        return self.config.variant_b if hash_value < threshold else self.config.variant_a

    def get_prompt_for_user(
        self, user_id: str, prompt_registry: dict
    ) -> str:
        """获取该用户应该使用的提示词版本"""
        variant = self.get_variant(user_id)
        return prompt_registry[variant]

# 使用示例
registry = {
    "v2.0.1": load_prompt("prompts/intent_classifier/v2.0.1.yaml"),
    "v2.1.0-rc1": load_prompt("prompts/intent_classifier/v2.1.0-rc1.yaml"),
}

experiment = ExperimentConfig(
    experiment_id="exp-2025-12-intent-v2.1",
    variant_a="v2.0.1",
    variant_b="v2.1.0-rc1",
    traffic_split=0.1
)

router = PromptABRouter(experiment)
prompt = router.get_prompt_for_user(user_id="user_12345", prompt_registry=registry)
```

**A/B测试的关键指标：**

| 指标类别 | 具体指标 | 说明 |
|---------|---------|------|
| **输出质量** | 意图分类准确率 | 需人工标注验证集作为ground truth |
| **输出质量** | Schema合规率 | JSON是否可解析、必填字段是否齐全 |
| **输出质量** | 人工介入率 | 是否减少了不必要的升级 |
| **用户体感** | 问题解决率 | 用户是否在后续消息中表示问题已解决 |
| **用户体感** | 用户满意度评分 | 对话结束后的评分 |
| **系统效率** | 平均解决轮次 | 从发起到解决需要的对话轮数 |
| **系统效率** | 平均延迟 | 提示词越长，延迟越高 |
| **系统效率** | Token消耗 | 输入+输出token数 |

**A/B测试的停止规则：**

```python
def should_stop_experiment(
    results_a: list[float],
    results_b: list[float],
    config: ExperimentConfig
) -> Optional[str]:
    """
    判断是否应该停止A/B测试。
    返回 None = 继续测试
    返回 "A" = A组显著更好，切换到A
    返回 "B" = B组显著更好，切换到B
    返回 "inconclusive" = 已收集足够样本但差异不显著
    """
    import numpy as np
    from scipy import stats

    n_a, n_b = len(results_a), len(results_b)

    # 样本量检查
    if n_a < config.min_sample_size or n_b < config.min_sample_size:
        return None  # 样本不足，继续收集

    # 独立样本t检验
    t_stat, p_value = stats.ttest_ind(results_a, results_b)

    # 效应量（Cohen's d）
    pooled_std = np.sqrt((np.std(results_a)**2 + np.std(results_b)**2) / 2)
    cohens_d = abs(np.mean(results_a) - np.mean(results_b)) / pooled_std

    ALPHA = 0.05  # 显著性水平
    MIN_EFFECT_SIZE = 0.2  # 最小有效效应量

    if p_value < ALPHA and cohens_d >= MIN_EFFECT_SIZE:
        winner = "A" if np.mean(results_a) > np.mean(results_b) else "B"
        return winner
    elif n_a > config.min_sample_size * 3:  # 超过3倍最小样本量
        return "inconclusive"  # 收集足够多数据，差异仍不显著

    return None  # 继续收集
```

### 灰度发布策略

除了A/B测试，灰度发布（Gradual Rollout）是降低风险的另一种常用策略：

```
阶段0：内部测试（0% 生产流量）
  └─ 在评估集上运行，确保所有指标不下降

阶段1：金丝雀发布（1% 生产流量，持续24小时）
  └─ 监控错误率、延迟、用户投诉

阶段2：小范围灰度（10% 生产流量，持续48小时）
  └─ 与旧版本做A/B对比

阶段3：半量灰度（50% 生产流量，持续24小时）
  └─ 确认大规模下无资源问题（如QPS瓶颈）

阶段4：全量发布（100% 生产流量）
  └─ 保留旧版本配置48小时，以备紧急回滚
```

## 总结

提示词的标准结构不是一套僵化的模板，而是一种**工程化思维**——将模糊的自然语言交互，转变为可设计、可测试、可迭代、可维护的技术系统。

回顾本文的核心要点：

- **系统指令**定义了"你是谁"——角色的精确设定远比"我是AI助手"能激发更精准的行为模式
- **用户指令**明确了"做什么"——用5W1H框架让你的需求无歧义地传达给模型
- **输出格式限定**确保了"怎么接收"——JSON Schema和结构化输出让AI的输出成为可靠的API
- **模板化**实现了"一次设计、多次使用"——变量插值、条件分支、模块化组合让提示词从手工艺品升级为工业化产品
- **版本管理和A/B测试**完成了"持续优化"——用数据驱动的方式迭代提示词，而不是凭感觉调参

结构化提示词的价值链是清晰的：

```
更清晰的意图表达
→ 更准确的模型输出
→ 更少的后处理修正
→ 更可靠的自动化流程
→ 更高效的人机协作
```

最后，记住一条黄金法则：**在你能用一句话把需求说清楚之前，不要写提示词。在你把提示词结构化之前，不要把它放到生产环境。** 结构化不是目的——持续产生高质量、可预期的AI输出才是。而结构化，是通向这个目的地的必经之路。
