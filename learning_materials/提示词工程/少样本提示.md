# 通过示例引导模型输出

## 引言

"给你看几个例子，你就明白了"——这是人类之间传递知识最高效的方式之一。对于AI模型来说，同样的原理也同样适用。**少样本提示（Few-Shot Prompting）**正是基于这一思想：通过在Prompt中提供几个示范性的输入-输出对，让模型"参照"这些例子来理解你的需求。

相比于单纯用文字描述你的期望（零样本提示），少样本提示往往能让模型输出更准确、格式更一致、风格更贴合。本文将带你深入掌握这项技术。

## 什么是少样本提示

少样本提示指的是在Prompt中嵌入若干个**示例（Examples）**，每个示例都展示了"给定什么样的输入，期望得到什么样的输出"的完整模式。模型通过分析这些示例，推断出其中的规律和模式，并将之应用到新的输入上。

一个基本的少样本提示结构如下：

```
[任务说明或上下文]

示例1:
输入：xxx
输出：xxx

示例2:
输入：xxx
输出：xxx

示例3:
输入：xxx
输出：xxx

现在请处理以下新输入：
输入：xxx
```

### 零样本 vs 单样本 vs 少样本

为了帮助理解，先厘清三个关键概念：

| 方式 | 示例数量 | 说明 | 示例 |
|------|---------|------|------|
| 零样本（Zero-Shot） | 0个 | 仅用指令描述任务 | "请将以下英文翻译成中文：Hello" |
| 单样本（One-Shot） | 1个 | 提供一个示例 | 先给一个翻译示例，再给待翻译内容 |
| 少样本（Few-Shot） | 2-5个 | 提供2-5个示例 | 先给3个翻译示例，再给待翻译内容 |

少样本提示之所以强于零样本提示，是因为示例为模型提供了**具体的行为模板**，消除了指令描述中的模糊性。例如，"请用正式的语言回复"——每个人对"正式"的理解不同，但如果你给出三个"正式回复"的示例，模型就能准确把握你的期望。

## 少样本提示为什么有效

### 1. 上下文学习（In-Context Learning）

少样本提示有效的基础是模型的一个关键能力——**上下文学习**。大语言模型不需要更新参数，就能根据上下文中的示例临时调整其行为。这种能力是模型在预训练中习得的：训练数据本身就包含了大量"多个例子 + 一个任务"的模式（如习题集、教程、FAQ等），模型学会了从示例中提取规律并加以应用。

### 2. 消解歧义

自然语言指令天然存在歧义。通过提供具体示例，你可以用"展示"代替"描述"，从根本上消除理解偏差。例如，"请用批判的语气评价这篇文章"——"批判"的程度和方式在不同场景下差异极大。而如果你给出了一个批判性评价的示例，模型就有了具体的参照标准。

### 3. 格式锚定

对于有特定输出格式需求的任务（如JSON、表格、特定标记语言），文字描述往往难以精确传达格式要求。而示例直观地展示了目标格式，模型可以直接"模仿"——比起"用JSON格式输出，包含name和age字段"，直接给出一个JSON示例要可靠得多。

### 4. 任务复杂性分解

对于复杂任务，仅用自然语言描述完整的行为规范可能会非常冗长且难以覆盖所有情况。少样本示例可以展示不同类型的输入应该分别如何处理，让模型自然地掌握分类处理的能力。

## 如何选择有效的示例

示例的质量直接决定了少样本提示的效果。以下是选择示例的关键原则。

### 1. 代表性

示例应该覆盖任务中可能遇到的主要情况类型。如果任务有多种输入变体，确保每种变体至少有一个示例。

**反面示例：** 一个情感分析任务，三个示例全部是"正面"评价。

**正面示例：**

```
示例1 — 正面：这个产品太好用了！ → 正面
示例2 — 负面：质量太差，用了三天就坏了 → 负面
示例3 — 中性：产品收到了，和描述一致 → 中性
```

### 2. 多样性

在有代表性的基础上，示例应该在语言风格、长度、复杂度等方面有所差异，帮助模型建立更鲁棒的映射模式。如果所有示例都是简短的陈述句，模型在面对长句子或疑问句时可能会不知所措。

### 3. 边界案例

包含至少一个"困难案例"或"边界案例"——这些案例最能帮助模型理解任务的精确边界。例如，在情感分析中，包含一个带有反讽或混合情感的例句，示范正确的标注方式。这会告诉模型："不是所有表面上看起来正面的评论都是正面评价"。

### 4. 正确性

这似乎不言自明，但值得强调：**错误的示例比没有示例更有害**。确保你提供的每一个示例都经过仔细验证，输入输出之间的映射关系是正确的。在实际项目中，建议对示例进行双重审核。

### 5. 贴近目标分布

示例应该与模型实际将要处理的新输入在风格和内容上保持相似。如果你将在电商评论上使用模型，用电商评论做示例，而不是新闻标题或社交媒体帖子。领域越贴近，迁移效果越好。

## 格式一致性

格式一致性是少样本提示中被低估但至关重要的因素。

### 输入输出格式要对齐

所有示例以及最终待处理输入应该使用**完全相同的结构**。如果示例中使用"输入：xxx"和"输出：xxx"的标签，那么最终任务提示也应使用相同的标签。模型对这种格式模式高度敏感——格式的不一致会让模型感到困惑。

### 使用清晰的分隔符

在示例之间使用明确的分隔符（如空行、`---`、`###`），帮助模型清晰地区分不同的示例。示例内部的输入和输出也应使用明确的标识。

**推荐的格式模板：**

```
任务：将用户评论分类为"正面"、"负面"或"中性"。

--- 示例开始 ---
评论：物流很快，包装也很精美，非常满意！
分类：正面
--- 示例结束 ---

--- 示例开始 ---
评论：收到后发现颜色和图片差很多，失望。
分类：负面
--- 示例结束 ---

--- 示例开始 ---
评论：还没有使用，看起来还行。
分类：中性
--- 示例结束 ---

现在请对以下评论进行分类：
评论：[新的待分类评论]
分类：
```

## 3-5个示例的"最佳区间"

在实践中，2-5个示例通常足以让模型掌握任务模式。为什么是这个数量？

- **1个示例**（单样本）：可能不足以展示任务的全部变化范围，模型可能过度"模仿"这个单一示例的风格和内容
- **2-3个示例**：能够展示基本的模式变化，对于简单任务的性价比最高
- **4-5个示例**：可以覆盖更多变体，是大多数中等复杂度任务的最佳选择
- **6+个示例**：边际收益递减——虽然更保险，但增加了Token消耗和处理时间，且大多数情况下效果提升有限

### 什么时候需要更多示例

在某些情况下，增加示例数量会有明显的收益：

- **任务规则复杂且隐含**：例如，某种特定的文风模仿、需要处理大量边缘情况的分类任务
- **输出格式非常严格**：例如，要求输出特定的JSON结构，字段名称和嵌套关系需要多个示例才能让模型稳定掌握
- **有非常接近但标注不同的案例**：需要多个示例来帮助模型区分这些"微妙差异"
- **模型在该任务上表现不佳**：更多示例可以提供更强的学习信号

## 零样本与少样本的选择

| 选择零样本（Zero-Shot） | 选择少样本（Few-Shot） |
|------------------------|----------------------|
| 任务简单且定义明确 | 任务存在歧义或难以用语言精确描述 |
| 标准输出格式即可 | 有非常特定的输出格式需求 |
| 上下文窗口紧张 | 有一定的上下文空间可分配 |
| 快速原型阶段 | 生产环境需要稳定一致的输出 |
| 模型已对该任务有充分训练 | 任务属于模型的弱项或边缘领域 |

一个实用策略是：**先用零样本快速测试**，如果效果不理想，再逐步增加1-2个示例观察改进效果，直到达到满意的输出质量。这样能避免过度使用示例带来的Token浪费。

## 标签平衡与去偏

### 标签不平衡的问题

如果分类任务的示例中，某一类别占据绝大多数，模型可能会产生偏向性——更倾向于输出该类别。

**问题示例：** 在5个情感分析示例中，4个是"正面"，1个是"中性"，0个是"负面"。

这种不平衡会向模型传递一个隐性信号：正面评论更常见或更"安全"，导致模型在面对模棱两可的输入时倾向于输出"正面"。这在需要精确分类的生产环境中可能导致系统性的偏差。

### 解决方案

- **均衡采样**：确保每个类别的示例数量大致相同
- **考虑真实分布**：如果真实数据确实是不平衡的（如欺诈检测中99%为正常交易），可以在说明中明确告知模型这种不平衡性，避免示例误导
- **混合策略**：示例保持均衡，但在任务说明中给出真实分布信息，让模型在"学习任务模式"和"了解数据特征"之间有所区分

## 实践建议与进阶技巧

### 1. 示例的排序策略

研究表明，示例的排列顺序会影响模型表现。一个经验性做法是：**将最"典型"或最清晰的示例放在最后**（最靠近待处理输入的位置），因为模型对最近看到的示例关注度更高。同时，最复杂的示例放在中间位置——开头和结尾留给相对简单清晰的示例。

### 2. 动态示例选择

当你有一个较大的示例库时，可以根据当前输入动态选择最相关的示例，而非每次都使用相同的固定示例——这与RAG的思路类似。例如，对于一条科技产品的评论，优先选择其他科技产品的评论作为示例。这种"最近邻示例"策略在大型任务中能带来显著的效果提升。

### 3. 否定示例的使用

除了正面的"应该这样做"的示例，有时也可以添加"不应该这样做"的示例——特别是需要纠正模型常见错误的时候：

```
错误示例（不要这样做）：
输入：这个手机还行吧
输出：中性/正面（模糊分类，不规范）

正确示例：
输入：这个手机还行吧
输出：根据用户措辞"还行吧"的迟疑语气，判断为中性偏正面，
但由于缺乏明确的正面词汇，最终分类为中性。
```

### 4. 示例中的解释增强

除了直接的输入输出对应，在示例中加入推理过程的解释（类似思维链），能进一步提升模型的理解深度和任务表现：

```
评论：等了半个月才收到，但产品确实不错。
分类：中性
理由：该评论包含负面因素（等待时间过长）和正面因素（产品质量好），
两者权重相当，整体情感倾向中性。
```

## 总结

少样本提示是连接"模糊意图"和"精确输出"之间的桥梁。它用直观的示例替代冗长的规则描述，让模型以"模仿学习"的方式快速适应你的需求。

掌握少样本提示的关键在于：**选择具有代表性、多样性和正确性的示例，保持严格的格式一致性，以及根据任务复杂度和上下文预算选择恰当数量的示例。** 当你下次发现模型无法准确理解你的期望时，试试给它看几个例子——往往，这就是从"差强人意"到"完全符合预期"的那一步。

对于AI智能体开发者来说，少样本提示是一项必备技能。在定义智能体的工具调用格式、输出规范、决策逻辑时，精心的示例设计往往比长篇大论的规则描述更加高效和可靠。
