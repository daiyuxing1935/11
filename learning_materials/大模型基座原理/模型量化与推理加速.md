# 模型量化与推理加速

## 引言

一个残酷的现实：你训练了一个惊艳的70亿参数大模型，在A100服务器上跑得飞快。但当你想把它部署到实际产品中时，发现：

- 模型加载需要28GB显存（FP32），一张A10只有24GB —— 放不下
- 即使用24GB的3090勉强加载，推理一个token需要200ms —— 用户体验不可接受
- 扩展到1000个并发用户 —— 需要的GPU数量让CTO直接否决了项目

这是大模型落地的核心挑战：**训练可以离线慢慢跑，但推理必须又快又省**。而解决这个问题的关键技术就是**模型量化**和**推理加速**。本文将深入讲解这些技术的原理、方法和实践。

## 为什么需要量化：大模型部署的数字鸿沟

### 模型大小的直观感受

```
LLaMA-2-7B（FP32）:
  70亿参数 × 4字节/参数 = 28 GB ← 需要A100 40G

LLaMA-2-13B（FP32）:
  130亿参数 × 4字节/参数 = 52 GB ← 需要A100 80G

LLaMA-2-70B（FP32）:
  700亿参数 × 4字节/参数 = 280 GB ← 需要4张A100 80G

GPT-3 175B（FP32）:
  1750亿参数 × 4字节/参数 = 700 GB ← 需要9张A100 80G
```

而典型的推理需求是：
- **延迟** < 100ms/token（实时对话）
- **吞吐量** > 100 tokens/s（多用户并发）
- **硬件**：单张消费级显卡，成本控制在几千元

这个鸿沟只能用模型量化和推理优化来填补。

### 量化的核心直觉

量化的基本思想非常简单：**用更少比特数来表示模型参数，从而减少内存和计算需求**。

```
类比：照片压缩

原始RAW照片（FP32）: 50MB/张  → 色彩最丰富，但太大了
PNG无损压缩（FP16）:   15MB/张  → 质量几乎一样，大小减半
JPEG高质量（INT8）:     5MB/张  → 质量略降，但肉眼难分辨
JPEG低质量（INT4）:     2MB/张  → 质量明显下降，但还能看清

模型的量化也类似——在"精度损失"和"资源节省"之间寻找最优平衡。
```

## 量化的基本原理

### 数据类型精度对比

| 数据类型 | 比特数 | 数值范围 | 内存占用 (7B模型) | 精度 |
|---------|--------|---------|------------------|------|
| FP32 | 32 bits | ±3.4×10^38 | 28 GB | 基准 |
| FP16 | 16 bits | ±65504 | 14 GB | 几乎无损 |
| BF16 | 16 bits | ±3.4×10^38 | 14 GB | 几乎无损 |
| INT8 | 8 bits | -128 ~ 127 | 7 GB | 轻微损失 |
| INT4 | 4 bits | -8 ~ 7 | 3.5 GB | 中等损失 |
| NF4 | 4 bits | 信息论最优 | 3.5 GB | 较小损失 |

### FP32 vs FP16：浮点数的基础结构

```
FP32（IEEE 754单精度浮点数）:
┌──────┬───────────┬────────────────────────────┐
│ 1bit │   8 bits  │         23 bits            │
│ sign │ exponent  │        mantissa            │
└──────┴───────────┴────────────────────────────┘

FP16（IEEE 754半精度浮点数）:
┌──────┬───────────┬──────────────┐
│ 1bit │   5 bits  │   10 bits    │
│ sign │ exponent  │  mantissa    │
└──────┴───────────┴──────────────┘

BF16（Brain Float 16）:
┌──────┬───────────┬──────────────┐
│ 1bit │   8 bits  │   7 bits     │
│ sign │ exponent  │  mantissa    │
└──────┴───────────┴──────────────┘
```

BF16的设计很聪明——它保留了和FP32一样的8位指数，只压缩了尾数。这意味着BF16的数值范围与FP32相同（不会溢出），只是精度降低。这也是BF16在大模型训练中广泛应用的原因。

### 从浮点到定点：INT8量化

量化的本质是**将连续的浮点数值映射到离散的整数值**。核心公式：

```
量化（浮点 → 整数）:
  q = round((x - offset) / scale)

反量化（整数 → 浮点）:
  x' = q × scale + offset

其中：
  x:   原始浮点值
  q:   量化后的整数值
  scale:   缩放因子（step size）
  offset:  零点偏移（zero point）
```

### 对称量化 vs 非对称量化

**对称量化（Symmetric Quantization）**：

```
假设原始权重范围：[-0.8, 1.0]
INT8范围：[-127, 127]

对称量化: offset = 0
  scale = max(|min|, |max|) / 127 = 1.0 / 127 ≈ 0.00787
  
映射关系（以零为中心）:
  -0.8 → -0.8 / 0.00787 ≈ -102
   0.5 →  0.5 / 0.00787 ≈  64
   1.0 →  1.0 / 0.00787 ≈ 127

可视化：
FP32:  -0.8 ...... 0 ...... 0.5 ...... 1.0
         ↓         ↓         ↓          ↓
INT8:  -102 ...... 0 ....... 64 ...... 127
```

**非对称量化（Asymmetric Quantization）**：

```
同样范围：[-0.8, 1.0]
INT8范围：[0, 255]（无符号）

非对称量化:
  scale = (max - min) / 255 = (1.0 - (-0.8)) / 255 ≈ 0.00706
  offset = round(-min / scale) = round(0.8 / 0.00706) ≈ 113

映射关系:
  -0.8 → (-0.8 / 0.00706) + 113 ≈ 0
   0.5 → (0.5 / 0.00706) + 113  ≈ 184
   1.0 → (1.0 / 0.00706) + 113  ≈ 255

可视化：
FP32:  -0.8 ...... 0 ...... 0.5 ...... 1.0
         ↓         ↓         ↓          ↓
INT8:    0 ......113 ......184 ...... 255
```

**对称 vs 非对称对比**：

| 特性 | 对称量化 | 非对称量化 |
|------|---------|-----------|
| 零点的处理 | FP32的0 → INT8的0 | FP32的0 → 非零的INT值 |
| 计算效率 | 更高（零即零，省去偏移计算） | 较低（需要额外的偏移） |
| 精度（对称分布数据） | 最优 | 略差（浪费表达能力） |
| 精度（偏斜分布数据） | 略差 | 更好（充分利用INT范围） |
| 典型应用 | 权重量化（通常对称分布） | 激活值量化（可能偏斜） |

### 量化粒度

量化可以在不同粒度上进行：

```
逐层量化（Per-Tensor）:
  整个权重矩阵共用一个scale和一个offset
  ← 最简单，精度损失较大

逐通道量化（Per-Channel）:
  权重矩阵的每一行（或列）有独立的scale和offset
  ← 标准选择，精度和效率的平衡

逐组量化（Per-Group）:
  每N个连续参数组成一组，每组独立的scale和offset
  ← 精度最高，但元数据开销大
```

现代量化方法通常采用**逐通道量化**作为权重量化的默认选择，因为它在几乎不增加开销的情况下显著提升了量化精度。

## 主流量化方法详解

### GPTQ：基于最优脑外科的权重量化

GPTQ（GPT Post-Training Quantization）由IST Austria在2023年提出，是目前最主流的INT4权重量化方法之一。

**核心思想**：GPTQ基于"最优脑外科医生（Optimal Brain Surgeon, OBS）"理论——在逐个量化权重的过程中，同时调整其余未量化的权重来**补偿量化误差**。

```
GPTQ算法流程（简化）：

输入：预训练的FP16权重矩阵 W
输出：量化后的INT4权重，以及用于补偿的scale

1. 对于权重矩阵的每一列（逐列处理）：
   a. 将该列量化为INT4: W_col_quantized = quantize(W_col)
   b. 计算量化误差: error = W_col - W_col_quantized
   c. 基于Hessian矩阵的逆，将误差补偿到尚未量化的列上:
      对于每个未量化的列j:
        W_col_j = W_col_j - error × H_inv[i,j] / H_inv[i,i]

2. 重复直到所有列都被量化
```

关键洞察：GPTQ不是简单地"量化完一列就忘了它"，而是利用二阶信息（Hessian矩阵）将量化误差"分摊"到后续的权重上。这让量化后的模型能够**全局保持原有的输出行为**。

**代码示例**（使用AutoGPTQ库）：

```python
from transformers import AutoTokenizer
from auto_gptq import AutoGPTQForCausalLM, BaseQuantizeConfig

# 1. 准备量化配置
quantize_config = BaseQuantizeConfig(
    bits=4,                    # INT4量化
    group_size=128,            # 每128个参数一组
    desc_act=False,            # 是否降序排列激活值（影响精度和速度）
    damp_percent=0.01,         # Hessian对角线的阻尼系数
)

# 2. 加载模型并量化
model = AutoGPTQForCausalLM.from_pretrained(
    "meta-llama/Llama-2-7b-hf",
    quantize_config=quantize_config,
)

# 3. 准备校准数据（用于计算量化参数）
tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-2-7b-hf")
calibration_texts = [
    "The capital of France is Paris.",
    "Machine learning is a subset of artificial intelligence.",
    # ... 约128条校准样本
]

# 4. 执行量化（这是计算密集型的步骤，可能需要几十分钟）
model.quantize(
    calibration_texts,          # 校准数据
    tokenizer=tokenizer,
    batch_size=1,
)

# 5. 保存量化模型
model.save_quantized("./llama2-7b-gptq-4bit")
tokenizer.save_pretrained("./llama2-7b-gptq-4bit")

# 6. 加载使用
model = AutoGPTQForCausalLM.from_quantized(
    "./llama2-7b-gptq-4bit",
    device="cuda:0",
)

inputs = tokenizer("法国的首都是", return_tensors="pt").to("cuda")
outputs = model.generate(**inputs, max_new_tokens=20)
print(tokenizer.decode(outputs[0]))
```

### AWQ：激活感知的权重量化

AWQ（Activation-aware Weight Quantization）由MIT在2023年提出，核心观察是：**不是所有权重对模型输出同等重要**——那些与较大激活值相乘的权重更为关键。

```
AWQ核心洞察：

对于加权和 y = Σ(w_i × x_i)，量化误差的影响取决于x_i的大小：
  - 如果x_i很小，即使w_i量化误差大，对y的影响也小
  - 如果x_i很大，即使w_i量化误差小，对y的影响也被放大

AWQ策略：
1. 找到激活值最大的1%通道（显著通道）
2. 对这些通道，在量化前将权重乘以s（>1），激活值除以s
   （数学上等价，但量化后精度更高）
3. 对剩余的99%通道，正常量化

效果：显著通道的权重被"提亮"，量化时保留更多有效信息
```

**AWQ vs GPTQ对比**：

```
+---------------------+---------------------+---------------------+
| 特性                | GPTQ                | AWQ                 |
+---------------------+---------------------+---------------------+
| 量化策略            | 基于Hessian的误差补偿| 基于激活值的缩放保护  |
| 校准数据需求        | 需要（128条）        | 需要（少量）          |
| 量化时间（7B模型）  | ~30分钟             | ~10分钟              |
| 需要GPU推理         | 是                  | 是                   |
| 精度（同等bit）     | 优秀                | 优秀（略优或相当）    |
| 社区支持            | 广泛                | 快速增长             |
+---------------------+---------------------+---------------------+
```

**AWQ代码示例**：

```python
from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

# 加载模型
model_path = "meta-llama/Llama-2-7b-hf"
quant_path = "./llama2-7b-awq-4bit"

# 量化
model = AutoAWQForCausalLM.from_pretrained(model_path)
tokenizer = AutoTokenizer.from_pretrained(model_path)

# AWQ量化配置
quant_config = {
    "zero_point": True,        # 使用非对称量化
    "q_group_size": 128,       # 组大小
    "w_bit": 4,                # 4-bit权重量化
    "version": "GEMM",         # GEMM内核（更快）
}

model.quantize(
    tokenizer,
    quant_config=quant_config,
)

# 保存
model.save_quantized(quant_path)
tokenizer.save_pretrained(quant_path)

# 推理
model = AutoAWQForCausalLM.from_quantized(quant_path, fuse_layers=True)
```

### GGUF：端侧推理的标准格式

GGUF（GPT-Generated Unified Format）由llama.cpp的开发者Georgi Gerganov设计，是目前最广泛使用的端侧/消费级GPU推理格式。

**GGUF的设计理念**：

```
GGUF = 单文件格式 + 多级量化方案 + CPU友好 + Metal/CUDA加速

设计目标：
1. 可扩展性：一个文件包含所有元数据（tokenizer、配置、权重）
2. 向前兼容：新版格式能被旧版读取器解析
3. 多级量化：从Q2到Q8的多种精度选项
4. 零依赖：不依赖Python、PyTorch等重量级库
```

**GGUF量化类型一览**：

| 类型 | 比特数 | 内存(7B) | 质量 | 适用场景 |
|------|--------|---------|------|---------|
| Q8_0 | 8-bit | ~7 GB | 几乎无损 | 高质量推理 |
| Q6_K | 6-bit | ~5.5 GB | 优秀 | 小幅压缩 |
| Q5_K_M | 5-bit | ~5 GB | 很好 | 推荐日常使用 |
| Q4_K_M | 4-bit | ~4.2 GB | 良好 | 消费级显卡首选 |
| Q3_K_M | 3-bit | ~3.5 GB | 可接受 | 极限压缩 |
| Q2_K | 2-bit | ~3 GB | 较差 | 仅用于极低资源 |

**代码示例**（使用llama-cpp-python）：

```python
from llama_cpp import Llama

# 转换模型为GGUF格式
# 首先需要用convert.py脚本（llama.cpp提供）将原始模型转换为GGUF
# python convert.py llama2-7b --outtype q4_k_m --outfile llama2-7b-q4_k_m.gguf

# 加载GGUF模型
llm = Llama(
    model_path="./llama2-7b-q4_k_m.gguf",
    n_ctx=4096,                # 上下文长度
    n_threads=8,               # CPU线程数
    n_gpu_layers=35,           # 卸载到GPU的层数（-1=全部）
    verbose=False,
)

# 生成文本
output = llm.create_chat_completion(
    messages=[
        {"role": "system", "content": "你是一个有帮助的助手。"},
        {"role": "user", "content": "请用一句话介绍中国。"},
    ],
    temperature=0.7,
    max_tokens=200,
)

print(output["choices"][0]["message"]["content"])

# 流式生成
for chunk in llm.create_chat_completion(
    messages=[{"role": "user", "content": "写一首关于AI的五言绝句"}],
    stream=True,
):
    if "content" in chunk["choices"][0]["delta"]:
        print(chunk["choices"][0]["delta"]["content"], end="", flush=True)
```

### bitsandbytes：即时量化加载

bitsandbytes提供了最简便的量化方式——**在加载模型时自动量化**，无需预先转换：

```python
import torch
from transformers import AutoModelForCausalLM, BitsAndBytesConfig

# 8-bit量化加载
model_8bit = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-7b-hf",
    load_in_8bit=True,              # 最简单的用法！
    device_map="auto",
)

# 4-bit量化加载（NF4，推荐）
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",      # NF4量化类型
    bnb_4bit_compute_dtype=torch.float16,  # 计算精度
    bnb_4bit_use_double_quant=True, # 双重量化（进一步节省0.4bit/参数）
)

model_4bit = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-7b-hf",
    quantization_config=bnb_config,
    device_map="auto",
)

# 查看内存使用
print(f"8-bit模型内存: {model_8bit.get_memory_footprint() / 1e9:.1f} GB")
print(f"4-bit模型内存: {model_4bit.get_memory_footprint() / 1e9:.1f} GB")
# 典型输出: 8-bit约7.5GB, 4-bit约4.5GB（包括额外开销）
```

### 量化方法选型决策树

```
需要量化一个模型吗？
│
├─ 有GPU，追求最高精度 → GPTQ (INT4) 或 AWQ (INT4)
│   └─ 推荐配置：group_size=128, desc_act=True (GPTQ)
│
├─ CPU推理或边缘设备 → GGUF
│   ├─ 消费级GPU (8-16GB) → Q4_K_M
│   ├─ 手机/树莓派 → Q3_K_M 或 Q2_K
│   └─ 服务器CPU → Q5_K_M
│
├─ 快速实验，不想等量化过程 → bitsandbytes (即时8bit/4bit)
│
└─ HuggingFace生态，即插即用 → bitsandbytes NF4
```

## 推理加速技术

量化为我们节省了显存，但推理速度还需要额外的优化技术。

### Flash Attention：IO感知的精确注意力

Flash Attention是2022年以来影响力最大的推理优化技术之一。它不改变注意力计算的数学结果，而是改变了**计算在GPU上的组织方式**。

**GPU内存层次结构的瓶颈**：

```
GPU内存层次（以A100为例）：
┌──────────────────────────────────────┐
│ HBM (High Bandwidth Memory) 80GB    │  ← 大但慢
│ 带宽：~2 TB/s                        │
├──────────────────────────────────────┤
│ L2 Cache  40MB                       │  ← 中间层
├──────────────────────────────────────┤
│ SRAM (On-chip)  192KB/SM × 108SM     │  ← 小但极快
│ 带宽：~19 TB/s                       │
└──────────────────────────────────────┘

标准注意力问题：
  计算 Q×K^T 时，每步计算都从HBM读取Q、K到SRAM
  然后写回S（注意力分数矩阵），再从HBM读取S，
  计算P=softmax(S)并写回，再从HBM读取P，
  计算O=P×V并写回...
  
  每次HBM读写都是瓶颈！（大量的冗余读写）
```

**Flash Attention的解决方案**：

```
Flash Attention核心技巧：
1. 分块（Tiling）：将Q、K、V分成小块，逐块在SRAM内计算
2. 重计算（Recomputation）：不存储中间注意力矩阵，反向传播时重算
3. Softmax的在线计算：通过维护running max/running sum，在单次遍历中
   完成softmax计算，无需写回完整矩阵

效果（相对于标准实现）：
  - 显存节省：O(n) 而非 O(n²)
  - 速度提升：2-4倍
  - 数学等价：结果与标准注意力完全一致
```

**Flash Attention的使用**：

```python
# PyTorch 2.0+ 内置支持
import torch

# 使用 scaled_dot_product_attention（自动调度到Flash Attention）
with torch.backends.cuda.sdp_kernel(
    enable_flash=True,
    enable_math=False,
    enable_mem_efficient=False,
):
    output = torch.nn.functional.scaled_dot_product_attention(
        query, key, value,
        attn_mask=attention_mask,
        dropout_p=0.0,
        is_causal=True,
    )

# HuggingFace Transformers 自动启用
model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-7b-hf",
    torch_dtype=torch.float16,
    attn_implementation="flash_attention_2",  # 显式指定
    device_map="auto",
)
```

### vLLM：PagedAttention与连续批处理

vLLM由UC Berkeley开发，是当前最流行的高吞吐量LLM推理引擎。

**PagedAttention的核心创新**：

```
传统KV Cache管理的问题：
  ┌─────────────────────────────┐
  │ 为每个序列预分配固定大小KV Cache │
  │ 问题1：内存碎片（不同序列长度不同） │
  │ 问题2：无法共享（同一prompt的KV）  │
  │ 问题3：无法动态扩展              │
  └─────────────────────────────┘

PagedAttention的解决方案：
  ┌────────────────┬────────────────┐
  │ 将KV Cache分成  │ 类似于操作系统中  │
  │ 固定大小的"页"  │ 虚拟内存的分页管理  │
  │ (Page)         │                │
  └────────────────┴────────────────┘

优势：
  - 消除内存碎片（所有页大小相同）
  - KV Cache共享（多个请求共享同一prompt的KV页）
  - 按需分配（仅在需要时分配新页）
  - 显存利用率提升2-4倍 → 吞吐量提升2-4倍
```

### KV Cache优化

KV Cache是Transformer推理时的标准优化，但也是显存杀手：

```
KV Cache显存计算公式：
  2（K和V）× 层数 × 头数 × 每头维度 × 序列长度 × 精度字节

以LLaMA-2-7B为例（FP16, 4096 tokens）：
  2 × 32层 × 32头 × 128维 × 4096 tokens × 2字节
  = 2 × 32 × 32 × 128 × 4096 × 2
  ≈ 2.15 GB

仅KV Cache就占用了2.15 GB显存！
```

**KV Cache优化方法**：

```
1. GQA/MQA（分组查询注意力/多查询注意力）
   ┌──────────────┬──────────────┐
   │ MHA（标准）   │ 每个头独立KV   │ ← KV Cache最大
   │ GQA          │ 多组Q共享一组KV│ ← 中间（LLaMA-2 70B使用）
   │ MQA          │ 所有Q共享一组KV│ ← KV Cache最小
   └──────────────┴──────────────┘

2. KV Cache量化（KV Cache也可以量化！）
   - INT8 KV Cache：2×压缩，几乎无损
   - FP8 KV Cache：新兴格式，硬件加速支持

3. 滑动窗口注意力
   - 只保留最近W个token的KV Cache
   - 丢弃超出窗口的历史KV
   - 适合流式对话场景
```

**使用GQA的模型**：

```python
# LLaMA-2-70B默认使用GQA（8组）
# 在模型配置中指定
from transformers import AutoConfig

config = AutoConfig.from_pretrained("meta-llama/Llama-2-70b-hf")
print(f"num_key_value_heads: {config.num_key_value_heads}")  # 8
print(f"num_attention_heads: {config.num_attention_heads}")  # 64
# 64个Q头，8个KV头 → 每8个Q头共享1组KV → KV Cache减少8倍
```

## 量化对模型性能的影响

### 关键发现：LLM对量化出人意料地鲁棒

研究一致表明：大语言模型（尤其是7B以上规模的）对INT8甚至INT4量化相当鲁棒：

```
LLaMA-2-7B在MMLU上的表现：

FP16（基准）:   45.3%
INT8:           45.1%  （↓0.2%）
INT4 (GPTQ):    44.5%  （↓0.8%）
INT4 (AWQ):     44.7%  （↓0.6%）
INT4 (GGUF Q4): 44.2%  （↓1.1%）
INT4 (NF4):     44.0%  （↓1.3%）
INT3:           42.1%  （↓3.2%）  ← 开始明显下降
INT2:           35.8%  （↓9.5%）  ← 严重下降
```

**经验规律**：

```
安全区：  INT8 ~ FP16       → 精度损失 <0.5%，放心使用
推荐区：  INT4 (GPTQ/AWQ)   → 精度损失 1-2%，日常使用完全OK
临界区：  INT3              → 精度损失 3-4%，仅在极限场景
危险区：  INT2              → 精度损失 >8%，不推荐
```

### 影响量化精度的因素

1. **模型规模**：越大的模型对量化越鲁棒。7B模型在INT4下损失约1%，而70B模型可能只有0.5%。（更大的模型有更多"冗余"信息维度）

2. **量化方法**：AWQ和GPTQ由于考虑了权重重要性，明显优于朴素的逐层量化

3. **组大小**：较小的组大小（64-128）通常精度更高，但元数据开销也更大

4. **校准数据**：GPTQ的校准数据与下游任务的相关性会影响量化质量

## 部署实践建议

### 典型部署配置推荐

```
场景1：单卡消费级推理（RTX 3090/4090 24GB）
  模型：LLaMA-2-7B 或 Qwen-7B
  方案：GPTQ INT4 (group_size=128) + Flash Attention 2
  显存占用：~6 GB（模型4.2GB + KV Cache ~1.8GB）
  推理速度：~50 tokens/s
  多LoRA支持：可以同时加载多个LoRA适配器

场景2：API服务（A10 24GB 或 A100 40GB）
  模型：LLaMA-2-13B 或 Qwen-14B
  方案：AWQ INT4 + vLLM（PagedAttention）
  显存占用：~12 GB / ~20 GB（含批处理KV Cache）
  吞吐量：~500 tokens/s（batch size 8-16）
  并发：支持10-30个同时请求

场景3：边缘设备（MacBook M2 16GB）
  模型：LLaMA-2-7B GGUF Q4_K_M
  方案：llama.cpp + Metal加速
  内存占用：~6 GB
  推理速度：~15 tokens/s（Metal GPU）
  特点：离线、隐私、零成本

场景4：手机端推理
  模型：Phi-3-mini / Gemma-2B GGUF Q4_K_S
  方案：llama.cpp Android/iOS binding
  内存占用：~1.5 GB
  推理速度：~5-10 tokens/s
```

### 性能调优清单

```
□ 1. 选择合适的精度：从INT8开始，不够再降级到INT4
□ 2. 开启Flash Attention：几乎零成本，2-3倍KV Cache节省
□ 3. 设置合理的上下文长度：max_length ≠ 4096（除非必要），减少KV Cache
□ 4. 使用GQA/MQA模型：如LLaMA-2-70B，天然节省KV Cache
□ 5. 批处理请求（vLLM）：动态批处理显著提升吞吐量
□ 6. 考虑投机解码（Speculative Decoding）：用小模型"猜"大模型的输出
□ 7. 量化KV Cache：在保持模型FP16的同时量化KV Cache到INT8
□ 8. 使用连续批处理：不要等整个batch完成才发下一个请求
```

### 性能优化决策树

```
需要LLM推理加速？
│
├─ 显存不够？ → 量化
│   ├─ GPU服务器 → GPTQ/AWQ
│   ├─ 消费级GPU → bitsandbytes NF4 / GGUF
│   └─ CPU/边缘 → GGUF
│
├─ 速度不够？
│   ├─ 单请求延迟高 → Flash Attention + TensorRT-LLM
│   ├─ 并发吞吐低 → vLLM (PagedAttention + 连续批处理)
│   └─ 首token太慢 → 量化KV Cache + prefix caching
│
└─ 需要更低延迟？
    ├─ 投机解码（小模型预测 + 大模型验证）
    ├─ 模型蒸馏（用小模型替代大模型）
    └─ 量化到INT4 + 编译优化 (TensorRT-LLM / MLC-LLM)
```

## 总结

模型量化和推理加速是大模型从"实验室"走向"生产环境"的关键桥梁。核心要点：

- **量化通过降低精度来减少显存和带宽需求**。INT8几乎无损，INT4是当前的最佳性价比选择，INT3及以下是极限场景的后备方案。

- **三种主流量化方法各有定位**：GPTQ精度最高但需要GPU推理，AWQ更快且在精度上可与GPTQ媲美，GGUF是CPU和边缘设备的最佳选择。

- **推理加速需要组合拳**：Flash Attention优化注意力计算，vLLM的PagedAttention优化KV Cache管理，GQA/MQA减少KV Cache大小——三项技术叠加可实现5-10倍的吞吐量提升。

- **量化对LLM的影响远小于直觉预期**。7B以上的模型在INT4量化下通常只损失1-2%的精度，而节省了75%的显存——这个交换几乎是任何生产环境的必然选择。

一句话总结：**INT4量化 + Flash Attention + vLLM**，是当前大模型推理部署的"黄金组合"。掌握了这三样，你就掌握了让大模型真正"跑起来"的能力。
