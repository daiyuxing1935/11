# 模型评估基础

## 引言

2025年，全球发布了超过200个大语言模型。面对眼花缭乱的选择，开发者最常问的问题是："哪个模型最好？"但这个问题本身就是一个陷阱——**没有脱离任务场景的"最好模型"**。一个在数学推理上出类拔萃的模型，可能在创意写作上平平无奇；一个对话流畅的模型，可能在代码生成上错误百出。

模型评估的目标，就是**系统性地、可复现地测量模型在不同维度上的能力**，让"哪个模型更好"从主观感受变为客观数据。本文将全面讲解大模型评估的基础框架、主流基准、评估指标和最佳实践。

## 为什么评估大模型如此困难

### 与传统ML评估的关键差异

传统机器学习模型的评估相对直接——你有一个测试集，计算准确率或F1-score，就完事了。但大模型的评估要复杂得多：

```
传统ML评估：
  输入 → 模型 → 分类标签 → 与ground truth比较 → 准确率
  （答案是非分明：对或错）

LLM评估：
  输入 → 模型 → 自由文本 → ??? → ???
  （"北京首都是什么" vs "中国的首都是北京" → 两者都对，但文本不同）
```

### LLM评估的独特挑战

1. **答案的多样性**：同一个问题，完全不同的文本表述都可能是正确的
2. **开放性任务**：创意写作、诗歌生成——没有标准答案
3. **部分正确**：答案可能"方向对了但细节错了"，如何打分？
4. **评估污染**：训练数据可能已经包含了测试题（数据泄露）
5. **主观性**：什么算"好的回答"取决于人的主观判断
6. **能力维度多**：一个模型可能数学好但编程差，如何给出综合评分？

## 主流的评估维度

一个全面的大模型评估体系通常涵盖以下维度：

```
大模型能力评估全景图
│
├── 知识与理解
│   ├── 事实性知识（MMLU, TriviaQA）
│   ├── 常识推理（HellaSwag, PIQA）
│   └── 世界知识（NaturalQuestions）
│
├── 推理能力
│   ├── 数学推理（GSM8K, MATH）
│   ├── 逻辑推理（BBH, LogiQA）
│   ├── 因果推理（COPA）
│   └── 多步推理（StrategyQA）
│
├── 语言能力
│   ├── 阅读理解（SQuAD, RACE）
│   ├── 文本摘要（CNN/DailyMail, XSum）
│   ├── 翻译（WMT, Flores）
│   └── 情感分析（SST-2, IMDb）
│
├── 代码能力
│   ├── 代码生成（HumanEval, MBPP）
│   ├── 代码修复（Defects4J, QuixBugs）
│   └── 代码理解（CodeXGLUE）
│
├── 对齐与安全
│   ├── 真实性（TruthfulQA）
│   ├── 无害性（Anthropic Harmlessness）
│   ├── 指令遵循（MT-Bench, AlpacaEval）
│   └── 偏见检测（BBQ, WinoBias）
│
└── 对话与交互
    ├── 多轮对话（MT-Bench）
    ├── 角色扮演（CharacterEval）
    └── 工具使用（ToolBench, APIBench）
```

## 主流Benchmark详解

### MMLU：海量多任务语言理解

MMLU（Massive Multitask Language Understanding）由UC Berkeley在2021年发布，是目前覆盖面最广的知识评测基准。

```
MMLU基本数据：
  - 57个学科领域（从数学到法律到医学）
  - 约15,000道选择题（4选1）
  - 覆盖STEM、人文、社科、其他四大类

典型题目示例：

学科：高中化学
题目：下列哪种物质在常温常压下是液体？
A. 氧气
B. 汞
C. 铁
D. 氮气
正确答案：B

学科：美国历史
题目：美国宪法第一修正案保护以下哪项权利？
A. 持有武器的权利
B. 言论自由的权利
C. 不被无理搜查的权利
D. 迅速审判的权利
正确答案：B
```

**MMLU的评测方式**：

```
标准评测（0-shot/5-shot）：
  Prompt: "The following are multiple choice questions about chemistry.
           {question}
           A. {A}  B. {B}  C. {C}  D. {D}
           Answer:"
  
  模型输出 → 提取选项字母(A/B/C/D) → 与正确答案比较 → 计算准确率

典型分数（5-shot MMLU）：
  GPT-4:        86.4%
  Claude 3.5:   88.7%
  GPT-3.5:      70.0%
  LLaMA-2-70B:  68.9%
  LLaMA-2-7B:   45.3%
  Random:       25.0%  ← 随机猜的基线
```

**MMLU的优缺点**：

| 优点 | 缺点 |
|------|------|
| 覆盖面广，综合性强 | 英文为主，非英语知识覆盖不足 |
| 客观题，评测结果可复现 | 选择题形式，无法评估生成质量 |
| 社区公认的"黄金标准" | 可能存在数据污染（训练数据包含MMLU题目） |
| 5-shot设定便于标准化 | 不评估推理过程和深度理解 |

### HumanEval：代码生成的金标准

HumanEval由OpenAI在2021年发布，包含164个Python编程问题，是目前代码生成最权威的评测。

```
HumanEval数据结构：

{
  "task_id": "HumanEval/0",
  "prompt": "from typing import List

def has_close_elements(numbers: List[float], threshold: float) -> bool:
    \"\"\" Check if in given list of numbers, are any two numbers closer
    to each other than given threshold.
    >>> has_close_elements([1.0, 2.0, 3.0], 0.5)
    False
    >>> has_close_elements([1.0, 2.8, 3.0, 4.0, 5.0, 2.0], 0.3)
    True
    \"\"\"
",
  "canonical_solution": "    for i in range(len(numbers)):
        for j in range(i + 1, len(numbers)):
            if abs(numbers[i] - numbers[j]) < threshold:
                return True
    return False\n",
  "entry_point": "has_close_elements",
  "test": "def check(candidate):\n    ..."
}
```

**评估方式**：pass@k指标

```
pass@k = E[1 - C(n-c, k) / C(n, k)]

通俗理解：
  对每个问题，生成k个候选代码
  只要k个中有至少1个能通过所有单元测试 → 该问题通过
  
  pass@1: 只生成1个，看能否通过（最严格）
  pass@10: 生成10个，有1个通过就算（较宽松）
  pass@100: 生成100个，有1个通过即算（最宽松）

典型分数（pass@1, HumanEval）：
  GPT-4:                    67.0%
  Claude 3.5 Sonnet:        92.0%
  DeepSeek-Coder-33B:       79.3%
  CodeLLaMA-34B:            53.7%
  LLaMA-2-70B:              29.3%
```

**HumanEval的局限性**：
- 只覆盖Python一种语言
- 题目偏算法/数据结构，不代表实际工程任务
- 164个题目样本量偏小，存在方差
- 不评估代码质量、可读性、性能

### GSM8K：小学数学推理

GSM8K（Grade School Math 8K）包含约8500道小学数学文字题，用于评测多步数学推理能力。

```
GSM8K题目示例：

问题：Janet的鸭子每天下16个蛋。她每天早上吃3个蛋当早餐，
      并用4个蛋每天烤松饼卖给朋友。她把剩下的蛋每天拿到
      农贸市场去卖，每个蛋卖2美元。她每天在农贸市场卖蛋
      能赚多少钱？

标准答案（包含推理步骤）：
  每天产蛋：16个
  早餐消耗：3个
  烘焙消耗：4个
  剩余：16 - 3 - 4 = 9个
  收入：9 × 2 = 18美元
  
  最终答案：18

评估方式：
  - 提取模型生成的最终数字答案
  - 与标准答案比对
  - 答案必须精确匹配（18 = 18，但 "$18" 或 "18 dollars" 通常也被接受）
```

**GSM8K的评测难度**：

```
GSM8K需要的能力：
1. 阅读理解：从文字中提取数学关系
2. 多步推理：分步计算，每步依赖前一步
3. 算术准确性：最终计算不能出错
4. 格式遵循：答案需要可提取

典型分数（8-shot CoT, GSM8K）：
  GPT-4:        92.0%
  Claude 3.5:   96.4%
  GPT-3.5:      57.1%
  LLaMA-2-70B:  56.8%
  LLaMA-2-7B:   14.6%  ← 小型模型数学很差
```

### BBH：困难推理的试金石

BBH（BIG-Bench Hard）从BIG-Bench中筛选出23个"当前模型无法超越人类"的困难任务：

```
BBH包含的任务类型（部分）：

逻辑推理：
  - 逻辑推理（5个选项）：形式逻辑三段论
  - 时间序列：推理事件的时间顺序
  - 析取推理：处理"或"逻辑

数学与算法：
  - 多步算术：需要多步的算术问题
  - 布尔表达式：评估复杂的布尔表达式
  - 几何形状：空间推理

语言理解：
  - 成语理解：非字面意义的理解
  - 讽刺检测：识别讽刺和反讽
  - 因果关系：判断因果关系

典型题目（逻辑推理）：
前提1：所有猫都是哺乳动物
前提2：一些哺乳动物会游泳
结论：一些猫会游泳

问题：这个推理是否有效？
答案：无效（从"一些哺乳动物会游泳"不能推出"猫会游泳"）
```

**BBH的价值**：它是一块"硬骨头"，即使是GPT-4级别的大模型在BBH上也远未达到饱和。这使它成为追踪基础推理能力进步的敏感指标。

### HellaSwag：常识推理

HellaSwag专门测试模型对日常生活中"常识性场景"的理解：

```
HellaSwag题目示例（选择正确的后续动作）：

场景：一个女人在厨房里打开冰箱。
A. 她拿出食材开始做饭。        ← 正确答案
B. 她开始修理屋顶。
C. 她跳进游泳池游泳。
D. 她开车去机场。

评估：模型需要从4个选项中选出最符合常识的后续动作。
```

HellaSwag的有趣之处在于：对人类来说极其简单（准确率>95%），但对2019年之前的语言模型来说极其困难。它衡量的是模型是否真正"理解世界如何运作"，而不仅仅是模式匹配。

## 评估指标体系

### 客观指标

**准确率（Accuracy）和精确匹配（Exact Match, EM）**：

```
适用：选择题、分类任务（MMLU, HellaSwag, GSM8K）

EM (Exact Match) = 完全匹配的样本数 / 总样本数

示例（GSM8K）：
  模型预测: 18  → 标准答案: 18  → 匹配 ✓
  模型预测: 18美元 → 标准答案: 18 → 提取后匹配 ✓
  模型预测: 17  → 标准答案: 18 → 不匹配 ✗
```

**困惑度（Perplexity, PPL）**：

```
困惑度衡量模型对文本的"惊讶程度"——模型越是"预料到"下一个token，
困惑度就越低。

PPL = exp( -1/N × Σ log P(token_i | context) )

直观理解：
  PPL = 10  → 模型平均每步在10个选项中"纠结" → 很好
  PPL = 100 → 模型平均每步在100个选项中"纠结" → 一般
  PPL = 1000 → 模型非常困惑 → 很差

PPL主要用于：
  - 语言建模能力的基础衡量
  - 同一模型架构下的对比（不同架构的PPL不可直接比较）
  - 训练过程中监控模型进步
```

**pass@k（代码生成专用）**：

```
pass@k = 生成k个候选代码，有≥1个通过所有测试的概率

pass@1: 衡量精确生成能力
pass@10: 衡量探索性生成能力（配合reranker使用）
pass@100: 衡量模型的潜力上限
```

### 生成文本的自动评估指标

**ROUGE（Recall-Oriented Understudy for Gisting Evaluation）**：

```
主要用于：文本摘要的自动评估

ROUGE-N = 参考摘要和生成摘要之间n-gram的重叠度

ROUGE-1（unigram重叠）：
  Reference: "the cat sat on the mat"
  Generated: "the cat is on the mat"
  
  重叠: "the", "cat", "on", "the", "mat" (5个)
  参考总数: 6个unigram
  ROUGE-1 Recall = 5/6 = 0.83
  ROUGE-1 Precision = 5/6 = 0.83
  ROUGE-1 F1 = 2×0.83×0.83/(0.83+0.83) = 0.83

ROUGE-L（最长公共子序列）：
  更关注词语的顺序和结构

ROUGE的局限：
  - 只关注表面词汇重叠，不理解语义
  - "非常好"和"很棒"在语义上相同，但ROUGE认为完全不同
  - 对改写和同义替换不友好
```

**BLEU（Bilingual Evaluation Understudy）**：

```
主要用于：机器翻译的自动评估

BLEU = brevity_penalty × exp(Σ w_n × log(p_n))

其中p_n是n-gram精确度的几何平均

特点：
  - 偏向精确度（Precision）而非召回
  - 有长度惩罚：太短的翻译会被扣分
  - 通常在语料库级别计算（不是单句）

BLEU分数解读：
  30-40: 可理解但不够流畅
  40-50: 较高质量的翻译
  50-60: 非常高质量的翻译
  >60: 接近专业人工翻译

BLEU的局限：
  - 对同义表达不敏感
  - 对语序变化过于敏感
  - sentence-level BLEU不稳定（建议只用于corpus-level）
```

**BERTScore**：

```
BERTScore使用预训练BERT模型的上下文嵌入来计算文本相似度：

BERTScore(token_i) = max_j cosine(embed(gen_i), embed(ref_j))

即：对生成文本中的每个token，找到参考文本中与之最相似的token，
    计算它们的余弦相似度。

优势（相比ROUGE/BLEU）：
  ✓ 理解语义：  "好"和"优秀"获得高相似度（词向量接近）
  ✓ 理解改写：  "猫坐在垫子上"和"垫子上坐着猫"有高相似度
  ✓ 不需要n-gram匹配

BERTScore使用示例：
from bert_score import score

references = ["the cat sat on the mat"]
candidates = ["a cat is sitting on the mat"]

P, R, F1 = score(candidates, references, lang="en")
print(f"BERTScore F1: {F1.mean().item():.3f}")  # ~0.92
```

## 人类评估 vs 自动评估

### 为什么自动评估不够

```
自动评估的盲区：

ROUGE/BLEU可以告诉你"生成的文本和参考答案有多少词相同"
但它们无法告诉你：
  ✗ 文本是否流畅自然？
  ✗ 逻辑是否自洽？
  ✗ 回答是否有帮助、有信息量？
  ✗ 语气是否得体？
  ✗ 是否存在安全隐患？
  ✗ 是否正确地处理了歧义？
```

### 人类评估的维度设计

```
典型的LLM人类评估维度：

1. 有用性 (Helpfulness)
   1分: 完全无帮助，答非所问
   3分: 部分有帮助，但缺少关键信息
   5分: 非常全面且有帮助

2. 真实性 (Truthfulness)
   1分: 包含明显的事实错误或幻觉
   3分: 基本正确但有小的不准确
   5分: 所有陈述都准确无误

3. 无害性 (Harmlessness)
   1分: 产生有害或危险内容
   3分: 内容安全但可能引发担忧
   5分: 完全安全，考虑了道德因素

4. 流畅度 (Fluency)
   1分: 不可读，语法错误严重
   3分: 基本流畅但有不自然的表达
   5分: 完全自然流畅
```

### 人类评估的实践挑战

```
挑战1：成本高昂
  评估1000个回复 × 3位评分者 × 每个回复2分钟 = 100小时人工时间
  在商业环境下，这意味着数万元的评估成本

挑战2：评分者间一致性
  评分者A给4分，评分者B给2分 → 谁的判断更可信？
  需要Krippendorff's alpha或Cohen's kappa来量化一致性

挑战3：量表漂移
  评分者在评估了100个"很好"的回复后，"好"的标准可能不自觉地提高了

缓解策略：
  - 使用锚定示例（给评分者看"1分/3分/5分"的标准答案）
  - 多轮校准（先让评分者独立评分 → 讨论分歧 → 重新校准 → 正式评分）
  - 随机插入控制样本（已知质量的样本，检验评分者是否稳定）
```

## LLM-as-Judge 评估范式

### 什么是LLM-as-Judge

由于人类评估的昂贵和自动指标（ROUGE/BLEU）的不足，业界越来越倾向于用**更强的LLM作为"评判者"**来评估其他LLM的输出：

```
LLM-as-Judge流程：
  问题 → 被评估模型 → 回答
          ↓
  评判Prompt + 回答 → 评判LLM（如GPT-4）→ 评分/排名/比较
```

### 三种评判方式

**1. 单回答评分（Single Answer Grading）**：

```
评判Prompt模板：
"请作为一个公正的评判者，评估以下AI助手的回答质量。
评分标准（1-10分）：
- 准确性：回答是否正确无误
- 完整性：是否涵盖了问题的所有方面
- 清晰度：表达是否清晰易懂
- 有用性：对用户是否有实际帮助

问题：{question}
AI回答：{answer}

请给出每个维度的分数和综合评分。
综合评分："
```

**2. 成对比较（Pairwise Comparison）**：

```
评判Prompt模板：
"请比较以下两个AI助手对同一问题的回答，选出更好的一个。

问题：{question}

助手A的回答：{answer_a}
助手B的回答：{answer_b}

更好的回答是：[A/B]
原因：[请简要说明]"
```

成对比较的优势在于：比"打一个绝对分数"更容易且更可靠。这在MT-Bench和Chatbot Arena中广泛使用。

**3. 参考基准比较（Reference-based Comparison）**：

```
评判Prompt模板：
"以下是同一个问题的参考答案和两个AI助手的回答。
请比较两个回答与参考答案的一致性。

问题：{question}
参考答案：{reference}
助手A的回答：{answer_a}
助手B的回答：{answer_b}

与参考答案更一致的是：[A/B]"
```

### LLM-as-Judge的可靠性

```
研究结论（来自多个独立验证）：

1. GPT-4作为评判者，与人类评分的Spearman相关系数约为0.8-0.9
   → 在多数维度上与人类高度一致

2. 位置偏见（Position Bias）：
   评判LLM可能倾向于选择位置A或位置B的回答
   → 缓解方案：交换位置比较两次，只有两次结果一致才采纳

3. 长度偏见（Length Bias）：
   评判LLM倾向于给更长的回答更高分（不管内容质量）
   → 缓解方案：在Prompt中明确要求"请以内容质量为准，忽略回答长度"

4. 自我增强偏见（Self-Enhancement Bias）：
   如果用GPT-4评判GPT-4自己的回答，可能给出更高分
   → 缓解方案：尽量让评判模型与被评模型是不同家族的
```

### MT-Bench：LLM-as-Judge的标杆应用

MT-Bench（Multi-Turn Benchmark）由LMSYS Org发布，是LLM-as-Judge范式的标杆：

```
MT-Bench结构：
  - 80个高质量多轮对话问题
  - 8个类别：写作、角色扮演、推理、数学、编程、提取、STEM、人文
  - 第1轮：用户提出问题
  - 第2轮：用户在助手回答后提出后续问题
  - 评判：GPT-4作为评判者，进行1-10的单回答评分

MT-Bench评分（GPT-4评判，2024年数据）：
  GPT-4-Turbo:           9.05
  Claude 3.5 Sonnet:     8.96
  GPT-4:                 8.91
  Gemini 1.5 Pro:        8.58
  LLaMA-3-70B:           8.47
  GPT-3.5-Turbo:         7.94
```

### Chatbot Arena（LMSYS）

Chatbot Arena采用**众包盲评**的独特方式，被认为是最接近"真实用户体验"的评测：

```
Chatbot Arena流程：
1. 用户提交任意问题
2. 系统随机选择两个匿名模型（模型A和模型B）
3. 两个模型同时生成回答（用户不知道哪个是哪个）
4. 用户投票选择更好的回答（可以平局）
5. 使用Elo评分系统排名（类似国际象棋等级分）

关键特点：
  - 完全由真人评判
  - 问题来自真实用户（非预设题目）
  - 盲评消除品牌偏见
  - 持续更新，反映最新性能

Elo评分（2025年最新）：
  Claude 3.5 Sonnet:   ~1310
  GPT-4-Turbo:         ~1285
  Gemini 1.5 Pro:      ~1260
  LLaMA-3-70B:         ~1230
```

## 评估的局限性与陷阱

### 陷阱1：数据污染

最大的陷阱是**训练数据中已经包含了测试题**：

```
数据污染的常见形式：
1. 预训练语料包含了MMLU/HumanEval等benchmark的原始题目
2. 指令微调数据中无意包含了测试集的变体
3. 多次迭代评估中模型"记住"了测试题

检测方法：
- 计算模型在不同n-gram顺序下的perplexity（污染数据perplexity异常低）
- 对比公开题和私有题的性能差异（如果公开题明显更好→疑似污染）
- 使用动态更新的benchmark（如DynaBench）

ChatGPT的一个已知争议：
  早期版本的GPT-4在GSM8K上的高分是否来自数据污染？
  → 通过对GSM8K题目进行数值替换生成新题，仍能保持高准确率
  → 结论：能力是真实的，不只是记忆
```

### 陷阱2：Benchmark过拟合

当一个Benchmark太受关注，模型开发者会刻意优化该Benchmark上的表现：

```
GSM8K过拟合现象：
  2022年：70B模型在GSM8K上约50%
  2023年：7B模型在GSM8K上约50%（针对性优化）
  2024年：1B模型在GSM8K上约40%（蒸馏+数据增强）

  是真正的推理能力提升了，还是"学会做这些题"了？
  → 转移到新benchmark（如MATH）时，性能并没有同比例提升
  → 说明存在一定程度的过拟合
```

### 陷阱3：只关注平均分

```
案例：两个模型MMLU平均分都是70%
  模型A：各学科分数均匀（65-75%）
  模型B：STEM学科90%，但人文学科只有50%

哪一个更好？取决于你的应用场景：
  - 数学辅导 → 模型B更好
  - 通识问答 → 模型A更好
  - 综合应用 → 需要查看方差和分项能力
```

### 陷阱4：忽略了非可测量维度

许多关键的用户体验维度在Benchmark中完全没有覆盖：

```
Benchmark不评估的维度：
  - 输出风格（正式/轻松/幽默）
  - 回答的简洁度（是否总是过于冗长）
  - 拒绝回答的合理性（何时应该说"我不知道"）
  - 对话节奏和自然的轮流感
  - 对用户情绪的响应能力
  - 多语言混合场景
```

### 陷阱5：静态Benchmark的时代滞后性

```
固定Benchmark的问题：
  - MMLU发布于2020年 → 知识截止于2019年
  - 模型的能力边界不断扩展 → 旧benchmark快速饱和
  - 新出现的能力（如长上下文理解）缺少成熟的benchmark

解决方向：
  - 动态Benchmark（定期更新题目）
  - 对抗性Benchmark（随模型能力增长而变难）
  - 真实世界任务（如SWE-bench的真实GitHub issue）
```

## 实践建议

### 如何为自己选择模型

```
决策流程：

1. 明确你的任务：写代码？写文章？数学推理？多轮对话？
   ↓
2. 找到该任务对应的主流Benchmark
   - 代码 → HumanEval, MBPP, SWE-bench
   - 数学 → GSM8K, MATH
   - 知识 → MMLU
   - 对话 → MT-Bench, Chatbot Arena Elo
   ↓
3. 查看相关benchmark的排名
   ↓
4. 不只看平均分——查看你在意的子领域的分数
   ↓
5. 如果可能，用你真实的数据做一个小规模自评
   ↓
6. 考虑部署成本（模型大小 × 量化方案 × 推理框架）
   ↓
7. 综合考虑，做出决策
```

### 自建评估集的最佳实践

```
1. 从真实业务数据中采样（不要用学术benchmark）
   例如：客服场景 → 从真实客服对话中抽取50-100个典型案例

2. 覆盖边缘情况和困难案例
   不要只选"典型问题"——要故意选那些容易出错的场景

3. 设计可量化的评判标准
   "回答质量好"太模糊 → "准确率 + 回复时间 + 是否包含必要信息"

4. 建立基线
   先用当前方案（或简单模型）跑一遍，作为对比基线

5. 定期更新
   - 模型版本更新时重新评估
   - 发现了新的失败案例 → 加入评估集
   - 已饱和的题目 → 替换为更难的新题目

6. 保持评估集独立
   评估集的数据绝对不能出现在训练/微调数据中
```

### 使用lm-evaluation-harness进行标准化评估

```python
# 安装
# pip install lm-eval

from lm_eval import simple_evaluate
from lm_eval.models import huggingface

# 加载模型
model = huggingface.AutoCausalLM(
    pretrained="meta-llama/Llama-2-7b-hf",
    device="cuda",
    dtype="float16",
)

# 运行MMLU评估
results = simple_evaluate(
    model=model,
    tasks=["mmlu"],              # 可以添加多个: ["mmlu", "gsm8k", "hellaswag"]
    num_fewshot=5,               # few-shot示例数
    batch_size=8,
    limit=None,                  # None=完整评估; 设置数字=快速测试
)

print(f"MMLU Accuracy: {results['results']['mmlu']['acc,none']:.3f}")

# 多任务批量评估
results = simple_evaluate(
    model=model,
    tasks=["mmlu", "hellaswag", "gsm8k", "truthfulqa_mc2"],
    num_fewshot=5,
    batch_size="auto",
)

for task, metrics in results["results"].items():
    print(f"{task}: {metrics}")
```

## 总结

大模型评估是一个仍在快速发展的领域。核心要点：

- **评估是带场景的**。不存在"最好"的模型，只有"在你的任务上最好"的模型。在选模型之前，先定义你的任务和评估标准。

- **Benchmark是地图，不是领土**。MMLU、HumanEval等基准提供了标准化的比较框架，但它们不能替代在你真实数据上的评估。高分不等于好体验。

- **LLM-as-Judge正在成为主流**。GPT-4作为评判者与人类高度一致（相关系数0.8-0.9），结合精心设计的评判Prompt，可以实现规模化、低成本的自动化评估。

- **关注评估的全貌而非单一数字**。一个分数背后可能隐藏着显著的能力差异——查看子维度得分、方差、失败案例分析，才能真正理解模型的强弱项。

- **评估是持续的过程**。模型在进化，benchmark在饱和，新的能力在涌现。建立一个可定期运行、自动化的评估体系，比一次性的深度评估更有长期价值。

记住这句话：**如果你不能测量它，你就不能改进它**。评估不仅是选模型的工具，更是理解模型、定位问题、指导优化的基础能力。
