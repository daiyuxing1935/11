# 知识检索增强：RAG技术在智能体中的应用

## 一、什么是RAG

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种将信息检索与文本生成相结合的技术。它的核心思想非常简单却强大：在让大语言模型回答问题之前，先从外部知识库中检索相关的信息，将这些信息作为"参考资料"一起提供给模型，让模型基于这些资料来生成答案。

为什么需要RAG？大语言模型有三大局限：

1. **知识截止日期**：模型训练完成后，知识就固定了，无法获取最新信息
2. **幻觉问题**：模型在面对不知道的内容时，可能会"编造"看似合理但实际上错误的答案
3. **私有知识盲区**：模型不了解企业的内部文档、专有数据和业务规则

RAG就像给模型配备了一个实时更新的"参考图书馆"。当模型不确定时，它可以先去图书馆查资料，然后再回答。

## 二、RAG的基本架构

一个标准的RAG系统包含四个核心阶段：

### 第一阶段：索引（Indexing）

将知识库中的文档预先处理好，建立便于快速检索的索引。

```
原始文档 → 文本清洗 → 文档分块（Chunking） → 向量化（Embedding） → 存入向量数据库
```

**关键步骤详解**：

**1. 文本清洗**：去除文档中的格式混乱、特殊字符、冗余内容

**2. 文档分块（Chunking）**：将长文档切分为适当大小的文本块。这是RAG性能的关键环节，下文会详细讨论。

**3. 向量化（Embedding）**：使用嵌入模型将每个文本块转换为高维向量。语义相近的文本在向量空间中距离也相近。

**4. 存入向量数据库**：将向量和对应的文本一起存储到专门的向量数据库中。

### 第二阶段：检索（Retrieval）

当用户提出问题时：

```
用户问题 → 向量化 → 在向量数据库中搜索 → 返回最相关的Top-K个文本块
```

检索的质量直接决定了RAG的效果。如果检索到的内容不相关，模型再怎么厉害也无法给出正确答案。

### 第三阶段：增强（Augmentation）

将检索到的文本块和用户问题组合成一个完整的Prompt：

```markdown
请根据以下参考资料回答用户的问题。如果参考资料中没有相关信息，请明确说明。

## 参考资料
{检索到的文本块1}
{检索到的文本块2}
{检索到的文本块3}

## 用户问题
{用户的问题}

## 回答要求
- 基于参考资料回答，不要编造信息
- 引用具体的资料来源
```

### 第四阶段：生成（Generation）

将组装好的Prompt发送给大语言模型，模型基于参考资料生成最终答案。

## 三、文档分块策略

文档分块（Chunking）是RAG中最容易被低估但实际非常重要的环节。分块太大，检索精度下降；分块太小，丢失上下文连贯性。

### 固定大小分块

最简单的方式：按固定的字符数或Token数切分。

```python
def fixed_size_chunk(text, chunk_size=500, overlap=50):
    """
    固定大小分块
    chunk_size: 每块的字符数
    overlap: 相邻块之间的重叠字符数（保持上下文连贯）
    """
    chunks = []
    start = 0
    while start < len(text):
        end = start + chunk_size
        chunk = text[start:end]
        chunks.append(chunk)
        start = end - overlap  # 有重叠地前进
    return chunks
```

**优点**：简单、容易实现
**缺点**：可能在句子中间断开，破坏语义完整性

### 语义分块

更智能的方式：按照语义边界（段落、句子）来分块。

```python
def semantic_chunk(text, max_chunk_size=1000):
    """
    按照自然段落和句子边界分块
    尽量在段落边界处分割，避免切断完整的语义单元
    """
    paragraphs = text.split('\n\n')
    chunks = []
    current_chunk = ""
    
    for para in paragraphs:
        if len(current_chunk) + len(para) < max_chunk_size:
            current_chunk += para + "\n\n"
        else:
            if current_chunk:
                chunks.append(current_chunk.strip())
            current_chunk = para + "\n\n"
    
    if current_chunk:
        chunks.append(current_chunk.strip())
    
    return chunks
```

### 递归分块

使用分隔符层级来递归地分割文本：

```python
def recursive_chunk(text, separators=None, chunk_size=500):
    """
    递归分块：优先使用大分隔符，不行再用小分隔符
    """
    if separators is None:
        separators = ["\n\n", "\n", "。", ".", " ", ""]
    
    for sep in separators:
        if sep == "":
            # 最终手段：按字符分割
            return fixed_size_chunk(text, chunk_size)
        
        if sep in text:
            parts = text.split(sep)
            if all(len(p) <= chunk_size for p in parts):
                return parts
            else:
                # 对较长的部分递归处理
                result = []
                for part in parts:
                    if len(part) <= chunk_size:
                        result.append(part)
                    else:
                        result.extend(recursive_chunk(part, separators[1:], chunk_size))
                return result
    
    return [text]
```

### 分块大小的最佳实践

| 应用场景 | 推荐分块大小 | 原因 |
|---------|------------|------|
| FAQ问答 | 200-500 Tokens | 答案通常较短，小块检索更精准 |
| 文档问答 | 500-1000 Tokens | 需要足够的上下文理解完整信息 |
| 长文总结 | 1000-2000 Tokens | 需要较长的上下文才能抓住主题 |
| 代码搜索 | 按函数/类分块 | 代码的语义单元是函数或类 |

## 四、嵌入模型与向量数据库

### 嵌入模型（Embedding Model）

嵌入模型将文本转换为固定维度的向量。选择一个好的嵌入模型非常重要：

- **通用嵌入模型**：OpenAI的text-embedding-3-small/large、Cohere的embed-v3等，适合中英文混合场景
- **中文优化模型**：BGE (BAAI General Embedding)、M3E等，在中文场景下表现出色
- **多语言模型**：multilingual-e5等，适合跨语言检索

选择嵌入模型时需要考虑：维度大小（影响存储和检索速度）、最大输入长度、支持的语言、是否有微调能力。

### 向量数据库（Vector Database）

向量数据库专门存储和检索高维向量。目前主流选择：

- **Chroma**：轻量级，适合原型开发和小规模应用
- **Pinecone**：全托管服务，运维成本低
- **Milvus**：开源、高性能，适合大规模生产环境
- **Weaviate**：支持混合搜索（向量+关键词）
- **FAISS**：Meta开源的高效向量搜索库，但本身不是完整的数据库

### 检索代码示例

```python
from chromadb import Client
from chromadb.utils import embedding_functions

# 1. 初始化
ef = embedding_functions.SentenceTransformerEmbeddingFunction(
    model_name="BAAI/bge-small-zh-v1.5"
)
client = Client()
collection = client.create_collection(
    name="knowledge_base",
    embedding_function=ef
)

# 2. 写入文档块
documents = ["文档块1的文本", "文档块2的文本", "..."]
ids = [f"doc_{i}" for i in range(len(documents))]
collection.add(documents=documents, ids=ids)

# 3. 检索
query = "如何使用Python连接MySQL数据库"
results = collection.query(query_texts=[query], n_results=5)

for doc in results['documents'][0]:
    print(f"检索到的文档块: {doc}")
```

## 五、混合搜索：稀疏+稠密

### 关键词搜索（稀疏检索/Sparse Retrieval）

使用BM25等传统算法，基于关键词匹配。优势是能精准匹配专有名词、产品型号等特定词汇。缺点是无法理解同义词和语义变体。

### 向量搜索（稠密检索/Dense Retrieval）

基于语义相似度搜索。优势是能理解"数据库"和"数据存储"是相关的概念。缺点是对于特别生僻的专有名词可能匹配不佳。

### 混合搜索的实现

结合两种方法的优势：

```python
def hybrid_search(query, collection, keyword_index, alpha=0.5):
    """
    混合搜索：结合向量搜索和关键词搜索
    alpha: 向量搜索的权重（0-1），1表示只用向量搜索
    """
    # 向量搜索结果
    vector_results = collection.query(
        query_texts=[query],
        n_results=10
    )
    
    # 关键词搜索结果（使用BM25）
    keyword_results = keyword_index.search(query, top_k=10)
    
    # 融合结果（使用倒数排名融合RRF）
    combined = reciprocal_rank_fusion(
        vector_results=vector_results,
        keyword_results=keyword_results,
        alpha=alpha
    )
    
    return combined

def reciprocal_rank_fusion(vector_results, keyword_results, alpha=0.5, k=60):
    """倒数排名融合（RRF）算法"""
    scores = {}
    
    # 向量搜索的得分
    for rank, result in enumerate(vector_results, start=1):
        scores[result.id] = scores.get(result.id, 0) + alpha / (k + rank)
    
    # 关键词搜索的得分
    for rank, result in enumerate(keyword_results, start=1):
        scores[result.id] = scores.get(result.id, 0) + (1 - alpha) / (k + rank)
    
    # 按融合得分排序
    sorted_results = sorted(scores.items(), key=lambda x: x[1], reverse=True)
    return sorted_results
```

## 六、高级RAG模式

### Self-RAG（自我反思RAG）

在标准RAG基础上，增加了一个"自我检查"环节。模型在生成答案后，会反思：
- 检索到的文档是否真的相关？
- 生成的答案是否忠实于检索到的信息？
- 是否需要补充检索？

```python
def self_rag(query, retriever, generator, reflector):
    # 检索
    docs = retriever.retrieve(query)
    
    # 生成
    answer = generator.generate(query, docs)
    
    # 反思
    reflection = reflector.evaluate(query, docs, answer)
    
    if reflection.needs_more_info:
        # 补充检索
        additional_docs = retriever.retrieve(reflection.follow_up_query)
        answer = generator.generate(query, docs + additional_docs)
    
    return answer
```

### Corrective RAG（纠正性RAG）

当检索结果质量不高时，不是简单地接受或拒绝，而是尝试"纠正"——使用更聪明的查询策略重新检索：

```python
def corrective_rag(query, retriever, generator, judge):
    docs = retriever.retrieve(query)
    
    # 判断检索质量
    quality = judge.evaluate_relevance(query, docs)
    
    if quality == "high":
        return generator.generate(query, docs)
    elif quality == "medium":
        # 用LLM优化查询后重新检索
        improved_query = generator.rewrite_query(query)
        new_docs = retriever.retrieve(improved_query)
        return generator.generate(query, docs + new_docs)
    else:
        # 检索质量差，尝试用不同策略
        # 比如分解问题、扩大搜索范围
        return fallback_strategy(query, retriever, generator)
```

## 七、RAG在智能体中的集成

将RAG集成到智能体中，通常有两种方式：

### 作为工具集成

将RAG检索作为一个工具注册到智能体：

```json
{
  "name": "search_knowledge_base",
  "description": "在内部知识库中搜索相关信息。当需要查询公司政策、产品文档、技术规范时使用。",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "搜索查询，尽量使用具体的关键词或问题"
      },
      "top_k": {
        "type": "integer",
        "description": "返回的最相关文档数量，默认为5"
      }
    },
    "required": ["query"]
  }
}
```

### 作为上下文注入

在每次智能体推理前，自动检索相关信息并注入到上下文中：

```python
class RAGAugmentedAgent:
    def __init__(self, llm, retriever):
        self.llm = llm
        self.retriever = retriever
    
    def process(self, user_message):
        # 自动检索相关知识
        relevant_docs = self.retriever.retrieve(user_message)
        
        # 将知识注入上下文
        context = "\n\n".join([f"[参考文档] {doc}" for doc in relevant_docs])
        augmented_message = f"{context}\n\n---\n用户问题：{user_message}"
        
        return self.llm.generate(augmented_message)
```

## 八、RAG效果优化清单

- [ ] 文档质量：原始文档是否清晰、准确、无歧义？
- [ ] 分块策略：分块大小是否适合你的使用场景？
- [ ] 嵌入模型：是否为你的语言和领域选择了合适的嵌入模型？
- [ ] 检索数量：Top-K的数量是否合理？太多引入噪音，太少可能遗漏信息
- [ ] 混合搜索：是否需要结合关键词搜索提高精度？
- [ ] 引用标注：答案中是否标注了信息来源，方便用户验证？
- [ ] 检索评估：是否有机制评估检索结果的相关性？
- [ ] 知识更新：知识库是否有定期更新机制？

RAG是AI智能体的"知识引擎"。一个设计良好的RAG系统，能让智能体拥有"过目不忘"的超能力，在企业知识问答、智能客服、法律咨询等场景中发挥巨大价值。
