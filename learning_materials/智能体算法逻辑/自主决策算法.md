# 自主决策算法：ReAct与Plan-Act决策算法对比

## 一、智能体决策的核心挑战

AI智能体的核心能力体现在"自主决策"——面对一个目标，智能体需要自己决定：先做什么、用什么工具、如何根据中间结果调整策略、以及何时停止。这个决策过程的质量直接决定了智能体的能力上限。

与传统的编程（if-else判断）不同，智能体的决策是在一个开放的、不确定的环境中进行的。你无法预先枚举所有可能的情况。因此，我们需要借助大语言模型的推理能力来实现灵活的自主决策。

目前在学术界和工业界，两种最主流的决策范式是**ReAct**和**Plan-Act**。它们代表了两种不同的决策哲学：是"边走边想"还是"想好再走"？本章将深入对比这两种方法，帮助你根据具体场景做出选择。

## 二、ReAct：思考-行动-观察循环

### ReAct的核心机制

**ReAct（Reasoning + Acting）** 由Google Research在2022年提出。它的核心是让模型在"推理"和"行动"之间交替进行：

1. **Thought（思考）**：分析当前状态，决定下一步该做什么
2. **Action（行动）**：执行具体的操作（调用工具、生成回复等）
3. **Observation（观察）**：获取行动的结果，更新对环境的认知
4. 回到第1步，开始新的一轮

这种"步步为营"的方式让ReAct非常灵活。因为每一步都可以根据最新的信息重新评估情况。

### ReAct的详细执行示例

让我们通过一个完整的例子来理解ReAct的工作方式：

**任务**："查找2024年诺贝尔物理学奖得主，并告诉我他们的主要贡献。"

```
第1轮：
  Thought: 我需要先查找2024年诺贝尔物理学奖得主是谁。
  Action: search_web("2024年诺贝尔物理学奖得主")
  Observation: 2024年诺贝尔物理学奖授予约翰·霍普菲尔德和杰弗里·辛顿，以表彰他们在人工神经网络和机器学习方面的基础性发现和发明。

第2轮：
  Thought: 已经知道得主是霍普菲尔德和辛顿。现在需要了解他们的具体贡献。
  Action: search_web("约翰·霍普菲尔德 诺贝尔奖 主要贡献")
  Observation: 霍普菲尔德因发明霍普菲尔德网络而获奖，这是一种能存储和重建信息的神经网络模型...

第3轮：
  Thought: 还需要查辛顿的贡献。
  Action: search_web("杰弗里·辛顿 诺贝尔奖 主要贡献")
  Observation: 辛顿因在深度学习和反向传播算法方面的开创性工作而获奖...

第4轮：
  Thought: 信息收集完毕，可以给出完整答案了。
  Action: FINAL_ANSWER
  Answer: 2024年诺贝尔物理学奖授予了约翰·霍普菲尔德和杰弗里·辛顿。霍普菲尔德的主要贡献是发明了霍普菲尔德网络...辛顿的主要贡献是深度学习领域的开创性工作...
```

**关键观察**：注意ReAct是如何根据前一步的搜索结果来决定下一步的——它不会一开始就计划好所有搜索，而是在每步完成后重新评估。

### ReAct的实现代码框架

```python
def react_agent(user_query, llm, tools, max_steps=10):
    messages = [{"role": "user", "content": user_query}]
    context = []  # 存储observation
    
    for step in range(max_steps):
        # 构建当前轮的Prompt
        prompt = build_react_prompt(user_query, context, step)
        
        # 让模型输出 Thought 和 Action
        response = llm.generate(prompt)
        thought, action = parse_react_response(response)
        
        # 记录思考过程
        print(f"Step {step + 1}: Thought: {thought}")
        
        # 检查是否是最终答案
        if action.type == "FINAL_ANSWER":
            return action.content
        
        # 执行工具调用
        observation = execute_tool(action.name, action.params)
        print(f"Step {step + 1}: Observation: {observation}")
        
        # 将观察结果加入上下文
        context.append({
            "step": step + 1,
            "thought": thought,
            "action": action,
            "observation": observation
        })
    
    return "任务超出最大步数限制"
```

## 三、Plan-Act：先规划后执行

### Plan-Act的核心机制

**Plan-Act** 采取完全不同的策略。它将任务分为两个清晰的阶段：

**规划阶段（Plan）**：在执行任何操作之前，先制定一个完整的计划。这个计划列出了所有需要的步骤、每一步的目标和预期结果。

**执行阶段（Act）**：严格按照计划执行每一步。执行过程中如果发现计划有误，可以触发重新规划。

### Plan-Act的详细示例

**任务**：与上面相同——"查找2024年诺贝尔物理学奖得主，并告诉我他们的主要贡献。"

```
规划阶段：
计划：
Step 1: 搜索"2024年诺贝尔物理学奖得主"，获取得主姓名列表
Step 2: 对于每位得主，搜索"[姓名] 诺贝尔奖 主要贡献"
Step 3: 整理所有信息，生成完整答案

执行阶段：
Step 1: search_web("2024年诺贝尔物理学奖得主")
结果: 得主为约翰·霍普菲尔德和杰弗里·辛顿 ✓

Step 2a: search_web("约翰·霍普菲尔德 诺贝尔奖 主要贡献")  
结果: 霍普菲尔德网络... ✓

Step 2b: search_web("杰弗里·辛顿 诺贝尔奖 主要贡献")
结果: 深度学习和反向传播... ✓

Step 3: 整理信息，生成答案 ✓
```

**关键观察**：Plan-Act在动手之前就明确了要做3个搜索（1次找名字 + 2次找贡献），并且提前识别出Step 2a和2b可以并行执行。

### Plan-Act的实现代码框架

```python
def plan_act_agent(user_query, llm, tools):
    # 阶段一：规划
    plan_prompt = f"""
    请为以下任务制定一个详细的执行计划。
    任务：{user_query}
    
    可用工具：{format_tools(tools)}
    
    请列出每一个执行步骤，包括：
    - 步骤描述
    - 所需工具
    - 预期产出
    """
    plan_text = llm.generate(plan_prompt)
    plan = parse_plan(plan_text)
    
    # 阶段二：执行
    results = []
    for step in plan.steps:
        # 执行步骤
        result = execute_step(step, tools)
        results.append(result)
        
        # 检查是否需要重新规划
        if result.status == "failed" and step.is_critical:
            print("关键步骤失败，触发重新规划...")
            plan = replan(plan, results, llm, tools)
        elif result.has_new_info:
            # 新信息可能影响后续步骤
            plan = adjust_plan(plan, results, llm)
    
    # 汇总结果
    return synthesize_results(results, user_query, llm)
```

## 四、ReAct vs Plan-Act：核心对比

| 维度 | ReAct | Plan-Act |
|-----|-------|----------|
| **决策方式** | 逐步推理，边走边看 | 先制定完整计划，再执行 |
| **灵活度** | 极高，每步可随时调整 | 较低，遵循预定计划 |
| **效率** | 较低，每步都需要推理 | 较高，计划明确后可高效执行 |
| **可预测性** | 低，行为不可预测 | 高，计划在手心中有数 |
| **Token消耗** | 较高（每步都需完整推理） | 中等（规划一次性消耗） |
| **适合任务** | 探索性、信息不完整的任务 | 流程明确、步骤清晰的确定性任务 |
| **错误恢复** | 容易（单步失败影响小） | 需要重新规划（成本较高） |
| **并行执行** | 难以并行（步骤是顺序的） | 可以识别并并行执行无依赖步骤 |

### 实际场景选择指南

**选择ReAct的场景**：
- 信息不完整，需要逐步探索（如调研一个你不熟悉的主题）
- 任务目标动态变化（如用户可能在对话中改变需求）
- 简单的、3-5步即可完成的任务（规划的过程本身就是浪费）
- 需要根据中间结果频繁调整策略

**选择Plan-Act的场景**：
- 流程固定、步骤清晰（如软件开发流程、报告生成流程）
- 需要可预测性（如自动化运维，你希望知道AI接下来会做什么）
- 有并行执行空间（如同时查询多个数据源）
- 需要人工审核执行计划（如涉及资金的关键操作）

## 五、混合方法：取长补短

在实际应用中，纯粹使用ReAct或Plan-Act往往不是最优方案。混合方法结合了两者的优势：

### 方案一：宏观Plan-Act + 微观ReAct

在宏观层面制定计划，但在执行每个步骤时使用ReAct进行灵活处理。

```python
def hybrid_agent(task, llm, tools):
    # 宏观规划
    plan = create_high_level_plan(task, llm, tools)
    
    # 微观执行（每个step内部使用ReAct）
    for step in plan.steps:
        # 每个步骤使用ReAct灵活执行
        step_result = react_execute(step, llm, tools)
        plan.update(step_result)
        
        # 根据执行结果决定是否调整宏观计划
        if step_result.significantly_deviated():
            plan = replan(plan, all_results, llm)
    
    return plan.final_result
```

### 方案二：ReAct为主 + 规划指导

让ReAct作为主执行引擎，但在每轮推理时参考一个动态更新的计划。计划不是刻板执行的脚本，而是一个灵活的行动指南。

```python
def guided_react(task, llm, tools):
    # 生成初始行动指南
    guide = generate_guide(task, llm)
    
    context = []
    # ReAct主循环
    while True:
        # 将行动指南注入上下文
        response = llm.react(task, context, guide=guide)
        thought, action = parse(response)
        
        if action.type == "FINAL":
            return action.answer
        
        observation = execute(action, tools)
        context.append({"thought": thought, "action": action, "observation": observation})
        
        # 动态更新行动指南
        guide = update_guide(guide, observation, llm)
```

### 方案三：基于复杂度的自适应选择

根据任务的复杂度自动选择策略：

```python
def adaptive_agent(task, llm, tools):
    complexity = assess_complexity(task)
    
    if complexity < 3:
        # 简单任务，直接ReAct
        return react_agent(task, llm, tools)
    elif complexity < 7:
        # 中等复杂度，Plan-Act
        return plan_act_agent(task, llm, tools)
    else:
        # 高复杂度，混合策略
        return hybrid_agent(task, llm, tools)
```

## 六、决策树在智能体中的应用

除了上述两种主流范式，**决策树（Decision Tree）** 在某些场景下也是一种有效的决策方式。

### 何时使用决策树

决策树适合**规则明确、边界清晰**的场景。例如，一个客服智能体的分流逻辑：

```
用户问题分类：
├── 是关于订单的？
│   ├── 查询订单状态 → 调用 order_status API
│   ├── 申请退款 → 先检查订单状态，再调用退款流程
│   └── 修改订单 → 检查是否在允许修改的时间内
├── 是关于产品的？
│   ├── 咨询产品规格 → 检索产品知识库
│   └── 反馈产品质量问题 → 记录问题 + 转接人工
└── 是投诉建议？→ 记录并转接人工
```

### 决策树的实现

```python
class AgentDecisionTree:
    def __init__(self, llm):
        self.llm = llm
        self.tree = self._build_tree()
    
    def _build_tree(self):
        # 可以手动定义规则，也可以用LLM辅助生成
        return {
            "condition": "classify_intent",
            "branches": {
                "订单相关": {
                    "condition": "classify_order_intent",
                    "branches": {
                        "查询": {"action": "query_order"},
                        "退款": {"action": "process_refund"},
                        "修改": {"action": "modify_order"}
                    }
                },
                "产品咨询": {"action": "search_knowledge_base"},
                "投诉建议": {"action": "escalate_to_human"}
            }
        }
    
    def decide(self, user_input):
        current_node = self.tree
        while "branches" in current_node:
            # 使用LLM来确定走哪个分支
            classification = self.llm.classify(
                user_input,
                list(current_node["branches"].keys())
            )
            current_node = current_node["branches"][classification]
        return current_node["action"]
```

### 决策树 vs LLM推理

决策树的优势是**确定性高**：对于同样的输入永远给出同样的决策路径。缺点是**灵活性差**：无法处理规则未覆盖的灰色地带。在智能体系统中，决策树通常作为第一层"路由器"使用——快速分流常见请求，复杂请求再交给ReAct或Plan-Act处理。

## 七、处理决策中的不确定性

无论使用哪种决策算法，不确定性都是无法避免的。以下是几种处理策略：

### 置信度评估

让模型在做出决策时，同时输出对决策的置信度：

```markdown
请做出决策，并给出你对这个决策的置信度（0-100%）。
如果置信度低于70%，请列出你的疑虑以及需要什么额外信息来消除不确定性。
```

### 多路径探索

当面对高不确定性的决策时，可以同时探索多个可能的路径，然后比较结果：

```python
def multi_path_explore(task, llm, tools, num_paths=3):
    # 生成多个可能的行动计划
    plans = []
    for i in range(num_paths):
        plan = generate_plan(task, llm, temperature=0.7 + 0.1 * i)
        plans.append(plan)
    
    # 每个计划执行几步尝试
    results = []
    for plan in plans:
        partial_result = execute_first_n_steps(plan, n=2)
        results.append(partial_result)
    
    # 选择最有前景的路径继续执行
    best_plan = evaluate_and_select(plans, results)
    return execute_remaining(best_plan)
```

### 安全兜底机制

对于高风险决策，设置"安全边界"：
- 设置决策后果的"回滚"机制（能撤销的操作才敢大胆尝试）
- 关键决策前请求人工确认
- 检测到不确定情况时，选择最保守的方案

```python
def safe_decide(options, risk_threshold=0.3):
    for option in options:
        if option.risk > risk_threshold and not option.reversible:
            # 高风险且不可逆的操作，请求人工确认
            return request_human_approval(option)
    
    # 选择风险最低的可行方案
    return min(options, key=lambda o: o.risk)
```

## 八、实践建议

1. **从小步开始**：如果你的智能体刚开始开发，建议先从ReAct入手。它的"步步为营"特性让你容易观察每一步发生了什么，方便调试。

2. **逐步引入规划**：当任务变得复杂（超过5-7步），引入Plan-Act的规划阶段。但保留执行过程中的灵活性。

3. **监控关键指标**：
   - 平均每任务的步数（步数越多，说明决策不够高效）
   - 失败率（ReAct和Plan-Act下的失败率对比）
   - Token消耗量（不同策略的成本对比）

4. **不要过度设计**：对于"今天天气怎么样"这样的简单查询，不需要任何复杂的决策算法。让决策系统的复杂度与任务的复杂度相匹配。

5. **善用日志**：记录智能体的每一次决策和对应的结果。这些日志是你优化决策策略最宝贵的资源——你可以看到哪些决策是明智的，哪些是多余的。

理解并掌握ReAct和Plan-Act，就像掌握了两种基本兵器——ReAct是匕首，灵活但需要步步为营；Plan-Act是长枪，需要准备但出手高效。优秀的智能体开发者知道在什么战场选择什么武器。
