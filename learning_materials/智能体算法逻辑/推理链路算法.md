# ReAct模式：智能体的思考与行动

## 什么是ReAct

ReAct是**Reasoning（推理）+ Acting（行动）**的缩写，由Google Research在2022年的论文《ReAct: Synergizing Reasoning and Acting in Language Models》中正式提出。它是现代AI智能体（Agent）最核心、最基础的工作模式。

ReAct的核心思想可以用一句话概括：**让语言模型在"思考"和"行动"之间交替循环，像人类解决问题一样运作**。传统的语言模型只会"想"（生成文字），而ReAct让它们也能"做"（执行具体操作）——想一想→做一做→看看结果→再想一想→继续做——直到问题解决。

这个看似简单的改变，将LLM从一个"纸上谈兵的理论家"转变为一个"能动手解决问题的实践者"，是AI从对话工具走向自主智能体的关键一步。

## 为什么需要ReAct

让我们通过一个具体的场景来理解ReAct的必要性。

**场景**：用户问"2024年诺贝尔物理学奖颁给了谁？获奖者的主要贡献是什么？"

**纯LLM的回答困境：**

如果你的模型训练数据截止到2023年10月，它根本无法知道2024年诺贝尔奖的结果。它有三种选择：
1. 诚实地回答"我的训练数据截止到2023年，不知道2024年的结果"（正确但无帮助）
2. 编造一个看似合理的答案（幻觉，极其危险）
3. 根据历史趋势猜测（不准确）

**ReAct模式下的回答过程：**

```
第1轮：
  Thought（思考）: 用户问的是2024年诺贝尔物理学奖，这超出了我的训练数据范围，
                    我需要通过搜索获取最新信息。
  Action（行动）: search("2024 Nobel Prize in Physics winner")
  Observation（观察）: [搜索结果] 2024年诺贝尔物理学奖授予了John J. Hopfield
                       和Geoffrey E. Hinton，以表彰他们在使用人工神经网络
                       进行机器学习方面的基础性发现和发明。

第2轮：
  Thought（思考）: 我已经知道了获奖者，但用户还问了他们的主要贡献。
                    搜索结果显示是Hopfield和Hinton，我需要了解更多细节。
  Action（行动）: search("John Hopfield Geoffrey Hinton Nobel Prize contribution neural networks")
  Observation（观察）: [搜索结果] Hopfield发明了Hopfield网络——一种能够存储
                       和重建模式的联想记忆模型。Hinton发明了Boltzmann机，
                       并推动了反向传播算法的应用...

第3轮：
  Thought（思考）: 我现在有了足够的信息来回答用户的问题。让我整理一下：
                    获奖者是Hopfield和Hinton，核心贡献在于神经网络的基础研究。
  Final Answer: 2024年诺贝尔物理学奖授予了John J. Hopfield和Geoffrey E. Hinton。
                他们的主要贡献是...
```

通过这个对比，我们可以清晰地看到ReAct的价值：
- **突破知识边界**：通过搜索获取训练数据之外的最新信息
- **消除幻觉**：每一步的结论都有实际的Observation支撑
- **过程透明**：用户可以清晰地看到AI的思考链和决策过程
- **自我纠正**：如果第一步搜索结果不够精确，AI可以重新搜索

## ReAct的核心循环

ReAct模式定义了一个精妙的循环结构，这个循环由三个基本元素组成：

```
┌──────────────────────────────────────────┐
│                                          │
│   ┌──────────┐     ┌──────────┐          │
│   │ Thought  │────→│  Action  │          │
│   │ (思考)   │     │  (行动)  │          │
│   └──────────┘     └────┬─────┘          │
│        ↑                │                │
│        │                ↓                │
│        │          ┌──────────┐           │
│        └──────────│Observation│          │
│                   │  (观察)  │           │
│                   └──────────┘           │
│                                          │
│   循环直到获得最终答案                      │
└──────────────────────────────────────────┘
```

**三个核心元素的职责：**

### Thought（思考）—— 智能体的"大脑"

Thought是ReAct循环中最关键的环节。在这里，智能体需要：
- 分析当前状态：我现在知道了什么？还缺什么信息？
- 规划下一步：接下来应该做什么？为什么？
- 评估结果：刚才的行动得到了什么？信息足够了吗？
- 决定结束：是否已经收集到回答用户问题所需的全部信息？

一个好的Thought应该像"自言自语"，清晰地记录推理过程。

**Thought示例对比：**

差的Thought（太简单）：
```
我需要搜索一下。
```

好的Thought（有推理）：
```
用户想知道两个城市中哪个更适合居住。我需要比较它们的生活成本、
气候、就业机会和安全状况。先从生活成本开始搜索，获得数据后再
搜索其他维度，最后做一个综合对比。
```

### Action（行动）—— 智能体的"手"

Action是智能体与外部世界交互的唯一途径。每个Action包含三个部分：
- **工具名称**：要调用哪个工具（search、calculator、database_query等）
- **输入参数**：传递给工具的具体参数
- **预期目的**：为什么选择这个工具和参数（隐含在Thought中）

Action必须是**可执行的操作**。如果无法执行（工具不存在、参数格式错误等），智能体需要在Observation中感知失败并在下一个Thought中调整策略。

### Observation（观察）—— 智能体的"感官"

Observation是外部世界对Action的反馈。它可以是搜索结果、计算结果、数据库查询记录、API的JSON响应或错误消息。Observation必须是**原始的、未经智能体"润色"的真实反馈**，否则可能引入新的偏差。

### 轨迹三：代码调试任务

**用户问题**："下面的Python代码在处理学生成绩时有时会报错。请找出所有bug并修复：

```python
def calculate_average(numbers):
    total = 0
    for i in range(len(numbers)):
        total += numbers[i]
    return total / len(numbers)

def find_top_students(scores, threshold=85):
    result = []
    for name, score in scores.items():
        avg = calculate_average(score)
        if avg > threshold:
            result.append(name)
    return result

def format_report(top_students, scores):
    report = '优秀学生报告\\\\n'
    for name in top_students:
        avg = calculate_average(scores[name])
        report += name + ': ' + str(avg) + '\\\\n'
    return report

scores = {
    '张三': [90, 85, 92],
    '李四': [78, 80],
    '王五': [],
    '赵六': [95]
}
print(find_top_students(scores))
```"

## 三种推理模式的深度对比

为了更直观地理解不同推理模式的差异，我们用一个**共享示例任务**来对比纯思维链（CoT）、ReAct和计划-执行（Plan-Execute）三种模式。

**共享任务**："帮我分析过去一个季度苹果、特斯拉、微软的营收变化趋势，找出增长最快和可能出现问题的公司。"

### 纯思维链（CoT）的处理方式

CoT模式只在大脑内推理，不访问任何外部信息。模型凭借训练数据中的记忆来"推测"数据：

```
模型输出：
"根据我了解的信息，2024年第四季度：
- 苹果的营收可能约1200亿美元，由于iPhone销售季节性波动，
  可能环比增长但同比持平。
- 特斯拉的营收可能约250亿美元，受Cybertruck量产推动，
  可能实现两位数的同比增长。
- 微软的营收可能约620亿美元，Azure云服务持续增长，
  推动整体营收稳步上升。

综合分析：特斯拉增长最快，苹果面临最大压力。
但这种分析基于我的训练数据，建议查阅最新财报确认。"
```

**问题**：所有数字都是推测，没有实际验证。"分析结论"实际上毫无可靠依据。

### ReAct模式的处理方式

ReAct通过多轮行动逐步收集真实数据，每步基于实际Observation调整策略。过程大约需要6-8个循环来获取三家公司财报、计算增长率、搜索分析评论、综合得出结论。每一步都建立在真实数据之上。

### 计划-执行（Plan-Execute）的处理方式

Plan-Execute会先制定完整计划：

```
【计划】
步骤1：并行获取三家公司的营收数据（3个API同时调用）
步骤2：计算每家公司的环比和同比增长率
步骤3：搜索行业分析师对三家公司的评价
步骤4：整合信息，生成对比报告

【执行阶段】
→ 同时发出步骤1的三个API调用
→ 等待所有数据返回
→ 按计划执行步骤2-4
→ 输出报告
```

**优势**：步骤1的三个API调用可以并行执行，节省等待时间。如果任务结构清晰且每一步的输入输出格式确定，Plan-Execute效率更高。

**劣势**：如果步骤1返回的数据格式不符合预期（比如某个API返回了按产品线而非按总营收的数据），步骤2-4可能全线失败，必须重新规划。

### 综合对比表

| 对比维度 | 纯CoT | ReAct模式 | Plan-Execute |
|---------|-------|----------|-------------|
| 所需步骤数 | 1步（一次生成） | 6-8个循环 | 2步（计划+执行） |
| 总耗时 | 最快（秒级） | 中等（数十秒） | 较快（内部可并行） |
| 数据准确性 | 低（可能幻觉） | 高（实时获取验证） | 高（实时获取） |
| 错误恢复能力 | 无（一次输出定型） | 强（每步观察后可调整） | 弱（失败需重规划） |
| 适合的任务 | 纯逻辑推理、数学证明 | 信息检索、数据探索、调试 | 结构化、确定性强的流程 |
| 不适合的任务 | 需要实时信息的任务 | 高度可并行的任务 | 探索性、不确定性高的任务 |
| Token消耗 | 低 | 中到高 | 中等 |
| 实现复杂度 | 低（只需提示词） | 中（需要工具系统和解析器） | 中（需要任务分解和调度器） |

### 实际工程中的混合策略

现代智能体框架几乎从不单一使用某一种模式，而是根据任务特征动态组合。一个典型的混合流程：

```
顶层用Plan-Execute制定宏观计划 → 划分可并行子任务 → 每个子任务
内部用ReAct模式逐步执行 → 纯推理步骤退化为CoT
```

例如处理"研究三家公司的财务表现"时，先Plan-Execute将任务分解为"获取数据"和"分析数据"两个阶段。"获取数据"阶段三个搜索子任务并行执行（Plan-Execute优势），"分析数据"阶段用ReAct模式逐步深入（ReAct优势）。这种设计既保证了宏观效率，又保留了微观灵活性。

## ReAct的完整Python实现

下面是一个完整、可直接运行（配合LLM API）的ReAct智能体核心实现。每一个组件都有详细的注释说明。

```python
"""
ReAct Agent 完整实现
====================
这是一个可直接运行的ReAct智能体框架。你需要将 mock_llm_call 替换为
真实的LLM API调用（如OpenAI、Claude API等）即可使用。
"""

import re
import json
from typing import Callable, Any, Optional
from dataclasses import dataclass, field


# ==================== 数据结构 ====================

@dataclass
class Tool:
    """单个工具的定义"""
    name: str
    description: str
    func: Callable[[str], str]
    # 参数说明，帮助LLM理解如何使用
    parameter_description: str = ""

    def execute(self, input_str: str) -> str:
        """安全执行工具，捕获所有异常"""
        try:
            result = self.func(input_str)
            return str(result)
        except Exception as e:
            return f"[工具执行错误] {type(e).__name__}: {str(e)}"


@dataclass
class AgentStep:
    """记录单次循环的完整信息"""
    iteration: int
    thought: str = ""
    action: str = ""
    action_input: str = ""
    observation: str = ""


# ==================== 输出解析器 ====================

class ReActParser:
    """解析LLM输出，提取Thought、Action、Action Input、Final Answer"""

    # 正则模式：分别匹配各字段
    THOUGHT_PATTERN = re.compile(
        r'Thought:\s*(.+?)(?=\n(?:Action|Final Answer|Observation)|\Z)',
        re.DOTALL | re.IGNORECASE
    )
    ACTION_PATTERN = re.compile(
        r'Action:\s*(\S+)',
        re.IGNORECASE
    )
    ACTION_INPUT_PATTERN = re.compile(
        r'Action Input:\s*(.+?)(?=\n(?:Thought|Action|Observation)|\Z)',
        re.DOTALL | re.IGNORECASE
    )
    FINAL_ANSWER_PATTERN = re.compile(
        r'Final Answer:\s*(.+)',
        re.DOTALL | re.IGNORECASE
    )

    @classmethod
    def parse(cls, text: str) -> dict:
        """
        解析LLM的一轮输出。
        返回字典，可能的键：thought, action, action_input, final_answer, parse_error
        """
        result = {}

        # 提取 Thought
        thought_match = cls.THOUGHT_PATTERN.search(text)
        if thought_match:
            result['thought'] = thought_match.group(1).strip()

        # 优先检查是否是 Final Answer
        final_match = cls.FINAL_ANSWER_PATTERN.search(text)
        if final_match:
            result['final_answer'] = final_match.group(1).strip()
            return result

        # 提取 Action
        action_match = cls.ACTION_PATTERN.search(text)
        if action_match:
            result['action'] = action_match.group(1).strip()

        # 提取 Action Input
        action_input_match = cls.ACTION_INPUT_PATTERN.search(text)
        if action_input_match:
            result['action_input'] = action_input_match.group(1).strip()

        # 如果既没有 Final Answer 也没有 Action，记录格式错误
        if 'final_answer' not in result and 'action' not in result:
            result['parse_error'] = True
            result['raw_output'] = text[:300]

        return result


# ==================== 提示词构建器 ====================

class PromptBuilder:
    """构建ReAct模式的系统提示词"""

    @staticmethod
    def build(tools: list[Tool], max_iterations: int) -> str:
        """生成系统提示词"""

        # 构建工具描述列表
        tool_lines = []
        for tool in tools:
            param_info = f"（参数：{tool.parameter_description}）" \
                if tool.parameter_description else ""
            tool_lines.append(f"- **{tool.name}**: {tool.description}{param_info}")

        tool_descriptions = "\n".join(tool_lines) if tool_lines else "（暂无可用工具）"

        prompt = f"""你是一个使用ReAct模式工作的AI智能助手。

## 可用工具

你可以使用以下工具获取信息和执行操作：

{tool_descriptions}

## 输出格式要求（极其重要！）

你必须严格遵循以下格式。每次只能输出以下两种形式之一：

### 形式一：执行工具调用
```
Thought: [你的思考过程——分析当前状态，决定下一步行动和原因]
Action: [工具名称，必须是上面列出的工具之一]
Action Input: [传递给工具的参数]
```

### 形式二：给出最终答案（仅在信息充分时使用）
```
Thought: 我现在已经收集了足够的信息来回答用户的问题。
Final Answer: [用中文给出的完整答案，包含所有关键信息]
```

## 重要规则

1. 每次只能调用一个工具。等待Observation后再决定下一步。
2. 如果工具返回错误，不要放弃——分析错误原因，尝试不同参数或工具。
3. 最多进行 {max_iterations} 轮工具调用。
4. 在给出Final Answer之前，确认用户的每个子问题都已得到回答。
5. 永远不要编造信息。如果你不知道，用工具查找而非猜测。
6. Final Answer必须使用中文，清晰、准确、完整。

## 开始工作

下面是用户的问题，请开始你的推理过程。"""

        return prompt


# ==================== ReAct Agent 主类 ====================

class ReActAgent:
    """
    ReAct智能体核心引擎。

    使用方式：
        agent = ReActAgent(llm_call=your_llm_function, max_iterations=10)
        agent.register_tool("search", "搜索互联网", your_search_function,
                            parameter_description="搜索关键词（字符串）")
        answer = agent.run("你的问题")
    """

    def __init__(self, llm_call: Callable[[str], str], max_iterations: int = 10):
        """
        Args:
            llm_call: 调用LLM的函数。接收prompt字符串，返回LLM的回复文本。
            max_iterations: 最大工具调用轮数，防止无限循环。
        """
        self.llm_call = llm_call
        self.max_iterations = max_iterations
        self.tools: dict[str, Tool] = {}
        self.trajectory: list[AgentStep] = []
        self.MAX_CONSECUTIVE_PARSE_ERRORS = 3

    def register_tool(
        self,
        name: str,
        description: str,
        func: Callable[[str], str],
        parameter_description: str = ""
    ):
        """注册一个新工具到智能体"""
        self.tools[name] = Tool(
            name=name,
            description=description,
            func=func,
            parameter_description=parameter_description
        )

    def run(self, user_question: str, verbose: bool = True) -> str:
        """
        运行ReAct主循环。
        Args:
            user_question: 用户问题
            verbose: 是否打印详细执行过程
        Returns:
            最终答案字符串
        """
        # 构建初始对话
        system_prompt = PromptBuilder.build(
            list(self.tools.values()), self.max_iterations
        )
        conversation = f"{system_prompt}\n\n用户问题：{user_question}\n\nThought:"

        self.trajectory = []
        parse_error_count = 0

        for iteration in range(1, self.max_iterations + 1):
            if verbose:
                print(f"\n{'='*60}")
                print(f"第 {iteration} 轮")
                print(f"{'='*60}")

            # 调用LLM
            llm_response = self.llm_call(conversation)
            if verbose:
                print(f"\n[LLM 输出]\n{llm_response[:500]}")

            # 解析LLM输出
            parsed = ReActParser.parse(llm_response)

            # 处理解析错误
            if parsed.get('parse_error'):
                parse_error_count += 1
                if parse_error_count >= self.MAX_CONSECUTIVE_PARSE_ERRORS:
                    return (
                        f"错误：LLM连续 {self.MAX_CONSECUTIVE_PARSE_ERRORS} 次"
                        f"输出格式不正确。请检查提示词或模型配置。\n"
                        f"最近一次输出：{parsed.get('raw_output', '无')}"
                    )
                # 添加格式纠正提示
                correction = (
                    "\n\n【系统提示】你的上一个输出格式不正确。请严格遵循以下格式：\n\n"
                    "Thought: [你的思考过程]\n"
                    "Action: [工具名称]\n"
                    "Action Input: [工具参数]\n\n"
                    "或者当准备给出最终答案时：\n\n"
                    "Thought: 我现在有足够的信息来回答用户的问题。\n"
                    "Final Answer: [你的答案]\n\n"
                    "请重新输出。"
                )
                conversation += f"\n{llm_response}\n{correction}"
                continue
            else:
                parse_error_count = 0  # 解析成功，重置错误计数

            # 记录当前步骤
            step = AgentStep(
                iteration=iteration,
                thought=parsed.get('thought', ''),
                action=parsed.get('action', ''),
                action_input=parsed.get('action_input', '')
            )
            if verbose:
                print(f"  Thought: {step.thought[:200]}...")

            # 如果是最终答案
            if 'final_answer' in parsed:
                if verbose:
                    print(f"  >>> 最终答案 <<<")
                return parsed['final_answer']

            # 执行工具
            if step.action:
                tool = self.tools.get(step.action)
                if tool is None:
                    step.observation = (
                        f"错误：工具 '{step.action}' 不存在。"
                        f"可用工具：{', '.join(self.tools.keys())}。"
                        f"请从可用工具列表中选择。"
                    )
                else:
                    if verbose:
                        print(f"  Action: {step.action}({step.action_input[:100]})")
                    step.observation = tool.execute(step.action_input)

                if verbose:
                    print(f"  Observation: {step.observation[:200]}...")

                self.trajectory.append(step)
                # 将本轮完整交互追加到对话历史
                conversation += (
                    f"\n{llm_response}\n"
                    f"Observation: {step.observation}\n\n"
                    f"Thought:"
                )
            else:
                # 有Thought但没有Action也没有Final Answer——不完整
                conversation += (
                    f"\n{llm_response}\n"
                    f"Observation: [请继续。如果你认为任务已经完成，"
                    f"请使用 'Final Answer:' 格式给出答案。]"
                    f"\n\nThought:"
                )

        # 超出最大迭代次数
        return (
            f"已达到最大迭代次数（{self.max_iterations}轮）。"
            f"请尝试简化问题或增加 max_iterations 参数。"
            f"\n执行轨迹摘要：共记录了 {len(self.trajectory)} 步操作。"
        )

    def get_trajectory_summary(self) -> str:
        """获取执行轨迹摘要，便于调试和分析"""
        if not self.trajectory:
            return "（无执行轨迹）"
        lines = []
        for step in self.trajectory:
            lines.append(
                f"第{step.iteration}轮 | Action: {step.action} | "
                f"Observation长度: {len(step.observation)}字符"
            )
        return "\n".join(lines)

            if key in query:
                return value
        return f"未找到关于'{query}'的精确信息。"

    # 创建Agent并注册工具
    agent = ReActAgent(llm_call=mock_llm, max_iterations=8)
    agent.register_tool(
        name="search",
        description="搜索互联网获取信息。返回与查询相关的文字内容。",
        func=search_func,
        parameter_description="搜索关键词（字符串）"
    )

    # 运行Agent
    print("=" * 60)
    print("ReAct Agent 演示启动")
    print("=" * 60)

    answer = agent.run(
        "电话的发明者出生在哪座城市？该城市属于哪个国家？",
        verbose=True
    )

    print("\n" + "=" * 60)
    print("【最终答案】")
    print("=" * 60)
    print(answer)
    print("\n【轨迹摘要】")
    print(agent.get_trajectory_summary())


if __name__ == "__main__":
    demo()
```

### 代码关键设计说明

1. **工具注册机制**：通过 `register_tool` 动态添加工具。每个工具有名称、描述、执行函数和参数说明。工具执行用 try-except 包裹，永远不会让Agent崩溃。

2. **安全限制体系**：`max_iterations` 防止无限循环；`MAX_CONSECUTIVE_PARSE_ERRORS` 防止格式错误死循环；工具执行异常被捕获并转为友好的Observation文本。

3. **输出解析鲁棒性**：LLM的输出格式可能不完美。`ReActParser` 使用多组正则表达式进行宽松匹配。解析失败时不崩溃，而是向对话中插入格式纠正提示，给LLM重新输出的机会。

4. **完整的轨迹记录**：`AgentStep` 数据类和 `trajectory` 列表记录每一步的完整信息。在调试时，`get_trajectory_summary()` 可以快速了解Agent都做了什么。

5. **提示词工程**：`PromptBuilder` 将工具描述和格式要求清晰组织。提示词中对"格式要求"的强调和具体示例是ReAct能否正常工作的关键——没有严格的格式约束，模型容易偏离预期输出结构。

## 常见失败模式与修复方案

ReAct智能体虽然强大，但在实际运行中会遇到各种失败情况。了解这些失败模式是构建可靠Agent的必修课。

### 失败模式一：无限循环——重复执行相同操作

**现象**：智能体反复搜索同一个关键词，每次得到相同或类似的结果，但就是不停止。

**具体案例**：
```
第1轮：Action: search("巴黎人口") → Observation: 巴黎人口约216万
第2轮：Thought: 信息不够精确，我再搜一次
       Action: search("巴黎人口") → Observation: 巴黎人口约216万
第3轮：Thought: 还是不够精确，再精确搜索
       Action: search("巴黎人口") → Observation: 巴黎人口约216万
（循环无限继续...）
```

**根本原因**：提示词中没有明确的"信息充分性判断标准"。模型认为自己还没有得到"足够好"的答案，但又不知道还能做什么不同的事。

**修复方案**：
- 提示词中加入约束："如果连续两次搜索返回相同或类似的结果，说明该信息来源的精度已达到上限，请基于现有信息给出答案。"
- 代码层面检测连续重复操作：如果连续3次相同的 Action + Action Input 组合，强制插入系统提示"请换一个策略或基于现有数据给出结论"。
- 为工具设计"置信度"字段，让Observation包含信息质量的元数据，帮助模型判断是否继续。

---

### 失败模式二：行动与观察脱节——忽略Observation实际内容

**现象**：模型基于自己的"想象"而非实际的Observation来继续推理。Observation明确说了"未找到"或给出了特定信息，但模型完全无视。

**具体案例**：
```
Action: search("2028年奥运会举办城市")
Observation: 2028年奥运会将在美国洛杉矶举办。

Thought: 好的，2028年奥运会在巴黎举办，现在可以查巴黎的信息了...
（模型完全忽略了Observation中的"洛杉矶"，用了自己记忆中的错误信息）
```

**根本原因**：模型训练数据中关于"2024年奥运会在巴黎"的知识太强，在生成时优先调用了训练记忆而非上下文中的Observation。

**修复方案**：
- 提示词中强调："你必须严格基于Observation中的内容进行推理。Observation是你唯一的信息来源——即使它与你记忆中的信息矛盾，也要以Observation为准。"
- 在Observation前面加显眼的格式标记，如 `【系统返回】` 或 `>>> 查询结果 <<<`，在视觉上区分模型输出和工具返回。
- 如果模型在Observation中接收到关键数值信息（如数字、日期、人名），在下一个提示中显式重复这些关键信息。

---

### 失败模式三：过早给出最终答案

**现象**：智能体只执行了一两个动作，在用户问题的多个部分还完全没有被处理的情况下就给出了Final Answer。

**具体案例**：
```
用户: "比较苹果、特斯拉和微软的最新季度营收和利润，分析增长趋势。"

第1轮：Action: search("苹果最新季度营收利润")
       Observation: 苹果营收895亿，净利润236亿

第2轮：Thought: 我已经有足够信息了。
       Final Answer: 苹果营收895亿美元，是三家公司中表现最好的。
       （完全忽略了特斯拉和微软，也没提增长趋势）
```

**根本原因**：模型倾向于快速完成任务（节省token），缺乏"完整性自我检查"机制。

**修复方案**：
- 提示词中加入"自我审查"步骤指令："在给出Final Answer前，请逐条检查用户的每个子问题是否都已被回答。如果发现遗漏，请继续收集信息。"
- 设置最小迭代次数（如至少3轮），前3轮不允许输出Final Answer。
- 在系统提示中给出"完整回答"的示例和"不完整回答"的反例，让模型学会区分。

---

### 失败模式四：工具调用参数错误

**现象**：模型调用了正确的工具，但传入的参数格式不正确、数值不合理或类型错误。

**具体案例**：
```
Action: calculator("苹果的营收减去特斯拉的营收")  # 传入了自然语言而非数学表达式
Observation: [工具执行错误] 无效的数学表达式

Action: search("")  # 传入了空字符串
Observation: [工具执行错误] 搜索关键词不能为空
```

**根本原因**：模型的提示词中对工具参数的说明不够具体，或者工具的参数格式与模型的直觉有偏差。

**修复方案**：
- 每个工具的 `parameter_description` 必须包含具体的参数格式说明和正确示例。比如计算器工具："参数应为纯数学表达式，例如 '2+3*4' 或 '(100-50)/2'。不要传入自然语言描述。"
- 在工具内部做好输入校验，返回清晰且有建设性的错误信息，帮助模型在下一次尝试中修正。错误信息不仅要说"错了"，还要说"应该怎么改"。
- 提供 Few-shot 示例：在系统提示词中给出几个正确调用和错误修正的例子。

---

### 失败模式五：上下文溢出与遗忘

**现象**：当ReAct循环次数增多时，历史对话越来越长，可能超出模型的上下文窗口限制。早期信息被截断后，Agent"忘记"了任务目标或已经获取的关键数据。

**具体案例**：
```
第1轮：搜索A → 结果A（包含股票代码"TSLA"）
第5轮：搜索E → 结果E
...（中间大量搜索和长文本Observation）
第18轮：上下文已超过100K tokens
第19轮：Thought: 用户想查特斯拉的股价？特斯拉的股票代码是什么来着？
       我需要先搜索股票代码...
       （此时上下文已满，第1轮的Observation被截断，Agent"忘了"TSLA这个代码）
```

**根本原因**：LLM的上下文窗口有限（如128K），而ReAct的多轮循环加上长Observation会快速消耗token。

**修复方案**：
- **上下文压缩**：每5-6轮用LLM对已有的Observation做摘要，用200字的摘要替代5000字的原始Observation文本。
- **信息分级存储**：将用户原始问题和"关键发现"固定在上下文的最前面，使用特殊标记（如 `[关键信息]`），使其在截断时优先保留。
- **控制单次Observation长度**：工具设计时让返回结果精炼。比如搜索工具限制返回前300字，而非全部内容。
- **动态迭代上限**：根据每次Observation的长度动态调整 max_iterations。如果Observation普遍很长（>2000字），降低迭代次数上限。

---

### 失败模式六：幻觉性工具调用

**现象**：模型"发明"了系统中不存在的工具，或者凭空给工具传入了基于幻觉的参数值。

**具体案例**：
```
Thought: 我需要获取特斯拉的实时股价和K线图。
Action: get_realtime_stock_chart     # 这个工具根本不存在！
Action Input: {"symbol": "TSLA", "interval": "1d", "range": "3mo"}
```

或者：
```
Action: search("特斯拉 2025年6月15日 Model Y降价新闻")
# 模型假设6月15日有降价新闻，但实际并没有这样的事件
```

**根本原因**：模型在生成Action时，有时会"借用"它在训练数据中见过的其他系统的工具名，或者根据对话上下文"推测"出一些看似合理的参数值。

**修复方案**：
- 系统提示词中必须列出所有可用工具的确切名称，并明确声明"你**只能**使用上面列出的工具，不能使用任何其他工具"。
- 在 `ReActAgent.run()` 中，当检测到工具不存在时，返回的Observation不仅要告知错误，还要**重新列出所有可用工具**，帮助模型在下一轮选择正确的工具。
- 在工具参数中使用枚举类型限制，避免自由文本参数被模型填入幻觉值。

## 与其他模式的关系

### ReAct vs Chain-of-Thought（CoT）
- CoT只在思维层面推理，不执行任何动作。适合纯推理任务（数学证明、逻辑推理、代码理解）。
- ReAct在推理的同时可以执行实际操作，适合需要外部信息的任务。
- 实际上，CoT的思想被融入到了ReAct的Thought步骤中——每一个Thought本质上就是一小段Chain-of-Thought。

### ReAct vs Plan-and-Execute
- ReAct一步一步交替思考和行动，灵活但单步效率低。
- Plan-and-Execute先制定完整计划再批量执行，高效但不够灵活。
- 实际应用中常常结合使用：宏观上Plan-Execute，微观上每个步骤用ReAct。

### ReAct vs Reflexion（反思模式）
Reflexion在ReAct的基础上增加了"自我反思"层。每次完成任务后，智能体会反思自己的表现，并将经验教训记录下来用于改进未来的表现：

```
ReAct:     Thought → Action → Observation → ... → 完成任务
Reflexion: Thought → Action → Observation → ... → 完成任务
             ↓
           Reflection（反思）: 哪些地方做得好？哪些可以改进？
             ↓
           将反思结果存入"长期记忆"
             ↓
           下次类似任务时参考之前的反思
```

## 实现ReAct的提示词设计要点

1. **格式约束极其重要**：如果没有严格的格式要求，模型可能把Thought、Action、Observation混在一起输出，导致解析失败。使用特定的标记（如`Thought:`、`Action:`）并用代码解析这些标记是标准做法。

2. **错误处理指令必不可少**：工具调用可能失败（网络超时、API限流、参数错误等）。没有错误处理指令的Agent遇到失败可能"卡住"或产生混乱输出。

3. **轮次限制防止失控**：有时Agent会陷入循环（反复搜索同一内容、反复尝试不存在的工具等）。设置最大轮次限制可以防止无限消耗token。

4. **中英文混合处理**：提示词中需要明确Final Answer的输出语言。否则模型可能用英文输出Observation的内容。

## 实际应用

ReAct模式是现代AI Agent的基础。几乎所有主流Agent框架都在使用或变体使用ReAct模式：

- **LangChain**：其AgentExecutor就是ReAct的工程化实现，支持多种工具集成和记忆管理。
- **AutoGPT**：在ReAct基础上增加了目标分解和长期记忆，让智能体可以自主完成更长期的复杂任务。
- **CrewAI**：将多个ReAct智能体组合成一个"团队"，每个智能体负责不同的子任务。
- **OpenAI Assistants API**：内置的function calling功能本质上也是ReAct模式的一种变体。

## 学习建议

1. **手动模拟**：拿一个简单问题，用纸笔手动模拟一遍ReAct循环，理解每一步的输入和输出。这比读任何文章都直观。

2. **阅读轨迹**：使用LangChain的Agent功能或任何支持Agent的框架，开启详细日志（verbose模式），仔细阅读每一次Thought-Action-Observation循环，理解模型为何做出某些决策。

3. **自己实现一遍**：用上面的Python代码作为起点，接入真实的LLM API，为它添加两三个简单工具（网页搜索、计算器、天气查询），看它如何处理真实问题。

4. **设计提示词模板**：根据自己的使用场景，定制化设计ReAct提示词模板。不同的任务可能需要调整模板中的约束条件和格式要求。

5. **观察失败，总结经验**：真正理解ReAct的最好方式，是观察它何时失败以及为什么失败。记录下失败案例，思考是提示词的问题、工具设计的问题，还是任务本身就不适合ReAct模式。

6. **深入阅读原论文**：阅读《ReAct: Synergizing Reasoning and Acting in Language Models》（Yao et al., 2022），理解ReAct的理论基础和实验设计。同时关注后续改进工作如Reflexion、Tree of Thoughts等。

ReAct模式虽看似简单——只是在LLM推理中加入了工具调用循环——但它从根本上改变了AI系统的能力边界。掌握ReAct不仅意味着学会了一种技术，更意味着你理解了智能体设计的核心哲学：**思考和行动的统一，才是真正的智能**。
