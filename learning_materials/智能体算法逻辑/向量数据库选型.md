# 向量数据库选型与相似度检索：从原理到实战

## 一、向量数据库基础：为什么传统数据库不够用

### 1.1 向量检索的本质

传统数据库的核心操作是精确匹配和范围查询：

```sql
-- 传统数据库擅长的事
SELECT * FROM users WHERE name = '张三';           -- 精确匹配
SELECT * FROM products WHERE price BETWEEN 100 AND 500;  -- 范围查询
SELECT * FROM articles WHERE title LIKE '%Python%';      -- 模糊匹配（效率低）
```

但当你需要回答"找出于'深度学习'最相关的10篇文章"时，SQL就力不从心了。这是因为：

- "最相关"是语义层面的概念，不是字符匹配
- "深度学习"和"神经网络"在字面上不同但语义高度相关
- B+树索引无法理解"距离"概念

向量数据库的出现解决了这个gap：

```
文本 → Embedding Model → 高维向量 → 向量数据库（按距离检索）
```

### 1.2 向量检索的核心挑战

| 挑战 | 说明 | 数据量级 |
|------|------|---------|
| 维度灾难 | 高维空间（768-4096维）中距离计算极其昂贵 | 维度越高，索引效率越低 |
| 近似 vs 精确 | 精确检索需要O(N*D)次计算 | 百万级向量，精确检索不可行 |
| 动态更新 | 插入/删除向量后索引需要更新 | 生产环境频繁更新 |
| 内存 vs 磁盘 | 向量数据量远超内存容量 | 需要磁盘支持和调度策略 |

### 1.3 向量数据库的演进历程

```
第一代：专用向量库（2017-2019）
  FAISS (Meta, 2017) — 高性能向量搜索库
  Annoy (Spotify, 2018) — 近似最近邻搜索
  ScaNN (Google, 2020) — 最大内积搜索优化

第二代：向量数据库（2019-2022）
  Milvus (2019) — 云原生分布式向量数据库
  Pinecone (2021) — 全托管向量数据库服务
  Weaviate (2019) — 开源向量搜索引擎（混合搜索）

第三代：多功能向量数据库（2022-至今）
  Qdrant (2021) — Rust实现的高性能向量数据库
  Chroma (2022) — 开发者友好的嵌入式向量数据库
  pgvector (2022) — PostgreSQL扩展，SQL+向量混合
  Elasticsearch (2023+) — 在搜索引擎上叠加向量能力
```

## 二、主流向量数据库深度对比

### 2.1 Chroma：轻量级开源首选

Chroma定位为"AI原生的开源向量数据库"，设计理念是简单易用、开发者友好。

**核心特点**：
- 纯Python实现，安装即用（`pip install chromadb`）
- 内建多种Embedding Function（Sentence-Transformers、OpenAI、Cohere等）
- 支持内存模式和持久化模式
- 提供元数据过滤（metadata filtering）

**架构概览**：

```
Chroma Architecture:
┌─────────────────────────────────────┐
│            Chroma Client             │
├─────────────────────────────────────┤
│ ┌─────────┐ ┌──────────┐ ┌────────┐│
│ │ Embedding│ │  Vector  │ │Metadata││
│ │ Function │ │  Store   │ │ Filter ││
│ └─────────┘ └──────────┘ └────────┘│
├─────────────────────────────────────┤
│   Storage Backend:                   │
│   In-Memory / SQLite / DuckDB       │
└─────────────────────────────────────┘
```

**适用场景**：
- 原型开发和POC验证
- 中小规模应用（百万级向量以内）
- 需要快速集成LangChain/LlamaIndex的项目
- 本地开发环境

**局限性**：
- 分布式能力有限，不适合超大规模场景
- 高并发性能不如专为生产设计的方案
- 社区相对年轻，生产踩坑案例较少

### 2.2 Pinecone：全托管云原生

Pinecone是向量数据库领域的SaaS标杆，提供完全托管、零运维的向量搜索服务。

**核心特点**：
- 全托管：无需管理服务器、索引、分片
- 弹性伸缩：自动根据数据量和请求量调整资源
- 实时索引：支持低延迟的向量写入和搜索
- 元数据过滤：支持基于元数据的预过滤和后过滤
- 命名空间隔离：多租户数据隔离

```python
import pinecone
from pinecone import Pinecone, ServerlessSpec

# Pinecone初始化
pc = Pinecone(api_key="your-api-key")

# 创建索引
pc.create_index(
    name="knowledge-base",
    dimension=1536,  # text-embedding-3-small的维度
    metric="cosine",
    spec=ServerlessSpec(
        cloud="aws",
        region="us-west-2"
    )
)

# 连接到索引
index = pc.Index("knowledge-base")

# 写入向量（批量）
vectors_to_upsert = [
    {
        "id": "doc_001",
        "values": [0.1, 0.2, ..., 0.5],  # 1536维向量
        "metadata": {"source": "manual.pdf", "page": 3, "category": "technical"}
    },
    # ... 更多向量
]
index.upsert(vectors=vectors_to_upsert, namespace="production")

# 查询
query_results = index.query(
    vector=[0.15, 0.23, ..., 0.48],  # 查询向量
    top_k=5,
    namespace="production",
    filter={"category": {"$eq": "technical"}},  # 元数据过滤
    include_metadata=True,
)

for match in query_results["matches"]:
    print(f"ID: {match['id']}, Score: {match['score']:.4f}")
```

**优势**：
- 零运维负担，适合不想管理基础设施的团队
- 可靠性高，SLA保障
- 生态集成好（LangChain、LlamaIndex官方支持）

**劣势**：
- 成本随数据量线性增长，大规模使用成本高
- 数据必须上传到云端，敏感数据场景受限
- 无法自定义索引算法和参数
- 离线/air-gapped环境不可用

**价格参考**（2024年数据）：

| 方案 | 价格 | 容量 | 适合 |
|------|------|------|------|
| Free | 免费 | 最多100K向量 | 学习和原型 |
| Standard | ~$0.33/GB/月 | 按使用计费 | 中小型生产 |
| Enterprise | 定制报价 | 无限 | 大型企业 |

### 2.3 Weaviate：混合搜索的标杆

Weaviate是一个开源向量数据库，最大的特色是对混合搜索（Hybrid Search）的原生支持——同时进行向量搜索和关键词搜索。

**核心特点**：
- 原生混合搜索：`hybrid` 参数一键融合向量+BM25
- GraphQL API：查询接口灵活强大
- 模块化架构：向量化模块、生成模块可插拔
- 支持多模态：文本、图片、音频向量统一存储

```python
import weaviate
from weaviate.classes.query import HybridFusion

# 连接Weaviate
client = weaviate.connect_to_local()

# 导入数据
collection = client.collections.get("Documents")
with collection.batch.dynamic() as batch:
    for doc in documents:
        batch.add_object(
            properties={
                "title": doc["title"],
                "content": doc["content"],
                "category": doc["category"],
            },
            # Weaviate自动向量化（如果配置了向量化模块）
        )

# 混合搜索：向量 + 关键词
response = collection.query.hybrid(
    query="Python异步编程的最佳实践",
    alpha=0.5,  # 0=纯关键词, 1=纯向量, 0.5=均等混合
    fusion_type=HybridFusion.RELATIVE_SCORE,
    limit=5,
    return_properties=["title", "content", "category"],
)

for obj in response.objects:
    print(f"标题: {obj.properties['title']}")
    print(f"内容: {obj.properties['content'][:100]}...\n")
```

**适用场景**：
- 需要混合搜索的场景（专有名词+语义理解）
- 多模态搜索（文本+图片）
- 需要灵活查询能力的场景（GraphQL支持复杂查询）
- 自托管的开源方案

**局限性**：
- 运维复杂度较高（需要管理JVM）
- 资源消耗较大（内存需求高）
- 学习曲线比Chroma陡峭

### 2.4 Milvus：高性能分布式

Milvus是LF AI & Data基金会旗下的开源向量数据库，专注于大规模、高性能的向量检索场景。

**核心特点**：
- 云原生架构：计算存储分离，支持弹性伸缩
- 多种索引类型：IVF_FLAT、IVF_SQ8、IVF_PQ、HNSW、ANNOY等
- 分布式：支持分片、副本、负载均衡
- 一致性可调：从强一致性到最终一致性
- GPU加速：支持GPU索引构建和搜索

```python
from pymilvus import (
    connections, Collection, FieldSchema,
    CollectionSchema, DataType
)

# 连接Milvus
connections.connect(host='localhost', port='19530')

# 定义Schema
fields = [
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True, auto_id=True),
    FieldSchema(name="text", dtype=DataType.VARCHAR, max_length=65535),
    FieldSchema(name="embedding", dtype=DataType.FLOAT_VECTOR, dim=768),
    FieldSchema(name="source", dtype=DataType.VARCHAR, max_length=256),
]

schema = CollectionSchema(fields, description="知识库文档")

# 创建Collection
collection = Collection(name="knowledge_base", schema=schema)

# 创建索引
index_params = {
    "metric_type": "IP",  # 内积(Inner Product)
    "index_type": "HNSW",
    "params": {"M": 16, "efConstruction": 200}
}
collection.create_index(field_name="embedding", index_params=index_params)

# 加载Collection到内存
collection.load()

# 插入数据
entities = [
    [0.1, 0.2, ..., 0.5],  # embeddings
    ["文档内容1", "文档内容2"],  # texts
    ["doc_a.pdf", "doc_b.pdf"],  # sources
]
collection.insert([entities[0], entities[1], entities[2]])

# 搜索
search_params = {"metric_type": "IP", "params": {"ef": 64}}
results = collection.search(
    data=[query_vector],
    anns_field="embedding",
    param=search_params,
    limit=5,
    output_fields=["text", "source"]
)

for hits in results:
    for hit in hits:
        print(f"ID: {hit.id}, Score: {hit.score:.4f}, Text: {hit.entity.get('text')[:50]}")
```

**适用场景**：
- 十亿级向量规模的超大规模应用
- 需要高并发、低延迟的生产环境
- 对检索精度和召回率有严格要求的场景
- 需要GPU加速的场景

**局限性**：
- 系统复杂度高，运维门槛高
- 资源消耗大（至少需要一定的内存）
- 小规模场景过于"重"

### 2.5 Qdrant：Rust实现的高性能方案

Qdrant采用Rust编写，在性能、内存效率方面有天然优势。

**核心特点**：
- Rust实现，内存安全和性能兼得
- Payload索引：对元数据建立独立索引，过滤效率高
- 量化支持：Scalar Quantization和Product Quantization减少内存占用
- 多种部署模式：本地Docker、Qdrant Cloud、Kubernetes

```python
from qdrant_client import QdrantClient
from qdrant_client.models import (
    Distance, VectorParams, PointStruct, Filter, FieldCondition, MatchValue
)

# 连接
client = QdrantClient(host="localhost", port=6333)

# 创建Collection
client.create_collection(
    collection_name="docs",
    vectors_config=VectorParams(size=768, distance=Distance.COSINE),
)

# 插入点
points = [
    PointStruct(
        id=i,
        vector=embedding,
        payload={"text": text, "category": "tech", "year": 2024}
    )
    for i, (embedding, text) in enumerate(zip(embeddings, texts))
]
client.upsert(collection_name="docs", points=points)

# 带过滤的搜索
results = client.search(
    collection_name="docs",
    query_vector=query_vector,
    limit=5,
    query_filter=Filter(
        must=[
            FieldCondition(key="category", match=MatchValue(value="tech")),
            FieldCondition(key="year", range={"gte": 2023}),
        ]
    ),
    with_payload=True,
)

for hit in results:
    print(f"Score: {hit.score:.4f}, Text: {hit.payload['text'][:50]}")
```

**适用场景**：
- 对性能有严格要求的场景（Rust的零成本抽象）
- 需要高精度元数据过滤
- 中等到大规模的向量检索
- 内存受限的环境（量化功能）

### 2.6 FAISS：Meta的高性能向量搜索库

FAISS不是完整的数据库，而是一个高效的向量相似度搜索库，通常被其他向量数据库用作底层引擎。

**核心特点**：
- 极致性能：C++实现，支持GPU加速
- 丰富的索引类型：超过20种索引结构
- 内存优化：支持Product Quantization等压缩技术
- 批处理优化：对大batch查询有特别优化

```python
import faiss
import numpy as np

# 准备数据
dimension = 768
num_vectors = 100000
vectors = np.random.random((num_vectors, dimension)).astype('float32')

# 方案A：精确搜索（Flat索引）
index_flat = faiss.IndexFlatIP(dimension)  # 内积(Inner Product)
index_flat.add(vectors)

# 方案B：倒排文件 + 乘积量化（适合大规模）
nlist = 100  # 聚类中心数
m = 8        # 量化子向量数
quantizer = faiss.IndexFlatL2(dimension)
index_ivfpq = faiss.IndexIVFPQ(quantizer, dimension, nlist, m, 8)
index_ivfpq.train(vectors)  # IVF需要训练
index_ivfpq.add(vectors)

# 方案C：HNSW（高精度、高性能）
index_hnsw = faiss.IndexHNSWFlat(dimension, 32)  # 32: M参数
index_hnsw.add(vectors)

# 搜索
k = 5
query = np.random.random((1, dimension)).astype('float32')
distances, indices = index_hnsw.search(query, k)

print(f"最近邻索引: {indices[0]}")
print(f"距离: {distances[0]}")

# 保存索引到磁盘
faiss.write_index(index_hnsw, "faiss_index.bin")

# 从磁盘加载
loaded_index = faiss.read_index("faiss_index.bin")

# GPU加速（如果可用）
res = faiss.StandardGpuResources()
gpu_index = faiss.index_cpu_to_gpu(res, 0, index_flat)  # 第0块GPU
```

**适用场景**：
- 对搜索性能有极致要求的场景
- 需要精细控制索引参数的研究场景
- 作为其他向量数据库的底层引擎
- GPU加速的批处理场景

**局限性**：
- 不是完整数据库：无CRUD、无持久化、无并发控制
- 需要自己管理ID到数据的映射
- 索引构建后无法增量更新（需要重建）
- 纯C++库，Python只是绑定

## 四、相似度度量：数学基础与选择

### 4.1 四种核心度量

**余弦相似度（Cosine Similarity）**

最常用的向量相似度度量，衡量两个向量方向的接近程度：

$$\text{Cosine}(A, B) = \frac{A \cdot B}{||A|| \times ||B||} = \frac{\sum_{i=1}^{n} A_i B_i}{\sqrt{\sum A_i^2} \sqrt{\sum B_i^2}}$$

值域：[-1, 1]，1表示完全同向，0表示正交，-1表示完全反向。

```python
import numpy as np

def cosine_similarity(a: np.ndarray, b: np.ndarray) -> float:
    """
    余弦相似度：已归一化的向量等价于点积
    """
    # 标准化方法
    a_norm = a / np.linalg.norm(a)
    b_norm = b / np.linalg.norm(b)
    return np.dot(a_norm, b_norm)

# 实战技巧：先归一化，再点积（性能优化）
def cosine_via_dot(a: np.ndarray, b: np.ndarray) -> float:
    a_norm = a / np.linalg.norm(a)
    b_norm = b / np.linalg.norm(b)
    return np.dot(a_norm, b_norm)
```

**适用场景**：文档检索、语义搜索 —— 文本embedding的范数往往与文本长度相关，余弦相似度能消除这种影响。

**欧氏距离（Euclidean Distance）**

最直观的距离度量，衡量空间中两点的直线距离：

$$L2(A, B) = \sqrt{\sum_{i=1}^{n} (A_i - B_i)^2}$$

值域：[0, +∞)，0表示完全重合。

```python
def euclidean_distance(a: np.ndarray, b: np.ndarray) -> float:
    return np.sqrt(np.sum((a - b) ** 2))

def euclidean_to_similarity(distance: float) -> float:
    """将欧氏距离转换为相似度（0-1，越大越相似）"""
    return 1.0 / (1.0 + distance)
```

**适用场景**：图像特征匹配、位置编码 —— 当向量范数本身包含有意义的信息时。

**内积/点积（Inner Product / Dot Product）**

$$Dot(A, B) = \sum_{i=1}^{n} A_i B_i$$

值域：(-∞, +∞)，绝对值越大表示越相关。

```python
def dot_product(a: np.ndarray, b: np.ndarray) -> float:
    return np.dot(a, b)
```

**适用场景**：推荐系统 —— 用户向量和物品向量的点积代表偏好强度。对于已归一化的向量，点积等价于余弦相似度。

**曼哈顿距离（Manhattan Distance / L1）**

$$L1(A, B) = \sum_{i=1}^{n} |A_i - B_i|$$

```python
def manhattan_distance(a: np.ndarray, b: np.ndarray) -> float:
    return np.sum(np.abs(a - b))
```

**适用场景**：高维稀疏向量 —— 当向量大部分维度为零时（如传统TF-IDF向量），L1距离比L2更稳定。

### 4.2 度量选择指南

| 场景 | 推荐度量 | 原因 |
|------|---------|------|
| 文档/文本检索 | 余弦相似度 | 消除文档长度偏差 |
| 图像特征匹配 | 欧氏距离 | 特征范数承载有意义的信息 |
| 推荐系统 | 内积 | 直接建模偏好强度 |
| 稀疏向量(BM25等) | 曼哈顿距离 | 对稀疏数据更鲁棒 |
| 已归一化的Embedding | 内积(=余弦) | 性能最优(无需除法) |

### 4.3 实战：多种度量的性能对比

```python
import numpy as np
import time

def benchmark_similarity_metrics(dim=768, n_vectors=10000, n_queries=100):
    """对比不同相似度度量在大批量计算时的性能"""
    # 生成模拟数据
    np.random.seed(42)
    database = np.random.randn(n_vectors, dim).astype('float32')
    queries = np.random.randn(n_queries, dim).astype('float32')
    
    # 余弦相似度（先归一化）
    start = time.time()
    db_norm = database / np.linalg.norm(database, axis=1, keepdims=True)
    q_norm = queries / np.linalg.norm(queries, axis=1, keepdims=True)
    cosine_results = np.dot(q_norm, db_norm.T)
    cosine_time = (time.time() - start) * 1000
    
    # 欧氏距离
    start = time.time()
    # 利用公式 |a-b|^2 = |a|^2 + |b|^2 - 2ab 加速计算
    a_sq = np.sum(queries**2, axis=1, keepdims=True)
    b_sq = np.sum(database**2, axis=1)
    ab = np.dot(queries, database.T)
    euclidean_results = np.sqrt(np.abs(a_sq + b_sq - 2 * ab))
    euclidean_time = (time.time() - start) * 1000
    
    # 内积
    start = time.time()
    dot_results = np.dot(queries, database.T)
    dot_time = (time.time() - start) * 1000
    
    print(f"维度: {dim}, 数据库向量数: {n_vectors:,}, 查询数: {n_queries}")
    print(f"{'度量':<16} {'耗时(ms)':<12} {'Top-1均值':<12} {'Top-10召回率':<12}")
    print("-" * 56)
    print(f"{'余弦相似度':<16} {cosine_time:<12.2f} {'--':<12} {'--':<12}")
    print(f"{'欧氏距离':<16} {euclidean_time:<12.2f} {'--':<12} {'--':<12}")
    print(f"{'内积':<16} {dot_time:<12.2f} {'--':<12} {'--':<12}")
    
    # 结论：内积最快（无需归一化），预先归一化后余弦=内积
    return cosine_results, euclidean_results, dot_results
```

## 五、索引算法深度解析

### 5.1 Flat（暴力搜索）

最原始的方法：遍历所有向量，计算距离，排序。

- **复杂度**：O(N * D)（N=向量数, D=维度）
- **精度**：100%（精确搜索）
- **适用**：N < 10万的小规模场景

### 5.2 IVF（倒排文件索引）

将向量空间划分为多个区域（Voronoi Cell），搜索时只搜索最相关的几个区域。

```
工作原理：
1. 训练阶段：用K-Means将所有向量聚类为 nlist 个中心
2. 索引阶段：每个向量分配给最近的中心
3. 搜索阶段：只搜索最近的 nprobe 个聚类

精度-速度权衡：
nprobe ↑ → 精度 ↑, 速度 ↓
nprobe ↓ → 精度 ↓, 速度 ↑
```

```python
# FAISS IVF索引示例
import faiss

dim = 768
nlist = 100  # 聚类中心数

# 量化器（用于分配向量到聚类）
quantizer = faiss.IndexFlatL2(dim)

# IVF索引
index = faiss.IndexIVFFlat(quantizer, dim, nlist, faiss.METRIC_L2)

# 训练（需要训练数据学习聚类中心）
train_vectors = np.random.random((10000, dim)).astype('float32')
index.train(train_vectors)

# 添加向量
index.add(database_vectors)

# 搜索时设置 nprobe
index.nprobe = 10  # 搜索10个最近的聚类
distances, indices = index.search(query_vectors, k=5)
```

### 5.3 HNSW（分层可导航小世界图）

目前最流行的近似最近邻算法，基于图结构实现高效检索。

```
HNSW图结构:
Layer 2:  ●────●      (顶层: 跳跃层，长距离连接)
          │
Layer 1:  ●──●──●──●   (中层: 中间连接)
          │  │  │  │
Layer 0:  ●──●──●──●──●──●  (底层: 全部节点，稠密连接)
```

**关键参数**：

| 参数 | 含义 | 增大效果 | 减小效果 |
|------|------|---------|---------|
| M | 每个节点的最大连接数 | 精度↑, 内存↑, 构建速度↓ | 精度↓, 内存↓, 构建速度↑ |
| ef_construction | 构建时搜索宽度 | 精度↑, 构建时间↑ | 精度↓, 构建时间↓ |
| ef_search | 搜索时搜索宽度 | 精度↑, 搜索时间↑ | 精度↓, 搜索时间↓ |

```python
# FAISS HNSW索引示例
index_hnsw = faiss.IndexHNSWFlat(dim, M=32)  # M越大精度越高
index_hnsw.hnsw.efConstruction = 200  # 构建时的搜索宽度
index_hnsw.hnsw.efSearch = 64         # 查询时的搜索宽度
index_hnsw.add(database_vectors)

# 搜索
distances, indices = index_hnsw.search(query_vectors, k=5)
```

### 5.4 PQ（乘积量化）

将高维向量压缩为短编码，大幅减少内存占用。

```
原始向量(768维 float32 = 3KB) 
  → 切分为 m=8 个子向量(每段96维)
  → 每段用256个聚类中心之一近似
  → 最终编码 = 8字节(每个子段1字节索引)
  
压缩比: 3KB → 8字节 = 384:1
```

```python
# FAISS PQ索引（内存优化）
m = 8       # 子向量数
nbits = 8   # 每个子向量的编码位数(8位=256个中心)

index_pq = faiss.IndexPQ(dim, m, nbits)
index_pq.train(train_vectors)  # 需要训练
index_pq.add(database_vectors)

# 内存对比
import sys
print(f"原始向量内存: {database_vectors.nbytes / 1024 / 1024:.1f} MB")
# flat索引 = 原始内存
# PQ索引 ≈ 原始内存 * (m * nbits / 8) / (dim * 4) 
```

### 5.5 索引算法对比总结

| 算法 | 检索精度 | 检索速度 | 内存效率 | 需训练 | 最佳规模 |
|------|---------|---------|---------|--------|---------|
| Flat | 100% | 慢 | 低 | 否 | <10万 |
| IVF | ~95% | 快 | 中 | 是 | 10万-1亿 |
| HNSW | ~98% | 很快 | 中 | 否 | 1万-1亿 |
| PQ | ~90% | 中 | 极高 | 是 | >1亿 |
| IVF+PQ | ~92% | 较快 | 很高 | 是 | >1亿 |

## 七、向量数据库性能优化清单

### 7.1 写入优化

- **批量写入**：单条写入是性能杀手，始终使用批量API（batch size建议100-500）
- **预计算Embedding**：不要依赖数据库内嵌的embedding模块，离线批量计算好再写入
- **索引延迟构建**：大数据量导入时，先导入数据再统一构建索引
- **连接池复用**：使用连接池避免频繁建立和销毁连接

### 7.2 查询优化

- **合理设置Top-K**：检索Top-K不是越大越好，过大引入噪声、增加延迟
- **利用元数据过滤**：将检索空间限制在相关子集内，显著提升速度和精度
- **预热索引**：Milvus/Qdrant需要load到内存后才能查询
- **选择合适的一致性级别**：非关键场景使用最终一致性换取延迟

### 7.3 成本优化

- **向量量化**：使用PQ/SQ减少存储和内存成本（精度损失通常<3%）
- **分层存储**：热数据在内存，冷数据在磁盘
- **命名空间规划**：合理使用命名空间实现多租户隔离，而非创建多个Collection

```python
# 批量写入最佳实践
def optimized_batch_insert(collection, documents, embedder, batch_size=200):
    """
    优化的批量插入：预计算Embedding + 分批写入
    """
    total = len(documents)
    
    for start in range(0, total, batch_size):
        end = min(start + batch_size, total)
        batch_docs = documents[start:end]
        
        # 预计算embedding（避免数据库内嵌计算）
        batch_texts = [doc.page_content for doc in batch_docs]
        batch_embeddings = embedder.embed_documents(batch_texts)
        
        # 准备数据
        ids = [f"doc_{i}" for i in range(start, end)]
        metadatas = [doc.metadata for doc in batch_docs]
        
        # 批量写入
        collection.add(
            ids=ids,
            embeddings=batch_embeddings,
            documents=batch_texts,
            metadatas=metadatas,
        )
        
        print(f"进度: {end}/{total} ({100*end//total}%)")
```

## 八、总结

向量数据库选型没有one-size-fits-all的答案。核心决策逻辑是：

1. **先明确需求**：数据规模、QPS要求、延迟预算、团队运维能力、预算
2. **从简到繁**：先用Chroma验证方案可行性，再根据实际瓶颈迁移到更适合的方案
3. **关注生态**：选择有良好LangChain/LlamaIndex集成的方案，降低开发成本
4. **预留迁移空间**：向量数据库的抽象层设计（如使用LangChain的VectorStore接口），便于未来切换

最终的建议是：**原型用Chroma，生产小规模用Pinecone，大规模自建用Milvus，需要混合搜索用Weaviate，追求极致可控用Qdrant。**
