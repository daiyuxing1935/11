# 规划-动态重规划

## 1. 概述：什么是动态重规划 (Dynamic Re-planning)

动态重规划（Dynamic Re-planning）是指智能体在执行任务过程中，根据**执行反馈（Execution Feedback）** 实时调整子目标（Sub-goal）的机制。与传统的"一次性规划、全程执行"不同，动态重规划允许智能体在遇到意外情况时，重新评估当前状态，修改或替换原有计划中的部分步骤，从而保证最终目标的达成。

核心思想：**Plan is a living document, not a static blueprint.**（计划是一份活的文件，而非静态蓝图。）

### 1.1 为什么需要动态重规划

在真实世界中，Agent 面临着高度不确定的环境：

- **环境变化（Environment Shift）**：外部状态与规划时的假设不一致。
- **资源不可用（Resource Unavailability）**：所需的 API、工具、数据源暂时或永久不可用。
- **执行失败（Execution Failure）**：某个子步骤执行失败但整体目标仍需达成。
- **新信息涌现（Emergent Information）**：在计划执行过程中获得了新的关键信息，原有计划不再最优。

```
+-------------------+       +-------------------+       +-------------------+
|   Initial Plan    | ----> |  Execute Step N   | ----> |  Monitor Feedback |
|  (初始计划生成)    |       |  (执行第 N 步)     |       |  (监控反馈)        |
+-------------------+       +-------------------+       +---------+---------+
                                                                  |
                                          +-----------------------+
                                          v
                              +-------------------+       +-------------------+
                              |  Deviation Detected| ----> |  Re-plan Sub-goals|
                              |  (检测到偏离)       |       |  (重规划子目标)    |
                              +-------------------+       +---------+---------+
                                                                  |
                                                                  v
                                                      +-------------------+
                                                      |  Resume Execution |
                                                      |  (恢复执行)        |
                                                      +-------------------+
```

### 1.2 动态重规划的核心挑战

| 挑战 (Challenge) | 描述 (Description) | 影响 (Impact) |
|---|---|---|
| **时机选择 (When to Re-plan)** | 何时触发重规划？过早导致过度反应，过晚导致资源浪费 | 效率与准确性的权衡 |
| **范围确定 (Scope of Re-plan)** | 是全盘重来 (Global) 还是局部修补 (Local)？ | 时间成本与质量的权衡 |
| **状态保持 (State Preservation)** | 已完成子目标的结果如何保存不被破坏 | 数据完整性与幂等性 |
| **一致性保证 (Consistency)** | 新计划与已完成部分之间是否存在逻辑冲突 | 计划可行性的根基 |
| **循环检测 (Loop Detection)** | 避免陷入反复规划-失败的无限循环 | 系统稳定性的保障 |

## 2. 执行监控：如何检测执行偏离 (Execution Monitoring)

执行监控（Execution Monitoring）是动态重规划的前置条件——必须首先感知到偏离，才能决定是否以及如何做出调整。

### 2.1 偏离检测机制 (Deviation Detection Mechanisms)

#### 2.1.1 预期-实际对比 (Expected vs. Actual)

最基础的监控方式：将执行结果与预期结果进行对比。

```python
# 预期-实际对比示意
expected_outcome = {
    "status": "success",
    "data": {"flight_number": "CA1234", "status": "confirmed"}
}
actual_outcome = {
    "status": "error",
    "data": {"message": "Flight CA1234 has been cancelled"}
}
# 对比结果：偏离检测 -> 触发 Re-planning
```

#### 2.1.2 异常检测维度

| 维度 (Dimension) | 检测内容 | 示例 |
|---|---|---|
| **状态偏离 (State Deviation)** | 执行后的世界状态与预期不一致 | 预计订到的航班被取消 |
| **资源偏离 (Resource Deviation)** | 所需资源不可用或性能下降 | API 超时、数据库不可达 |
| **约束违反 (Constraint Violation)** | 执行结果违反了预设约束 | 总预算超支、时间超限 |
| **置信度下降 (Confidence Drop)** | Agent 对当前计划的信心显著降低 | LLM 输出的不确定性升高 |
| **进度异常 (Progress Anomaly)** | 执行进度与预期时间线不符 | 某步骤耗时超过预期的 3 倍 |

### 2.2 监控架构 (Monitoring Architecture)

```
+------------------------------------------------------------------+
|                     Execution Monitor (执行监控器)                  |
|                                                                  |
|  +------------------+  +------------------+  +------------------+ |
|  | State Validator  |  | Resource Monitor |  | Constraint       | |
|  | (状态校验器)      |  | (资源监控器)      |  | Checker          | |
|  |                  |  |                  |  | (约束检查器)      | |
|  +--------+---------+  +--------+---------+  +--------+---------+ |
|           |                     |                     |           |
|           +---------------------+---------------------+           |
|                                 |                                 |
|                       +---------v---------+                       |
|                       | Anomaly Aggregator|                       |
|                       | (异常聚合器)       |                       |
|                       +---------+---------+                       |
|                                 |                                 |
|                       +---------v---------+                       |
|                       | Re-plan Trigger   |                       |
|                       | (重规划触发器)      |                       |
|                       +-------------------+                       |
+------------------------------------------------------------------+
```

### 2.3 偏离等级分类 (Deviation Severity Levels)

| 等级 | 名称 | 描述 | 响应策略 |
|---|---|---|---|
| **L0** | 无偏离 (None) | 执行结果完全符合预期 | 继续执行下一步 |
| **L1** | 轻微偏离 (Minor) | 结果与预期有小差异，但不影响后续步骤 | 记录日志，继续执行 |
| **L2** | 局部影响 (Local) | 当前步骤失败，但后续步骤可独立进行 | 触发 Local Re-planning |
| **L3** | 连锁影响 (Cascade) | 当前失败影响了多个后续步骤的可行性 | 触发 Global Re-planning |
| **L4** | 目标不可达 (Unreachable) | 无论如何调整都无法达成目标 | 向用户报告，请求人工干预 |

## 3. 本地重规划 vs 全局重规划 (Local vs Global Re-planning)

### 3.1 概念对比

```
Global Re-planning (全局重规划):
  丢弃所有未执行步骤，从当前状态重新生成完整计划。

  [Step1: Done] [Step2: Done] [Step3: Failed] [Step4: Pending] [Step5: Pending]
       |              |              |               |               |
       v              v              v               v               v
  [Step1: Done] [Step2: Done] [Step3': New] [Step4': New] [Step5': New]
                                 ^~~~~~~~~~~~~^~~~~~~~~~~~~^~~~~~~~~~~~~^
                                           全部重新规划


Local Re-planning (本地重规划):
  仅替换失败的步骤，保持其他未执行步骤不变（或最小修改）。

  [Step1: Done] [Step2: Done] [Step3: Failed] [Step4: Pending] [Step5: Pending]
       |              |              |               |               |
       v              v              v               v               v
  [Step1: Done] [Step2: Done] [Step3': New] [Step4: Keep] [Step5: Keep]
                                 ^~~~~~~~~~~~^
                                   仅替换失败步骤
```

### 3.2 适用场景对比

| 维度 | Local Re-planning (本地) | Global Re-planning (全局) |
|---|---|---|
| **适用场景** | 单步失败，其他步骤独立性强 | 多步失败，步骤间耦合度高 |
| **典型例子** | 换一个航班号，行程其他不变 | 航班取消导致酒店、接机全需调整 |
| **计算成本 (Cost)** | 低，LLM 调用次数少 | 高，需完整重新推理 |
| **时间延迟 (Latency)** | 快，毫秒到秒级 | 慢，可能需要数秒到数十秒 |
| **结果质量 (Quality)** | 局部最优，可能非全局最优 | 全局最优（假如推理充分） |
| **状态保留** | 大部分状态保留 | 需要迁移/重建状态 |
| **一致性风险** | 旧步骤与新步骤可能不一致 | 一致性由全新规划保证 |
| **调用次数 (LLM Calls)** | 1-2 次 | 多次（重新分解 + 重新推理） |

### 3.3 决策流程 (Decision Flow)

```
执行步骤 N 失败
      |
      v
+-----------+    是     +-------------------+
| L2 以下？  | -------> | Local Re-planning |
+-----------+          | (本地重规划)        |
      | 否             +-------------------+
      v
+-----------+    否     +-------------------+
| 依赖图     | -------> | Global Re-planning|
| 中度耦合？  |          | (全局重规划)        |
+-----------+          +-------------------+
      | 是
      v
+-----------+    是     +-------------------+
| 已完成步骤 | -------> | Hybrid Re-planning|
| 的成果可复用？|        | (混合重规划)        |
+-----------+          | 保留已完成 + 全局重排|
      | 否             +-------------------+
      v
+-------------------+
| Ask Human         |
| (请求人工决策)      |
+-------------------+
```

### 3.4 成本量化对比 (Cost Comparison)

假设一个旅游订票 Agent 执行 5 步计划，LLM 每 1K token 成本 $0.01：

| 策略 | Token 消耗 | 延迟 (s) | 成功率 |
|---|---|---|---|
| **无重规划 (No Re-plan)** | ~2K (仅一次规划) | 2 | 仅理想环境 |
| **Local Re-planning（替换 1 步）** | ~3K (原规划 + 局部修复) | 3 | 85% |
| **Global Re-planning（全盘重来）** | ~8K (原规划 + 完整重规划) | 8 | 95% |
| **Hybrid（保留 3 步已完，重排剩余）** | ~5K | 5 | 92% |

## 4. 回滚策略：如何保留已完成子目标的状态 (Rollback Strategy)

### 4.1 核心问题

当触发 Global Re-planning 时，已完成步骤的结果不能简单地丢弃——它们可能包含了重要的**副作用（Side Effects）**，如已预订的酒店、已支付的费用等。回滚策略需要回答：**已完成的工作中，哪些需要保留？哪些需要撤销？哪些需要修改？**

### 4.2 状态分类 (State Classification)

```
                    已完成步骤 (Completed Steps)
                              |
            +-----------------+-----------------+
            |                 |                 |
            v                 v                 v
   +-----------+      +-----------+      +-----------+
   | Immutable |      | Mutable   |      | Revertible|
   | (不可变)   |      | (可变)     |      | (可撤销)   |
   +-----------+      +-----------+      +-----------+
         |                 |                 |
    示例：             示例：             示例：
    - 已完成的支付     - 已选但未支付的    - 已添加的购物车
    - 已发送的邮件      酒店选项           - 临时创建的 Session
    - 已确认的订单     - 计划中的时间槽位  - 内存中的变量
         |                 |                 |
         v                 v                 v
      保留               尽量保留           可以丢弃
      (Preserve)        (Preserve if       (Discardable)
                         Compatible)
```

### 4.3 回滚策略表

| 策略名称 | 描述 | 适用场景 | 复杂度 | 风险 |
|---|---|---|---|---|
| **快照恢复 (Snapshot Restore)** | 在执行前创建完整状态快照，失败后整体回滚 | 短周期、无外部副作用的任务 | 低 | 丢失所有进度 |
| **增量提交 (Incremental Commit)** | 每步执行后独立保存中间结果，新计划可以复用 | 步骤独立性高的任务 | 中 | 步骤间状态可能冲突 |
| **补偿事务 (Compensating Transaction)** | 为每个步骤定义一个"撤销操作"，失败后逆序执行 | 有外部副作用的任务（如支付） | 高 | 补偿本身也可能失败 |
| **Saga 模式** | 长事务拆分为多个本地事务，每个有对应的补偿操作 | 分布式、多服务的复杂任务 | 很高 | 实现复杂，需编排引擎 |
| **乐观保留 (Optimistic Preserve)** | 尽量保留已完成步骤，仅重规划兼容部分 | AI Agent 常见场景 | 中 | 可能存在隐式冲突 |

### 4.4 状态保留的校验机制

重规划后，新计划必须与已保留的已完成步骤进行**兼容性校验（Compatibility Check）**：

```
+-------------------+        +-------------------+        +-------------------+
| Preserved State   |        | New Plan          |        | Compatibility     |
| (已保留状态)       | -----> | (新计划)           | -----> | Check (兼容性检查)  |
| [Step1: booked]   |        | [Step3': ...]     |        |                   |
| [Step2: paid]     |        | [Step4': ...]     |        | - 逻辑依赖检查     |
+-------------------+        +-------------------+        | - 约束冲突检查     |
                                                         | - 时间线一致性     |
                                                         +--------+----------+
                                                                  |
                                                  +---------------+
                                                  v
                                        +-------------------+
                                        | Pass? (通过?)      |
                                        +---+-------+-------+
                                            |       |
                                           是       否
                                            |       |
                                            v       v
                                    继续执行    标记冲突，
                                               请求解决
```

## 5. 实战场景演练：航班取消后的动态重规划

### 5.1 场景设定 (Scenario)

用户请求 Agent："帮我安排一趟从北京到上海的出差：订 7 月 10 日的机票、酒店、以及接机服务。"

**初始计划 (Initial Plan)：**

```
Step 1: Search flights  (PEK -> SHA, 7月10日)
Step 2: Book flight CA1234 (08:00-10:30)
Step 3: Search hotels near Pudong
Step 4: Book hotel JinJiang (Check-in: 7月10日 12:00)
Step 5: Book airport pickup (Arrival: 10:30, to Hotel)
Step 6: Generate itinerary summary
```

### 5.2 意外事件：航班取消

执行到 Step 2 时，系统返回：**Flight CA1234 has been cancelled due to weather.**

执行监控器检测到 **L2 级别偏离** — 当前步骤失败但后续步骤尚未执行。

### 5.3 动态重规划过程

#### 阶段 1：状态快照

```
State Snapshot at Step 2 Failure:
+------------------------------+
| Completed Steps:             |
|   [Step 1: Done]              |
|     - flights_found: [...]   |
|     - search_completed: True |
|                              |
| Failed Step:                 |
|   [Step 2: FAILED]           |
|     - error: CANCELLED       |
|     - flight: CA1234         |
|                              |
| Pending Steps:               |
|   [Step 3, 4, 5, 6]          |
+------------------------------+
```

#### 阶段 2：影响分析 (Impact Analysis)

```
Flight CA1234 Cancelled
      |
      v
+----------------------------+
| Downstream Dependencies:   |
|                            |
| Step 4 (Hotel):            |
|   - 入住时间原定为12:00     |
|   - 依赖到达时间 10:30      |
|   - 影响：MEDIUM           |
|                            |
| Step 5 (Pickup):           |
|   - 接机时间 10:30          |
|   - 航班号 CA1234          |
|   - 影响：HIGH             |
|                            |
| Step 6 (Summary):          |
|   - 汇总所有信息            |
|   - 影响：LOW（仅文本输出）  |
+----------------------------+
```

#### 阶段 3：策略选择

由于航班取消影响了多个后续步骤（Pickup 强依赖，Hotel 中等依赖），且已完成步骤仅 Step 1 的结果可复用，Agent 决定采用 **Hybrid Re-planning**：

- **保留** Step 1 的搜索结果（其他可选航班列表）
- **重规划** Step 2-6

#### 阶段 4：执行重规划

```
Re-planned Steps:
Step 2': Book flight MU5102 (09:30-12:00)  [替代航班]
Step 3': Search hotels near Pudong           [保留，但时间段调整]
Step 4': Book hotel JinJiang (Check-in: 7月10日 14:00) [时间推迟]
Step 5': Book airport pickup (Arrival: 12:00, to Hotel) [更新到达时间]
Step 6': Generate itinerary summary           [保留]
```

### 5.4 完整的执行流程图

```
时间线 ────────────────────────────────────────────────────────────>

  T0           T1             T2              T3            T4
  |            |              |               |             |
  v            v              v               v             v
+------+   +------+      +---------+    +---------+    +---------+
|Plan  |-->|Exec  |----->|Monitor  |--->|Re-plan |--->|Resume   |
|Gen   |   |Step1 |      |Step2    |    |Hybrid  |    |Exec     |
|      |   |OK    |      |FAILED!  |    |        |    |Step2'-6'|
+------+   +------+      +----+----+    +----+----+    +---------+
                               |              |
                               | Detected     | New plan:
                               | L2+L3        | MU5102 flight
                               | deviation    | adjusted times
                               v              v
                          Flight CA1234   Preserved Step1
                          CANCELLED       results reused
```

## 6. 完整 Python 实现：可运行的 Re-planner

以下是一个完整的、可运行的 Python 实现，模拟了一个带有执行监控和动态重规划能力的 Agent。

```python
"""
动态重规划系统 (Dynamic Re-planning System)
============================================
一个完整的可运行示例，演示 Agent 如何在执行过程中检测偏离、
选择重规划策略、保留已完成状态，并进行局部或全局重规划。

运行方式: python replanner.py
"""

from __future__ import annotations
from dataclasses import dataclass, field
from enum import Enum, auto
from typing import Any, Callable, Dict, List, Optional, Tuple
import copy
import random
import time


# ============================================================================
# 第一部分：基础数据结构 (Basic Data Structures)
# ============================================================================

class StepStatus(Enum):
    """步骤执行状态"""
    PENDING   = "PENDING"     # 待执行
    RUNNING   = "RUNNING"     # 执行中
    SUCCEEDED = "SUCCEEDED"   # 执行成功
    FAILED    = "FAILED"      # 执行失败
    SKIPPED   = "SKIPPED"     # 被跳过（重规划时被替换）


class DeviationLevel(Enum):
    """偏离严重程度等级"""
    NONE   = 0   # 无偏离
    MINOR  = 1   # 轻微偏离 — 不影响后续
    LOCAL  = 2   # 局部影响 — 触发 Local Re-planning
    CASCADE = 3  # 连锁影响 — 触发 Global Re-planning
    FATAL  = 4   # 目标不可达 — 需人工介入


class ReplanStrategy(Enum):
    """重规划策略类型"""
    NONE   = auto()   # 不做重规划
    LOCAL  = auto()   # 本地重规划：仅替换失败步骤
    GLOBAL = auto()   # 全局重规划：从当前状态重新生成完整计划
    HYBRID = auto()   # 混合重规划：保留已完成的，重排未执行的


@dataclass
class SubGoal:
    """
    子目标 / 计划步骤

    Attributes:
        step_id: 步骤唯一标识符
        description: 步骤的人类可读描述
        action: 执行该步骤的可调用函数
        dependencies: 当前步骤依赖的其他 step_id 列表
        status: 当前步骤的执行状态
        result: 执行结果（成功时填充）
        error: 错误信息（失败时填充）
        data: 步骤执行过程中产生的任意数据
        is_immutable: True 表示该步骤的结果不可撤销（如已支付的订单）
    """
    step_id: str
    description: str
    action: Optional[Callable[[Dict[str, Any]], Dict[str, Any]]] = None
    dependencies: List[str] = field(default_factory=list)
    status: StepStatus = StepStatus.PENDING
    result: Optional[Dict[str, Any]] = None
    error: Optional[str] = None
    data: Dict[str, Any] = field(default_factory=dict)
    is_immutable: bool = False  # 不可变步骤标记


@dataclass
class ExecutionContext:
    """
    执行上下文：保存全局状态

    Attributes:
        original_goal: 用户的原始目标描述
        world_state: 模拟的"世界状态"，在执行过程中会变化
        step_history: 所有步骤的历史记录（用于审计和回滚）
        max_replan_attempts: 最大重规划尝试次数，防止无限循环
        replan_count: 当前已执行的重规划次数
        preserved_state: 重规划时需要保留的已完成步骤状态快照
    """
    original_goal: str
    world_state: Dict[str, Any] = field(default_factory=dict)
    step_history: List[Dict[str, Any]] = field(default_factory=list)
    max_replan_attempts: int = 3
    replan_count: int = 0
    preserved_state: Dict[str, Any] = field(default_factory=dict)


# ============================================================================
# 第二部分：执行监控器 (Execution Monitor)
# ============================================================================

class ExecutionMonitor:
    """
    执行监控器：负责检测执行偏离并评估严重等级。

    工作原理：
    1. 在每一步执行后，对比预期结果和实际结果 (Expected vs. Actual)
    2. 调用一组 Checker 来检测不同维度的异常
    3. 汇总所有异常，输出综合偏离等级
    4. 如果偏离等级 >= LOCAL，触发重规划建议
    """

    def __init__(self):
        # 注册的异常检测器列表，每个检测器是一个 (名称, 检测函数) 对
        self.checkers: List[Tuple[str, Callable[[Dict[str, Any], Dict[str, Any]], Optional[DeviationLevel]]]] = []

    def register_checker(
        self,
        name: str,
        checker_fn: Callable[[Dict[str, Any], Dict[str, Any]], Optional[DeviationLevel]]
    ) -> None:
        """
        注册一个异常检测器。

        Args:
            name: 检测器名称
            checker_fn: 检测函数，接收 (expected, actual) 两个字典，
                        返回 DeviationLevel 或 None（无偏离）
        """
        self.checkers.append((name, checker_fn))

    def evaluate(
        self,
        expected: Dict[str, Any],
        actual: Dict[str, Any]
    ) -> Tuple[DeviationLevel, List[str]]:
        """
        评估执行偏离等级。

        Args:
            expected: 预期的执行结果
            actual: 实际的执行结果

        Returns:
            (最高偏离等级, 检测到的异常描述列表)
        """
        max_level = DeviationLevel.NONE
        anomalies: List[str] = []

        for checker_name, checker_fn in self.checkers:
            level = checker_fn(expected, actual)
            if level is not None and level.value > DeviationLevel.NONE.value:
                anomalies.append(f"[{checker_name}] Deviation L{level.value}: "
                                 f"expected={expected.get('status')}, "
                                 f"actual={actual.get('status')}")
                if level.value > max_level.value:
                    max_level = level

        return max_level, anomalies


# 预定义的检测器函数
def status_checker(expected: Dict[str, Any], actual: Dict[str, Any]) -> Optional[DeviationLevel]:
    """
    状态偏离检测：对比执行状态。

    检测逻辑：
    - status 一致 -> 无偏离
    - status 为 "error" 但数据可容错 -> L1 (Minor)
    - status 为 "error" 且数据不可用 -> L2 (Local)
    - status 为 "fatal_error" -> L3 (Cascade)
    - 目标明确不可达 -> L4 (Fatal)
    """
    exp_status = expected.get("status", "success")
    act_status = actual.get("status", "success")

    if exp_status == act_status:
        return DeviationLevel.NONE
    if act_status == "fatal_error":
        return DeviationLevel.FATAL
    if act_status == "error" and actual.get("recoverable", True):
        return DeviationLevel.LOCAL
    if act_status == "error":
        return DeviationLevel.CASCADE
    return DeviationLevel.MINOR


def resource_checker(expected: Dict[str, Any], actual: Dict[str, Any]) -> Optional[DeviationLevel]:
    """
    资源偏离检测：检测所需资源是否可用。

    检测逻辑：
    - 如果实际结果中包含 "resource_unavailable" 标记 -> L2
    - 如果包含 "timeout" -> L1 (可能临时故障)
    """
    if actual.get("resource_unavailable"):
        return DeviationLevel.LOCAL
    if actual.get("timeout"):
        return DeviationLevel.MINOR
    return None


def constraint_checker(expected: Dict[str, Any], actual: Dict[str, Any]) -> Optional[DeviationLevel]:
    """
    约束违反检测：检测执行结果是否违反了预设约束。

    检测逻辑：
    - 预算超支 > 20% -> L3
    - 时间超限 > 30% -> L2
    - 预算/时间轻微超限 -> L1
    """
    budget_overrun = actual.get("budget_overrun_percent", 0)
    time_overrun = actual.get("time_overrun_percent", 0)

    if budget_overrun > 20 or time_overrun > 50:
        return DeviationLevel.CASCADE
    if budget_overrun > 10 or time_overrun > 30:
        return DeviationLevel.LOCAL
    if budget_overrun > 0 or time_overrun > 0:
        return DeviationLevel.MINOR
    return None


# ============================================================================
# 第三部分：规划器 (Planner) 与 重规划器 (Re-planner)
# ============================================================================

class Replanner:
    """
    重规划器：核心组件，负责根据执行反馈动态调整计划。

    职责：
    1. 接收来自 ExecutionMonitor 的偏离检测结果
    2. 决定采用哪种重规划策略 (Local / Global / Hybrid)
    3. 保留已完成步骤中不可变或可复用的部分
    4. 生成新的执行计划
    5. 校验新计划与已保留状态的一致性
    """

    def __init__(self, monitor: ExecutionMonitor):
        self.monitor = monitor

    def decide_strategy(
        self,
        deviation_level: DeviationLevel,
        failed_step: SubGoal,
        all_steps: List[SubGoal],
        context: ExecutionContext
    ) -> ReplanStrategy:
        """
        根据偏离等级、失败步骤、当前计划状态决定重规划策略。

        决策规则：
        - L0: NONE — 无偏离，继续执行
        - L1: NONE — 轻微偏离，记录但不重规划
        - L2: LOCAL — 仅当前步骤失败，依赖关系单一
        - L3: GLOBAL 或 HYBRID — 多步受影响
        - L4: 中止并请求人工介入

        Args:
            deviation_level: 检测到的偏离等级
            failed_step: 失败的步骤
            all_steps: 所有步骤（包括已完成和未执行的）
            context: 执行上下文

        Returns:
            推荐的重规划策略
        """
        if deviation_level == DeviationLevel.NONE:
            return ReplanStrategy.NONE
        if deviation_level == DeviationLevel.MINOR:
            return ReplanStrategy.NONE  # 轻微偏离不触发重规划
        if deviation_level == DeviationLevel.FATAL:
            # L4 无法自动恢复，输出所有已完成结果供用户参考
            print(f"\n[FATAL] 目标不可达: {context.original_goal}")
            print(f"[FATAL] 失败步骤: {failed_step.step_id} - {failed_step.error}")
            print(f"[FATAL] 已完成步骤: {[s.step_id for s in all_steps if s.status == StepStatus.SUCCEEDED]}")
            return ReplanStrategy.NONE

        # L2: 检查是否有多个步骤依赖此失败步骤
        dependent_steps = [
            s for s in all_steps
            if failed_step.step_id in s.dependencies
            and s.status in (StepStatus.PENDING, StepStatus.RUNNING)
        ]

        if deviation_level == DeviationLevel.LOCAL and len(dependent_steps) == 0:
            # L2 且无下游依赖 — 本地重规划即可
            return ReplanStrategy.LOCAL

        if deviation_level == DeviationLevel.CASCADE or len(dependent_steps) > 1:
            # L3 或有多个下游依赖 — 检查已完成步骤的复用情况
            succeeded_steps = [s for s in all_steps if s.status == StepStatus.SUCCEEDED]
            if len(succeeded_steps) > 0:
                # 有已完成步骤，使用混合策略以保留成果
                return ReplanStrategy.HYBRID
            else:
                # 没有已完成步骤，全盘重来
                return ReplanStrategy.GLOBAL

        return ReplanStrategy.LOCAL

    def preserve_state(
        self,
        all_steps: List[SubGoal],
        context: ExecutionContext
    ) -> Dict[str, Any]:
        """
        提取并保留已完成步骤的状态快照。

        保留策略：
        - Immutable 步骤（如已支付订单）：必须保留
        - 非 Immutable 但已成功的步骤：尽量保留，标记为可协商
        - 失败步骤和未执行步骤：不保留

        Args:
            all_steps: 所有步骤
            context: 执行上下文

        Returns:
            保留的状态快照，key 为 step_id，value 为步骤结果
        """
        preserved: Dict[str, Any] = {}
        for step in all_steps:
            if step.status == StepStatus.SUCCEEDED:
                preserved[step.step_id] = {
                    "result": copy.deepcopy(step.result),
                    "data": copy.deepcopy(step.data),
                    "description": step.description,
                    "immutable": step.is_immutable,
                }
                if step.is_immutable:
                    print(f"  [PRESERVE-IMMUTABLE] {step.step_id}: {step.description}")
                else:
                    print(f"  [PRESERVE-MUTABLE]   {step.step_id}: {step.description}")
        return preserved

    def compatibility_check(
        self,
        new_plan: List[SubGoal],
        preserved_state: Dict[str, Any],
        context: ExecutionContext
    ) -> Tuple[bool, List[str]]:
        """
        校验新计划与已保留状态之间的兼容性。

        检查项：
        1. 依赖检查：新计划的步骤是否依赖于已被标记为"跳过"的旧步骤
        2. 约束一致性：新计划的时间/资源约束是否与已保留结果冲突
        3. Immutable 冲突：新计划是否试图修改不可变步骤的结果

        Args:
            new_plan: 新生成的计划
            preserved_state: 已保留的状态
            context: 执行上下文

        Returns:
            (是否通过, 冲突描述列表)
        """
        conflicts: List[str] = []

        # 1. 依赖检查
        preserved_ids = set(preserved_state.keys())
        for step in new_plan:
            for dep in step.dependencies:
                if dep not in preserved_ids and dep not in {s.step_id for s in new_plan}:
                    conflicts.append(
                        f"依赖缺失: {step.step_id} 依赖 {dep}，但 {dep} 不在保留状态"
                        f"和新计划中"
                    )

        # 2. Immutable 冲突检查
        for step in new_plan:
            for dep in step.dependencies:
                if dep in preserved_state and preserved_state[dep].get("immutable"):
                    # 如果有新步骤依赖一个不可变步骤 → 允许（只读依赖）
                    pass
            # 检查新步骤是否试图覆盖不可变步骤的 ID
            if step.step_id in preserved_state and preserved_state[step.step_id].get("immutable"):
                conflicts.append(
                    f"Immutable 冲突: {step.step_id} 是不可变步骤，不能被重规划覆盖"
                )

        # 3. 时间一致性检查（模拟）
        for step in new_plan:
            if "time_constraint" in step.data:
                tc = step.data["time_constraint"]
                for preserved_id, pstate in preserved_state.items():
                    if "time_constraint" in pstate.get("data", {}):
                        ptc = pstate["data"]["time_constraint"]
                        # 简化：如果新步骤的时间约束与保留步骤冲突
                        if tc.get("before") and ptc.get("after"):
                            if tc["before"] < ptc["after"]:
                                conflicts.append(
                                    f"时间冲突: {step.step_id} 需要在 {tc['before']} 之前，"
                                    f"但 {preserved_id} 在 {ptc['after']} 之后"
                                )

        return len(conflicts) == 0, conflicts

    def replan(
        self,
        deviation_level: DeviationLevel,
        failed_step: SubGoal,
        all_steps: List[SubGoal],
        context: ExecutionContext,
        plan_generator: Callable[[str, Dict[str, Any], List[SubGoal]], List[SubGoal]]
    ) -> Tuple[List[SubGoal], ReplanStrategy]:
        """
        执行重规划的主入口。

        完整流程：
        1. 决定策略
        2. 保留已完成状态
        3. 根据策略生成新计划
        4. 兼容性校验
        5. 返回最终计划

        Args:
            deviation_level: 偏离等级
            failed_step: 失败的步骤
            all_steps: 当前所有步骤
            context: 执行上下文
            plan_generator: 计划生成函数（模拟 LLM 规划能力）

        Returns:
            (新计划步骤列表, 采用的策略)
        """
        strategy = self.decide_strategy(deviation_level, failed_step, all_steps, context)
        print(f"\n{'='*60}")
        print(f"[RE-PLAN] 触发重规划! 策略: {strategy.name}")
        print(f"[RE-PLAN] 失败步骤: {failed_step.step_id} ({failed_step.error})")
        print(f"[RE-PLAN] 偏离等级: L{deviation_level.value}")
        print(f"{'='*60}")

        if strategy == ReplanStrategy.NONE:
            return all_steps, strategy

        # Step 1: 保留已完成步骤的状态
        preserved_state = self.preserve_state(all_steps, context)
        context.preserved_state = preserved_state
        print(f"\n  已保留 {len(preserved_state)} 个已完成步骤的状态")

        # Step 2: 根据策略生成新计划
        if strategy == ReplanStrategy.LOCAL:
            new_plan = self._local_replan(failed_step, all_steps, context, plan_generator)
        elif strategy == ReplanStrategy.GLOBAL:
            new_plan = self._global_replan(all_steps, context, plan_generator)
        elif strategy == ReplanStrategy.HYBRID:
            new_plan = self._hybrid_replan(failed_step, all_steps, context, plan_generator)
        else:
            new_plan = all_steps

        # Step 3: 兼容性校验
        passed, conflicts = self.compatibility_check(new_plan, preserved_state, context)
        if not passed:
            print(f"\n  [WARNING] 兼容性校验未通过，检测到 {len(conflicts)} 个冲突:")
            for conflict in conflicts:
                print(f"    - {conflict}")
            # 在真实系统中，这里会触发冲突解决流程或人工介入
            # 简单演示：将冲突步骤标记为需要人工确认
            for step in new_plan:
                if any(conflict for conflict in conflicts if step.step_id in conflict):
                    step.data["needs_human_approval"] = True
        else:
            print(f"\n  [OK] 兼容性校验通过")

        context.replan_count += 1
        return new_plan, strategy

    def _local_replan(
        self,
        failed_step: SubGoal,
        all_steps: List[SubGoal],
        context: ExecutionContext,
        plan_generator: Callable
    ) -> List[SubGoal]:
        """
        本地重规划：仅替换失败步骤。

        保持其他步骤不变，只为失败步骤生成替代方案。
        在真实系统中，plan_generator 会调用 LLM 来生成替代步骤。
        """
        print(f"\n  [LOCAL REPLAN] 仅重规划步骤: {failed_step.step_id}")
        # 标记失败步骤为 SKIPPED
        failed_step.status = StepStatus.SKIPPED
        # 生成替代步骤（模拟：基于失败信息和上下文重新规划）
        alternative_steps = plan_generator(
            f"替代方案 for {failed_step.description}",
            context.preserved_state,
            [failed_step]
        )
        # 替换：将替代步骤插入原位置
        new_plan = []
        for step in all_steps:
            if step.step_id == failed_step.step_id:
                # 用替代步骤替换
                for alt in alternative_steps:
                    alt.status = StepStatus.PENDING
                    # 继承原步骤的依赖关系
                    alt.dependencies = list(set(alt.dependencies + failed_step.dependencies))
                new_plan.extend(alternative_steps)
                print(f"    替换为: {[s.step_id for s in alternative_steps]}")
            else:
                new_plan.append(step)
        return new_plan

    def _global_replan(
        self,
        all_steps: List[SubGoal],
        context: ExecutionContext,
        plan_generator: Callable
    ) -> List[SubGoal]:
        """
        全局重规划：从当前状态重新生成完整计划。

        将所有未完成步骤标记为 SKIPPED，然后从零开始生成新计划。
        保留已完成步骤的引用，以便新计划可以复用它们的数据。
        """
        print(f"\n  [GLOBAL REPLAN] 从当前状态重新生成完整计划")
        # 标记所有未完成步骤
        for step in all_steps:
            if step.status in (StepStatus.PENDING, StepStatus.RUNNING):
                step.status = StepStatus.SKIPPED
        # 基于当前世界状态和已完成的结果，重新生成计划
        new_plan = plan_generator(
            context.original_goal,
            context.preserved_state,
            all_steps  # 传入历史供参考
        )
        for step in new_plan:
            step.status = StepStatus.PENDING
        return new_plan

    def _hybrid_replan(
        self,
        failed_step: SubGoal,
        all_steps: List[SubGoal],
        context: ExecutionContext,
        plan_generator: Callable
    ) -> List[SubGoal]:
        """
        混合重规划：保留已完成步骤，重排所有未执行步骤。

        这是最实用的策略：
        - 保留 SUCCEEDED 步骤不动
        - 将所有 PENDING/RUNNING 步骤（包括失败步骤）标记为 SKIPPED
        - 基于保留状态重新生成剩余计划
        """
        print(f"\n  [HYBRID REPLAN] 保留已完成步骤，重排剩余部分")
        succeeded_steps = [s for s in all_steps if s.status == StepStatus.SUCCEEDED]
        # 标记所有未完成步骤
        for step in all_steps:
            if step.status not in (StepStatus.SUCCEEDED, StepStatus.SKIPPED):
                step.status = StepStatus.SKIPPED
        # 基于保留状态生成剩余计划
        remaining_plan = plan_generator(
            context.original_goal,
            context.preserved_state,
            all_steps
        )
        for step in remaining_plan:
            step.status = StepStatus.PENDING
        # 合并：已完成步骤 + 新生成步骤
        return succeeded_steps + remaining_plan

# 第四部分：模拟执行引擎 (Execution Engine)
# ============================================================================

class ExecutionEngine:
    """
    执行引擎：负责按顺序执行计划步骤，调用监控器，触发重规划。

    核心循环：
    while 有计划步骤待执行:
        1. 取出下一步
        2. 执行 (调用 step.action)
        3. 监控 (对比预期与实际)
        4. 如果偏离 -> 触发 Replanner
        5. 如果重规划 -> 更新计划 -> 继续循环
    """

    def __init__(self, monitor: ExecutionMonitor, replanner: Replanner):
        self.monitor = monitor
        self.replanner = replanner

    def execute_plan(
        self,
        plan: List[SubGoal],
        context: ExecutionContext,
        plan_generator: Callable
    ) -> ExecutionContext:
        """
        执行计划的主循环。

        关键特性：
        - 循环检测：replan_count 超过最大值时终止
        - 状态追踪：每一步都记录到 step_history
        - 渐进式日志：实时输出执行进度

        Args:
            plan: 初始计划
            context: 执行上下文
            plan_generator: 用于重规划时的计划生成函数

        Returns:
            更新后的执行上下文
        """
        current_plan = list(plan)
        step_index = 0

        print(f"\n{'#'*60}")
        print(f"# 开始执行计划: {context.original_goal}")
        print(f"# 初始步骤数: {len(current_plan)}")
        print(f"{'#'*60}")

        while step_index < len(current_plan):
            # 循环检测
            if context.replan_count >= context.max_replan_attempts:
                print(f"\n[ABORT] 已达最大重规划次数 ({context.max_replan_attempts})，终止执行")
                break

            step = current_plan[step_index]

            # 跳过已成功的步骤和已标记跳过的步骤
            if step.status == StepStatus.SUCCEEDED:
                print(f"\n  [SKIP] {step.step_id}: 已完成，跳过")
                step_index += 1
                continue
            if step.status == StepStatus.SKIPPED:
                step_index += 1
                continue

            # 检查依赖是否满足
            deps_met = all(
                any(s.step_id == dep and s.status == StepStatus.SUCCEEDED
                    for s in current_plan)
                for dep in step.dependencies
            )
            if not deps_met:
                unmet = [dep for dep in step.dependencies
                         if not any(s.step_id == dep and s.status == StepStatus.SUCCEEDED
                                    for s in current_plan)]
                print(f"\n  [WAIT] {step.step_id}: 依赖未满足 {unmet}，跳过")
                step_index += 1
                continue

            # ---- 执行步骤 ----
            print(f"\n  >>> 执行: [{step.step_id}] {step.description}")
            step.status = StepStatus.RUNNING
            context.step_history.append({
                "step_id": step.step_id,
                "action": "start",
                "timestamp": time.time()
            })

            expected = {"status": "success", "data": step.data}
            actual: Dict[str, Any]

            if step.action is not None:
                try:
                    actual = step.action(context.world_state)
                except Exception as e:
                    actual = {"status": "error", "error": str(e), "recoverable": False}
            else:
                actual = {"status": "success", "message": "no action defined"}

            # ---- 监控 ----
            deviation_level, anomalies = self.monitor.evaluate(expected, actual)

            if deviation_level.value > DeviationLevel.NONE.value:
                print(f"  [!] 检测到偏离 L{deviation_level.value}:")
                for anomaly in anomalies:
                    print(f"      {anomaly}")

            # 根据偏离等级更新步骤状态
            if deviation_level in (DeviationLevel.NONE, DeviationLevel.MINOR):
                # 成功（或轻微偏离但可接受）
                step.status = StepStatus.SUCCEEDED
                step.result = actual
                context.world_state[step.step_id] = actual
                print(f"  [OK] {step.step_id}: 完成")
                step_index += 1

            elif deviation_level == DeviationLevel.FATAL:
                step.status = StepStatus.FAILED
                step.error = actual.get("error", "Fatal error")
                print(f"  [FATAL] {step.step_id}: 不可恢复的错误，终止")
                break

            else:
                # L2 或 L3 — 触发重规划
                step.status = StepStatus.FAILED
                step.error = actual.get("error", "Unknown error")
                step.result = actual

                # 调用重规划器
                new_plan, strategy = self.replanner.replan(
                    deviation_level, step, current_plan, context, plan_generator
                )

                if strategy == ReplanStrategy.NONE:
                    print(f"  [STOP] 重规划策略为 NONE，终止执行")
                    break

                # 更新计划并重置索引
                current_plan = new_plan
                print(f"\n  新计划步骤: {[s.step_id + ':' + s.status.value for s in current_plan]}")
                step_index = 0  # 从头重新遍历新计划

        # 汇总
        print(f"\n{'#'*60}")
        print(f"# 执行结束")
        succeeded = [s for s in current_plan if s.status == StepStatus.SUCCEEDED]
        failed = [s for s in current_plan if s.status == StepStatus.FAILED]
        print(f"# 成功: {len(succeeded)} 步, 失败: {len(failed)} 步")
        print(f"# 重规划次数: {context.replan_count}")
        print(f"{'#'*60}")

        return context

def demonstrate_rollback_strategies():
    """
    独立演示回滚策略的各个模式。
    展示不可变步骤的保留、状态快照和补偿事务。
    """

    print("\n\n")
    print("=" * 70)
    print("  回滚策略演示 (Rollback Strategy Demo)")
    print("=" * 70)

    # 场景：模拟一个多步骤的电商下单流程
    print("""
场景：电商下单流程
  Step 1: 验证库存  (可撤销)
  Step 2: 锁定优惠券 (可撤销 — 有补偿操作)
  Step 3: 创建订单   (不可变 — 一旦创建不能删除，只能取消)
  Step 4: 发起支付   (不可变 — 有补偿操作：退款)
  Step 5: 发送确认邮件 (不可变)

Step 4 支付失败 → 需要回滚 Step 2 和 Step 3
""")

    # 模拟补偿事务
    compensation_log: List[str] = []

    def compensate_coupon():
        """补偿操作：释放锁定的优惠券"""
        compensation_log.append("释放优惠券 COUPON-001")
        return True

    def compensate_order():
        """补偿操作：取消已创建的订单"""
        compensation_log.append("取消订单 ORD-12345")
        return True

    # 定义补偿操作映射
    compensations = {
        "S2": ("释放优惠券锁定", compensate_coupon),
        "S3": ("取消已创建订单", compensate_order),
    }

    # 模拟：Step 1-3 成功，Step 4 失败
    completed_steps = {
        "S1": {"desc": "验证库存", "immutable": False, "compensatable": False},
        "S2": {"desc": "锁定优惠券", "immutable": False, "compensatable": True},
        "S3": {"desc": "创建订单", "immutable": True, "compensatable": True},
    }

    print("回滚执行日志:")
    print("-" * 40)

    # 逆序执行补偿（S3 → S2）
    rollback_order = ["S3", "S2"]
    for step_id in rollback_order:
        if step_id in completed_steps and completed_steps[step_id]["compensatable"]:
            comp_name, comp_fn = compensations[step_id]
            success = comp_fn()
            status = "成功" if success else "失败"
            print(f"  补偿 [{step_id}] {comp_name}: {status}")

    print(f"\n  已完成的不可变步骤 (不会回滚):")
    for sid, info in completed_steps.items():
        if info["immutable"] and sid not in rollback_order:
            print(f"    [{sid}] {info['desc']} (已保留)")

    print(f"\n  补偿日志: {compensation_log}")
