# Agent设计模式集

## 概述

在构建基于LLM的智能应用时，单一Agent往往无法完成复杂的多步骤任务。经过大量工程实践和学术研究，社区总结出了四种经典的设计模式，用于编排多个LLM调用以完成复杂工作流。这四种模式并非互斥，在实际项目中常常组合使用。

本文深入剖析每种模式的设计原理、适用场景、架构模式和完整代码实现。

### 四种模式概览

```
                        设计模式集
        ┌─────────┬─────────┼─────────┬─────────┐

  顺序链   按意图  分角色  Sectioning Voting  代码审查

## 模式1：提示链（Prompt Chaining）

### 1.1 场景描述

**提示链**是最基础的LLM编排模式——将复杂任务拆解为多个顺序执行的步骤，每一步的输出作为下一步的输入，形成一条处理流水线。

**典型场景：文档处理流水线** —— 输入一篇英文技术文档，依次完成：提取关键信息 → 翻译为中文 → 生成摘要 → 格式化为Markdown。

### 1.2 架构图（文字描述）

│                    Prompt Chaining                    │
│                                                      │
│  ┌─────────┐     ┌─────────┐     ┌─────────┐       │
│  │  原始    │     │  提取后  │     │  翻译后  │       │
│  │  文档    │     │  信息    │     │  中文    │       │
│  └────┬────┘     └────┬────┘     └────┬────┘       │
│       │               │               │             │
│       ▼               ▼               ▼             │
│  ┌─────────┐     ┌─────────┐     ┌─────────┐       │
│  │ Step 1  │────►│ Step 2  │────►│ Step 3  │       │
│  │ 信息提取 │     │ 中文翻译 │     │ 生成摘要 │       │
│  │ (LLM)   │     │ (LLM)   │     │ (LLM)   │       │
│  └─────────┘     └─────────┘     └─────────┘       │
│                                        │             │
│                                        ▼             │
│                                  ┌─────────┐       │
│                                  │ Step 4  │       │
│                                  │ 格式化   │       │
│                                  │ (规则)   │       │
│                                  └────┬────┘       │
│                                       │             │
│                                       ▼             │
│                                  ┌─────────┐       │
│                                  │ 最终输出 │       │
│                                  │ Markdown│       │

└──────────────────────────────────────────────────────┘

#### 关键特征

- **线性依赖**：每一步严格依赖前一步的输出
- **可观测性**：每个中间结果都可以检查和调试
- **可插拔**：每个步骤可以独立替换或优化
- **门控机制**：可以在每个步骤后添加质量检查，不达标则重试

### 1.3 完整代码实现

```python
"""
提示链（Prompt Chaining）模式 — 文档处理流水线
流程：提取关键信息 → 翻译为中文 → 生成摘要 → 格式化为Markdown

本示例使用OpenAI兼容API，可替换为任意LLM服务。
"""
import os
import json
from typing import Dict, List, Optional
from dataclasses import dataclass
from openai import OpenAI

# ============================================================
# 配置
# ============================================================
client = OpenAI(
    api_key=os.environ.get("OPENAI_API_KEY", "sk-demo"),
    base_url=os.environ.get("OPENAI_BASE_URL", "https://api.openai.com/v1"),
)

# ============================================================
# 数据结构定义
# ============================================================
@dataclass
class PipelineContext:
    """流水线上下文 —— 贯穿整个处理流程的数据载体"""
    original_text: str
    extracted_info: Optional[str] = None
    translated_text: Optional[str] = None
    summary_text: Optional[str] = None
    final_markdown: Optional[str] = None
    errors: List[str] = None

    def __post_init__(self):
        if self.errors is None:
            self.errors = []


# ============================================================
# 核心LLM调用函数
# ============================================================
def call_llm(system_prompt: str, user_prompt: str,
             model: str = "gpt-4", temperature: float = 0.3) -> str:
    """
    统一的LLM调用接口
    - system_prompt: 系统角色定义
    - user_prompt: 用户输入
    - 返回模型的文本输出
    """
    response = client.chat.completions.create(
        model=model,
        temperature=temperature,
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_prompt},
        ],
    )
    return response.choices[0].message.content.strip()


# ============================================================
# Step 1: 信息提取
# ============================================================
def step_extract_info(ctx: PipelineContext) -> PipelineContext:
    """
    第一步：从原始文档中提取关键信息
    提取：标题、作者、核心技术点、关键数据、结论
    """
    system_prompt = """你是一位专业的技术文档分析师。请从给定的技术文档中提取以下信息：
1. 文档标题（如果有的话）
2. 核心技术概念和术语（列出3-5个）
3. 关键数据和指标（如果有的话）
4. 主要结论或发现（1-2句话）
5. 文档的目标读者

请以JSON格式输出，确保所有字段都有值。如果某个字段在原文档中未出现，标注为"未提及"。
输出格式示例：
{
  "title": "...",
  "core_concepts": ["概念1", "概念2"],
  "key_data": ["数据1"],
  "conclusions": "结论...",
  "target_audience": "..."
}"""

    try:
        result = call_llm(system_prompt, ctx.original_text)
        ctx.extracted_info = result
        print(f"[Step 1 完成] 信息提取成功")
        print(f"  提取结果预览: {result[:150]}...")
    except Exception as e:
        ctx.errors.append(f"Step1 提取失败: {e}")

    return ctx


# ============================================================
# Step 2: 中文翻译
# ============================================================
def step_translate(ctx: PipelineContext) -> PipelineContext:
    """
    第二步：将提取的信息翻译为中文
    输入的 ctx.extracted_info 是第一步的输出
    """
    if not ctx.extracted_info:
        ctx.errors.append("Step2 跳过：缺少上一步输出")
        return ctx

    system_prompt = """你是一位专业的技术翻译专家，精通中英文技术翻译。
请将用户提供的英文/混合内容翻译为流畅、专业的中文。
要求：
1. 专业术语保留英文原文（括号标注中文）
2. 保持原文的技术准确性
3. 中文表达自然流畅，符合技术文档的书面风格
4. 保持JSON格式不变，仅翻译字符串值"""

    try:
        result = call_llm(system_prompt, ctx.extracted_info)
        ctx.translated_text = result
        print(f"[Step 2 完成] 翻译完成")
        print(f"  翻译结果预览: {result[:150]}...")
    except Exception as e:
        ctx.errors.append(f"Step2 翻译失败: {e}")

    return ctx


# ============================================================
# Step 3: 生成摘要
# ============================================================
def step_summarize(ctx: PipelineContext) -> PipelineContext:
    """
    第三步：基于中文翻译生成简洁摘要
    """
    if not ctx.translated_text:
        ctx.errors.append("Step3 跳过：缺少上一步输出")
        return ctx

    system_prompt = """你是一位资深技术编辑。请为以下技术内容生成一段专业的摘要。
要求：
1. 摘要长度：100-150字
2. 突出核心技术和关键发现
3. 语言简洁有力，适合放在文档开头
4. 仅输出摘要文本，不要加任何前缀（不要写"摘要："等）"""

    try:
        result = call_llm(system_prompt, ctx.translated_text)
        ctx.summary_text = result
        print(f"[Step 3 完成] 摘要生成")
        print(f"  摘要: {result}")
    except Exception as e:
        ctx.errors.append(f"Step3 摘要生成失败: {e}")

    return ctx


# ============================================================
# Step 4: 格式化为Markdown
# ============================================================
def step_format_markdown(ctx: PipelineContext) -> PipelineContext:
    """
    第四步（非LLM步骤）：将前面所有步骤的结果组装为Markdown
    这一步是纯规则处理，不需要调用LLM
    """
    if not ctx.summary_text:
        ctx.errors.append("Step4 跳过：缺少上一步输出")
        return ctx

    # 尝试解析提取的信息（如果是JSON格式）
    try:
        # 从 ctx.extracted_info 或 ctx.translated_text 中提取结构化数据
        # 尝试解析JSON
        extracted_json = json.loads(ctx.extracted_info) if ctx.extracted_info else {}
    except json.JSONDecodeError:
        extracted_json = {}

    title = extracted_json.get("title", "技术文档分析报告")
    concepts = extracted_json.get("core_concepts", [])
    conclusions = extracted_json.get("conclusions", "")

    # 构建Markdown
    md_parts = [
        f"# {title}",
        "",
        "---",
        "",
        "## 摘要",
        "",
        ctx.summary_text,
        "",
        "---",
        "",
        "## 核心技术概念",
        "",
    ]
    for concept in concepts:
        md_parts.append(f"- **{concept}**")

    md_parts.extend([
        "",
        "## 提取的详细信息",
        "",
        ctx.translated_text or ctx.extracted_info or "（无内容）",
        "",
        "---",
        "",
        f"*本报告由AI自动生成 | 分析结论: {conclusions}*",
    ])

    ctx.final_markdown = "\n".join(md_parts)
    print(f"[Step 4 完成] Markdown 格式化完成")
    return ctx


# ============================================================
# 流水线编排器
# ============================================================
class PromptChainPipeline:
    """
    提示链流水线编排器
    按照预定义的顺序执行各个步骤，管理上下文传递和错误处理
    """

    def __init__(self):
        # 定义流水线步骤（有序）
        self.steps = [
            ("信息提取", step_extract_info),
            ("中文翻译", step_translate),
            ("生成摘要", step_summarize),
            ("Markdown格式化", step_format_markdown),
        ]

    def run(self, input_text: str) -> PipelineContext:
        """
        执行完整的流水线
        """
        ctx = PipelineContext(original_text=input_text)
        print(f"\n{'='*60}")
        print(f"Prompt Chaining Pipeline 开始执行")
        print(f"输入文本长度: {len(input_text)} 字符")
        print(f"{'='*60}")

        for step_name, step_func in self.steps:
            print(f"\n>>> 执行步骤: {step_name}")
            ctx = step_func(ctx)
            if ctx.errors and ctx.errors[-1].startswith(f"Step"):
                print(f"  ⚠ 步骤 {step_name} 出现错误，流水线继续执行")

        # 输出执行状态
        error_count = len([e for e in ctx.errors if e])
        print(f"\n{'='*60}")
        print(f"流水线执行完成。错误数: {error_count}")
        if ctx.errors:
            for err in ctx.errors:
                print(f"  - {err}")
        print(f"{'='*60}")

        return ctx


# ============================================================
# 示例执行入口
# ============================================================
if __name__ == "__main__":
    # 模拟输入：一篇英文技术文档
    sample_document = """
    Technical Report: Deep Learning for Natural Language Processing

    Abstract:
    This report examines state-of-the-art deep learning architectures for NLP tasks,
    including Transformer, BERT, and GPT models. Our experiments on sentiment analysis
    and named entity recognition tasks demonstrate that fine-tuned Transformer models
    achieve 94.2% accuracy on the SST-2 benchmark and 91.5% F1 on CoNLL-2003.

    Key Findings:
    1. Pre-training on domain-specific corpora improves downstream task performance
       by an average of 3.7 percentage points.
    2. Model distillation reduces inference latency by 60% while maintaining 97% of
       the original model's accuracy.
    3. Attention visualization reveals that deeper layers capture syntactic structure
       while early layers focus on local word dependencies.

    Conclusions:
    Deep learning has fundamentally transformed NLP. Fine-tuned pre-trained models
    are now the standard approach for most NLP tasks. Future work should focus on
    reducing model size while maintaining performance, and improving cross-lingual
    transfer capabilities.

    Target Audience: ML Engineers, NLP Researchers, Technical Decision Makers
    """

    # 执行流水线
    pipeline = PromptChainPipeline()
    final_context = pipeline.run(sample_document)

    # 打印最终输出
    if final_context.final_markdown:
        print("\n\n" + "=" * 60)
        print("最终输出 (Markdown)")
        print("=" * 60)
        print(final_context.final_markdown)
```

## 模式2：路由（Routing）

### 2.1 场景描述

**路由模式**的核心是"分类-分发"：首先由一个分类器分析用户输入的意图，然后将任务路由到对应的专业处理器。这允许一个系统同时处理多种类型的请求，每种请求由最合适的LLM或处理逻辑来处理。

**典型场景：客服系统** —— 用户发来消息后，系统判断意图类型："技术问题"发给技术专家、"账单问题"发给财务专家、"投诉"升级为人工客服、"一般咨询"由通用Agent处理。

### 2.2 架构图（文字描述）

│                    Routing Pattern                    │
│                                                      │
│                    ┌─────────────┐                   │
│      用户输入 ─────►│   Router    │                   │
│                    │ (意图分类器) │                   │
│                    └──┬──┬──┬──┬─┘                   │
│                       │  │  │  │                     │
│             ┌─────────┘  │  │  └─────────┐          │
│             ▼            ▼  ▼            ▼          │
│        ┌────────┐  ┌────────┐  ┌────────┐  ┌──────┐│
│        │技术专家 │  │财务专家 │  │投诉升级 │  │通用  ││
│        │Handler │  │Handler │  │人工处理 │  │Handler││
│        └───┬────┘  └───┬────┘  └───┬────┘  └──┬───┘│
│            │            │           │           │    │
│            └────────────┴───────────┴───────────┘    │
│                              │                        │
│                              ▼                        │
│                        ┌─────────┐                   │
│                        │ 统一响应 │                   │
│                        │ 格式化   │                   │
│                        └─────────┘                   │
└─────────────────────────────────────────────────────┘
```

#### 关键特征

- **关注点分离**：不同类型的请求由专门的处理器处理，互不干扰
- **可扩展**：新增业务类型只需添加新的Handler和路由规则
- **降级策略**：无法分类的请求可以放入通用Handler处理
- **确定性路由 + 语义路由**：简单规则匹配或LLM语义理解均可

### 2.3 完整代码实现

```python
"""
路由（Routing）模式 — 智能客服系统
根据用户输入意图，将请求分发到不同的专业处理器

支持的路由：
- "technical"  → 技术专家Handler
- "billing"    → 财务/账单专家Handler
- "complaint"  → 人工客服升级
- "general"    → 通用信息Handler
"""
import os
import enum
from typing import Dict, Tuple, Optional
from dataclasses import dataclass, field
from openai import OpenAI

client = OpenAI(
    api_key=os.environ.get("OPENAI_API_KEY", "sk-demo"),
    base_url=os.environ.get("OPENAI_BASE_URL", None),
)


# ============================================================
# 数据结构定义
# ============================================================
class Intent(enum.Enum):
    """意图枚举"""
    TECHNICAL = "technical"    # 技术问题
    BILLING = "billing"        # 账单/付费问题
    COMPLAINT = "complaint"   # 投诉
    GENERAL = "general"        # 一般咨询


@dataclass
class RoutingRequest:
    """路由请求"""
    user_message: str
    user_id: str = "anonymous"
    context: Dict = field(default_factory=dict)  # 可选的上下文信息


@dataclass
class RoutingResponse:
    """路由响应"""
    intent: Intent
    handler_name: str
    response_text: str
    confidence: float
    escalation_needed: bool = False


# ============================================================
# 意图分类器（LLM Router）
# ============================================================
class IntentClassifier:
    """
    基于LLM的意图分类器
    分析用户消息，输出对应的Intent枚举值
    """

    # 意图分类的系统提示词
    CLASSIFICATION_PROMPT = """你是一个客服系统的意图分类器。请分析用户消息，判断其意图类别。

类别定义：
- technical: 技术问题（产品使用、功能咨询、Bug报告、API问题、安装配置等）
- billing: 账单/付费问题（退款、发票、订阅、价格咨询、付款失败等）
- complaint: 投诉（对产品、服务、人员的不满，要求投诉升级等）
- general: 一般咨询（产品介绍、公司信息、合作咨询等，不属于以上分类的）

请严格按照以下JSON格式输出，不要输出其他内容：
{"intent": "technical|billing|complaint|general", "confidence": 0.0-1.0, "reason": "简短分类理由"}"""

    def classify(self, message: str) -> Tuple[Intent, float, str]:
        """
        对用户消息进行意图分类
        返回: (Intent, 置信度, 分类理由)
        """
        response = client.chat.completions.create(
            model="gpt-4",
            temperature=0.0,  # 分类任务使用0温度提高一致性
            messages=[
                {"role": "system", "content": self.CLASSIFICATION_PROMPT},
                {"role": "user", "content": message},
            ],
        )

        # 解析JSON响应
        import json
        result = json.loads(response.choices[0].message.content)

        intent = Intent(result["intent"])
        confidence = float(result["confidence"])
        reason = result.get("reason", "")

        return intent, confidence, reason


# ============================================================
# Handler 基类和具体实现
# ============================================================
class BaseHandler:
    """Handler基类 —— 所有专业处理器的父类"""
    def __init__(self, name: str):
        self.name = name

    def can_handle(self, intent: Intent) -> bool:
        """判断是否能处理该意图"""
        raise NotImplementedError

    def handle(self, request: RoutingRequest) -> str:
        """处理请求并返回响应文本"""
        raise NotImplementedError


class TechnicalHandler(BaseHandler):
    """技术问题处理器"""

    def __init__(self):
        super().__init__("技术专家")

    def can_handle(self, intent: Intent) -> bool:
        return intent == Intent.TECHNICAL

    def handle(self, request: RoutingRequest) -> str:
        system_prompt = """你是一位资深技术支持工程师。
请根据用户的问题提供专业、详细的解答。要求：
1. 如果问题是Bug类，提供排查步骤
2. 如果是功能咨询，说明使用方法并给出示例
3. 如果问题较复杂，提供多个解决方案，从易到难排列
4. 使用友好、耐心的口吻
请用中文回答。"""

        response = client.chat.completions.create(
            model="gpt-4",
            temperature=0.5,
            messages=[
                {"role": "system", "content": system_prompt},
                {"role": "user", "content": request.user_message},
            ],
        )
        return response.choices[0].message.content


class BillingHandler(BaseHandler):
    """账单/付费问题处理器"""

    def __init__(self):
        super().__init__("财务专家")

    def can_handle(self, intent: Intent) -> bool:
        return intent == Intent.BILLING

    def handle(self, request: RoutingRequest) -> str:
        system_prompt = """你是一位专业的账单客服专员。

class CustomerServiceRouter:
    """
    客服路由系统
    整合意图分类和Handler分发
    """

    def __init__(self):
        self.classifier = IntentClassifier()
        # 注册所有Handler
        self.handlers: Dict[Intent, BaseHandler] = {
            Intent.TECHNICAL: TechnicalHandler(),
            Intent.BILLING: BillingHandler(),
            Intent.COMPLAINT: ComplaintHandler(),
            Intent.GENERAL: GeneralHandler(),
        }
        # 置信度阈值：低于此值的分类结果会降级为GENERAL
        self.CONFIDENCE_THRESHOLD = 0.7

    def route(self, request: RoutingRequest) -> RoutingResponse:
        """
        路由入口：分类 → 分发 → 处理 → 返回
        """
        # Step 1: 意图分类
        intent, confidence, reason = self.classifier.classify(request.user_message)
        print(f"[Router] 意图分类: {intent.value} (置信度: {confidence:.2f})")
        print(f"[Router] 分类理由: {reason}")

        # Step 2: 低置信度降级处理
        if confidence < self.CONFIDENCE_THRESHOLD:
            print(f"[Router] 置信度过低({confidence:.2f})，降级为 GENERAL")
            intent = Intent.GENERAL

        # Step 3: 获取对应的Handler
        handler = self.handlers.get(intent, self.handlers[Intent.GENERAL])

        # Step 4: Handler处理
        print(f"[Router] 分发到: {handler.name}")
        response_text = handler.handle(request)

        # Step 5: 判断是否需要升级
        escalation_needed = (intent == Intent.COMPLAINT and confidence > 0.5)

        return RoutingResponse(
            intent=intent,
            handler_name=handler.name,
            response_text=response_text,
            confidence=confidence,
            escalation_needed=escalation_needed,
        )

## 模式3：并行化（Parallelization）

### 3.1 场景描述

**并行化**模式适用于一个大任务可被分解为多个互不依赖的子任务，并行执行后合并结果。根据任务特性和合并策略，分为两种子模式：

- **Sectioning（分块并行）**：将输入数据拆分为多个独立块，分别用相同的处理逻辑并行处理，最后合并。如：长篇文档的多段落分析。
- **Voting（多模型投票）**：将同一个任务发送给多个不同的模型/提示，收集结果后进行投票或择优。如：代码审查中多个审查员的评审意见合并。

"""
并行化（Parallelization）模式 — Sectioning + Voting

子模式1: Sectioning — 长文档分块并行分析
子模式2: Voting    — 多审查员并行审查代码
"""
import os
import time
import json
from concurrent.futures import ThreadPoolExecutor, as_completed
from typing import List, Dict, Tuple
from dataclasses import dataclass
from openai import OpenAI

client = OpenAI(
    api_key=os.environ.get("OPENAI_API_KEY", "sk-demo"),
    base_url=os.environ.get("OPENAI_BASE_URL", None),
)


# ============================================================
# 通用LLM调用工具
# ============================================================
def call_llm(system_prompt: str, user_prompt: str,
             model: str = "gpt-4", temperature: float = 0.3) -> str:
    """统一的LLM调用接口"""
    response = client.chat.completions.create(
        model=model,
        temperature=temperature,
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_prompt},
        ],
    )
    return response.choices[0].message.content.strip()


# ============================================================
# 子模式1: Sectioning — 长文档分块并行分析
# ============================================================
@dataclass
class SectionResult:
    """单个分块的分析结果"""
    section_id: int
    summary: str
    key_points: List[str]
    sentiment: str  # positive / negative / neutral
    important_entities: List[str]


class SectioningProcessor:
    """
    分块并行处理器
    流程：分块 → 并行分析 → 结果合并
    """

    def __init__(self, max_workers: int = 4):
        self.max_workers = max_workers

    def split_text(self, text: str, chunk_size: int = 3) -> List[Tuple[int, str]]:
        """
        将长文本按段落分块
        chunk_size: 每个分块包含的段落数
        """
        paragraphs = [p.strip() for p in text.split("\n\n") if p.strip()]
        chunks = []
        for i in range(0, len(paragraphs), chunk_size):
            chunk_text = "\n\n".join(paragraphs[i:i + chunk_size])
            chunk_id = i // chunk_size + 1
            chunks.append((chunk_id, chunk_text))
        print(f"[Sectioning] 文本分块: 共 {len(chunks)} 个块（每块 {chunk_size} 段）")
        return chunks

    def analyze_chunk(self, chunk_id: int, chunk_text: str) -> SectionResult:
        """
        分析单个分块（可在独立线程中执行）
        """
        print(f"  [Chunk {chunk_id}] 开始分析 ({len(chunk_text)} 字符)...")
        start_time = time.time()

        system_prompt = """你是一位文本分析专家。请分析给定文本块，以JSON格式输出：
{
  "summary": "2-3句话总结该部分的核心内容",
  "key_points": ["要点1", "要点2", "要点3"],
  "sentiment": "positive/negative/neutral",
  "important_entities": ["实体1", "实体2"]
}
仅输出JSON，不要输出其他内容。"""

        try:
            result_text = call_llm(system_prompt, chunk_text, temperature=0.2)
            # 尝试清洗和解析JSON
            result_text = result_text.strip()
            if result_text.startswith("```"):
                result_text = result_text.split("\n", 2)[-1].rsplit("```", 1)[0]
            data = json.loads(result_text)

            elapsed = time.time() - start_time
            print(f"  [Chunk {chunk_id}] 完成 ({elapsed:.1f}s)")

            return SectionResult(
                section_id=chunk_id,
                summary=data.get("summary", ""),
                key_points=data.get("key_points", []),
                sentiment=data.get("sentiment", "neutral"),
                important_entities=data.get("important_entities", []),
            )
        except Exception as e:
            print(f"  [Chunk {chunk_id}] 错误: {e}")
            return SectionResult(
                section_id=chunk_id,
                summary=f"分析失败: {e}",
                key_points=[],
                sentiment="neutral",
                important_entities=[],
            )

    def merge_results(self, results: List[SectionResult]) -> str:
        """
        合并所有分块的分析结果，调用LLM生成综合报告
        """
        # 构建分块结果摘要
        chunks_summary = ""
        for r in sorted(results, key=lambda x: x.section_id):
            chunks_summary += f"\n--- 第{r.section_id}部分 ---\n"
            chunks_summary += f"摘要: {r.summary}\n"
            chunks_summary += f"要点: {', '.join(r.key_points)}\n"
            chunks_summary += f"情感: {r.sentiment}\n"

        system_prompt = """你是一位报告撰写专家。请根据以下多个文本分块的分析结果，
生成一份结构清晰的综合报告。要求：
1. 开头是一个总体概述（3-4句话）
2. 按主题组织内容（而非按分块编号）
3. 提炼出贯穿全文的关键主题
4. 结尾给出整体结论或建议
使用Markdown格式。请用中文。"""

        merged_report = call_llm(system_prompt, chunks_summary, temperature=0.5)
        return merged_report

    def process(self, long_text: str) -> str:
        """
        主流程：分块 → 并行分析 → 合并
        """
        print("\n" + "=" * 60)
        print("Sectioning 并行处理开始")
        print("=" * 60)

        # Step 1: 分块
        chunks = self.split_text(long_text)
        if not chunks:
            return "输入的文本为空，无法分析"

        # Step 2: 并行分析各分块
        results: List[SectionResult] = []
        with ThreadPoolExecutor(max_workers=self.max_workers) as executor:
            # 提交所有分块分析任务
            future_to_chunk = {
                executor.submit(self.analyze_chunk, cid, ctext): cid
                for cid, ctext in chunks
            }
            # 收集结果（按完成顺序）
            for future in as_completed(future_to_chunk):
                result = future.result()
                results.append(result)

        print(f"\n[Sectioning] 所有分块分析完成，共 {len(results)} 个结果")

        # Step 3: 合并结果
        final_report = self.merge_results(results)
        return final_report

# 子模式2: Voting — 多审查员并行审查
# ============================================================
class VotingProcessor:
    """
    多模型投票处理器
    多个"审查员"（不同视角/提示）并行审查同一段代码，
    综合各方意见给出最终评审结果。
    """

    # 定义不同的审查角色及其审查重点
    REVIEWERS = [
        {
            "name": "安全审查员",
            "focus": """你是一位安全专家。请审查以下代码的安全性：
- 是否存在SQL注入、XSS、命令注入等安全漏洞
- 敏感数据是否正确加密和脱敏
- 认证和授权逻辑是否完善
- 是否存在不安全的依赖或配置""",
        },
        {
            "name": "性能审查员",
            "focus": """你是一位性能优化专家。请审查以下代码的性能：
- 是否存在N+1查询问题
- 是否有不必要的循环或重复计算
- 数据库查询是否能使用索引
- 内存使用是否合理""",
        },
        {
            "name": "代码规范审查员",
            "focus": """你是一位代码规范专家。请审查以下代码的可维护性：
- 命名是否清晰（变量、函数、类名）
- 函数是否符合单一职责原则
- 是否有合适的注释和文档
- 代码结构是否清晰，易于理解和修改""",
        },
        {
            "name": "逻辑审查员",
            "focus": """你是一位逻辑验证专家。请审查以下代码的逻辑正确性：
- 边界条件是否处理正确（空输入、null、极端值）
- 异常处理和错误恢复是否完善
- 是否存在竞态条件或死锁风险
- 业务逻辑是否正确""",
        },
    ]

    def single_review(self, reviewer: Dict, code: str) -> Tuple[str, str]:
        """
        单个审查员审查代码
        返回: (审查员名称, 审查意见)
        """
        print(f"  [{reviewer['name']}] 开始审查...")
        start_time = time.time()

        system_prompt = f"""{reviewer['focus']}

请以JSON格式输出审查结果：
{{
  "score": 1-10（总体评分）,
  "issues": [
    {{"severity": "critical|major|minor", "description": "问题描述", "suggestion": "改进建议"}}
  ],
  "strengths": ["优点1", "优点2"],
  "verdict": "approve|approve_with_changes|reject",
  "summary": "总体评价（1-2句话）"
}}

仅输出JSON，不要输出其他内容。"""

        try:
            result_text = call_llm(system_prompt, code, temperature=0.3)
            elapsed = time.time() - start_time
            print(f"  [{reviewer['name']}] 完成 ({elapsed:.1f}s)")
            return reviewer["name"], result_text
        except Exception as e:
            return reviewer["name"], json.dumps({
                "score": 0, "issues": [],
                "strengths": [], "verdict": "error",
                "summary": f"审查失败: {str(e)}"
            })

    def aggregate_results(self, review_results: List[Tuple[str, str]]) -> str:
        """
        聚合多个审查员的意见，生成最终评审报告
        """
        # 构建聚合提示
        reviews_text = ""
        for name, result in review_results:
            reviews_text += f"\n### {name}\n```json\n{result}\n```\n"

        system_prompt = """你是一位首席技术审查官（CTO）。请根据多位专家的代码审查意见，
生成一份综合评审报告。要求：
1. 评分汇总：各位审查员的评分和平均分
2. 关键问题（Critical/Major级别的问题必须列出）
3. 共识建议：多位审查员共同指出的问题
4. 最终决断：批准(approve)、有条件批准(approve_with_changes)、或驳回(reject)
5. 改进优先级清单（从最重要的开始）
使用Markdown格式。请用中文。"""

        final_report = call_llm(system_prompt, reviews_text, temperature=0.4)
        return final_report

    def review_code(self, code: str, max_workers: int = 4) -> str:
        """
        主流程：并行审查 → 聚合
        """
        print("\n" + "=" * 60)
        print("Voting 多审查员并行审查开始")
        print("=" * 60)

        results = []
        with ThreadPoolExecutor(max_workers=max_workers) as executor:
            future_to_reviewer = {
                executor.submit(self.single_review, reviewer, code): reviewer["name"]
                for reviewer in self.REVIEWERS
            }
            for future in as_completed(future_to_reviewer):
                results.append(future.result())

        print(f"\n[Voting] 所有审查员完成审查，共 {len(results)} 份意见")
        return self.aggregate_results(results)

## 模式4：自我反思（Reflection）

### 4.1 场景描述

**自我反思模式**让Agent能够审视自己的输出，发现问题并迭代改进。这种模式模拟了人类"先写草稿，再检查修改"的工作方式。Agent首先生成初始输出，然后切换到一个"审查者"角色来评估输出质量，发现问题后回到"生成者"角色进行修改，如此循环直到满足质量标准或达到最大迭代次数。

**典型场景：代码生成与审查循环** —— Agent生成代码 → Agent审查代码（发现漏洞、性能问题） → Agent修改代码 → 再次审查 → ... → 通过后输出最终版本。

#### 关键特征

- **自我改进**：不需要外部反馈，Agent通过角色切换实现自我提升
- **质量门控**：定义明确的质量标准作为退出条件
- **迭代收敛**：每轮反思都会改进输出质量，逐步逼近最优解
- **成本权衡**：每轮迭代都会消耗Token，需要在质量和成本之间平衡

### 4.3 完整代码实现

```python
"""
自我反思（Reflection）模式 — 代码生成与审查循环

流程：
1. Generator Agent 根据需求生成代码
2. Reviewer Agent 审查代码质量
3. 如果评分低于阈值，将审查反馈返回给 Generator 修改
4. 重复直到评分达标或达到最大迭代次数
"""
import os
import json
from typing import Dict, List, Tuple, Optional
from dataclasses import dataclass, field
from openai import OpenAI

client = OpenAI(
    api_key=os.environ.get("OPENAI_API_KEY", "sk-demo"),
    base_url=os.environ.get("OPENAI_BASE_URL", None),
)


# ============================================================
# 数据结构定义
# ============================================================
@dataclass
class ReflectionState:
    """反思循环的状态"""
    requirement: str                       # 原始需求
    current_code: str = ""                # 当前版本的代码
    iteration: int = 0                    # 当前迭代次数
    review_history: List[Dict] = field(default_factory=list)  # 历史审查记录
    max_iterations: int = 3               # 最大迭代次数
    quality_threshold: float = 7.0        # 质量阈值（0-10）
    final_approved: bool = False          # 是否最终通过


# ============================================================
# 工具函数
# ============================================================
def call_llm(system_prompt: str, user_prompt: str,
             model: str = "gpt-4", temperature: float = 0.3) -> str:
    """统一的LLM调用"""
    response = client.chat.completions.create(
        model=model,
        temperature=temperature,
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_prompt},
        ],
    )
    return response.choices[0].message.content.strip()


# ============================================================
# Generator：代码生成器
# ============================================================
class CodeGenerator:
    """
    代码生成Agent
    根据需求和审查反馈生成/修改代码
    """

    GENERATOR_SYSTEM_PROMPT = """你是一位资深Python开发者，擅长编写高质量、安全、高效的代码。

编写代码时需要遵循以下原则：
1. **安全性**：防止SQL注入、XSS、命令注入，敏感数据不硬编码
2. **性能**：避免N+1查询，使用合适的算法和数据结构
3. **可维护性**：清晰的命名、适当的注释、单一职责函数
4. **健壮性**：处理边界条件（空输入、null、极端值），完善的异常处理
5. **规范性**：遵循PEP 8，使用类型提示

请直接输出代码，代码外使用```python```代码块包裹。"""

    def generate_initial(self, requirement: str) -> str:
        """根据需求生成初始代码"""
        print("[Generator] 生成初始代码...")
        prompt = f"请根据以下需求编写Python代码，确保完整可运行：\n\n{requirement}"
        code = call_llm(self.GENERATOR_SYSTEM_PROMPT, prompt, temperature=0.5)
        return self._extract_code(code)

    def revise(self, requirement: str, current_code: str,
               review_feedback: str) -> str:
        """根据审查反馈修改代码"""
        print("[Generator] 根据反馈修改代码...")
        prompt = f"""原始需求：
{requirement}

当前代码：
```python
{current_code}
```

审查反馈：
{review_feedback}

请根据以上审查反馈，修改代码。解决所有指出的问题，保持代码的完整性和可运行性。
改进后的代码："""
        code = call_llm(self.GENERATOR_SYSTEM_PROMPT, prompt, temperature=0.4)
        return self._extract_code(code)

    @staticmethod
    def _extract_code(text: str) -> str:
        """从LLM输出中提取代码块"""
        if "```python" in text:
            start = text.find("```python") + len("```python")
            end = text.find("```", start)
            return text[start:end].strip() if end > 0 else text
        elif "```" in text:
            start = text.find("```") + 3
            end = text.find("```", start)
            return text[start:end].strip() if end > 0 else text
        return text.strip()


# ============================================================
# Reviewer：代码审查器
# ============================================================
class CodeReviewer:
    """
    代码审查Agent
    评估代码质量并给出改进建议
    """

    REVIEWER_SYSTEM_PROMPT = """你是一位资深代码审查专家。请仔细审查以下代码，
从以下五个维度评分并给出改进建议：

评分维度（每个维度1-10分）：
1. security: 安全性 —— 是否有安全漏洞（SQL注入、XSS、硬编码密钥等）
2. performance: 性能 —— 算法复杂度、数据库查询效率、内存使用
3. maintainability: 可维护性 —— 命名、注释、代码结构、函数职责
4. robustness: 健壮性 —— 异常处理、边界条件、输入验证
5. correctness: 正确性 —— 逻辑是否正确、是否满足需求

请以严格的JSON格式输出，每个字段都必须有值：
{
  "security": {"score": 1-10, "issues": ["问题1", "问题2"], "suggestions": ["建议1"]},
  "performance": {"score": 1-10, "issues": ["问题1"], "suggestions": ["建议1"]},
  "maintainability": {"score": 1-10, "issues": [], "suggestions": []},
  "robustness": {"score": 1-10, "issues": [], "suggestions": []},
  "correctness": {"score": 1-10, "issues": [], "suggestions": []},
  "overall_score": 1-10（加权平均分）,
  "critical_issues": ["必须修复的关键问题"],
  "summary": "总体评价（1-2句话）",
  "decision": "approve|revise|reject"
}

仅输出JSON，不要输出其他内容。"""

    def review(self, requirement: str, code: str) -> Dict:
        """审查代码并返回结构化的审查结果"""
        prompt = f"""原始需求：
{requirement}

待审查代码：
```python
{code}
```

请从多维度审查以上代码。"""

        result_text = call_llm(self.REVIEWER_SYSTEM_PROMPT, prompt, temperature=0.2)
        # 清洗并解析JSON
        result_text = result_text.strip()
        if result_text.startswith("```"):
            result_text = result_text.split("\n", 2)[-1].rsplit("```", 1)[0]

        try:
            return json.loads(result_text)
        except json.JSONDecodeError as e:
            print(f"[Reviewer] JSON解析失败: {e}")
            return {
                "overall_score": 0,
                "critical_issues": [f"审查结果解析失败: {str(e)}"],
                "summary": f"审查出错: {result_text[:200]}",
                "decision": "revise",
            }

    def format_feedback(self, review_result: Dict) -> str:
        """将结构化审查结果格式化为人类可读的反馈文本"""
        lines = ["=== 代码审查报告 ===\n"]
        lines.append(f"综合评分: {review_result.get('overall_score', 'N/A')}/10")

        for dim in ["security", "performance", "maintainability", "robustness", "correctness"]:
            dim_data = review_result.get(dim, {})
            score = dim_data.get("score", "N/A")
            issues = dim_data.get("issues", [])
            lines.append(f"\n【{dim.upper()}】评分: {score}/10")
            if issues:
                for issue in issues:
                    lines.append(f"  - {issue}")

        critical = review_result.get("critical_issues", [])
        if critical:
            lines.append(f"\n【关键问题（必须修复）】")
            for c in critical:
                lines.append(f"  ❌ {c}")

        lines.append(f"\n总结: {review_result.get('summary', '')}")
        lines.append(f"决定: {review_result.get('decision', 'unknown')}")

        return "\n".join(lines)


# ============================================================
# Reflection Loop：反思循环编排器
# ============================================================
class ReflectionLoop:
    """
    反思循环编排器
    协调 Generator 和 Reviewer，迭代改进代码质量
    """

    def __init__(self, max_iterations: int = 3, quality_threshold: float = 7.0):
        self.generator = CodeGenerator()
        self.reviewer = CodeReviewer()
        self.max_iterations = max_iterations
        self.quality_threshold = quality_threshold

    def run(self, requirement: str) -> ReflectionState:
        """
        执行反思循环
        """
        state = ReflectionState(
            requirement=requirement,
            max_iterations=self.max_iterations,
            quality_threshold=self.quality_threshold,
        )

        print("=" * 60)
        print("Reflection Loop 启动")
        print(f"需求: {requirement[:80]}...")
        print(f"最大迭代次数: {self.max_iterations}")
        print(f"质量阈值: {self.quality_threshold}")
        print("=" * 60)

        # Step 1: 生成初始代码
        state.current_code = self.generator.generate_initial(requirement)
        print(f"\n[第 {state.iteration} 版] 初始代码生成完成 "
              f"({len(state.current_code)} 字符)")

        # Step 2: 反思循环
        while state.iteration < self.max_iterations:
            state.iteration += 1
            print(f"\n{'─' * 50}")
            print(f"第 {state.iteration} 轮反思")
            print(f"{'─' * 50}")

            # 2a: 审查当前代码
            review_result = self.reviewer.review(requirement, state.current_code)
            overall_score = review_result.get("overall_score", 0)
            decision = review_result.get("decision", "revise")
            feedback = self.reviewer.format_feedback(review_result)

            # 记录审查历史
            state.review_history.append({
                "iteration": state.iteration,
                "score": overall_score,
                "decision": decision,
                "feedback": feedback,
            })

            print(f"\n[Reviewer] 评分: {overall_score}/10, 决策: {decision}")
            print(f"[Reviewer] 关键问题数: {len(review_result.get('critical_issues', []))}")
            print(feedback)

            # 2b: 判断是否达到质量标准
            if overall_score >= self.quality_threshold and decision == "approve":
                state.final_approved = True
                print(f"\n✅ 代码质量达标！评分 {overall_score} >= 阈值 {self.quality_threshold}")
                break

            # 2c: 未达标 — 反馈给Generator修改
            if decision == "reject":
                print(f"\n❌ 审查员判定: 驳回。代码需要大幅重写。")
            else:
                print(f"\n🔄 评分未达标 ({overall_score} < {self.quality_threshold})，继续修改...")

            # 提取关键反馈
            critical_feedback = review_result.get("summary", "")
            if review_result.get("critical_issues"):
                critical_feedback += "\n\n关键问题：\n" + "\n".join(
                    f"- {iss}" for iss in review_result["critical_issues"]
                )

            # Generator修改代码
            state.current_code = self.generator.revise(
                requirement, state.current_code, critical_feedback
            )
            print(f"[Generator] 修改完成 ({len(state.current_code)} 字符)")

        # Step 3: 输出结果
        if not state.final_approved and state.iteration >= self.max_iterations:
            print(f"\n⚠ 达到最大迭代次数 ({self.max_iterations})，输出当前最佳版本")

        print(f"\n{'=' * 60}")
        print("Reflection Loop 结束")
        print(f"最终迭代次数: {state.iteration}")
        print(f"最终审批状态: {'通过' if state.final_approved else '未通过（强制退出）'}")
        print(f"{'=' * 60}")

        return state

## 5. 四种模式对比总结

| 维度 | 提示链 | 路由 | 并行化 | 自我反思 |
|------|--------|------|--------|---------|
| **核心机制** | 顺序传递上下文 | 意图分类+分发 | 并行执行+聚合 | 生成→审查→迭代 |
| **任务依赖** | 强依赖（串行） | 无依赖（分支） | 无依赖（并行） | 有依赖（循环） |
| **延迟特性** | 累加延迟（最慢） | 单个Handler延迟 | 最慢子任务延迟 | N倍单次延迟 |
| **成本特性** | N次LLM调用 | 1+N次LLM调用 | N次并行LLM调用 | N*M次LLM调用 |
| **复杂度** | 低 | 中 | 中-高 | 高 |
| **适用任务** | 文档处理流水线、ETL | 客服、任务分发 | 大文件分析、多视角审查 | 代码审查、写作润色 |
| **错误处理** | 单步失败影响全局 | 单个Handler隔离 | 部分失败可容忍 | 循环可控退出 |
| **扩展方式** | 插入新步骤 | 添加新Handler | 增加并行度 | 调整迭代策略 |
| **典型场景** | 提取→翻译→摘要 | 客服问题分类处理 | 长文档分块分析 | 代码生成+审查 |

## 6. 小结与选择指南

### 选择决策流程

当面对一个任务时，通过以下问题来决定使用哪种模式：

1. **任务是否可以分解为多个独立的子任务？**
   - 是 → 考虑**并行化**（Sectioning）
   - 否 → 继续下一个问题

2. **任务的步骤之间是否有严格的先后依赖？**
   - 是 → **提示链**
   - 否 → 继续下一个问题

3. **是否需要根据不同输入类型做不同的处理？**
   - 是 → **路由**
   - 否 → 继续下一个问题

4. **任务是否需要多轮迭代改进才能达到满意质量？**
   - 是 → **自我反思**
   - 否 → 单一LLM调用即可，不需要设计模式

### 实践建议

| 阶段 | 模式 | 建议 |
|------|------|------|
| 初学者 | 提示链 | 从简单的两步骤链条开始，逐步增加步骤 |
| 进阶 | 路由 + 并行化 | 先实现路由，再为高延迟Handler添加并行化 |
| 高级 | 自我反思 + 组合模式 | 在提示链的关键环节插入自我反思步骤 |
| 生产 | 全部四种 | 根据场景灵活组合，关注成本控制和超时处理 |

### 关键原则

- **不要过度设计**：简单的单次LLM调用就能解决的问题，不需要引入复杂模式
- **关注成本**：每次LLM调用都有成本（$和延迟），模式选择要与预算匹配
- **监控与回退**：每种模式都需要考虑失败场景和降级策略
- **人类审核节点**：在生产环境中，在关键步骤（如最终输出前）设置人工审核
