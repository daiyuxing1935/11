# 长期记忆：向量数据库与语义检索

## 一、从"记住一切"到"记住对的"

在《记忆模块》一文中，我们讨论了智能体记忆系统的整体架构：短期缓冲区负责当前对话，长期记忆负责跨会话的关键信息。但长期记忆面临一个核心挑战——当智能体积累了成千上万条记忆之后，如何在海量信息中快速找到"与当前任务相关的几条"？

这就是向量数据库和语义检索要解决的问题。

### 传统检索 vs 语义检索

传统检索（如关键词匹配、全文搜索）的局限非常明显：它们只能找到字面上匹配的内容，无法理解语义上的关联。比如用户说"我想学开车"，关键词搜索不会返回"驾照考试流程"或"汽车驾驶技巧"——因为字面上没有重合。而语义检索能理解"学开车"与"驾照"、"驾驶"之间的语义相关性，返回真正有用的内容。

```
传统关键词检索：
"学开车" --关键词匹配--> "开车注意事项" ✓  "驾驶技巧" ✗

语义向量检索：
"学开车" --Embedding--> [0.12, -0.34, ...] --相似度--> "驾驶技巧入门" ✓  "驾照考试流程" ✓
```

## 二、Embedding：文本如何变成向量

### 什么是Embedding

Embedding（嵌入）是将文本、图片、音频等非结构化数据转换为固定长度的数值向量的技术。这些向量不是随机的——它们被训练成能够**在向量空间中反映语义关系**：语义相近的文本，其向量之间的距离也近；语义无关的文本，向量距离则远。

可以把Embedding想象为一种"语义坐标系统"。在这个系统中，"猫"和"狗"的坐标很接近（都是宠物），而"猫"和"汽车"的坐标则相距甚远。

### Embedding模型的训练原理

Embedding模型通常通过大量文本语料训练得到，核心训练目标可以归纳为：

**对比学习（Contrastive Learning）**：让语义相关的文本对在向量空间中拉近，让无关的文本对推远。

```
训练样本：
正例对（语义相近）：("今天天气真好", "阳光明媚的一天")  → 向量距离应小
负例对（语义无关）：("今天天气真好", "数据库连接超时")  → 向量距离应大

损失函数：minimize(正例距离) + maximize(负例距离)
```

### 常用的Embedding模型

| 模型 | 维度 | 特点 | 适用场景 |
|------|------|------|----------|
| OpenAI text-embedding-3-small | 512/1536 | 性价比高，中英文均可 | 通用语义检索 |
| OpenAI text-embedding-3-large | 256/1024/3072 | 精度最高，支持维度截断 | 高精度检索 |
| BGE-M3 (BAAI) | 1024 | 开源，多语言，支持稠密+稀疏 | 中文场景首选 |
| text2vec-large-chinese | 1024 | 中文优化，本地部署 | 纯中文离线场景 |
| Cohere Embed v3 | 1024 | 多语言，支持分类/检索模式 | 企业级应用 |
| Jina Embeddings v3 | 1024 | 支持任务特定LoRA适配 | 多任务微调 |

选择Embedding模型时需要权衡三点：**精度**（检索质量）、**速度**（推理延迟）、**成本**（API费用或GPU资源）。

### Embedding的代码实战

```python
import numpy as np
from openai import OpenAI

# 使用OpenAI Embedding API
client = OpenAI()

def get_embedding(text, model="text-embedding-3-small"):
    """将文本转换为向量"""
    text = text.replace("\n", " ")
    response = client.embeddings.create(
        input=[text],
        model=model
    )
    return response.data[0].embedding

# 示例：计算两段文本的语义相似度
text1 = "智能体需要长期记忆来存储用户偏好"
text2 = "AI助手应该记住用户的个性化设置"

vec1 = get_embedding(text1)
vec2 = get_embedding(text2)

# 余弦相似度
similarity = np.dot(vec1, vec2) / (np.linalg.norm(vec1) * np.linalg.norm(vec2))
print(f"语义相似度: {similarity:.4f}")  # 输出约 0.85+

# 对比无关文本
text3 = "今天中午吃红烧排骨"
vec3 = get_embedding(text3)
similarity_irrelevant = np.dot(vec1, vec3) / (np.linalg.norm(vec1) * np.linalg.norm(vec3))
print(f"无关文本相似度: {similarity_irrelevant:.4f}")  # 输出约 0.3-
```

## 三、相似度度量：如何定义"相关"

有了向量之后，我们需要一种方法来衡量两个向量之间的"距离"或"相似程度"。以下是三种最常用的度量方式。

### 余弦相似度（Cosine Similarity）

衡量两个向量在方向上的相似程度，取值范围[-1, 1]。值越接近1表示方向越一致（越相似）。

```
cos(θ) = (A · B) / (|A| × |B|)

         A · B
= ─────────────────
  sqrt(ΣAi²) × sqrt(ΣBi²)
```

```python
def cosine_similarity(vec_a, vec_b):
    """余弦相似度：关注方向，忽略长度"""
    dot_product = np.dot(vec_a, vec_b)
    norm_a = np.linalg.norm(vec_a)
    norm_b = np.linalg.norm(vec_b)
    return dot_product / (norm_a * norm_b)

# 余弦相似度的特点：
# - 不受向量长度影响（归一化后比较）
# - 适合文本语义比较（文本长度差异大时依然有效）
# - 是Embedding检索中最常用的度量
```

### 欧氏距离（Euclidean Distance）

衡量两个向量在空间中的直线距离。值越小表示越接近。

```
d(A, B) = sqrt( Σ(Ai - Bi)² )
```

```python
def euclidean_distance(vec_a, vec_b):
    """欧氏距离：关注绝对位置差异"""
    return np.sqrt(np.sum((np.array(vec_a) - np.array(vec_b)) ** 2))

# 欧氏距离的特点：
# - 受向量长度影响（长文本的向量范数通常更大）
# - 适合向量已经过L2归一化的场景
# - L2归一化后，欧氏距离与余弦相似度单调相关
```

### 点积（Dot Product）

两个向量对应元素乘积之和。在向量已归一化时等价于余弦相似度。

```python
def dot_product(vec_a, vec_b):
    """点积：简单高效"""
    return np.dot(vec_a, vec_b)
```

### 三种度量的对比与选择

| 度量 | 计算公式 | 值域 | 受长度影响 | 推荐场景 |
|------|---------|------|-----------|---------|
| 余弦相似度 | A·B / (\|A\|\|B\|) | [-1, 1] | 否 | 文本语义检索（首选） |
| 欧氏距离 | sqrt(Σ(Ai-Bi)²) | [0, ∞) | 是 | 图像特征匹配 |
| 点积 | Σ(Ai×Bi) | (-∞, ∞) | 是 | 需高吞吐的近似检索 |

在生产环境中，绝大多数向量数据库默认使用**余弦相似度**，这也是文本语义检索的最佳选择。如果你使用OpenAI的Embedding，它的向量已经做了归一化，此时点积就等于余弦相似度，可以用点积替代以节省计算开销。

## 四、向量索引：大海捞针的艺术

当记忆只有几百条时，暴力扫描所有向量计算相似度完全可行。但当规模增长到百万、亿级时，逐条比对就变得不可接受。向量索引技术通过牺牲少量精度来换取数量级的查询速度提升。

### 暴力搜索（Flat / Brute Force）

```
查询向量Q → 依次与数据库中每条向量的计算相似度 → 排序 → 返回Top-K

时间复杂度：O(N × D)  其中N为数据量，D为维度
适用规模：< 10万条
优点：100%精度（召回率=1.0）
缺点：线性扫描，大数据量下太慢
```

### IVF（Inverted File Index，倒排索引）

IVF的核心思想是"先聚类，再搜索"。在索引构建阶段，用K-Means将向量空间划分为多个区域（聚类中心）。查询时，只需要在与查询向量最近的几个聚类中进行搜索。

```
索引构建阶段：
所有向量 → K-Means聚类 → N个聚类中心（centroids）

查询阶段：
查询向量Q → 找到最近的nprobe个聚类中心 → 仅在这些聚类的向量中搜索 → 返回Top-K

时间复杂度：O((nprobe/N) × N × D) ≈ O(nprobe × avg_cluster_size × D)
```

```python
# IVF索引的伪代码实现
class IVFFlatIndex:
    def __init__(self, nlist=100, nprobe=10):
        self.nlist = nlist      # 聚类数量
        self.nprobe = nprobe    # 查询时探测的聚类数
        self.centroids = []     # 聚类中心
        self.inverted_lists = {} # 倒排列表: {cluster_id: [vector_ids]}
    
    def train(self, vectors):
        """K-Means聚类，生成倒排列表"""
        self.centroids = kmeans(vectors, k=self.nlist)
        for i, vec in enumerate(vectors):
            nearest_cluster = argmin(distance(vec, c) for c in self.centroids)
            self.inverted_lists.setdefault(nearest_cluster, []).append(i)
    
    def search(self, query_vec, top_k=10):
        """只在最近的nprobe个聚类中搜索"""
        # 找到最近的nprobe个聚类
        cluster_scores = [(i, distance(query_vec, c)) 
                          for i, c in enumerate(self.centroids)]
        nearest_clusters = sorted(cluster_scores, key=lambda x: x[1])[:self.nprobe]
        
        # 在候选聚类中暴力搜索
        candidates = []
        for cluster_id, _ in nearest_clusters:
            for vec_id in self.inverted_lists.get(cluster_id, []):
                score = similarity(query_vec, self.vectors[vec_id])
                candidates.append((vec_id, score))
        
        return sorted(candidates, key=lambda x: x[1], reverse=True)[:top_k]
```

**IVF参数调优**：
- `nlist`（聚类数）：建议设为 `4 × sqrt(N)`。太大会增加查询时需要探测的聚类数，太小则每个聚类过大
- `nprobe`（探测数）：越大召回率越高，但速度越慢。生产环境通常设为 `nlist` 的 1%-10%

### HNSW（Hierarchical Navigable Small World）

HNSW是目前工业界综合表现最好的向量索引结构之一，同时也是最精妙的。它构建了一个多层图结构，灵感来自于"六度分隔理论"——在社交网络中，任意两个人之间平均只需要六步就能建立联系。

```
HNSW的多层结构：

Layer 2 (稀疏，长距离边):  ● ─────────────── ●
                             │                  │
Layer 1 (中等密度):         ● ──── ● ──── ● ── ●
                             │     /│      │     │
Layer 0 (最密，全连接):     ●──●──●──●──●──●──●──●

搜索过程（从顶层开始）：
1. 在最高层找到距离查询最近的入口点
2. 逐层下降，每层都做局部贪心搜索
3. 在Layer 0（底层）进行最终的精搜
```

HNSW的核心数据结构是**NSW图（Navigable Small World）**，每个节点与若干"邻居"节点相连。搜索时采用贪心策略：从入口点出发，反复跳到与查询更近的邻居节点，直到无法再接近为止。这种搜索的复杂度是O(log N)。

多个层次的设计让HNSW兼具"快速跳跃"和"精细搜索"的能力——高层图稀疏，边很长，可以快速跨越大片空间；低层图密集，边短，能精确定位到最近的向量。

```python
# HNSW索引的简化实现（仅Layer 0的单层NSW图）
class SimpleHNSW:
    def __init__(self, M=16, ef_construction=200, ef_search=50):
        self.M = M                      # 每个节点的最大连接数
        self.ef_construction = ef_construction  # 构建时的搜索宽度
        self.ef_search = ef_search      # 查询时的搜索宽度
        self.nodes = {}                 # {node_id: vector}
        self.graph = {}                 # {node_id: [neighbor_ids]}
        self.entry_point = None
    
    def insert(self, node_id, vector):
        """插入新节点并建立连接"""
        self.nodes[node_id] = vector
        
        if self.entry_point is None:
            self.entry_point = node_id
            self.graph[node_id] = []
            return
        
        # 搜索最近的M个邻居
        neighbors = self._search_layer(vector, self.ef_construction)
        
        # 建立双向连接
        self.graph[node_id] = neighbors[:self.M]
        for neighbor_id in neighbors[:self.M]:
            self.graph[neighbor_id].append(node_id)
            # 如果邻居连接数超过M，剪枝保留最近的
            if len(self.graph[neighbor_id]) > self.M:
                self._prune_connections(neighbor_id)
    
    def search(self, query_vec, top_k=10):
        """搜索最近的top_k个节点"""
        candidates = self._search_layer(query_vec, self.ef_search)
        return candidates[:top_k]
    
    def _search_layer(self, query_vec, ef):
        """在单层图中执行贪心搜索"""
        visited = set()
        candidates = [(self.entry_point, 
                       cosine_similarity(query_vec, self.nodes[self.entry_point]))]
        results = list(candidates)
        
        while candidates:
            current_id, current_dist = candidates.pop(0)
            
            # 探索当前节点的所有邻居
            for neighbor_id in self.graph.get(current_id, []):
                if neighbor_id in visited:
                    continue
                visited.add(neighbor_id)
                
                dist = cosine_similarity(query_vec, self.nodes[neighbor_id])
                
                # 如果距离更近，加入候选和结果集
                if dist > results[-1][1] or len(results) < ef:
                    candidates.append((neighbor_id, dist))
                    results.append((neighbor_id, dist))
                    results.sort(key=lambda x: x[1], reverse=True)
                    results = results[:ef]
        
        return [r[0] for r in results]
    
    def _prune_connections(self, node_id):
        """剪枝：只保留最近的M个邻居"""
        vec = self.nodes[node_id]
        neighbors = self.graph[node_id]
        scored = [(n, cosine_similarity(vec, self.nodes[n])) for n in neighbors]
        scored.sort(key=lambda x: x[1], reverse=True)
        self.graph[node_id] = [n for n, _ in scored[:self.M]]
```

### 主流向量索引对比

| 索引类型 | 查询速度 | 召回率 | 内存占用 | 构建速度 | 是否支持增量 |
|---------|---------|--------|---------|---------|------------|
| Flat | 基准(1x) | 100% | 低 | 即时 | 是 |
| IVF_FLAT | 快(10-50x) | 90-98% | 中（需存储聚类中心） | 较快 | 需要重建 |
| IVF_PQ | 很快(50-200x) | 85-95% | 低（乘积量化压缩） | 较慢 | 需要重建 |
| HNSW | 很快(50-300x) | 95-99% | 高（需存储图结构） | 慢 | 是 |
| DiskANN | 快(10-100x) | 95-99% | 低（磁盘存储） | 慢 | 离线构建 |

**选择建议**：
- 数据量 < 10万：Flat 暴力搜索，简单高效
- 10万 - 100万 + 内存充裕：HNSW，查询速度最优
- 100万 + 内存有限：IVF_PQ，用精度换空间
- 亿级 + 成本敏感：DiskANN，充分利用廉价磁盘

## 五、向量数据库实战：ChromaDB / Milvus / Pinecone

向量数据库不是普通的数据库——它们是专门为高维向量检索优化的存储系统，在传统数据库的基础上集成了向量索引、相似度计算、元数据过滤等功能。

### ChromaDB：轻量级本地向量库

ChromaDB是Python生态中最容易上手的向量数据库，适合开发阶段和中小规模应用。它可以直接在进程中运行（嵌入式模式），也可以在C/S模式下部署。

```python
import chromadb
from chromadb.utils import embedding_functions
import uuid

# ============================================
# 初始化ChromaDB
# ============================================
# 嵌入式模式（数据持久化到本地目录）
client = chromadb.PersistentClient(path="./chroma_data")

# 或使用内存模式（仅用于测试）
# client = chromadb.Client()

# 定义Embedding函数（使用开源模型，无需API Key）
ef = embedding_functions.SentenceTransformerEmbeddingFunction(
    model_name="BAAI/bge-base-zh-v1.5"  # 中文Embedding模型
)

# 创建或获取集合
collection = client.get_or_create_collection(
    name="agent_long_term_memory",
    embedding_function=ef,
    metadata={"hnsw:space": "cosine"}  # 使用余弦相似度
)

# ============================================
# 存储记忆
# ============================================
def store_memory(content, memory_type="fact", importance=0.5, tags=None):
    """将一段内容存入长期记忆"""
    doc_id = str(uuid.uuid4())
    collection.add(
        ids=[doc_id],
        documents=[content],
        metadatas=[{
            "type": memory_type,
            "importance": importance,
            "tags": ",".join(tags or []),
            "timestamp": str(datetime.datetime.now())
        }]
    )
    return doc_id

# 存储示例记忆
store_memory("用户张三的数据库连接字符串是 postgresql://localhost:5432/mydb",
             memory_type="fact", importance=0.9, tags=["database", "config"])
store_memory("用户偏好使用FastAPI框架开发后端服务",
             memory_type="preference", importance=0.7, tags=["tech_stack"])
store_memory("上次部署时遇到了Docker内存限制问题，最终通过增加swap解决",
             memory_type="experience", importance=0.8, tags=["deployment", "troubleshooting"])
store_memory("用户团队的代码规范要求使用Black格式化，行宽120",
             memory_type="fact", importance=0.6, tags=["coding_standard"])

# ============================================
# 语义检索
# ============================================
def retrieve_memories(query, top_k=5, memory_type=None):
    """根据语义检索相关记忆，支持类型过滤"""
    where_filter = None
    if memory_type:
        where_filter = {"type": memory_type}
    
    results = collection.query(
        query_texts=[query],
        n_results=top_k,
        where=where_filter
    )
    
    memories = []
    for i, doc in enumerate(results["documents"][0]):
        memories.append({
            "id": results["ids"][0][i],
            "content": doc,
            "metadata": results["metadatas"][0][i],
            "distance": results["distances"][0][i]
```

## 六、检索策略优化：不止于向量相似度

单纯依赖向量相似度检索存在天花板——有时候语义上不相关但业务上强相关的记忆会被漏掉，而语义相关但对当前任务无用的记忆却被返回。优秀的检索系统需要多层次策略。

### 混合检索（Hybrid Search）

将向量语义检索与传统关键词检索结合起来，弥补各自的短板。

```python
import numpy as np
from rank_bm25 import BM25Okapi
import jieba

class HybridRetriever:
    """混合检索器：结合向量语义 + 关键词匹配"""
    
    def __init__(self, vector_store, alpha=0.7):
        self.vector_store = vector_store  # 向量数据库
        self.alpha = alpha  # 向量相似度的权重
        self.documents = []  # 文档原文（用于构建BM25索引）
        self.bm25 = None
    
    def index_documents(self, documents):
        """构建关键词检索索引"""
        self.documents = documents
        # 中文分词后构建BM25
        tokenized = [list(jieba.cut(doc)) for doc in documents]
        self.bm25 = BM25Okapi(tokenized)
    
    def search(self, query, top_k=10):
        """混合检索"""
        # 1. 向量语义检索
        vector_results = self.vector_store.search(query, top_k=top_k * 2)
        vector_scores = {r["id"]: r["score"] for r in vector_results}
        
        # 2. 关键词检索（BM25）
        tokenized_query = list(jieba.cut(query))
        bm25_scores = self.bm25.get_scores(tokenized_query)
        bm25_normalized = bm25_scores / (bm25_scores.max() + 1e-8)
        
        # 3. 融合分数：加权求和
        fused = []
        for i, doc_id in enumerate(self.documents.keys()):
            vec_score = vector_scores.get(doc_id, 0.0)
            kw_score = float(bm25_normalized[i])
            
            # RRF（Reciprocal Rank Fusion）融合算法
            fused_score = self.alpha * vec_score + (1 - self.alpha) * kw_score
            fused.append((doc_id, fused_score))
        
        fused.sort(key=lambda x: x[1], reverse=True)
        return fused[:top_k]
```

### 重排序（Re-ranking）

用更强大的模型对初步检索结果进行二次排序，提升最终返回结果的精度。

```python
def rerank_with_cross_encoder(query, candidate_docs):
    """
    使用Cross-Encoder对候选文档进行精排。
    
    与Bi-Encoder（Embedding模型）不同，Cross-Encoder将
    (查询, 文档)作为一对输入，直接输出相关性分数。
    精度更高但计算量更大，因此只对Top-N候选使用。
    """
    from sentence_transformers import CrossEncoder
    
    model = CrossEncoder("BAAI/bge-reranker-v2-m3")
    
    # 构造(查询, 文档)对
    pairs = [(query, doc) for doc in candidate_docs]
    
    # 批量计算相关性分数
    scores = model.predict(pairs)
    
    # 按分数重排
    ranked = sorted(
        zip(candidate_docs, scores), 
        key=lambda x: x[1], 
        reverse=True
    )
    
    return [doc for doc, score in ranked]
```

### 时间衰减与重要性加权

记忆的价值不仅取决于语义相关性，还与时间和重要性有关。

```python
def compute_memory_relevance(memory, query_vec, current_time):
    """
    综合评分 = 语义相似度 × 时间衰减 × 重要性
    
    Returns:
        float: 0-1之间的综合相关性分数
    """
    # 语义相似度（余弦相似度）
    semantic_score = cosine_similarity(query_vec, memory.embedding)
    
    # 时间衰减：记忆越久远，衰减越严重
    age_hours = (current_time - memory.timestamp) / 3600
    time_decay = np.exp(-0.01 * age_hours)  # 衰减系数可调
    # 记忆存储时的原始重要性
    importance = memory.importance  # 0.0 - 1.0
    
    # 综合评分（权重可调）
    final_score = (
        0.5 * semantic_score +
        0.2 * time_decay +
        0.3 * importance
    )
    
    return final_score
```

### 查询改写（Query Rewriting）

用户的原始查询可能不是检索的最佳形式。查询改写可以有效提升召回率。

```python
class QueryRewriter:
    """查询改写器：生成多个视角的查询以提升召回率"""
    
    def rewrite(self, original_query, llm):
        """使用LLM生成查询变体"""
        prompt = f"""
        你是一个搜索查询优化器。请将以下查询改写为3个不同角度的变体，
        以提升检索的召回率。每个变体关注原始查询的不同方面。
        
        原始查询：{original_query}
        
        输出格式（JSON列表）：
        ["变体1", "变体2", "变体3"]
        """
        
        response = llm.generate(prompt)
        variants = json.loads(response)
        
        return [original_query] + variants
    
    def multi_query_retrieve(self, original_query, vector_store, top_k=5, llm=None):
        """
        多查询检索：
        1. 生成多个查询变体
        2. 对每个变体独立检索
        3. 合并去重结果
        """
        # Step 1: 查询改写
        if llm:
            queries = self.rewrite(original_query, llm)
        else:
            # 降级方案：使用同义词替换等规则
            queries = [original_query]
        
        # Step 2: 多路召回
        all_results = {}
        for query in queries:
            results = vector_store.search(query, top_k=top_k)
            for r in results:
                doc_id = r["id"]
                if doc_id not in all_results:
                    all_results[doc_id] = r
                else:
                    # 取最高分
                    all_results[doc_id]["score"] = max(
                        all_results[doc_id]["score"], r["score"]
                    )
        
        # Step 3: 去重排序
        merged = sorted(all_results.values(), 
                       key=lambda x: x["score"], 
                       reverse=True)
        return merged[:top_k]
```

### 上下文窗口分配策略

检索回来的记忆如何放入模型的上下文窗口也是有讲究的。

```python
def format_memories_for_context(memories, max_tokens=2000):
    """
    将检索到的记忆格式化为上下文，智能分配Token预算。
    
    策略：
    - 高相关度记忆获得更多Token配额
    - 每条记忆截断到合理长度
    - 总Token不超出限制
    """
    if not memories:
        return ""
    
    # 按相关性分数归一化，作为Token分配权重
    scores = np.array([m["score"] for m in memories])
    weights = scores / scores.sum()
    
    context_parts = ["## 相关历史记忆\n"]
    remaining_tokens = max_tokens
    
    for i, mem in enumerate(memories):
        # 该记忆的Token预算
        token_budget = int(weights[i] * max_tokens * 0.8)  # 留20%余量
        
        content = mem["content"]
        # 截断到预算范围（简单按字符估算，1 token ≈ 1.5 中文字符）
        char_limit = int(token_budget * 1.5)
        if len(content) > char_limit:
            content = content[:char_limit] + "..."
        
        entry = f"- [{mem['metadata'].get('type', 'memory')}] {content}\n"
        
        # 粗略Token估算
        estimated_tokens = len(entry) // 2
        if remaining_tokens - estimated_tokens < 50:
            break
        
        context_parts.append(entry)
        remaining_tokens -= estimated_tokens
    
    return "\n".join(context_parts)
```

```python
import datetime

import numpy as np
from typing import List, Dict, Optional

class LongTermMemorySystem:
    """

    记忆生命周期：
    [编码] → [存储] → [检索] → [整合] → [遗忘]
    """
    
    def __init__(self, vector_db, embedding_model, llm=None):

        self.embedding_model = embedding_model  # Embedding函数

    def remember(self, content: str, memory_type: str = "general",
                 importance: float = 0.5, tags: List[str] = None,
                 source: str = "conversation"):
        """
        将信息编码并存入长期记忆。
        
        Args:
            content: 记忆内容
            memory_type: 类型（fact/preference/experience/knowledge）

        embedding = self.embedding_model(content)
        
        self.vector_db.store(
            id=doc_id,
            vector=embedding,
            content=content,
            metadata={
                "type": memory_type,
                "importance": importance,
                "tags": ",".join(tags or []),
                "source": source,
                "timestamp": datetime.datetime.now().isoformat(),
                "access_count": 0

               memory_type: str = None,
               min_importance: float = 0.0,
               use_rewriting: bool = True,
               use_reranking: bool = True) -> List[Dict]:
        """
        检索与当前查询相关的记忆。
        
        完整检索流水线：
        Query改写 → 多路召回 → 元数据过滤 → 时间/重要性加权 → 重排序
        """
        # Step 1: 查询改写（可选）
        queries = [query]
        if use_rewriting:
            queries = self.rewriter.rewrite(query, self.llm)
        
        # Step 2: 多路召回
        all_candidates = {}
        for q in queries:
            candidates = self.vector_db.search(
                query=q,
                top_k=top_k * 3,  # 多召回一些用于后续精排
                memory_type=memory_type
            )
            for c in candidates:
                if c["id"] not in all_candidates:
                    all_candidates[c["id"]] = c
                else:
                    all_candidates[c["id"]]["score"] = max(
                        all_candidates[c["id"]]["score"], c["score"]
                    )
        
        candidates = list(all_candidates.values())
        
        # Step 3: 重要性过滤
        if min_importance > 0:
            candidates = [c for c in candidates 
                         if c["metadata"].get("importance", 0) >= min_importance]
        
        # Step 4: 时间衰减加权（非精确匹配模式）
        now = datetime.datetime.now()
        for c in candidates:
            ts = datetime.datetime.fromisoformat(c["metadata"]["timestamp"])
            age_days = (now - ts).days
            time_weight = np.exp(-0.005 * age_days)  # 每200天衰减至-0.37
            c["score"] = c["score"] * time_weight
        
        # Step 5: 重排序（可选）
        if use_reranking and len(candidates) > top_k:
            candidates = self._rerank(query, candidates)
        
        # 排序并截断
        candidates.sort(key=lambda x: x["score"], reverse=True)
        return candidates[:top_k]
    
    def _rerank(self, query, candidates):
        """使用Cross-Encoder精排"""
        from sentence_transformers import CrossEncoder
        model = CrossEncoder("BAAI/bge-reranker-v2-m3")
        pairs = [(query, c["content"]) for c in candidates]
        scores = model.predict(pairs)
        for c, s in zip(candidates, scores):
            c["score"] = float(s)
        return candidates
    
    # ---- 记忆整合到上下文 ----
    def get_context_for_llm(self, query: str, max_tokens: int = 2000) -> str:
        """检索记忆并格式化为LLM上下文"""
        memories = self.recall(query, top_k=5)
        
        if not memories:
            return ""
        
        lines = ["<long_term_memory>",
                  "以下是智能体长期记忆中与当前对话相关的信息："]
        for i, mem in enumerate(memories, 1):
            meta = mem["metadata"]
            lines.append(
                f"{i}. [{meta.get('type', 'general')}] "
                f"{mem['content']} "
                f"(重要性: {meta.get('importance', 0):.1f})"
            )
        lines.append("</long_term_memory>")
        
        return "\n".join(lines)
    
    # ---- 记忆巩固（短期 → 长期） ----
    def consolidate(self, conversation_messages: List[Dict]):
        """
        从对话中提取值得长期保存的信息。
        这是记忆巩固的核心——不是所有对话都值得长期记忆。
        """
        dialogue_text = "\n".join([
            f"{m['role']}: {m['content']}" for m in conversation_messages
        ])
        
        prompt = f"""
        从以下对话中提取需要长期保存的关键信息。只提取以下类型：
        - fact: 事实性信息（如配置、数据、业务规则）
        - preference: 用户偏好（如风格偏好、技术选择）
        - experience: 经验教训（如问题解决方案、注意事项）
        
        忽略日常寒暄、临时性讨论。以JSON数组输出。
        
        对话内容：
        {dialogue_text}
        
        输出格式：
        [
          {{"content": "...", "type": "fact", "importance": 0.9}},
          {{"content": "...", "type": "preference", "importance": 0.7}}
        ]
        """
        
        response = self.llm.generate(prompt)
        extracted = json.loads(response)
        
        for item in extracted:
            self.remember(
                content=item["content"],
                memory_type=item.get("type", "general"),
                importance=item.get("importance", 0.5)
            )
        
        return len(extracted)
    
    # ---- 记忆遗忘 ----
    def forget(self, strategy: str = "automatic", **params):
        """
        执行记忆遗忘策略。
        
        策略：
        - age: 删除超过指定天数的记忆
        - capacity: 超过容量上限时删除最不重要的
        - access: 删除长期未被访问的记忆
        """
        if strategy == "age":
            max_age_days = params.get("max_age_days", 90)
            cutoff = (datetime.datetime.now() - 
                     datetime.timedelta(days=max_age_days)).isoformat()
            self.vector_db.delete_by_filter(
                lambda m: m["timestamp"] < cutoff
            )
        
        elif strategy == "capacity":
            max_count = params.get("max_count", 10000)
            all_memories = self.vector_db.list_all()
            if len(all_memories) > max_count:
                # 按重要性 + 最后访问时间排序，删除分数最低的
                def forget_score(m):
                    imp = m["metadata"].get("importance", 0)
                    access = m["metadata"].get("access_count", 0)
                    return imp * 0.3 + min(access / 10, 1.0) * 0.7
                
                all_memories.sort(key=forget_score)
                to_delete = all_memories[:len(all_memories) - max_count]
                for m in to_delete:
                    self.vector_db.delete(m["id"])
        
        elif strategy == "update":
            # 新信息覆盖旧信息
            old_id = params.get("old_id")
            new_content = params.get("new_content")
            if old_id and new_content:
                self.vector_db.update(old_id, new_content)
```

## 八、最佳实践与常见陷阱

### 最佳实践

1. **选择合适的Embedding维度**：高维度（1024/1536/3072）精度更高但存储和计算成本成倍增加。对于大多数文本检索场景，512-1024维是性价比最优区间。使用OpenAI的`text-embedding-3-small`并截断到512维通常足够。

2. **为记忆添加元数据**：不要只存储向量和原文。元数据（类型、时间戳、重要性、标签）是后续过滤、排序和遗忘的基础。没有元数据的向量数据库只是一个黑盒搜索引擎。

3. **分层检索策略**：先用向量相似度快速召回Top-N（粗排），再用Cross-Encoder精排（精排），最后通过LLM判断相关性（终排）。三层漏斗既能保证速度，又能保证精度。

4. **定期巩固与清理**：设置定时任务，定期（如每天一次）运行记忆巩固和遗忘流程。不要让记忆无限膨胀。每条长期记忆都应该回答一个问题："六个月后，这条信息还有用吗？"

5. **记忆可解释性**：在返回检索结果时，附带来源信息（何时、何种场景记录），让模型（和用户）理解"为什么这条记忆被召回"。

6. **分用户隔离**：使用向量数据库的partition或namespace功能按用户ID隔离数据，避免A用户的记忆污染B用户的检索结果。

### 常见陷阱

1. **Embedding模型与检索场景不匹配**：通用Embedding模型在垂直领域可能表现糟糕。如果你的场景涉及大量专业术语（如医疗、法律、金融），考虑使用领域微调的Embedding模型或在通用模型基础上做适配。

2. **忽视Embedding的对称性问题**：查询和文档的Embedding用同一个模型编码不一定最优。对于短查询 vs 长文档的非对称场景，使用专门的非对称模型（如E5、Instructor）效果更好。

3. **向量维度与索引参数不调优**：高维向量（>2048）在使用IVF索引时需要注意"维度灾难"——高维空间中最近邻和随机点的距离趋于相同，聚类效果会退化。此时HNSW是更安全的选择。

4. **仅依赖语义相似度**：前面提到过的"混合检索"章节已经说明了这个问题。对于包含数字、代码、ID等精确信息的检索需求，必须结合关键词匹配。

5. **忽视冷启动问题**：新部署的记忆系统没有数据，前几次检索结果会很糟糕。可以预先注入种子记忆（如常见问题、基础配置），或在新系统上线初期降低检索阈值。

6. **引用失效（Citation Dilution）**：随着记忆增长，早期高价值记忆可能被不断新增的中低价值记忆"挤"出Top-K结果。解决方案是结合重要性评分做加权，而不是纯依赖相似度排序。
