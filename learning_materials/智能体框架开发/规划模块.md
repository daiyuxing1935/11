# 规划模块：任务分解与执行规划

## 一、智能体为什么需要规划

如果让你"写一份年度市场分析报告"，你不会不假思索地打开Word就开始敲字。你会先拆解任务：收集数据→分析趋势→制作图表→撰写正文→排版检查。这个过程就是**规划（Planning）**。

对于AI智能体来说，规划同样至关重要。设想一个智能体接收到"帮我预订下周三去上海的机票和酒店"这个指令。如果它不先规划，可能会随机执行——先查酒店、再查机票但不考虑时间衔接、最后发现预算不足。有了规划，它会先理解需求、然后并行查询机票和酒店、对比后做出推荐、最后确认预订。

**规划的三大价值**：

1. **降低复杂性**：将大任务拆分为可控的小步骤
2. **提高成功率**：提前发现潜在的依赖和冲突
3. **提升效率**：识别可以并行执行的子任务

## 二、自上而下与自下而上规划

### 自上而下规划（Top-Down）

从最终目标出发，逐层分解为子目标，直到每个子任务都可以直接执行。

```
目标：发布一篇技术博客
├── 子任务1：确定选题和技术方向
├── 子任务2：编写初稿
│   ├── 2.1 写引言
│   ├── 2.2 写核心内容
│   └── 2.3 写总结
├── 子任务3：添加代码示例
├── 子任务4：审校和修改
└── 子任务5：发布和推广
```

这种方式的优势是全局视角清晰，不容易遗漏关键步骤。适合目标明确、结构清晰的任务。

### 自下而上规划（Bottom-Up）

从当前可用的信息和工具出发，逐步构建出通向目标的路径。

```
可用信息：用户偏好、历史数据
可用工具：搜索引擎、数据分析工具、文档生成器
→ 先获取数据 → 分析规律 → 发现主题 → 撰写内容
```

这种方式更加灵活，适合探索性任务——目标可能不够明确，需要边走边看。

### 双向规划

在实际智能体系统中，通常是两种方式的结合：先用自上而下的方式制定高层框架，再用自下而上的方式在执行过程中灵活调整。

## 三、层次化任务分解

**层次化任务分解（Hierarchical Task Decomposition）** 是将复杂任务组织为树状结构，不同层级有不同的抽象程度。

```markdown
**以"开发一个网页应用"为例的三层分解：**

第一层（战略层）：
- 需求分析
- 系统设计
- 编码实现
- 测试部署

第二层（战术层，以"编码实现"为例）：
- 搭建项目框架
- 实现用户认证模块
- 实现核心业务逻辑
- 实现前端界面

第三层（操作层，以"实现用户认证"为例）：
- 创建用户数据表
- 编写注册接口
- 编写登录接口
- 添加JWT Token验证中间件
```

层次化分解的好处在于，智能体可以根据需要选择从哪个层级开始执行。如果高层计划有变，只需要调整受影响的分支，而不需要全部推倒重来。

### 实现层次化分解的技巧

1. **使用编号系统**：如1.0、1.1、1.1.1，体现层次关系
2. **每层控制在3-7个子任务**：太少说明分解不够，太多说明层级可以再细分
3. **叶子节点可执行**：最底层的子任务应该足够具体，可以直接分配执行

## 四、Plan-and-Execute模式

**Plan-and-Execute（先规划后执行）** 是一种经典的智能体行为模式。它的核心思想是：在动手执行之前，先花时间做一个完整的计划。

### 工作流程

```
阶段一：规划（Plan）
输入：用户的任务描述
处理：LLM分析任务，生成详细的执行计划
输出：步骤列表，包含每个步骤的目标、工具、预期结果

阶段二：执行（Execute）
输入：执行计划
处理：按计划逐步执行，监控每步结果
输出：每一步的执行结果

阶段三：（可选）总结
将各步结果整合为最终输出
```

### Plan-and-Execute的规划Prompt示例

```markdown
你是一个任务规划助手。请为以下任务制定一个详细的执行计划。

任务：{user_task}

请按以下格式输出计划：
1. **任务分析**：简要分析任务的核心需求和挑战
2. **执行步骤**：列出每一个执行步骤，包括：
   - 步骤目标：这一步要达成什么
   - 所需工具/资源：需要调用什么工具或访问什么数据
   - 预期产出：这一步完成后应该得到什么
3. **依赖关系**：标注哪些步骤之间有先后依赖关系
4. **风险评估**：指出可能出现的问题及应对方案
```

### Plan-and-Execute的优缺点

**优点**：
- 思路清晰，执行过程有条不紊
- 适合需要多步骤协调的复杂任务
- 计划可以人工审查后再执行

**缺点**：
- 对动态变化的适应性较差
- 如果初始计划有误，可能浪费大量步骤
- 不适合需要频繁根据中间结果调整方向的任务

## 五、ReAct模式

**ReAct（Reasoning + Acting）** 是另一种行为模式，它将推理和行动交替进行，而不是先规划再执行。

### ReAct循环

```
Thought: 我需要知道用户所在城市的天气才能给出穿衣建议
Action: 调用 get_weather("上海")
Observation: 上海今天晴，15-22°C
Thought: 我知道天气了，现在可以给出建议
Action: 查询穿衣建议知识库(温度范围="15-22°C", 天气="晴")
Observation: 建议穿长袖T恤加薄外套
Thought: 信息足够了，可以给出最终建议
Answer: 上海今天15-22°C，晴天。建议穿长袖T恤配一件薄外套，早晚可能需要。
```

ReAct的每一步都遵循**思考-行动-观察**的模式，像一个不断思考的探索者。

### ReAct的优缺点

**优点**：
- 灵活适应，能根据中间结果实时调整
- 适合信息不完整、需要逐步探索的任务
- 出错时可以及时发现并纠正

**缺点**：
- 可能偏离主要目标
- 缺乏全局视角，容易陷入局部优化
- 执行效率可能较低（反复思考和调整）

## 六、何时使用哪种模式

| 场景特征 | 推荐模式 | 原因 |
|---------|---------|------|
| 目标明确、步骤清晰 | Plan-and-Execute | 提前规划避免走弯路 |
| 信息不完整、需要探索 | ReAct | 灵活调整，边做边看 |
| 步骤之间有强依赖 | Plan-and-Execute | 需要全局规划依赖顺序 |
| 环境动态变化 | ReAct | 能适应变化的条件 |
| 需要人工审查 | Plan-and-Execute | 可以在执行前审核计划 |
| 简单快速的查询 | ReAct | 直接推理-行动更高效 |

### 混合策略

在复杂系统中，最好的做法往往是将两者结合：

```python
def hybrid_agent(task):
    # 第一阶段：宏观规划
    plan = llm.plan(task)
    
    # 第二阶段：逐步执行（每个步骤内部使用ReAct）
    results = []
    for step in plan.steps:
        result = react_execute(step)  # 每个步骤内部灵活执行
        results.append(result)
        
        # 检查是否需要调整剩余计划
        if need_replan(result):
            plan = llm.replan(plan, results)
    
    return synthesize_results(results)
```

## 七、处理计划失败与重新规划

计划赶不上变化。当执行过程中遇到问题时，智能体需要能够重新规划。

### 触发重新规划的常见情况

- 某个步骤执行失败且不可跳过
- 发现了初始计划未考虑的重要信息
- 用户中途改变了需求
- 外部条件发生变化

### 重新规划的Prompt设计

```markdown
原始任务：{original_task}
原始计划：{original_plan}
已完成步骤及结果：{completed_steps}
遇到问题：{problem_description}

请基于以上信息重新规划剩余任务的执行方案。如果需要，可以调整未执行步骤的顺序、内容，甚至提出全新的方案。
```

### 本地修复 vs 全局重规划

- **本地修复**：只有当某个步骤失败时，尝试为该步骤寻找替代方案，不改动其他步骤。代价小，适合局部问题
- **全局重规划**：当问题影响到整体计划的可行性时，从当前状态出发完全重新规划剩余部分。代价大，但有时是必要的

## 八、规划模块的实现要点

1. **计划的结构化存储**：使用JSON或类似格式存储计划，而不是纯文本。这样方便检查和修改计划中的特定步骤

2. **进度追踪**：每个步骤设置状态标记（pending/in_progress/completed/failed），方便智能体了解当前进度

3. **超时和步数限制**：设置计划执行的最大步数和时间限制，防止智能体陷入无限循环

4. **人机协作**：在关键步骤执行前，允许人工确认。特别是涉及资金、数据删除等敏感操作

5. **计划的可视化和日志**：将计划的执行过程记录下来，方便事后分析和优化

规划能力是智能体从"简单响应者"升级为"自主行动者"的标志。一个好的规划模块能让智能体处理那些需要10步、20步才能完成的复杂任务，而不仅仅是"一问一答"的简单交互。
