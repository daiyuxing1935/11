# 记忆增强：知识图谱与混合检索

## 一、从向量记忆到结构化记忆

基于向量嵌入的语义记忆在模糊语义匹配时很强大，但在**精确的关系推理**上存在短板。例如用户问"上次和产品组开会提到的性能优化方案，后来是谁负责跟进的？"——向量检索只能找到语义相似的内容，无法理解"产品组开会"、"性能优化方案"、"负责跟进"之间的结构化关系。

**知识图谱（Knowledge Graph）** 以结构化方式存储实体及其关系，让智能体不仅能"模糊回忆"，还能"精确推理"。

| 维度 | 向量语义记忆 | 知识图谱记忆 |
|------|------------|------------|
| 存储形式 | 高维浮点数向量 | 实体-关系-实体三元组 |
| 检索方式 | 余弦相似度 | 图遍历 / Cypher查询 |
| 优势场景 | 模糊语义匹配、长文本理解 | 精确关系推理、多跳查询 |
| 推理能力 | 弱（仅相似度排序） | 强（路径遍历、图算法） |
| 可解释性 | 低（"黑盒"向量） | 高（可视化关系链路） |

两者是互补关系——这也正是本章的核心：**知识图谱 + 向量数据库的混合检索**。

## 二、知识图谱基础：实体-关系三元组

知识图谱的最小单元是**三元组（Triple）**，格式为 `(主体)--[谓词]-->(客体)`，即经典的 RDF 模型：

```
(张三) --[就职于]--> (XX科技公司)
(XX科技公司) --[位于]--> (北京)
(张三) --[参与项目]--> (智能客服系统)
(智能客服系统) --[使用框架]--> (LangGraph)
```

大量三元组聚合为**有向图**：实体（Entity）是节点（Node），关系（Relation）是边（Edge），属性（Attribute）附加在节点或边上。

```
                    ┌─────────────┐
                    │  XX科技公司   │
                    └──────┬──────┘
              ┌────────────┼────────────┐
              │ 位于        │            │ 拥有部门
              ▼            ▼            ▼
         ┌────────┐  ┌─────────┐  ┌──────────┐
         │  北京   │  │就职于──▶│  │  技术部   │
         └────────┘  │         │  └────┬─────┘
                     │         │       │ 属于
                     ▼         │       ▼
              ┌──────────┐    │  ┌──────────┐
              │   张三    │◀───┘  │   李四    │
              └────┬─────┘       └────┬─────┘
                   │ 参与项目          │ 参与项目
                   ▼                  ▼
              ┌──────────────────────────────┐
              │        智能客服系统            │
              └──────────┬───────────────────┘
                         │ 使用框架
                         ▼
                   ┌────────────┐
                   │  LangGraph  │
                   └────────────┘
```

通过这张图，智能体可回答："张三在哪个城市工作？"（两跳）、"张三和李四的共同点？"（共同参与智能客服系统）、"智能客服系统用了什么框架？"（单跳）。这些多跳推理查询正是向量数据库难以高效完成的。

## 三、图数据库：Neo4j实战

关系型数据库也能存三元组，但多跳查询需要多次 JOIN，复杂度随跳数指数增长，而图数据库原生支持图遍历。**Neo4j** 使用 **Cypher** 查询语言，用 `(node)-[relation]->(node)` 表示路径。

```python
from neo4j import GraphDatabase

class Neo4jMemoryStore:
    def __init__(self, uri, user, password):
        self.driver = GraphDatabase.driver(uri, auth=(user, password))
    
    def add_triple(self, s, r, o):
        with self.driver.session() as sess:
            sess.run("""MERGE (s:Entity {name: $s}) MERGE (o:Entity {name: $o})
                        MERGE (s)-[r:RELATION {type: $r}]->(o)""", s=s, r=r, o=o)
    
    def query_relation(self, subject, relation):
        with self.driver.session() as sess:
            result = sess.run("""MATCH (s:Entity {name: $s})-[r:RELATION {type: $r}]->(o)
                                 RETURN o.name AS t""", s=subject, r=relation)
            return [r["t"] for r in result]
    
    def find_common(self, a, b):
        with self.driver.session() as sess:
            result = sess.run("""MATCH (a:Entity {name: $a})-[r1]->(c)<-[r2]-(b:Entity {name: $b})
                                 RETURN c.name AS c, r1.type AS ra, r2.type AS rb""", a=a, b=b)
            return [dict(r) for r in result]

# 使用：多跳查询
store = Neo4jMemoryStore("bolt://localhost:7687", "neo4j", "password123")
store.add_triple("张三", "就职于", "XX科技公司")
store.add_triple("XX科技公司", "位于", "北京")
store.add_triple("张三", "参与项目", "智能客服系统")
store.add_triple("李四", "参与项目", "智能客服系统")

company = store.query_relation("张三", "就职于")
location = store.query_relation(company[0], "位于")
print(f"张三在{location[0]}工作")  # 输出：张三在北京工作
```

## 四、从对话中抽取知识

知识图谱不会自动构建——需要从非结构化文本中提取三元组，这个过程称为**知识抽取（Knowledge Extraction）**。

### 基于LLM的知识抽取

```python
class KnowledgeExtractor:
    def __init__(self, llm):
        self.llm = llm
        self.prompt = """从对话中抽取(主体, 关系, 客体)三元组。
规则：只抽取事实性信息；主体和客体是具体实体；关系简洁明确（"就职于"、"参与"、"使用"）；
用户偏好用"偏好"、"要求"作为关系。每行一个三元组，格式：主体 | 关系 | 客体

对话：{dialogue}
输出三元组（每行一个，|分隔）："""
    
    def extract(self, dialogue):
        resp = self.llm.generate(self.prompt.format(dialogue=dialogue))
        triples = []
        for line in resp.strip().split("\n"):
            parts = [p.strip() for p in line.split("|")]
            if len(parts) == 3:
                triples.append(tuple(parts))
        return triples

# 示例
extractor = KnowledgeExtractor(llm)
dialogue = """用户：我叫王芳，是后端开发工程师，在阿里巴巴工作。
用户：我们团队在做一个叫"凤凰"的微服务治理平台，用Go语言。"""
triples = extractor.extract(dialogue)
# [('王芳', '就职于', '阿里巴巴'), ('王芳', '职位', '后端开发工程师'),
#  ('王芳', '参与项目', '凤凰微服务治理平台'), ('凤凰微服务治理平台', '使用语言', 'Go')]
```

### 知识冲突与更新

新知识可能与已有知识冲突（同一主体同一关系但不同客体），需要解决策略：

```python
class KnowledgeResolver:
    def resolve(self, graph, s, r, new_o):
        existing = graph.query_relation(s, r)
        if not existing:
            graph.add_triple(s, r, new_o)
            return {"action": "added"}
        if new_o in existing:
            return {"action": "skipped"}
        # 冲突：保留最新，旧关系归档为"曾{relation}"
        for old in existing:
            graph.add_triple(s, f"曾{r}", old)
        graph.delete_relation(s, r, existing)
        graph.add_triple(s, r, new_o)
        return {"action": "updated", "old": existing, "new": new_o}
```

## 五、知识图谱 + 向量数据库混合检索

**混合检索（Hybrid Search）** 将图谱的精确推理与向量数据库的语义理解结合。

### 混合检索架构

```
                      用户查询
                         │
              ┌──────────┴──────────┐
              │    查询分析与路由     │
              └──────────┬──────────┘
         ┌───────────────┼───────────────┐
         ▼               ▼               ▼
   ┌──────────┐   ┌──────────┐   ┌──────────┐
   │ 向量检索  │   │ 图谱检索  │   │ 混合融合  │
   │(Chroma)  │   │(Neo4j)   │   │(两者都查) │
   └────┬─────┘   └────┬─────┘   └────┬─────┘
        └───────────────┼──────────────┘
                        ▼
                 ┌─────────────┐
                 │ 结果融合排序  │
                 └──────┬──────┘
                        ▼
                   最终检索结果
```

### 代码实现

```python
import uuid
from chromadb import Client

class HybridMemoryStore:
    def __init__(self, neo4j_store, chroma_client, embed_fn):
        self.graph = neo4j_store
        self.embed = embed_fn
        self.col = chroma_client.get_or_create_collection("agent_memory")
    
    def store(self, content, relations=None):
        did = str(uuid.uuid4())
        self.col.add(ids=[did], embeddings=[self.embed(content)], documents=[content])
        if relations:
            for s, r, o in relations:
                self.graph.add_triple(s, r, o)
    
    def hybrid_search(self, query, top_k=5):
        vr = self.col.query(query_embeddings=[self.embed(query)], n_results=top_k)
        gr = self._graph_search(query)
        return self._fuse(vr, gr)[:top_k]
    
    def _graph_search(self, q):
        with self.graph.driver.session() as sess:
            result = sess.run("""MATCH (n:Entity)-[r:RELATION]-(m:Entity)
                WHERE n.name CONTAINS $kw RETURN n.name AS e, r.type AS r, m.name AS m LIMIT 20""", kw=q)
            return [(r["e"], r["r"], r["m"]) for r in result]
    
    def _fuse(self, vr, gr):
        fused = {}
        for doc in vr.get("documents", [[]])[0]:
            fused[doc] = {"c": doc, "s": 0.7, "g": 0.0}
        for e, r, m in gr:
            text = f"{e}的{r}是{m}"
            if text in fused:
                fused[text]["g"] = 0.5
            else:
                fused[text] = {"c": text, "s": 0.1, "g": 0.8}
        for v in fused.values():
            v["f"] = 0.5 * v["s"] + 0.5 * v["g"]
        return [i["c"] for i in sorted(fused.values(), key=lambda x: x["f"], reverse=True)]
```

### 三种混合检索模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **串行模式** | 先图谱检索，不足再向量补充 | 查询有明确实体指向 |
| **并行模式** | 同时查两边，结果融合排序 | 查询意图不明确 |
| **级联模式** | 向量粗筛，图谱精排验证 | 需要高精度结果 |

## 六、知识的动态更新与维护

知识会过时、关系会变化。一个静态图谱很快变成"死图"，需要完整的生命周期管理：

```
   ┌─────────┐   验证通过   ┌─────────┐
   │  候选态  │ ──────────▶ │  活跃态  │
   └────┬────┘             └────┬────┘
        │ 失败                  │ 长期未访问/过时
        ▼                       ▼
   ┌─────────┐             ┌─────────┐
   │  已拒绝  │             │  归档态  │
   └─────────┘             └────┬────┘
                                │ 确认过期
                                ▼
                           ┌─────────┐
                           │  已删除  │
                           └─────────┘
```

```python
from datetime import datetime, timedelta

class KnowledgeLifecycle:
    def __init__(self, graph_store):
        self.graph = graph_store
    
    def add_candidate(self, s, r, o, confidence=0.5):
        """低置信度知识以候选态存入"""
        with self.graph.driver.session() as sess:
            sess.run("""MERGE (s:Entity {name: $s}) MERGE (o:Entity {name: $o})
                MERGE (s)-[r:RELATION {type: $r}]->(o)
                SET r.status='candidate', r.confidence=$c,
                    r.created_at=datetime(), r.last_accessed=datetime()""",
                s=s, r=r, o=o, c=confidence)
    
    def promote(self, s, r, o):
        with self.graph.driver.session() as sess:
            sess.run("""MATCH (s:Entity {name: $s})-[r:RELATION {type: $r}]->(o:Entity {name: $o})
                        SET r.status='active', r.confidence=1.0""", s=s, r=r, o=o)
    
    def decay_check(self, days=90):
        with self.graph.driver.session() as sess:
            t = (datetime.now() - timedelta(days=days)).isoformat()
            return sess.run("""MATCH ()-[r:RELATION]->()
                WHERE r.last_accessed < $t AND r.status='active'
                SET r.status='archived' RETURN count(r) AS cnt""", t=t).single()["cnt"]
```

## 七、在Agent中的应用场景

### 场景一：个性化记忆Agent

用户长期交互后，图谱积累偏好信息，每次对话开始时检索并注入System Prompt：

```
(用户) --[偏好]--> (简洁回答风格)
(用户) --[使用]--> (Python)
(用户) --[关注项目]--> (智能客服系统)
```

### 场景二：企业知识助手（GraphRAG）

传统 RAG 基于向量检索存在"语义碎片化"问题——检索片段缺乏上下文关联。**GraphRAG** 加入图谱层：向量检索获取 Top-K 片段后，再通过实体识别在图谱中查询关系上下文，合并后交给 LLM 生成。

```python
class EnterpriseKnowledgeAgent:
    def __init__(self, graph_store, llm):
        self.graph = graph_store
        self.llm = llm
    
    def answer(self, question):
        with self.graph.driver.session() as sess:
            result = sess.run("""MATCH (n:Entity)-[r:RELATION]-(m:Entity)
                WHERE any(w IN split($q,' ') WHERE n.name CONTAINS w)
                RETURN n.name AS e, r.type AS r, m.name AS m LIMIT 20""", q=question)
            ctx = "\n".join(f"- {r['e']} {r['r']} {r['m']}" for r in result)
        return self.llm.generate(f"基于知识图谱回答：\n{ctx}\n\n问题：{question}")
```

### 场景三：多Agent共享记忆与上下文追踪

不同Agent掌握不同信息片段，知识图谱充当共享记忆总线：

```
Agent A: 用户反馈微服务治理平台高峰响应慢
  → 存入: (微服务治理平台) --[存在问题]--> (高峰响应慢)

Agent B: 查询图谱发现未解决问题，排查后存入
  → (高峰响应慢) --[原因为]--> (数据库连接池不足)

Agent C: 查询本周问题+解决方案 → 自动生成周报
```

此外，在单次复杂对话中，知识图谱可跟踪涉及的实体和关系变化。当用户切换话题时，Agent 自动关联到之前讨论过的实体，保持对话的连贯性。

## 八、设计知识图谱记忆的最佳实践

1. **从简单开始，逐步丰富**：不需要一开始就构建完美的本体论（Ontology）。从简单三元组开始，让图谱随使用演化出实体类型和关系类型

2. **知识抽取需要质量控制**：LLM抽取的三元组质量参差不齐。引入置信度评分，高置信度自动入库，低置信度进入审核队列

3. **图谱不是越大越好**：大型图谱查询性能会下降，需要合理的索引和分区策略

4. **向量和图谱互补而非替代**：事实性、关系型信息走图谱；描述性、叙事性内容走向量。两者各司其职

5. **注意知识时效性**：为每条知识添加时间戳和有效期。周期性运行过期检查，清理或归档过时知识

6. **实体消歧是持续挑战**：同一事物有不同名称。利用LLM辅助判断两个实体是否指向同一事物，定期合并

7. **隐私与权限控制**：知识图谱可能包含敏感信息。实现细粒度访问控制，明确哪些Agent可读写哪些实体/关系

8. **可观测性**：记录知识来源（哪次对话）、变更历史。当Agent答案不符合预期时，能追溯到具体是哪条知识出了问题

---

知识图谱让智能体的记忆从"模糊的印象"升级为"清晰的知识"。向量检索告诉智能体"这可能是相关的"，知识图谱告诉它"这就是你要找的，而且它和X有关，又影响了Y"。两者结合，智能体的记忆系统才真正接近了人类记忆的完整能力——既能模糊联想，又能精确推理。
