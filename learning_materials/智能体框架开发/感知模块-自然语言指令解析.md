# 感知模块：自然语言指令的精准解析

## 一、感知模块在智能体架构中的角色

如果把智能体比作一个人，那么**感知模块（Perception Module）** 就是它的"耳朵"和"阅读理解能力"。用户说出的每一句话、输入的每一段文字，都需要经过感知模块的处理，才能转化为智能体可以理解和执行的结构化信息。

没有精准的感知，智能体就像一个"听不懂人话"的助手——你让它"查一下苹果的股价"，它可能去搜索水果价格；你说"帮我订昨天的那个餐厅"，它不知道"那个"指的是什么。

感知模块的核心任务可以概括为四个关键能力：

```
用户原始输入（自然语言）
        ↓
┌───────────────────────────────────────┐
│           感 知 模 块                  │
│                                       │
│  ┌─────────┐  ┌─────────┐            │
│  │意图识别 │  │槽位填充 │            │
│  └─────────┘  └─────────┘            │
│                                       │
│  ┌─────────┐  ┌─────────┐            │
│  │实体抽取 │  │指代消解 │            │
│  └─────────┘  └─────────┘            │
│                                       │
└───────────────────────────────────────┘
        ↓
结构化语义表示（意图 + 槽位 + 实体）
        ↓
    传递给下游模块（规划 / 工具调用）
```

## 二、意图识别：理解用户到底想做什么

**意图识别（Intent Recognition / Intent Classification）** 是感知模块的起点。它的任务是回答一个问题：用户在说什么类型的指令？

### 2.1 意图的本质

意图是一句话的"动作类型"。例如：

| 用户输入 | 意图 | 置信度 |
|---------|------|--------|
| "今天天气怎么样" | `query_weather` | 0.97 |
| "帮我订一张去北京的机票" | `book_flight` | 0.95 |
| "我的订单到哪了" | `track_order` | 0.92 |
| "你们支持微信支付吗" | `ask_payment_method` | 0.88 |
| "太贵了，有没有便宜的" | `filter_by_price` | 0.79 |

意图识别本质上是一个**短文本分类**任务。在智能体场景中，意图体系通常由业务方提前定义好。一般来说，意图数量控制在20-50个之间比较合理——太少覆盖不全，太多容易混淆。

### 2.2 基于LLM的意图识别

在大模型时代，意图识别不再需要单独训练分类模型。你可以在Prompt中定义意图体系，让LLM直接判断：

```python
class IntentRecognizer:
    """基于LLM的意图识别器"""
    
    def __init__(self, llm):
        self.llm = llm
        self.intent_schema = """
你是一个意图识别助手。请分析用户输入，判断其意图。

可用意图列表：
- query_weather: 查询天气信息
- book_flight: 预订机票
- book_hotel: 预订酒店
- track_order: 查询订单状态
- ask_faq: 询问常见问题
- cancel_order: 取消订单
- modify_reservation: 修改预订
- chitchat: 闲聊

请以JSON格式返回：
{
  "intent": "意图名称",
  "confidence": 0.0-1.0,
  "reasoning": "判断依据"
}
"""
    
    def recognize(self, user_input: str) -> dict:
        prompt = f"{self.intent_schema}\n\n用户输入：{user_input}"
        response = self.llm.generate(prompt)
        return json.loads(response)
    
    def recognize_with_fallback(self, user_input: str) -> str:
        """带歧义处理的意图识别"""
        result = self.recognize(user_input)
        
        # 如果置信度低于阈值，触发澄清对话
        if result["confidence"] < 0.7:
            return self._generate_clarification(user_input, result)
        
        return result["intent"]
    
    def _generate_clarification(self, user_input: str, result: dict) -> str:
        """让LLM生成一个澄清问题"""
        prompt = f"""
用户输入"{user_input}"的意图不明确（当前判断为{result['intent']}，
置信度{result['confidence']}）。请生成一个简短友好的澄清问题。
"""
        return self.llm.generate(prompt)
```

### 2.3 多意图处理

用户的一句话可能包含多个意图。例如："帮我查一下明天的天气，如果下雨的话就推迟会议"。这句话包含两个意图：`query_weather`（查天气）和 `reschedule_meeting`（改会议时间），且两者有条件依赖关系。

```python
class MultiIntentRecognizer:

- intent: 意图名称
- span: 对应的原文片段
- dependency: 对其他意图的依赖关系（如有）

用户输入：{user_input}

请以JSON数组格式返回。
"""
        response = self.llm.generate(prompt)
        return json.loads(response)
```

## 三、槽位填充：从指令中提取结构化参数

如果把意图比作"函数名"，那么**槽位（Slot）** 就是"函数参数"。槽位填充的任务是在用户输入的文本中，找到意图所需的每一个参数值。

### 3.1 槽位定义

每个意图都有一套预定义的槽位。以`book_flight`为例：


### 3.2 基于LLM的槽位填充

```python
class SlotFiller:
    """从用户输入中提取槽位值"""
    
    def __init__(self, llm):
        self.llm = llm
    
    def fill_slots(
        self, user_input: str, intent: str, slot_schema: dict
    ) -> dict:
        """
        填充指定意图的槽位。
        返回 {slot_name: value} 的字典，未填充的槽位值为 None。
        """
        schema_desc = json.dumps(slot_schema, ensure_ascii=False, indent=2)
        
        prompt = f"""
意图：{intent}
槽位定义：{schema_desc}

请从以下用户输入中提取槽位值。只提取有明确信息的槽位，
不要猜测或编造任何值。对于未提及的槽位，值设为 null。

用户输入：{user_input}

请以JSON格式返回，key为槽位名称，value为提取到的值或null。
"""
        response = self.llm.generate(prompt)
        extracted = json.loads(response)
        return extracted
    
    def check_completeness(self, filled_slots: dict, slot_schema: dict) -> list:
        """检查哪些必填槽位尚未填充，返回缺失槽位列表"""
        missing = []
        for slot_name, slot_def in slot_schema.items():
            if slot_def.get("required") and not filled_slots.get(slot_name):
                missing.append({
                    "slot": slot_name,
                    "prompt": slot_def.get("prompt", f"请提供{slot_def['description']}")
                })
        return missing
    
    def multi_turn_fill(self, user_input: str, intent: str, 
                        slot_schema: dict, history_slots: dict = None) -> dict:
        """支持多轮对话的槽位填充"""
        # 1. 从当前输入提取槽位
        current_slots = self.fill_slots(user_input, intent, slot_schema)
        
        # 2. 合并历史槽位（当前值优先）
        merged = {**history_slots} if history_slots else {}
        for key, value in current_slots.items():
            if value is not None:
                merged[key] = value
        
        # 3. 检查完整性，返回缺失提示
        missing = self.check_completeness(merged, slot_schema)
        
        return {
            "slots": merged,
            "missing_slots": missing,
            "is_complete": len(missing) == 0
        }
```

### 3.3 槽位规范化

从自然语言中提取的值往往是口语化的，需要规范化为结构化格式。

```python
class SlotNormalizer:
    """将槽位值规范化为标准格式"""
    
    def normalize_date(self, raw_date: str) -> str:
        """日期规范化"""
        prompt = f"""
将以下自然语言日期转换为 "YYYY-MM-DD" 格式。
如果表达不完整（如"下周三"），假设今天是2026-07-09进行推算。

输入：{raw_date}
输出：只返回日期字符串。
"""
        return self.llm.generate(prompt).strip()
    
    def normalize_city(self, raw_city: str) -> dict:
        """城市名规范化（消歧义）"""
        prompt = f"""
将以下自然语言中的城市名转换为标准格式。
输出JSON：{{"city": "标准名", "city_code": "三字码", "country": "国家"}}

输入：{raw_city}
"""
        response = self.llm.generate(prompt)
        return json.loads(response)
    
    def normalize(self, slot_name: str, raw_value: str) -> str:
        """统一规范化入口"""
        normalizers = {
            "departure_date": self.normalize_date,
            "return_date": self.normalize_date,
            "departure_city": lambda v: self.normalize_city(v)["city_code"],
            "arrival_city": lambda v: self.normalize_city(v)["city_code"],
        }
        normalizer = normalizers.get(slot_name)
        return normalizer(raw_value) if normalizer else raw_value
```

## 四、实体抽取：识别文本中的关键信息单元

**实体抽取（Named Entity Recognition, NER）** 和槽位填充有重叠但侧重点不同。槽位填充是"面向意图的参数提取"，而实体抽取是"面向文本的通用信息提取"，不依赖于具体意图。

### 4.1 实体类型体系

常见的实体类型包括：

```python
ENTITY_TYPES = {
    "PERSON": "人名",
    "ORG": "组织/公司名",
    "LOC": "地点",
    "DATE": "日期",
    "TIME": "时间",
    "MONEY": "金额",
    "PERCENT": "百分比",
    "PRODUCT": "产品名",
    "PHONE": "电话号码",
    "EMAIL": "邮箱地址",
    "URL": "网址",
    "ID": "订单号/证件号等标识符",
    "QUANTITY": "数量",
}
```

### 4.2 基于LLM的NER实现

```python
class EntityExtractor:
    def __init__(self, llm):
        self.llm = llm
    
    def extract(self, text: str) -> list:
        prompt = f"""
从以下文本中提取所有命名实体。对于每个实体，标注：
- text: 原文中的实体文本
- type: 实体类型（PERSON / ORG / LOC / DATE / TIME / MONEY / PRODUCT / ID）
- start: 在原文本中的起始位置（字符索引）
- end: 在原文本中的结束位置（字符索引）

文本：{text}

以JSON数组格式返回。
"""
        response = self.llm.generate(prompt)
        return json.loads(response)
    
    def extract_with_relations(self, text: str) -> dict:
        """不仅提取实体，还抽取实体间的关系"""
        prompt = f"""
从以下文本中提取实体及其关系。

文本：{text}

返回格式：
{{
  "entities": [
    {{"id": "E1", "text": "...", "type": "..."}},
  ],
  "relations": [
    {{"subject": "E1", "predicate": "works_at", "object": "E2"}},
  ]
}}
"""
        response = self.llm.generate(prompt)
        return json.loads(response)
```

### 4.3 实体的应用场景

实体信息不只是为了提取而提取。在智能体系统中，实体的价值体现在：

- **工具参数构造**：将`PRODUCT`类型的实体填入搜索工具的query参数
- **上下文增强**：识别到的`PERSON`实体可用于查询用户的历史信息
- **知识图谱联动**：识别出的`ORG`实体可以触发公司信息查询
- **安全过滤**：对`PHONE`、`EMAIL`等敏感实体进行脱敏处理

## 五、指代消解：理解"它""那个""这个"到底指什么

**指代消解（Coreference Resolution）** 是感知模块中最容易被忽视但最影响体验的能力。

### 5.1 指代消解的必要性

用户在一段对话中很少每次都完整表述。他们会说：

```
用户：帮我查一下苹果公司的最新财报。             ← 引入实体
用户：它上个季度的营收是多少？                    ← "它" 指"苹果公司"
用户：和微软的那个比一下。                         ← "微软的那个"指"微软的最新财报"
```

如果智能体不理解"它"指的是"苹果公司"，对话就无法继续。这就是指代消解要解决的问题。

### 5.2 基于对话历史的指代消解

```python
class CoreferenceResolver:
    def __init__(self, llm):
        self.llm = llm
        self.entity_memory = []  # 对话中引入过的实体
    
    def resolve(self, user_input: str, conversation_history: list) -> str:
        """
        将用户输入中的指代词替换为明确的实体。
        返回消解后的文本。
        """
        if not self._has_reference(user_input):
            return user_input
        
        prompt = f"""
请将以下用户消息中的指代词（如"它""他""她""这个""那个""其"等）
替换为对话历史中对应的明确实体。如果某个指代无法确定所指对象，
保留原样并标注不确定性。

对话历史：
{self._format_history(conversation_history)}

用户当前消息：{user_input}

请直接返回消解后的消息文本，不要添加额外解释。
"""
        resolved = self.llm.generate(prompt).strip()
        
        # 更新实体记忆
        self._update_entity_memory(resolved)
        
        return resolved
    
    def _has_reference(self, text: str) -> bool:
        """检测文本中是否包含指代词"""
        reference_words = ["它", "他", "她", "这个", "那个", "这些", "那些",
                          "其", "该", "此", "前者", "后者", "上面", "前面"]
        return any(word in text for word in reference_words)
    
    def _format_history(self, history: list) -> str:
        return "\n".join(
            f"[{msg['role']}]: {msg['content']}" 
            for msg in history[-10:]  # 只取最近10轮
        )
    
    def _update_entity_memory(self, resolved_text: str):
        """从消解后的文本中提取并更新实体记忆"""
        entities = self._extract_key_entities(resolved_text)
        self.entity_memory.extend(entities)
        # 限制记忆大小
        if len(self.entity_memory) > 50:
            self.entity_memory = self.entity_memory[-50:]
    
    def _extract_key_entities(self, text: str) -> list:
        """提取文本中的关键实体"""
        prompt = f"""
提取以下文本中引入的关键实体（人名、组织、产品、专有名词等）。
每个实体返回 {{"name": "实体名", "type": "类型"}}。

文本：{text}
"""
        response = self.llm.generate(prompt)
        return json.loads(response)
```

### 5.3 零指代与省略恢复

中文对话中还存在大量**零指代（Zero Anaphora）**现象——主语或宾语被省略：

```
用户：打开微信。       ← 省略主语（你/系统）
用户：然后发一条消息。  ← 省略主语（你/系统）和工具上下文（在微信里）
```

```python
def resolve_zero_anaphora(user_input: str, last_action: dict) -> str:
    """恢复中文中的零指代省略"""
    prompt = f"""
上一个操作：{json.dumps(last_action, ensure_ascii=False)}

用户的当前输入省略了部分上下文。请补充完整：
当前输入：{user_input}

返回补充后的完整输入。
"""
    return llm.generate(prompt).strip()

# 示例
last_action = {"tool": "open_app", "app": "微信"}
user_input = "然后发一条消息"
resolved = resolve_zero_anaphora(user_input, last_action)
# 结果："然后在微信中发一条消息"
```

## 六、NLU Pipeline：组装完整的理解流程

将上述四个模块串联起来，构成一条完整的**自然语言理解流水线（NLU Pipeline）**：

```python
class NLUPipeline:
    """自然语言理解流水线"""
    
    def __init__(self, llm):
        self.llm = llm
        self.intent_recognizer = IntentRecognizer(llm)
        self.slot_filler = SlotFiller(llm)
        self.entity_extractor = EntityExtractor(llm)
        self.coref_resolver = CoreferenceResolver(llm)
        
        # 意图-槽位映射表
        self.intent_slot_map = {
            "book_flight": FLIGHT_BOOKING_SLOTS,
            "book_hotel": HOTEL_BOOKING_SLOTS,
            "query_weather": WEATHER_QUERY_SLOTS,
        }
    
    def process(
        self, 
        user_input: str, 
        conversation_history: list = None
    ) -> dict:
        """
        完整的NLU处理流程。
        
        参数：
            user_input: 用户原始输入
            conversation_history: 对话历史
        
        返回：
            结构化的语义理解结果
        """
        history = conversation_history or []
        
        # Step 1: 指代消解
        resolved_input = self.coref_resolver.resolve(
            user_input, history
        )
        print(f"[NLU] 指代消解: {user_input} -> {resolved_input}")
        
        # Step 2: 意图识别
        intent_result = self.intent_recognizer.recognize(resolved_input)
        intent = intent_result["intent"]
        confidence = intent_result["confidence"]
        print(f"[NLU] 意图识别: {intent} (置信度: {confidence})")
        
        # Step 3: 实体抽取
        entities = self.entity_extractor.extract(resolved_input)
        print(f"[NLU] 实体抽取: {len(entities)} 个实体")
        
        # Step 4: 槽位填充
        slot_schema = self.intent_slot_map.get(intent, {})
        slot_result = self.slot_filler.multi_turn_fill(
            resolved_input, intent, slot_schema
        )
        print(f"[NLU] 槽位填充: {slot_result['is_complete']}")
        
        # Step 5: 构建统一的结构化输出
        return {
            "original_input": user_input,
            "resolved_input": resolved_input,
            "intent": {
                "name": intent,
                "confidence": confidence,
                "reasoning": intent_result.get("reasoning", "")
            },
            "entities": entities,
            "slots": slot_result["slots"],
            "missing_slots": slot_result["missing_slots"],
            "is_executable": slot_result["is_complete"] and confidence > 0.7,
            "needs_clarification": (
                confidence < 0.7 or
```

## 七、多语言与跨语言指令处理

在全球化的应用场景中，智能体需要处理来自不同语言的指令。

### 7.1 处理策略对比

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **翻译后处理** | 先翻译为统一语言（如英文），再做NLU | 复用已有的单语言NLU pipeline | 翻译错误会级联放大 |
| **多语言Prompt** | 在Prompt中列出多语言的同义表达 | 实现简单 | Prompt过长，难以维护 |
| **多语言模型** | 使用天然支持多语言的base model | 端到端处理，无需中间翻译 | 低资源语言效果差 |
| **语言检测+路由** | 先检测语言，再路由到对应的处理模块 | 每种语言可独立优化 | 增加系统复杂度 |

### 7.2 语言检测与路由实现

```python
class MultilingualNLUPipeline:
    def __init__(self, llm):
        self.llm = llm
        self.pipelines = {
            "zh": NLUPipeline(llm, lang="zh"),
            "en": NLUPipeline(llm, lang="en"),
            "ja": NLUPipeline(llm, lang="ja"),
        }
        self.fallback = NLUPipeline(llm, lang="en")  # 降级方案
    
    def detect_language(self, text: str) -> str:
        """检测输入文本的语言"""
        prompt = f"""
检测以下文本的语言。只返回ISO 639-1语言代码
（如 zh, en, ja, ko, fr, de, es 等）。

文本：{text}
"""
        return self.llm.generate(prompt).strip()
    
    def process(self, user_input: str, history=None) -> dict:
        lang = self.detect_language(user_input)
        pipeline = self.pipelines.get(lang, self.fallback)
        print(f"[MultiLang] 检测到语言: {lang}, 使用pipeline: {type(pipeline).__name__}")
        return pipeline.process(user_input, history)
```

### 8.2 歧义检测与澄清策略

```python
class AmbiguityHandler:
    def __init__(self, llm):
        self.llm = llm
    
    def detect_and_resolve(
        self, user_input: str, nlu_result: dict
    ) -> dict:
        """
        检测并处理歧义。
        策略优先级：上下文消歧 > 主动澄清 > 多假设并行
        """
        
        # 策略1: 用已确认的上下文消歧
        if self._can_disambiguate_with_context(user_input, nlu_result):
            return self._context_disambiguation(user_input, nlu_result)
        
        # 策略2: 判断歧义是否影响任务执行
        ambiguities = self._detect_ambiguities(user_input, nlu_result)
        critical = [a for a in ambiguities if a["impact"] == "critical"]
        
        if critical:
            # 策略3: 生成澄清问题
            return self._generate_clarification(user_input, critical)
        
        # 策略4: 对于非关键歧义，并行探索多种解释
        return self._parallel_hypotheses(user_input, ambiguities)
    
    def _detect_ambiguities(self, user_input: str, nlu_result: dict) -> list:
        """使用LLM检测所有可能的歧义点"""
        prompt = f"""
分析以下用户输入是否存在歧义。对于每个歧义点，标注：
- span: 有歧义的原文段落
- interpretations: 可能的多种理解
- impact: critical（影响任务执行）或 minor（不影响）
- resolution_hint: 如何解决这个歧义的建议

用户输入：{user_input}
当前NLU结果：{json.dumps(nlu_result, ensure_ascii=False)}

以JSON数组格式返回所有歧义点。
"""
        response = self.llm.generate(prompt)
        return json.loads(response)
    
    def _can_disambiguate_with_context(
        self, user_input: str, nlu_result: dict
    ) -> bool:
        """检查是否有足够的上下文来消歧"""
        # 例如：历史中提到过"Apple公司"，则"苹果"优先理解为公司
        return nlu_result.get("context_entities") is not None
    
    def _context_disambiguation(
        self, user_input: str, nlu_result: dict
    ) -> dict:
        """利用上下文进行消歧"""
        prompt = f"""
上下文信息：
- 用户画像：{nlu_result.get('user_profile', {})}
- 对话历史主题：{nlu_result.get('conversation_topic', '')}
- 已确认实体：{nlu_result.get('context_entities', [])}

用户输入：{user_input}

请基于上下文选择最合理的理解，返回消歧后的NLU结果（JSON格式）。
"""
        response = self.llm.generate(prompt)
        return json.loads(response)
    
    def _generate_clarification(
        self, user_input: str, ambiguities: list
    ) -> dict:
        """生成友好的澄清问题"""
        ambiguity_desc = "\n".join(
            f"- \"{a['span']}\" 可能指 {', '.join(a['interpretations'])}"
            for a in ambiguities
        )
        
        prompt = f"""
用户输入存在歧义：
{ambiguity_desc}

请生成一个简短、自然的澄清问题，帮助用户明确意图。
问题应该友好、不唐突、易于回答。
"""
        question = self.llm.generate(prompt).strip()
        
        return {
            "status": "needs_clarification",
            "ambiguities": ambiguities,
            "clarification_question": question,
            "nlu_result": None
        }
    
    def _parallel_hypotheses(
        self, user_input: str, ambiguities: list
    ) -> dict:
        """
        对非关键歧义，生成多个理解假设，并行处理。
        适合"先做再看"的场景。
        """
        return {
            "status": "multiple_hypotheses",
            "ambiguities": ambiguities,
            "hypotheses": [
                {"interpretation": interp, "action": "explore"}
                for a in ambiguities
                for interp in a["interpretations"]
            ]
        }
```

### 8.3 澄清对话的设计原则

澄清不是"承认失败"，而是提升体验的机会。设计良好的澄清对话应该：

1. **具体而非笼统**：不说"我不明白"，而说"您说的'苹果'是指苹果公司还是水果？"
2. **提供选项而非开放式提问**：给出2-3个最可能的选项，让用户一键选择
3. **记住澄清结果**：澄清后的信息要持久化，避免重复询问
4. **允许模糊继续**：如果歧义不影响核心任务，可以先执行再澄清

```python
def generate_smart_clarification(ambiguities: list) -> dict:
    """生成智能澄清——提供可点击的选项"""
    options = []
    for a in ambiguities[:2]:  # 一次最多澄清2个歧义点
        for i, interp in enumerate(a["interpretations"]):
            options.append({
                "id": f"amb_{a['span']}_{i}",
                "label": interp,
                "resolves": a["span"]
            })
    
    return {
        "type": "clarification",
        "message": f"您说的\"{ambiguities[0]['span']}\"是指？",
        "options": options,
        "allow_skip": len([a for a in ambiguities if a["impact"] == "minor"]) > 0
    }
```

## 九、最佳实践

### 9.1 感知模块的设计原则

1. **渐进式理解**：不要试图在一次调用中完成所有理解。先做意图识别，确定方向后再做深度解析，节省Token的同时提高精度。

2. **置信度阈值分级处理**：
   - `confidence >= 0.9`：直接执行
   - `0.7 <= confidence < 0.9`：执行但附加不确定提示
   - `confidence < 0.7`：触发澄清对话

3. **可解释性优先**：每个NLU决策都要有reasoning字段，方便调试和优化。当智能体行为不符合预期时，你能追溯到底是"理解错了"还是"执行错了"。

4. **槽位定义即文档**：槽位的description就是给LLM看的"使用说明"。写得越详细，模型填充越准确。

5. **Pipeline可观测性**：为每个步骤记录输入输出和耗时，方便定位瓶颈。

```python
class ObservableNLUPipeline(NLUPipeline):
    def process(self, user_input, history=None):
        trace = {"input": user_input, "steps": []}
        
        for step_name, step_fn in [
            ("coref_resolution", self._step_coref),
            ("intent_recognition", self._step_intent),
            ("entity_extraction", self._step_entity),
            ("slot_filling", self._step_slot),
        ]:
            start = time.time()
            result = step_fn(user_input, history, trace)
            elapsed = time.time() - start
            trace["steps"].append({
                "name": step_name,
                "duration_ms": elapsed * 1000,
                "result": result
            })
        
        trace["total_duration_ms"] = sum(
            s["duration_ms"] for s in trace["steps"]
        )
        return trace
```

6. **多语言降级策略**：当目标语言的NLU效果不佳时，自动降级为翻译+主流语言处理的方案，确保基本可用。

7. **指代消解要保守**：宁可保留指代词并标记"未消解"，也比错误消解导致后续操作错误要好。错误的消解比不消解危害更大。

8. **用户反馈闭环**：当用户纠正了智能体的理解（"不是这个意思"），这个反馈应该回写进系统，用于后续的消歧和实体记忆更新。

### 9.2 常见陷阱与避免方法

| 陷阱 | 表现 | 避免方法 |
|------|------|---------|
| 过早提交 | 槽位不完整就触发执行 | 严格执行完整性检查 |
| 过度询问 | 每个不确定都让用户确认 | 只对关键歧义进行澄清 |
| 意图膨胀 | 意图体系无限增长 | 定期审查和合并相似意图 |
| 实体漂移 | 对话中实体含义悄悄改变 | 每轮都重新做实体校验 |
| Prompt泄漏 | NLU的Prompt被用户注入 | 用分隔符隔离用户输入和系统指令 |
| 长对话退化 | 对话越长理解越差 | 定期压缩历史，重置不活跃的实体 |
