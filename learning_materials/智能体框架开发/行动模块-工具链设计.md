# 行动模块：工具链设计——编排、依赖与容错

## 一、从单个工具到工具链

在前面的章节中，我们学习了如何定义工具 Schema、如何让模型调用单个工具。但在真实场景中，用户的需求往往无法通过一次工具调用来解决。

考虑这样一个用户请求：**"帮我整理上周的销售数据，生成一份 PDF 报告，然后用邮件发给王总。"**

这个任务至少涉及三个工具：

1. `query_sales_data` —— 从数据库查询销售数据
2. `generate_pdf_report` —— 将数据生成 PDF 报告
3. `send_email` —— 发送邮件

这三个工具不是独立的——它们之间存在明确的先后依赖关系：数据查询的结果是报告生成的输入，报告生成的结果又是邮件发送的输入。这就是**工具链（Tool Chain）**的典型场景。

**工具链**，本质上是一组工具按照特定逻辑关系编排而成的有向无环图（DAG）。它描述了工具之间"谁先谁后"、"谁依赖谁"、"谁与谁并行"、"出错了怎么办"等一系列问题。

## 二、工具链的三种编排模式

工具间的编排关系主要可以分为三种：串联、并行和条件分支。

### 2.1 串联（Sequential）

串联是最基础的编排模式——工具按顺序依次执行，前一个工具的输出是后一个工具的输入。

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│  Tool A  │────>│  Tool B  │────>│  Tool C  │
│ 查询数据  │     │ 处理数据  │     │ 存储结果  │
└──────────┘     └──────────┘     └──────────┘
```

**代码示例：**

```python
class SequentialChain:
    """串联工具链：按顺序执行工具，前一个的输出传给后一个"""

    def __init__(self, steps: list):
        """
        steps: 列表，每个元素为 (tool_name, param_mapping)
        param_mapping 是一个函数，接收上一步的输出，返回当前步的参数
        """
        self.steps = steps

    def run(self, initial_input: dict) -> dict:
        current_output = initial_input
        execution_log = []

        for i, (tool_name, param_mapping) in enumerate(self.steps):
            tool = ToolRegistry.get(tool_name)
            params = param_mapping(current_output)
            print(f"[Step {i + 1}] 执行 {tool_name}，参数: {params}")
            
            result = tool.execute(**params)
            execution_log.append({
                "step": i + 1,
                "tool": tool_name,
                "params": params,
                "result": result
            })
            
            if not result.get("success"):
                print(f"[Step {i + 1}] 工具 {tool_name} 执行失败: {result.get('error')}")
                return {"success": False, "log": execution_log, "error": result.get("error")}
            
            current_output = result

        return {"success": True, "log": execution_log, "final_output": current_output}


# 实际使用
chain = SequentialChain(steps=[
    ("query_sales_data", lambda prev: {"date_range": "last_week"}),
    ("analyze_trends",    lambda prev: {"raw_data": prev["data"]}),
    ("generate_report",   lambda prev: {"analysis": prev["insights"]}),
])

result = chain.run(initial_input={})
```

### 2.2 并行（Parallel）

当多个工具之间没有数据依赖时，它们可以同时执行以提升效率。

```
              ┌──────────┐
              │  Tool A  │────> 结果A
              │ 查询股价  │
              └──────────┘
  初始输入 ──┤                      ├──> 汇总
              ┌──────────┐
              │  Tool B  │────> 结果B
              │ 查询新闻  │
              └──────────┘
```

**代码示例：**

```python
import concurrent.futures

class ParallelChain:
    """并行工具链：同时执行多个无依赖关系的工具"""

    def __init__(self, tasks: list):
        """
        tasks: 列表，每个元素为 (tool_name, params_dict)
        """
        self.tasks = tasks

    def run(self) -> dict:
        results = {}
        errors = []

        with concurrent.futures.ThreadPoolExecutor(max_workers=len(self.tasks)) as executor:
            future_to_tool = {}

            for tool_name, params in self.tasks:
                tool = ToolRegistry.get(tool_name)
                future = executor.submit(tool.execute, **params)
                future_to_tool[future] = tool_name

            for future in concurrent.futures.as_completed(future_to_tool):
                tool_name = future_to_tool[future]
                try:
                    results[tool_name] = future.result(timeout=30)
                except Exception as e:
                    errors.append({"tool": tool_name, "error": str(e)})

        return {
            "success": len(errors) == 0,
            "results": results,
            "errors": errors
        }


# 实际使用——同时获取股票价格和相关新闻
parallel_chain = ParallelChain(tasks=[
    ("get_stock_price",  {"symbol": "AAPL"}),
    ("get_company_news", {"symbol": "AAPL", "days": 7}),
    ("get_market_index", {"index": "NASDAQ"}),
])

result = parallel_chain.run()
```

### 2.3 条件分支（Conditional Branch）

根据中间结果动态决定下一步执行哪个工具。这是工具链"智能化"的关键——链不是死的，而是能根据实际情况自适应调整。

```
                          ┌──────────┐
                          │  Tool A  │
                          │ 风险评估  │
                          └────┬─────┘
                               │
                    风险等级 >= 7? 
                    /            \
                   是             否
                   ↓              ↓
            ┌──────────┐   ┌──────────┐
            │  Tool B  │   │  Tool C  │
            │ 高级审查  │   │ 自动通过  │
            └──────────┘   └──────────┘
```

**代码示例：**

```python
class ConditionalChain:
    """条件分支工具链：根据前一步结果动态选择后续工具"""

    def __init__(self):
        self.branches = {}  # {(tool_name, condition): next_chain}

    def add_branch(self, from_tool: str, condition, next_chain):
        """
        condition: 函数，接收 from_tool 的输出，返回 True/False
        next_chain: 当 condition 为 True 时执行的子链
        """
        self.branches[(from_tool, condition)] = next_chain

    def add_default_branch(self, from_tool: str, next_chain):
        """当所有条件都不满足时的默认分支"""
        self.default_branches = getattr(self, "default_branches", {})
        self.default_branches[from_tool] = next_chain

    def run(self, start_tool: str, params: dict) -> dict:
        current_tool = start_tool
        current_params = params
        final_result = None

        while current_tool:
            tool = ToolRegistry.get(current_tool)
            result = tool.execute(**current_params)
            final_result = result

            # 寻找匹配的分支条件
            matched = False
            for (from_tool, condition), next_chain in self.branches.items():
                if from_tool == current_tool and condition(result):
                    current_tool = next_chain.start_tool
                    current_params = next_chain.map_params(result)
                    matched = True
                    break

            if not matched:
                # 尝试默认分支
                default = self.default_branches.get(current_tool)
                if default:
                    current_tool = default.start_tool
                    current_params = default.map_params(result)
                else:
                    break  # 无更多分支，结束

        return final_result


# 实际使用——风险评估流程
risk_chain = ConditionalChain()

# 高风险分支：进入高级审查
risk_chain.add_branch(
    from_tool="assess_risk",
    condition=lambda r: r["risk_score"] >= 7,
    next_chain=SequentialChain(steps=[
        ("deep_review",    lambda r: {"case_id": r["case_id"]}),
        ("flag_compliance", lambda r: {"review_result": r["findings"]}),
    ])
)

# 低风险分支：自动通过
risk_chain.add_default_branch(
    from_tool="assess_risk",
    next_chain=SequentialChain(steps=[
        ("auto_approve", lambda r: {"case_id": r["case_id"]}),
    ])
)

risk_chain.run(start_tool="assess_risk", params={"application_id": "APP-042"})
```

### 2.4 三种模式的对比

| 维度 | 串联 (Sequential) | 并行 (Parallel) | 条件分支 (Conditional) |
|------|-------------------|-----------------|------------------------|
| 执行顺序 | 严格顺序执行 | 同时并发执行 | 动态路由选择 |
| 数据流向 | 上游输出 -> 下游输入 | 各自独立，最后汇聚 | 上游输出 -> 路由决策 -> 下游输入 |
| 适用场景 | 有强依赖关系的任务 | 相互独立的子任务 | 需要根据结果做决策的任务 |
| 复杂度 | 低 | 中（需处理并发） | 高（需定义路由规则） |
| 失败影响 | 失败点之后全部中断 | 某路失败不影响其他路 | 取决于分支结构 |
| 效率 | 总耗时 = 各步之和 | 总耗时 = 最慢的一路 | 总耗时 = 所选路径之和 |

## 三、依赖管理：工具A的输出是工具B的输入

工具链中最复杂的问题之一就是**数据依赖管理**。当链中有五个甚至更多工具时，清晰地追踪"哪个数据从哪来、要到哪去"就变得至关重要。

### 3.1 显式参数映射

最直接的方式是在定义链时明确声明参数如何从前一步的结果中提取。

```python
class DataFlowManager:
    """管理工具链中的数据流动"""

    def __init__(self):
        self.context = {}  # 全局上下文，存储所有中间结果

    def store(self, tool_name: str, result: dict):
        """将工具执行结果存入上下文"""
        self.context[tool_name] = result
        # 同时按路径展开存储，方便精确引用
        self._flatten_store(tool_name, result)

    def resolve(self, mapping: dict) -> dict:
        """
        解析参数映射，将模板参数替换为实际值。
        
        支持的引用语法：
        - "@step_name.field"  引用某步骤结果的字段
        - "@step_name"        引用某步骤的完整结果
        - "@step_name.data.items[0]"  引用嵌套路径
        """
        resolved = {}
        for key, value in mapping.items():
            if isinstance(value, str) and value.startswith("@"):
                resolved[key] = self._resolve_ref(value)
            else:
                resolved[key] = value
        return resolved

    def _resolve_ref(self, ref: str) -> any:
        """解析引用字符串，从上下文中取值"""
        path = ref[1:].split(".")  # 去掉@前缀，按.拆分路径
        current = self.context
        for segment in path:
            if isinstance(current, dict):
                current = current.get(segment)
            elif isinstance(current, list):
                current = current[int(segment.strip("[]"))]
            else:
                return None
        return current


# 使用示例
dfm = DataFlowManager()

# Step 1: 查询数据
step1_result = {"success": True, "data": {"sales": 150000, "orders": 320}}
dfm.store("query_sales", step1_result)

# Step 2: 使用显式映射获取参数
step2_params = dfm.resolve({
    "total_sales": "@query_sales.data.sales",
    "total_orders": "@query_sales.data.orders",
    "report_format": "pdf"
})
# 结果: {"total_sales": 150000, "total_orders": 320, "report_format": "pdf"}
```

### 3.2 依赖图的拓扑排序

当工具链非常复杂时，可以用**有向无环图（DAG）**来建模依赖关系，并通过拓扑排序自动确定执行顺序。

```python
from collections import deque

class ToolDAG:
    """用 DAG 管理工具的依赖关系"""

    def __init__(self):
        self.nodes = {}       # {tool_name: ToolNode}
        self.dependencies = {}  # {tool_name: [依赖的工具名列表]}

    def add_tool(self, name: str, tool_func, depends_on: list = None):
        """
        添加一个工具节点。
        depends_on: 该工具依赖的其他工具名称列表
        """
        self.nodes[name] = tool_func
        self.dependencies[name] = depends_on or []

    def topological_order(self) -> list:
        """拓扑排序，返回合法的执行顺序"""
        in_degree = {name: 0 for name in self.nodes}
        children = {name: [] for name in self.nodes}

        # 构建依赖关系
        for node, deps in self.dependencies.items():
            in_degree[node] = len(deps)
            for dep in deps:
                if dep not in children:
                    children[dep] = []
                children[dep].append(node)

        # Kahn 算法
        queue = deque([n for n, d in in_degree.items() if d == 0])
        order = []

        while queue:
            current = queue.popleft()
            order.append(current)
            for child in children.get(current, []):
                in_degree[child] -= 1
                if in_degree[child] == 0:
                    queue.append(child)

        if len(order) != len(self.nodes):
            raise ValueError("检测到循环依赖，无法生成合法的执行顺序")

        return order

    def find_parallel_groups(self) -> list:
        """
        找出可以并行执行的工具组。
        同一组内的工具入度为0（互不依赖），可以同时执行。
        """
        order = self.topological_order()
        # 简化实现：将拓扑顺序中相互之间无依赖的工具归为一组
        groups = []
        visited = set()
        for tool in order:
            if tool not in visited:
                # 找到所有不依赖当前组内任何工具的节点
                group = [tool]
                visited.add(tool)
                for other in order:
                    if other not in visited:
                        # 检查 other 是否与 group 中任何工具有依赖
                        deps = self.dependencies.get(other, [])
                        if not any(d in group for d in deps):
                            group.append(other)
                            visited.add(other)
                groups.append(group)
        return groups


# 使用示例：构建一个数据分析工具链
dag = ToolDAG()
dag.add_tool("fetch_from_db",    fetch_func,  depends_on=[])
dag.add_tool("fetch_from_api",   api_func,    depends_on=[])
dag.add_tool("merge_data",       merge_func,  depends_on=["fetch_from_db", "fetch_from_api"])
dag.add_tool("analyze",          analyze_func, depends_on=["merge_data"])
dag.add_tool("generate_chart",   chart_func,  depends_on=["analyze"])
dag.add_tool("generate_report",  report_func,  depends_on=["analyze"])
dag.add_tool("send_notification", notify_func, depends_on=["generate_report"])

print("执行顺序:", dag.topological_order())
# ['fetch_from_db', 'fetch_from_api', 'merge_data', 'analyze', 'generate_chart', 'generate_report', 'send_notification']

print("并行组:", dag.find_parallel_groups())
# [['fetch_from_db', 'fetch_from_api'], ['merge_data'], ['analyze'], ['generate_chart', 'generate_report'], ['send_notification']]
```

### 3.3 依赖冲突的检测与解决

当两个工具对同一个数据字段有不同要求时，会产生依赖冲突。

```python
class DependencyConflictResolver:
    """检测并解决工具间的依赖冲突"""

    @staticmethod
    def detect_conflicts(chain_def: dict) -> list:
        """
        检测链定义中的潜在冲突。
        返回冲突列表，每个冲突包含冲突字段和涉及的工具。
        """
        conflicts = []
        field_writers = {}

        for tool_name, tool_def in chain_def.items():
            # 检查输出字段是否被多个工具写入
            for output_field in tool_def.get("outputs", []):
                if output_field in field_writers:
                    conflicts.append({
                        "field": output_field,
                        "writers": [field_writers[output_field], tool_name],
                        "type": "write_conflict",
                        "suggestion": "考虑使用不同的字段名，或在链定义中明确字段的覆盖规则"
                    })
                field_writers[output_field] = tool_name

        return conflicts

    @staticmethod
    def resolve_by_rename(chain_def: dict, conflict_field: str, rename_map: dict):
        """通过重命名字段来消解冲突"""
        for tool_name, new_field in rename_map.items():
            # 在工具的输出定义中重命名
            chain_def[tool_name]["outputs"] = [
                new_field if f == conflict_field else f
                for f in chain_def[tool_name]["outputs"]
            ]
        return chain_def
```

## 四、错误传递与恢复

### 4.1 错误传递模型

工具链中任何一个工具失败，都可能影响整个链的执行。错误传递需要回答三个核心问题：

1. **哪个工具出错了？**——精确的故障定位
2. **错误的影响范围是什么？**——哪些后续工具无法执行
3. **如何恢复？**——回滚、重试还是跳过

```python
from enum import Enum
from dataclasses import dataclass
from typing import Optional, Callable

class ErrorSeverity(Enum):
    RETRYABLE = "retryable"    # 可重试（如网络超时）
    SKIPPABLE = "skippable"    # 可跳过（非关键步骤）
    FATAL = "fatal"            # 致命错误，整个链必须终止

@dataclass
class ChainError:
    tool_name: str
    step_index: int
    error_message: str
    severity: ErrorSeverity
    context: dict               # 出错时的上下文数据
    affected_downstream: list   # 受影响的后续工具列表

class ErrorPropagationHandler:
    """工具链的错误传递处理器"""

    def __init__(self):
        self.error_handlers = {}   # {(tool_name, error_type): handler_func}
        self.rollback_stack = []   # 已执行步骤的逆序记录，用于回滚

    def register_handler(self, tool_name: str, error_type: type, 
                         handler: Callable, severity: ErrorSeverity = ErrorSeverity.FATAL):
        """为特定工具的错误注册处理函数"""
        self.error_handlers[(tool_name, error_type)] = (handler, severity)

    def handle(self, tool_name: str, step_index: int, 
               exception: Exception, context: dict, downstream: list) -> ChainError:
        """处理工具执行过程中抛出的异常"""
        error_key = (tool_name, type(exception))
        
        if error_key in self.error_handlers:
            handler, severity = self.error_handlers[error_key]
            handler(exception, context)
        else:
            # 默认：网络相关错误可重试，其他为致命
            if isinstance(exception, (TimeoutError, ConnectionError)):
                severity = ErrorSeverity.RETRYABLE
            else:
                severity = ErrorSeverity.FATAL

        return ChainError(
            tool_name=tool_name,
            step_index=step_index,
            error_message=str(exception),
            severity=severity,
            context=context,
            affected_downstream=downstream
        )

    def add_rollback_step(self, rollback_func: Callable, context: dict):
        """记录一个回滚步骤"""
        self.rollback_stack.append((rollback_func, context))

    def rollback(self):
        """按逆序执行回滚操作"""
        errors_during_rollback = []
        while self.rollback_stack:
            rollback_func, context = self.rollback_stack.pop()
            try:
                rollback_func(context)
                print(f"[回滚] 成功执行: {rollback_func.__name__}")
            except Exception as e:
                errors_during_rollback.append(str(e))
                print(f"[回滚失败] {rollback_func.__name__}: {e}")
        return errors_during_rollback
```

### 4.2 回滚策略

回滚是工具链容错的核心机制。当链执行到一半失败时，需要撤销已执行步骤产生的副作用。

```python
class RollbackManager:
    """管理工具链的回滚操作"""

    def __init__(self):
        self.compensation_actions = []  # 补偿操作列表（逆序）

    def record_action(self, tool_name: str, compensation_fn, context: dict):
        """
        记录一个补偿操作。
        compensation_fn: 撤销该工具副作用的函数
        """
        self.compensation_actions.append({
            "tool": tool_name,
            "compensate": compensation_fn,
            "context": context
        })

    def rollback_all(self) -> dict:
        """执行所有已记录步骤的回滚"""
        results = {"rolled_back": [], "failed": []}
        
        for action in reversed(self.compensation_actions):
            try:
                action["compensate"](action["context"])
                results["rolled_back"].append(action["tool"])
                print(f"[回滚] {action['tool']} 的副作用已撤销")
            except Exception as e:
                results["failed"].append({
                    "tool": action["tool"],
                    "error": str(e)
                })
                print(f"[回滚失败] {action['tool']}: {e}")

        return results


# 实际案例：一个包含数据库操作的工具链回滚
rollback_mgr = RollbackManager()

def insert_order(order_data):
    order_id = db.insert("orders", order_data)
    # 记录补偿操作：删除刚插入的订单
    rollback_mgr.record_action(
        tool_name="insert_order",
        compensation_fn=lambda ctx: db.delete("orders", ctx["order_id"]),
        context={"order_id": order_id}
    )
    return {"success": True, "order_id": order_id}

def deduct_inventory(order_data):
    db.update("inventory", {"stock": stock - order_data["quantity"]})
    # 记录补偿操作：恢复库存
    rollback_mgr.record_action(
        tool_name="deduct_inventory",
        compensation_fn=lambda ctx: db.update(
            "inventory", 
            {"stock": ctx["original_stock"]}
        ),
        context={"original_stock": stock, "item_id": order_data["item_id"]}
    )
    return {"success": True}
```

### 4.3 重试机制

```python
import time

class RetryPolicy:
    """工具链的重试策略"""

    def __init__(self, max_retries: int = 3, backoff_factor: float = 1.5,
                 retryable_exceptions: tuple = (TimeoutError, ConnectionError)):
        self.max_retries = max_retries
        self.backoff_factor = backoff_factor
        self.retryable_exceptions = retryable_exceptions

    def execute_with_retry(self, tool_func, *args, **kwargs) -> dict:
        """带指数退避重试的工具执行器"""
        last_exception = None

        for attempt in range(self.max_retries + 1):
            try:
                result = tool_func(*args, **kwargs)
                return {"success": True, "result": result, "attempts": attempt + 1}
            except self.retryable_exceptions as e:
                last_exception = e
                if attempt < self.max_retries:
                    wait_time = self.backoff_factor ** attempt
                    print(f"[重试] 第 {attempt + 1} 次失败，{wait_time:.1f}秒后重试... ({e})")
                    time.sleep(wait_time)
            except Exception as e:
                # 不可重试的异常，直接返回失败
                return {"success": False, "error": str(e), "attempts": attempt + 1}

        return {
            "success": False,
            "error": f"重试 {self.max_retries} 次后仍失败: {last_exception}",
            "attempts": self.max_retries + 1
        }


# 使用重试策略包装工具
retry_policy = RetryPolicy(max_retries=3)

def robust_api_call(endpoint, params):
    return retry_policy.execute_with_retry(
        lambda: requests.get(endpoint, params=params, timeout=5)
    )
```

## 五、工具链的 DSL 描述

当工具链变得复杂时，纯代码定义会变得冗长且难以维护。引入一个简单的 **DSL（Domain Specific Language，领域特定语言）** 可以大幅提升可读性和可维护性。

### 5.1 DSL 设计原则

一个好的工具链 DSL 应该满足：
- **声明式**：描述"要什么"，而不是"怎么做"
- **可读性**：非技术人员也能大致理解链的逻辑
- **可验证**：能在执行前对链定义做静态校验

### 5.2 YAML 格式的工具链 DSL

```yaml
# 工具链定义: 销售数据分析与报告流程
chain:
  name: sales_report_pipeline
  version: "1.0"
  description: "查询销售数据 → 分析趋势 → 生成报告 → 邮件发送"
  
  # 全局配置
  config:
    timeout: 60           # 单步超时（秒）
    on_error: rollback    # 全局错误策略：rollback | skip | abort
    max_retries: 2        # 默认重试次数

  # 工具声明
  tools:
    query_sales:
      type: database_query
      config:
        database: sales_db
        query_template: "SELECT * FROM sales WHERE date BETWEEN '{start_date}' AND '{end_date}'"
      outputs: [raw_data]

    analyze_trends:
      type: llm_analysis
      config:
        model: gpt-4
        prompt_template: "分析以下销售数据中的关键趋势：{raw_data}"
      inputs:
        raw_data: "@query_sales.raw_data"
      outputs: [analysis_result]

    generate_charts:
      type: chart_generator
      config:
        chart_types: [bar, line, pie]
      inputs:
        data: "@query_sales.raw_data"
        analysis: "@analyze_trends.analysis_result"
      outputs: [chart_files]

    generate_pdf:
      type: pdf_renderer
      config:
        template: sales_report_v2
      inputs:
        data: "@query_sales.raw_data"
        analysis: "@analyze_trends.analysis_result"
        charts: "@generate_charts.chart_files"
      outputs: [pdf_path]

    send_email:
      type: email_sender
      config:
        smtp_server: smtp.company.com
      inputs:
        recipient: "{recipient_email}"      # 花括号表示外部传入的参数
        subject: "销售周报 - {report_date}"
        attachment: "@generate_pdf.pdf_path"
      outputs: [send_status]

  # 执行流程定义
  flow:
    # 第一步：并行查询数据和生成图表（图表依赖数据，异步启动）
    - parallel:
        - tool: query_sales
          params:
            start_date: "{start_date}"
            end_date: "{end_date}"
        - tool: analyze_trends
          depends_on: [query_sales]     # analyze_trends 依赖 query_sales
      on_error: retry

    # 第二步：并行生成图表和 PDF
    - parallel:
        - tool: generate_charts
        - tool: generate_pdf
      depends_on: [query_sales, analyze_trends]

    # 第三步：发送邮件
    - tool: send_email
      depends_on: [generate_pdf]
      on_error:
        type: fallback
        fallback_tool: send_slack_notification   # 邮件失败时改用 Slack 通知
```

### 5.3 DSL 解析器实现

```python
import yaml
from typing import Dict, List, Any

class ChainDSLParser:
    """解析 YAML DSL 定义，构建可执行的工具链"""

    def __init__(self, dsl_path: str):
        with open(dsl_path, "r", encoding="utf-8") as f:
            self.definition = yaml.safe_load(f)
        self.chain = self.definition["chain"]
        self._validate()

    def _validate(self):
        """静态校验 DSL 定义的正确性"""
        tool_names = set(self.chain["tools"].keys())
        
        # 检验 1: flow 中引用的工具都在 tools 中声明过
        flow_tools = self._collect_flow_tools(self.chain["flow"], tool_names)
        undeclared = flow_tools - tool_names
        if undeclared:
            raise ValueError(f"DSL 校验失败: flow 中引用了未声明的工具: {undeclared}")

        # 检验 2: tools 的 inputs 引用的输出源存在
        for tool_name, tool_def in self.chain["tools"].items():
            for input_key, input_ref in tool_def.get("inputs", {}).items():
                if isinstance(input_ref, str) and input_ref.startswith("@"):
                    source_tool = input_ref.split(".")[0][1:]   # 去掉@前缀
                    if source_tool not in tool_names:
                        raise ValueError(
                            f"DSL 校验失败: {tool_name}.inputs.{input_key} "
                            f"引用了不存在的工具: {source_tool}"
                        )

        # 检验 3: 检查循环依赖
        self._check_circular_dependency()

        print(f"[DSL 校验通过] 工具链 '{self.chain['name']}' 定义合法")

    def _collect_flow_tools(self, flow: list, tool_names: set) -> set:
        """收集 flow 中所有引用的工具名"""
        names = set()
        for step in flow:
            if "tool" in step:
                names.add(step["tool"])
            if "parallel" in step:
                for sub_step in step["parallel"]:
                    names.add(sub_step["tool"])
            if "fallback_tool" in step.get("on_error", {}):
                names.add(step["on_error"]["fallback_tool"])
        return names

    def _check_circular_dependency(self):
        """检测 depends_on 中是否存在循环依赖"""
        # 构建邻接表
        adj = {}
        for tool_name, tool_def in self.chain["tools"].items():
            deps = []
            for input_ref in tool_def.get("inputs", {}).values():
                if isinstance(input_ref, str) and input_ref.startswith("@"):
                    source = input_ref.split(".")[0][1:]
                    deps.append(source)
            adj[tool_name] = deps

        # DFS 检测环
        WHITE, GRAY, BLACK = 0, 1, 2
        color = {t: WHITE for t in adj}

        def dfs(node):
            color[node] = GRAY
            for neighbor in adj.get(node, []):
                if color.get(neighbor) == GRAY:
                    raise ValueError(f"DSL 校验失败: 检测到循环依赖，涉及工具 '{node}' 和 '{neighbor}'")
                if color.get(neighbor) == WHITE:
                    dfs(neighbor)
            color[node] = BLACK

        for tool in adj:
            if color[tool] == WHITE:
                dfs(tool)

    def build_executable_chain(self, external_params: dict = None) -> "ExecutableChain":
        """将 DSL 定义编译为可执行的工具链对象"""
        external_params = external_params or {}
        return ExecutableChain(self.chain, external_params)
```

### 5.4 DSL 执行引擎

```python
class ExecutableChain:
    """执行由 DSL 编译而来的工具链"""

    def __init__(self, chain_def: dict, external_params: dict):
        self.chain_def = chain_def
        self.external_params = external_params
        self.context = DataFlowManager()
        self.rollback_mgr = RollbackManager()
        self.retry_policy = RetryPolicy(
            max_retries=chain_def.get("config", {}).get("max_retries", 2)
        )

    def run(self) -> dict:
        """执行整个工具链"""
        flow = self.chain_def["flow"]
        error_policy = self.chain_def["config"].get("on_error", "abort")

        for step in flow:
            if "parallel" in step:
                result = self._execute_parallel(step)
            else:
                result = self._execute_single(step)

            if not result["success"]:
                if error_policy == "rollback":
                    self.rollback_mgr.rollback_all()
                return {"success": False, "failed_at": step, "error": result["error"]}

        return {"success": True, "final_context": self.context.context}

    def _execute_single(self, step: dict) -> dict:
        """执行单个工具步骤"""
        tool_name = step["tool"]
        tool_def = self.chain_def["tools"][tool_name]

        # 解析参数：先应用外部参数，再解析内部引用
        params = {}
        for key, value in step.get("params", {}).items():
            params[key] = value.format(**self.external_params)
        params = self.context.resolve(params)

        # 带重试的执行
        tool_func = ToolRegistry.get(tool_name)
        result = self.retry_policy.execute_with_retry(tool_func, **params)

        if result["success"]:
            self.context.store(tool_name, result["result"])

        # 错误处理
        if not result["success"] and "on_error" in step:
            error_config = step["on_error"]
            if isinstance(error_config, dict) and error_config.get("type") == "fallback":
                fallback_tool = error_config["fallback_tool"]
                print(f"[回退] {tool_name} 失败，改用 {fallback_tool}")
                return self._execute_single({"tool": fallback_tool, "params": params})

        return result

    def _execute_parallel(self, step: dict) -> dict:
        """执行并行步骤组"""
        tasks = []
        for sub_step in step["parallel"]:
            tasks.append((sub_step["tool"], sub_step.get("params", {})))

        parallel_chain = ParallelChain(tasks=tasks)
        result = parallel_chain.run()

        if result["success"]:
            for tool_name, tool_result in result["results"].items():
                self.context.store(tool_name, tool_result)

        return result
```

## 七、设计工具链的最佳实践

1. **从简单开始，逐步演进**：不要一开始就设计完美的 DSL 和容错机制。先用硬编码实现串联链，验证逻辑正确后再逐步引入并行、条件分支、DSL 等高级特性

2. **工具链的可观测性至关重要**：每一步执行都要记录日志，包括入参、出参、耗时、是否成功。推荐使用 OpenTelemetry 或类似的追踪框架，为每个步骤生成 Span

3. **补偿操作要趁早记录**：在执行一个工具有副作用的操作（写数据库、发消息、扣库存）之前，先确保它的补偿操作已被记录。遵循"先记录补偿，再执行操作"的原则

4. **区分业务异常和技术异常**：技术异常（网络超时、服务不可用）可以重试；业务异常（库存不足、余额不够）不应重试，而应走对应的业务分支

5. **DSL 与代码的边界要清晰**：DSL 负责描述"做什么"（声明式），代码负责实现"怎么做"。不要在 DSL 中嵌入复杂的业务逻辑

6. **设置全局超时和步数上限**：防止某一步卡死导致整个链永远无法完成。建议设置全局超时（如 5 分钟）和最大步骤数（如 50 步）

7. **测试回滚**：回滚逻辑往往是最容易被忽略的测试点。在测试环境中，故意让中间步骤失败，验证回滚是否完整执行——库存是否恢复、已发送的消息是否有撤回机制

8. **工具链的幂等性**：尽量让每个工具支持幂等调用（多次执行结果一致），这样重试时不会产生副作用。例如，使用唯一键（idempotency key）标识每次请求

9. **文档化工具链的依赖图**：复杂的工具链应该有可视化的依赖图，让团队成员能快速理解数据流向和步骤间的关系

10. **关注安全边界**：工具链中的敏感操作（扣款、发送短信、删除数据）应在 DSL 层面标记为 `sensitive: true`，执行引擎在遇到这些步骤时可以暂停等待人工确认

---

工具链设计是智能体"行动模块"中最体现工程能力的部分。一个精心设计的工具链，让智能体能够可靠地完成多步骤、有依赖、需要容错的复杂任务——这正是从"Demo"走向"生产环境"的关键跨越。
