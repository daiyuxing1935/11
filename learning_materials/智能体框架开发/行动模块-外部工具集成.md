# 行动模块：外部工具集成

## 一、为什么需要外部工具集成

智能体的核心能力在于"思考+行动"。前面的章节讨论了工具调用（Tool Use）的基本概念——模型可以请求调用函数。但模型本身不执行任何代码，真正"动手"的是**外部工具**。这些工具可能是你公司内部的REST API、第三方SaaS服务的SDK、数据库连接、甚至是物理设备的控制接口。

**外部工具集成**解决的核心问题是：如何将各种异构的外部能力，以一种统一、可靠、安全的方式接入智能体系统，让LLM能够"看到"它们，理解它们的能力边界，并在正确的时机正确地调用它们。

一个设计良好的工具集成层，要做到以下三点：

1. **让模型"看得见"**：通过清晰的工具描述，让模型准确理解每个工具能做什么
2. **让调用"跑得通"**：处理认证、参数转换、网络通信、错误重试等工程细节
3. **让系统"管得住"**：提供工具注册、权限控制、调用审计等治理能力

## 二、REST API调用集成

REST API是目前最普遍的外部工具形式。绝大多数SaaS服务和内部系统都通过RESTful接口暴露能力。将REST API封装为智能体可调用的工具，是最常见的集成场景。

### 基本HTTP请求封装

一个健壮的API调用封装需要考虑以下要素：请求构造、认证注入、响应解析、错误映射。

```python
import httpx
from typing import Any, Optional
import json

class APIToolClient:
    """封装REST API调用的基础客户端"""

    def __init__(self, base_url: str, timeout: int = 30):
        self.base_url = base_url.rstrip("/")
        self.client = httpx.Client(
            timeout=timeout,
            headers={"Content-Type": "application/json"}
        )

    def _build_headers(self, extra_headers: Optional[dict] = None) -> dict:
        """构建请求头，统一注入认证信息"""
        headers = {}
        if self.auth_token:
            headers["Authorization"] = f"Bearer {self.auth_token}"
        if extra_headers:
            headers.update(extra_headers)
        return headers

    def call(self, method: str, path: str, params: dict = None,
             body: dict = None) -> dict:
        """
        统一的API调用入口

        Args:
            method: HTTP方法 (GET/POST/PUT/DELETE)
            path: API路径 (如 /users/123)
            params: URL查询参数
            body: 请求体 (JSON)

        Returns:
            {"success": bool, "data": ..., "error": str}
        """
        url = f"{self.base_url}{path}"
        try:
            response = self.client.request(
                method=method,
                url=url,
                params=params,
                json=body
            )
            response.raise_for_status()
            return {
                "success": True,
                "data": response.json(),
                "status_code": response.status_code
            }
        except httpx.HTTPStatusError as e:
            return {
                "success": False,
                "error": f"HTTP {e.response.status_code}: {e.response.text[:500]}",
                "status_code": e.response.status_code
            }
        except httpx.TimeoutException:
            return {
                "success": False,
                "error": "请求超时，请在稍后重试",
                "status_code": 408
            }
        except Exception as e:
            return {
                "success": False,
                "error": f"请求失败: {str(e)}",
                "status_code": 500
            }
```

### 将API封装为LLM工具

有了底层的API调用能力，下一步是将其包装为LLM可以理解的工具定义：

```python
def create_weather_tool(api_client: APIToolClient) -> dict:
    """将一个天气API封装为工具定义+执行函数"""

    tool_definition = {
        "name": "get_weather",
        "description": (
            "查询指定城市的实时天气信息。"
            "当用户询问天气、气温、是否下雨等与天气相关的问题时使用此工具。"
            "返回温度、湿度、天气状况和风力信息。"
        ),
        "parameters": {
            "type": "object",
            "properties": {
                "city": {
                    "type": "string",
                    "description": "城市名称，使用中文，如'北京'、'上海'。支持地级市及以上级别城市"
                }
            },
            "required": ["city"]
        }
    }

    def execute(city: str) -> dict:
        """实际执行天气查询"""
        return api_client.call(
            method="GET",
            path="/weather/current",
            params={"city": city}
        )

    return {"definition": tool_definition, "execute": execute}
```

这种"定义+执行函数"的组合模式，让工具描述和实际执行逻辑紧密绑定但各自独立，便于测试和维护。

### 请求重试与连接池

对于生产环境，简单的单次调用是不够的。网络抖动、服务瞬时不可用是常态。

```python
from tenacity import retry, stop_after_attempt, wait_exponential

class RobustAPIToolClient(APIToolClient):

    def __init__(self, base_url: str, max_retries: int = 3):
        super().__init__(base_url)
        self.max_retries = max_retries

    @retry(
        stop=stop_after_attempt(3),
        wait=wait_exponential(multiplier=1, min=1, max=10),
        retry=lambda e: isinstance(e, (httpx.TimeoutException,
                                        httpx.HTTPStatusError))
        and (getattr(e, 'response', None) is None
             or e.response.status_code >= 500)
    )
    def call_with_retry(self, method: str, path: str, **kwargs) -> dict:
        """带自动重试的API调用 —— 仅在超时或5xx错误时重试"""
        return self.call(method, path, **kwargs)
```

关键策略：**只对幂等操作和服务器错误进行重试**。4xx错误（如参数错误、权限不足）重试没有意义。

## 三、SDK集成模式

很多时候外部服务提供了官方的SDK（如AWS boto3、GitHub PyGithub、Slack SDK）。直接在工具中调用SDK比构造HTTP请求更方便，但也带来了紧耦合的问题。

### 直接SDK封装

最简单的方式——在工具执行函数中直接调用SDK：

```python
from github import Github

def create_github_issue_tool(github_token: str) -> dict:
    g = Github(github_token)

    def execute(repo: str, title: str, body: str, labels: list = None) -> dict:
        try:
            repo_obj = g.get_repo(repo)
            issue = repo_obj.create_issue(
                title=title,
                body=body,
                labels=labels or []
            )
            return {
                "success": True,
                "issue_url": issue.html_url,
                "issue_number": issue.number
            }
        except Exception as e:
            return {"success": False, "error": str(e)}

    return {
        "definition": {
            "name": "create_github_issue",
            "description": (
                "在指定的GitHub仓库中创建一个Issue。"
                "当用户要求记录Bug、创建任务、提交需求时使用此工具。"
            ),
            "parameters": {
                "type": "object",
                "properties": {
                    "repo": {"type": "string", "description": "仓库全名，格式为 owner/repo，如 'octocat/hello-world'"},
                    "title": {"type": "string", "description": "Issue标题"},
                    "body": {"type": "string", "description": "Issue的正文内容"},
                    "labels": {"type": "array", "items": {"type": "string"}, "description": "标签列表"}
                },
                "required": ["repo", "title", "body"]
            }
        },
        "execute": execute
    }
```

### 统一抽象层设计

当系统集成了数十个外部服务时，每个工具各自管理SDK实例会导致资源浪费和配置分散。一个统一的抽象层可以解决这个问题：

```python
class ToolProvider(ABC):
    """工具提供者的抽象基类"""

    @abstractmethod
    def get_tools(self) -> list[dict]:
        """返回该提供者的所有工具定义和执行函数"""
        pass

    @abstractmethod
    def health_check(self) -> bool:
        """检查外部服务是否可用"""
        pass

class GitHubToolProvider(ToolProvider):
    def __init__(self, token: str):
        self.client = Github(token)  # SDK实例在Provider级别统一管理

    def get_tools(self) -> list:
        return [
            self._create_issue_tool(),
            self._search_repos_tool(),
            self._get_pr_tool(),
        ]

    def health_check(self) -> bool:
        try:
            self.client.get_user().login
            return True
        except Exception:
            return False

    def _create_issue_tool(self) -> dict:
        # ... 工具定义与执行
        pass
```

这种架构带来的好处：

| 方面 | 直接封装 | 统一抽象层 |
|------|---------|-----------|
| SDK实例管理 | 每个工具独立创建 | Provider级别统一管理 |
| 配置管理 | 分散在各工具中 | 集中在Provider中 |
| 健康检查 | 难以统一实现 | 每个Provider自带 |
| 工具发现 | 每次手动注册 | 遍历Provider自动收集 |
| 测试复杂度 | 工具和SDK紧耦合 | 可以Mock Provider |

## 四、OAuth认证流程

许多第三方服务（Google、Microsoft、Slack等）使用OAuth 2.0进行授权。让智能体工具支持OAuth，意味着用户需要先授权智能体访问他们的数据。

### OAuth 2.0授权码流程

```
用户                智能体系统              第三方服务(如Google)
 |                     |                          |
 |  1. 请求使用工具      |                          |
 |-------------------->|                          |
 |                     |  2. 生成授权URL           |
 |                     |------------------------->|
 |  3. 重定向到授权页面   |                          |
 |<----------------------------------------------->|
 |  4. 用户确认授权      |                          |
 |<----------------------------------------------->|
 |                     |  5. 回调 + Authorization Code
 |                     |<-------------------------|
 |                     |  6. 用Code换取Access Token |
 |                     |------------------------->|
 |                     |  7. 返回Access Token      |
 |                     |<-------------------------|
 |                     |  8. 存储Token，工具就绪    |
 |  9. 后续工具调用      |                          |
 |-------------------->|                          |
 |                     |  10. 使用Access Token调用API
 |                     |------------------------->|
 |                     |  11. API返回结果          |
 |                     |<-------------------------|
 |  12. 返回结果        |                          |
 |<--------------------|                          |
```

### Token管理与刷新

OAuth的Access Token有过期时间（通常1小时），需要Refresh Token来续期。工具执行时必须考虑Token过期的情况：

```python
import time
from datetime import datetime, timedelta

class OAuthTokenManager:
    def __init__(self, client_id: str, client_secret: str,
                 token_url: str):
        self.client_id = client_id
        self.client_secret = client_secret
        self.token_url = token_url
        self._access_token: Optional[str] = None
        self._refresh_token: Optional[str] = None
        self._expires_at: float = 0

    def is_token_valid(self) -> bool:
        """检查当前Token是否有效（留60秒缓冲）"""
        return (self._access_token is not None
                and time.time() < self._expires_at - 60)

    def get_access_token(self) -> str:
        """获取有效的Access Token，必要时自动刷新"""
        if not self.is_token_valid():
            if self._refresh_token:
                self._refresh_access_token()
            else:
                raise ValueError("Token已过期且无Refresh Token，需要重新授权")
        return self._access_token

    def exchange_code_for_token(self, code: str) -> None:
        """用Authorization Code换取Token"""
        response = httpx.post(self.token_url, data={
            "grant_type": "authorization_code",
            "code": code,
            "client_id": self.client_id,
            "client_secret": self.client_secret,
        })
        data = response.json()
        self._access_token = data["access_token"]
        self._refresh_token = data.get("refresh_token")
        self._expires_at = time.time() + data.get("expires_in", 3600)

    def _refresh_access_token(self) -> None:
        """使用Refresh Token刷新Access Token"""
        response = httpx.post(self.token_url, data={
            "grant_type": "refresh_token",
            "refresh_token": self._refresh_token,
            "client_id": self.client_id,
            "client_secret": self.client_secret,
        })
        data = response.json()
        self._access_token = data["access_token"]
        self._expires_at = time.time() + data.get("expires_in", 3600)
```

### 在工具中集成OAuth

将OAuth能力注入到具体工具中，让模型感知到授权状态：

```python
class OAuthAwareTool:
    def __init__(self, token_manager: OAuthTokenManager):
        self.token_manager = token_manager

    def get_definition(self) -> dict:
        """返回工具定义，如果未授权则说明需要授权"""
        definition = {
            "name": "google_calendar_list_events",
            "description": "查询用户的Google日历事件列表",
            "parameters": {
                "type": "object",
                "properties": {
                    "days": {
                        "type": "integer",
                        "description": "查询未来多少天的事件"
                    }
                }
            }
        }
        if not self.token_manager.is_token_valid():
            definition["description"] += " [注意：当前未授权，需要用户先完成Google账号授权]"
        return definition

    def execute(self, days: int = 7) -> dict:
        try:
            token = self.token_manager.get_access_token()
            # 使用token调用Google Calendar API
            response = httpx.get(
                "https://www.googleapis.com/calendar/v3/events",
                headers={"Authorization": f"Bearer {token}"},
                params={"maxResults": 50}
            )
            return {"success": True, "events": response.json().get("items", [])}
        except ValueError as e:
            return {
                "success": False,
                "error": str(e),
                "auth_required": True,
                "auth_url": self._generate_auth_url()
            }
```

## 五、工具注册与发现机制

当系统中有几十上百个工具时，需要一套机制来管理它们——哪些工具可用、哪些已废弃、哪些需要特殊权限。

### 工具注册表设计

```python
from dataclasses import dataclass, field
from typing import Callable
import inspect

@dataclass
class RegisteredTool:
    """注册表中的工具条目"""
    name: str
    description: str
    parameters: dict
    execute_fn: Callable
    category: str                    # 工具分类，如 "database", "external_api"
    provider: str                    # 来源提供者名称
    requires_auth: bool = False      # 是否需要用户授权
    rate_limit: int = 60             # 每分钟最大调用次数
    enabled: bool = True             # 是否启用
    metadata: dict = field(default_factory=dict)

class ToolRegistry:
    """全局工具注册表"""

    def __init__(self):
        self._tools: dict[str, RegisteredTool] = {}
        self._categories: dict[str, list[str]] = {}

    def register(self, tool: RegisteredTool) -> None:
        """注册一个工具"""
        if tool.name in self._tools:
            raise ValueError(f"工具 '{tool.name}' 已存在，请先取消注册")
        self._tools[tool.name] = tool
        self._categories.setdefault(tool.category, []).append(tool.name)

    def unregister(self, name: str) -> None:
        """移除一个工具"""
        tool = self._tools.pop(name, None)
        if tool and tool.category in self._categories:
            self._categories[tool.category].remove(name)

    def get_for_llm(self, names: list[str] = None) -> list[dict]:
        """
        返回LLM可以使用的工具定义列表。
        只返回已启用的工具，可按名称筛选。
        """
        target_names = names or list(self._tools.keys())
        return [
            {
                "name": tool.name,
                "description": tool.description,
                "parameters": tool.parameters
            }
            for name in target_names
            if (tool := self._tools.get(name)) and tool.enabled
        ]

    def get_by_category(self, category: str) -> list[RegisteredTool]:
        """按分类获取工具"""
        names = self._categories.get(category, [])
        return [self._tools[n] for n in names if n in self._tools]

    def execute(self, name: str, args: dict) -> dict:
        """执行指定的工具"""
        tool = self._tools.get(name)
        if not tool:
            return {"success": False, "error": f"工具 '{name}' 未注册"}
        if not tool.enabled:
            return {"success": False, "error": f"工具 '{name}' 已被禁用"}
        return tool.execute_fn(**args)

# 全局单例
tool_registry = ToolRegistry()
```

### 动态工具发现

对于大型系统，工具不应该只在启动时静态注册。动态发现机制可以让系统在运行时加载新工具：

```python
import importlib
import pkgutil

class PluginBasedDiscovery:
    """基于插件的工具发现 —— 扫描指定包下的工具提供者"""

    def __init__(self, plugin_package: str = "agent_tools.plugins"):
        self.plugin_package = plugin_package

    def discover(self, registry: ToolRegistry) -> int:
        """扫描并注册所有发现的工具"""
        package = importlib.import_module(self.plugin_package)
        count = 0
        for _, module_name, _ in pkgutil.iter_modules(
            package.__path__, package.__name__ + "."
        ):
            module = importlib.import_module(module_name)
            # 查找模块中的 ToolProvider 子类
            for attr_name in dir(module):
                attr = getattr(module, attr_name)
                if (isinstance(attr, type)
                        and issubclass(attr, ToolProvider)
                        and attr is not ToolProvider):
                    provider = attr()
                    for tool in provider.get_tools():
                        registry.register(tool)
                        count += 1
        return count
```

这意味着你只需要在`agent_tools/plugins/`目录下新建一个Python文件，实现一个`ToolProvider`子类，工具就会被自动发现并注册。

### 分层工具权限

不是所有工具都应该对所有场景开放。引入权限层级：

```python
@dataclass
class ToolPermission:
    level: str  # "public", "authenticated", "admin", "dangerous"

def get_allowed_tools(registry: ToolRegistry, user_role: str) -> list[dict]:
    """根据用户角色过滤可用工具"""
    permission_map = {
        "guest": ["public"],
        "user": ["public", "authenticated"],
        "admin": ["public", "authenticated", "admin", "dangerous"],
    }
    allowed_levels = permission_map.get(user_role, ["public"])
    allowed_names = [
        name for name, tool in registry._tools.items()
        if tool.metadata.get("permission", "public") in allowed_levels
    ]
    return registry.get_for_llm(allowed_names)
```

## 六、工具描述的最佳实践

工具描述是LLM理解工具的唯一途径。一个写得不清晰的工具描述，直接导致模型选错工具或传错参数。写好工具描述是工具集成的核心工作之一。

### 描述应该包含哪些信息

一个高质量的工具描述应该回答以下三个问题：

1. **这个工具做什么**（功能描述）
2. **什么情况下应该用它**（使用场景，这是最关键的部分）
3. **有什么限制或注意事项**（边界条件）

```python
# 好的描述
"description": (
    "在公司的产品数据库中搜索商品信息。"
    "当用户询问产品价格、库存、规格、评价时使用此工具。"
    "当用户笼统地问'你们有什么产品'时也应使用此工具进行搜索。"
    "注意：不支持模糊搜索，需要至少提供产品名称的关键词。"
    "返回结果最多20条。"
)

# 不好的描述
"description": "搜索产品"  # 太模糊，模型不知道什么时候该用
```

### 参数描述是"说明书"不是"类型声明"

LLM通过参数的自然语言描述来理解参数含义，JSON Schema的type只是辅助校验。参数描述要写明"这个参数代表什么，从哪里获取，有什么格式要求"：

```python
# 好的参数描述
"date": {
    "type": "string",
    "description": (
        "要查询的日期，格式为YYYY-MM-DD，如'2025-01-15'。"
        "如果用户说'今天''明天''昨天'，请转换为对应的具体日期。"
        "如果用户没有明确指定日期，默认为今天。"
    )
}

# 不好的参数描述
"date": {
    "type": "string",
    "description": "日期"  # 没有告诉模型格式和默认行为
}
```

### 用正例和反例引导模型

在描述中给出典型用例，能显著提升模型选择的准确率：

```python
"description": (
    "发送一封电子邮件。"
    "使用场景示例："
    "- 用户说'帮我发邮件给张三' -> 使用此工具"
    "- 用户说'帮我写一封邮件' -> 不要用此工具（用户只是要内容，不是真的发送）"
    "- 用户说'查看我的邮件' -> 不要用此工具，应使用 search_emails 工具"
    "注意：此工具会真实发送邮件，请在用户明确确认后使用。"
)
```

这种写法将"工具选择"这个模糊的判断问题，转化为模型擅长的"模式匹配"问题。

### 工具数量与粒度

| 工具数量 | 效果 | 建议 |
|---------|------|------|
| 1-5个 | 选择准确率高，但能力有限 | 简单场景 |
| 5-15个 | 能力和准确率的平衡点 | 推荐范围 |
| 15-30个 | 选择准确率开始下降 | 考虑使用分类筛选 |
| 30+个 | 模型容易选错工具 | 必须使用动态筛选 |

当工具数量超过15个时，建议按场景分组，每次只给模型发送相关的工具子集：

```python
def select_tools_for_context(registry: ToolRegistry,
                              user_message: str) -> list[dict]:
    """根据用户消息的意图，筛选相关工具"""
    # 简单的关键词匹配（实际可用embedding做语义匹配）
    category_keywords = {
        "email": ["邮件", "email", "发送", "收件"],
        "calendar": ["日历", "日程", "会议", "提醒"],
        "database": ["查询", "数据", "报表", "统计"],
    }
    matched_categories = set()
    for category, keywords in category_keywords.items():
        if any(kw in user_message for kw in keywords):
            matched_categories.add(category)

    if not matched_categories:
        return registry.get_for_llm()  # 返回全部

    tools = []
    for cat in matched_categories:
        tools.extend(registry.get_by_category(cat))
    return [t for t in tools if t.enabled]
```

## 七、错误处理与超时管理

外部工具调用是智能体系统中最不可靠的环节——网络波动、服务宕机、API限流、认证过期……任何问题都可能导致调用失败。如果没有妥善的错误处理，智能体可能拿到一个错误信息后"编造"一个结果（幻觉），或者陷入反复重试的死循环。

### 错误分类与处理策略

将工具调用的错误分为三类，分别处理：

```python
from enum import Enum

class ErrorType(Enum):
    RETRYABLE = "retryable"          # 可重试：网络超时、503
    NON_RETRYABLE = "non_retryable"  # 不可重试：参数错误、404
    AUTH_REQUIRED = "auth_required"  # 需要认证：Token过期

def classify_error(error: Exception, status_code: int = None) -> ErrorType:
    """根据错误类型分类"""
    if isinstance(error, httpx.TimeoutException):
        return ErrorType.RETRYABLE
    if status_code == 401 or status_code == 403:
        return ErrorType.AUTH_REQUIRED
    if status_code and status_code >= 500:
        return ErrorType.RETRYABLE
    if status_code and 400 <= status_code < 500:
        return ErrorType.NON_RETRYABLE
    return ErrorType.NON_RETRYABLE
```

针对不同类型，执行不同的处理策略：

```python
def execute_tool_with_recovery(tool_name: str, args: dict,
                                registry: ToolRegistry,
                                max_retries: int = 2) -> dict:
    """执行工具并处理各类错误"""
    last_error = None

    for attempt in range(max_retries + 1):
        try:
            result = registry.execute(tool_name, args)
            if result.get("success"):
                return result

            # 工具执行层返回了失败信息
            error_msg = result.get("error", "")

            # 如果是参数错误，不要在description里重复报错，给出友好提示
            if "invalid" in error_msg.lower() or "missing" in error_msg.lower():
                return {
                    "success": False,
                    "error": error_msg,
                    "suggestion": "请检查参数是否正确，修正后重试"
                }

            # 其他错误，尝试重试
            if attempt < max_retries:
                time.sleep(2 ** attempt)
                continue
            return result

        except Exception as e:
            last_error = str(e)
            error_type = classify_error(e)

            if error_type == ErrorType.AUTH_REQUIRED:
                # 认证问题不能重试，直接返回
                return {
                    "success": False,
                    "error": f"认证失败: {last_error}。请重新授权后重试。",
                    "auth_required": True
                }

            if error_type == ErrorType.RETRYABLE and attempt < max_retries:
                wait = 2 ** attempt
                time.sleep(wait)
                continue

            # 不可重试或重试次数耗尽
            return {
                "success": False,
                "error": f"工具执行失败(已重试{attempt}次): {last_error}",
                "retries_exhausted": attempt >= max_retries
            }

    return {"success": False, "error": f"重试{max_retries}次后仍然失败: {last_error}"}
```

### 超时管理

不同的工具有不同的合理超时时间。一个统一的超时配置机制可以避免"一刀切"的问题：

```python
@dataclass
class TimeoutConfig:
    """工具超时配置"""
    connect_timeout: int = 10     # 建立连接的超时（秒）
    read_timeout: int = 30       # 等待响应的超时（秒）
    total_timeout: int = 60      # 整个调用的总超时（秒）

class TimeoutRegistry:
    """管理各工具的超时配置"""
    _defaults = TimeoutConfig()

    _overrides = {
        "search_database": TimeoutConfig(total_timeout=10),         # 快查询
        "generate_report": TimeoutConfig(total_timeout=300),       # 报表生成很慢
        "send_email": TimeoutConfig(connect_timeout=5, total_timeout=15),
        "upload_file": TimeoutConfig(total_timeout=120),           # 大文件上传
    }

    @classmethod
    def get(cls, tool_name: str) -> TimeoutConfig:
        return cls._overrides.get(tool_name, cls._defaults)
```

在调用工具时应用超时配置：

```python
def execute_with_timeout(tool_name: str, args: dict,
                          registry: ToolRegistry) -> dict:
    """带超时保护的工具执行"""
    config = TimeoutRegistry.get(tool_name)

    try:
        with httpx.Client(
            timeout=httpx.Timeout(
                connect=config.connect_timeout,
                read=config.read_timeout,
                pool=config.total_timeout
            )
        ):
            result = registry.execute(tool_name, args)
            return result
    except httpx.TimeoutException:
        return {
            "success": False,
            "error": (
                f"工具 '{tool_name}' 执行超时"
                f"（限制 {config.total_timeout} 秒）。"
                f"请尝试缩小查询范围后重试。"
            )
        }
```

### 断路器模式

当某个外部服务持续不可用时，反复重试只会浪费资源并拖慢整个智能体。断路器（Circuit Breaker）模式可以避免这个问题：

```python
import threading
from collections import defaultdict

class CircuitBreaker:
    """简单的断路器实现"""

    def __init__(self, failure_threshold: int = 5,
                 recovery_timeout: int = 60):
        self.threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self._failures: dict[str, int] = defaultdict(int)
        self._last_failure_time: dict[str, float] = {}
        self._lock = threading.Lock()

    def is_open(self, tool_name: str) -> bool:
        """检查断路器是否打开（工具暂时不可用）"""
        with self._lock:
            failures = self._failures[tool_name]
            if failures < self.threshold:
                return False
            # 达到阈值后，检查是否过了恢复时间
            elapsed = time.time() - self._last_failure_time.get(tool_name, 0)
            if elapsed > self.recovery_timeout:
                # 尝试恢复 —— 重置计数，放行一次
                self._failures[tool_name] = 0
                return False
            return True

    def record_failure(self, tool_name: str) -> None:
        with self._lock:
            self._failures[tool_name] += 1
            self._last_failure_time[tool_name] = time.time()

    def record_success(self, tool_name: str) -> None:
        with self._lock:
            self._failures[tool_name] = 0

# 使用示例
circuit_breaker = CircuitBreaker(failure_threshold=5, recovery_timeout=120)

def execute_with_circuit_breaker(tool_name: str, args: dict,
                                  registry: ToolRegistry) -> dict:
    if circuit_breaker.is_open(tool_name):
        return {
            "success": False,
            "error": f"工具 '{tool_name}' 暂时不可用，已自动熔断。请稍后重试。"
        }

    result = execute_tool_with_recovery(tool_name, args, registry)

    if result.get("success"):
        circuit_breaker.record_success(tool_name)
    else:
        circuit_breaker.record_failure(tool_name)

    return result
```

断路器的状态流转：

```
    正常调用成功
  [CLOSED] ──────────> [CLOSED]
      │                      ^
      │ 连续失败5次           │
      v                      │ 过了120秒，放行一次
  [OPEN]  ──────────> [HALF-OPEN]
                              │
                              │ 调用失败
                              v
                           [OPEN]
```

## 八、综合实战：构建一个完整的工具集成框架

将前面讨论的各部分整合起来，构建一个完整的工具集成框架：

```python
class AgentToolFramework:
    """智能体工具集成框架 —— 整合注册、认证、超时、熔断"""

    def __init__(self):
        self.registry = ToolRegistry()
        self.token_managers: dict[str, OAuthTokenManager] = {}
        self.circuit_breaker = CircuitBreaker()

    def register_provider(self, provider: ToolProvider) -> int:
        """注册一个工具提供者及其所有工具"""
        count = 0
        for tool_meta in provider.get_tools():
            tool = RegisteredTool(
                name=tool_meta["name"],
                description=tool_meta["description"],
                parameters=tool_meta["parameters"],
                execute_fn=tool_meta["execute"],
                category=tool_meta.get("category", "general"),
                provider=provider.__class__.__name__,
                requires_auth=tool_meta.get("requires_auth", False),
                metadata=tool_meta.get("metadata", {})
            )
            self.registry.register(tool)
            count += 1
        return count

    def get_tools_for_agent(self, user_message: str = None,
                             user_role: str = "user") -> list[dict]:
        """获取当前上下文下智能体可用的工具定义"""
        tools = get_allowed_tools(self.registry, user_role)
        if user_message:
            # 根据用户意图筛选相关工具
            tools = self._filter_by_intent(tools, user_message)
        return tools

    def execute_tool(self, name: str, args: dict) -> dict:
        """执行工具 —— 带完整的错误处理和熔断保护"""
        if self.circuit_breaker.is_open(name):
            return {
                "success": False,
                "error": f"工具 '{name}' 暂时不可用（已熔断），请稍后重试"
            }

        # 检查OAuth（如果需要）
        tool = self.registry._tools.get(name)
        if tool and tool.requires_auth:
            provider_key = tool.metadata.get("oauth_provider")
            if provider_key and provider_key in self.token_managers:
                tm = self.token_managers[provider_key]
                if not tm.is_token_valid():
                    return {
                        "success": False,
                        "error": "需要授权，请完成OAuth认证流程",
                        "auth_required": True
                    }

        # 执行工具
        result = execute_tool_with_recovery(name, args, self.registry)

        # 更新断路器状态
        if result.get("success"):
            self.circuit_breaker.record_success(name)
        else:
            self.circuit_breaker.record_failure(name)

        return result

    def _filter_by_intent(self, tools: list[dict],
                           user_message: str) -> list[dict]:
        """基于用户意图的动态工具筛选（简化版）"""
        # 实际项目中可用embedding做语义相似度匹配
        return tools  # 简化：返回全部
```

## 九、设计工具集成层的最佳实践

1. **统一错误格式**：所有工具返回 `{"success": bool, "data": ..., "error": str}`，让LLM能可靠地判断成功或失败

2. **幂等性优先**：对于可能重试的操作，确保多次调用不会产生副作用。使用幂等键（idempotency key）是一个好习惯

3. **工具描述要"面向LLM编写"**：不要从开发者的角度写描述，要从"LLM阅读这段文字后能否正确判断使用场景"的角度来写

4. **分层超时配置**：快查询给短超时，慢操作给长超时。不要让一个慢工具拖垮整个智能体循环

5. **熔断是最后一道防线**：在重试之上加上断路器，防止故障扩散。记录熔断事件，触达告警渠道

6. **审计日志不可或缺**：记录每一次工具调用——谁、什么时间、调用了什么工具、传了什么参数、返回了什么结果。这对于调试和安全审计至关重要

7. **Token是消耗品，工具描述要精简**：每个工具的description都会占用上下文窗口的Token。在清晰的前提下尽量简洁。对于非常复杂的工具，可以通过"帮助工具"来提供详细文档，而不是把所有细节都塞进description

8. **定期测试工具可用性**：外部服务随时可能变更API或下线。建立健康检查机制，在工具不可用时及时从注册表中移除，而不是等用户调用了才发现

9. **考虑工具调用的成本**：每次工具调用都有时间成本和金钱成本（如果调用的是付费API）。在工具定义中可以加入预估成本信息，让LLM在需要时告知用户

10. **安全永远第一**：对于涉及资金、数据删除、权限变更的工具，加入人工确认环节。在工具定义中使用 `requires_confirmation: true` 标记，执行前暂停智能体循环等待用户确认

外部工具集成是智能体系统的"四肢"。一个设计不良的工具集成层会让智能体变成"笨手笨脚"的助手——选错工具、乱传参数、遇到错误不知所措。而一个设计良好的工具集成层，则让智能体真正成为可靠的"行动者"，把大语言模型的推理能力转化为真实世界的生产力。
