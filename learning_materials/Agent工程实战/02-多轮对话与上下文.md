## 二、核心规则

模型本身不会自动记住上一次 HTTP 请求。每一轮都要把历史重新传入：

```text
system → user1 → assistant1 → user2 → assistant2
```

## 三、创建 memory_chat.py

复用上一节的模型初始化代码，然后加入：

```python
messages = [
    {"role": "system", "content": "你是课程助教。信息不足时先追问，不要编造。"}
]

while True:
    user_text = input("你：").strip()
    if user_text.lower() in {"exit", "quit"}:
        break
    if not user_text:
        print("请输入内容")
        continue

    messages.append({"role": "user", "content": user_text})
    response = model.invoke(messages)
    answer = str(response.content)
messages.append({"role": "assistant", "content": answer})
print("AI：", answer)
```

把历史更新与裁剪整理为 `solution.py` 中的 `append_turn_and_trim`，保持输入列表不被原地修改。

<!-- lab-check:implementation -->

先输入“我叫小林”，再问“我叫什么”。如果第二轮只发送当前问题，模型无法回答；发送完整 `messages` 后，它能引用前文。

### 本节的 LangChain 调用重点

- `model.invoke(messages)` 每执行一次，都会把当时的完整 `messages` 重新发送给模型；LangChain 不会替普通 `ChatModel` 自动保存上一轮历史。
- 返回值仍是 `AIMessage`。把 `response.content` 作为 `assistant` 消息追加回历史，下一轮模型才能同时看到“用户问了什么”和“自己答了什么”。
- 消息中的 `role` 是模型协议，不是随意标签。`system` 放稳定规则，`user` 放用户输入，`assistant` 放模型历史回答。角色错位会直接改变模型如何理解内容。
- 本节没有引入新的 LangChain 包；核心变化在“如何组织传给框架的消息状态”，所以不重复解释 `while`、`append` 等 Python 语法。

## 四、加入上下文上限

历史不能无限增长。先保留 system，再保留最近 10 条普通消息：

```python
def trim_messages(messages, max_messages=11):
    if len(messages) <= max_messages:
        return list(messages)
    system = messages[:1] if messages and messages[0].get("role") == "system" else []
    remaining = max_messages - len(system)
    return system + messages[-remaining:]
```

在每次调用模型前执行：

```python
messages = trim_messages(messages)
```

然后在 `app.py` 中导入该函数并接入聊天循环，确认第二轮问题确实携带第一轮历史。

## 五、为什么要返回新列表

直接删除调用方传入的历史，会让日志、重试和测试难以复现。工程代码优先返回新值，让状态变化发生在明确的一行。

## 七、常见错误

1. 只追加 user、不追加 assistant：下一轮缺少模型自己的回答。
2. 裁剪时把 system 删除：模型角色会漂移。
3. 把一轮对话拆掉一半：尽量使用奇数上限（1 条 system + 偶数条对话消息）。
4. 历史里混入空内容：在入口统一校验。
