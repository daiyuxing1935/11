## 二、核心决策：做深而不是做广

### 2.1 战略选择

```
选项A: 做广 — 助教+助学+助研全场景覆盖
  风险: 精力分散，每个场景都浅，沦为功能堆砌
  结果: 平庸的"全场景平台"，评委看腻了

选项B: 做深 — 聚焦一个真实痛点，打穿打透  ← 我们的选择
  风险: 场景覆盖面窄
  结果: 在一个痛点上做到极致，让评委记住你
```

### 2.2 为什么做深是正确的

比赛评审维度中，"创意实用度(20分)"要求：
> "针对教学中的知识传授、技能训练、答疑辅导，或科研中的实验分析、数据处理、课题攻坚等**真实场景痛点**，提出**新颖且可落地**的解决方案"

关键词：**真实场景痛点**、**新颖**、**可落地**

当前项目的问题：做了一个"更好的刷题+答疑平台"——100个人都在做同样的事。

## 三、选定赛道：直击中国计算机教育最深的两个痛点

### 痛点一：AI代写泛滥 → "伪装的强大"
> 大一学生靠AI跑出大四的项目效果，但"手上拿着核弹不知道保险销在哪"

### 痛点二：只会用不会造 → "能力的空心化"
> 54.83%企业认为传统编程能力重要性下降，但53.85%认为毕业生系统架构能力缺失

### 我们的差异化主张

```
市面上的助学平台: "帮你更好地学知识"  (Knowledge Transfer)
我们的平台:       "帮你在AI时代不可替代" (Competence Building)
```

## 四、深度化方案：从"AI助学工具"到"AI能力锻造系统"

### 核心产品定位
**面向计算机学科的AI能力锻造与真实性评估系统**

一句话：**不让AI代写毁了计算机教育，把AI从"作弊工具"变成"能力锻造工具"**

### 模块一：学科知识库 + RAG引擎（基础层，对标比赛要求1-4）

**这是比赛要求中最核心的技术基础，当前项目完全没有。**

```
建设内容:
├── 教材知识库: CS核心教材的结构化提取(数据结构/OS/网络/编译/...)
├── Agent知识库: 论文+框架文档+最佳实践
├── 代码知识库: GitHub优质项目+标准库实现+设计模式
├── 题库知识库: 已有题库的结构化升级,关联知识图谱
└── 学术论文库: 顶会论文的摘要/方法/结论提取

RAG实现:
- 向量化: text-embedding-3-large 或 讯飞Embedding API
- 检索: 混合检索(语义+关键词+知识图谱)
- 生成: 检索结果注入Prompt → 带来源标注的回答
- 可追溯: 每个回答附引用来源(教材章节/论文DOI/文档段落)
```

**对比赛要求的满足**:
- ✅ 知识库构建(结构化、体系化)
- ✅ RAG(解决幻觉)
- ✅ 知识权威可信
- ✅ 内容可追溯

### 模块二：代码过程溯源与真实性评估（创新层，直击痛点1）

**这是我们的"杀手级"差异化功能，没有任何助学平台在做。**

```
核心理念: 不看最终代码，看代码是怎么写出来的

技术方案:
1. 浏览器端CodeMirror/Monaco Editor事件捕获
   - 每次按键的时间戳
   - 每次粘贴操作
   - 每次大段删除
   - 每次函数重命名
   - 光标移动轨迹
   - 切换窗口/标签页事件

2. 行为模式分析模型
   ┌─────────────────┬──────────────────┐
   │ 人类编码特征     │ AI代写特征        │
   ├─────────────────┼──────────────────┤
   │ 渐进式构建       │ 大段粘贴          │
   │ 频繁微调         │ 一次成型          │
   │ 回退修改         │ 无回退            │
   │ 中间态不完整     │ 直接完美          │
   │ 逐步调试         │ 无调试过程        │
   │ 分段测试         │ 全量提交          │
   └─────────────────┴──────────────────┘

3. 多维度真实性评分(Process Authenticity Score, PAS)
   - 过程分(50%): 编码节奏、修改模式、调试行为
   - 答辩分(30%): 系统追问学生的理解深度
   - 代码分(20%): 与传统检测工具的交叉验证

4. 分层诚信策略
   ├── L1 基础阶段: 强制"无AI模式"，做实打实训练
   ├── L2 进阶阶段: 允许AI辅助但需过程记录+标注
   └── L3 高阶阶段: 人机协作，评估"驾驭AI的能力"
```

**对比赛要求的满足**:
- ✅ 创新性: 唯一的过程溯源评估系统
- ✅ 深度贴合场景: 编程作业/实验课的真实痛点
- ✅ 商业化潜力: 可推广到所有编程类课程
- ✅ 复杂问题求解: 系统自身就是复杂AI应用

### 模块三：系统设计能力锻造工坊（创新层，直击痛点2）

**不做"刷题平台"，做"造轮子工坊"。**

```
渐进式能力锻造路径:

Level 1: 使用层 → 调用API完成功能
  └── 评估: 功能是否正确？边界情况是否考虑？

Level 2: 理解层 → 解释框架/库的内部原理
  └── 评估: "这段代码为什么这样设计？"追问引擎

Level 3: 修改层 → 修改开源框架源码实现新功能
  └── 评估: 修改是否正确？是否破坏了原有设计？

Level 4: 重建层 → 从零实现核心组件
  └── 评估: 自建 vs 标准实现的diff对比+性能对比

Level 5: 设计层 → 自主设计新架构解决开放问题
  └── 评估: 架构合理性、trade-off分析、可扩展性

核心机制: "为什么"追问引擎
- 对每个提交，系统自动生成追问:
  "你为什么选择这个数据结构而不是X？复杂度差异是多少？"
  "如果数据量从1万变成1亿，你的方案还成立吗？"
  "这个设计模式的替代方案是什么？各自适用场景？"
- 学生必须在限定时间内回答
- 系统评估回答的深度和准确性
- 追问难度根据学生水平动态调整
```

**对比赛要求的满足**:
- ✅ 复杂问题求解: 系统追问+开放设计问题
- ✅ 个性化与自适应: 五级能力路径
- ✅ 深度贴合场景: 直面"只会用不会造"的核心矛盾

### 模块四：教师透视面板（助教场景补充）

```
面向教师的"班级真相仪表盘":

1. 学情热力图
   - 全班知识掌握度分布(不是分数,是真实理解度)
   - 每个学生的认知模式标签(概念混淆/记忆缺失/逻辑断裂/迁移困难)

2. 真实性警示
   - 标注"高度疑似AI代写"的作业
   - 展示过程真实性评分(PAS)异常的个体
   - 发现"代码完美但答不出追问"的能力-表现不一致

3. 教学干预建议
   - 自动生成分层教学方案
   - 标记需要额外关注的"高风险"学生
   - 推荐针对性练习
```
